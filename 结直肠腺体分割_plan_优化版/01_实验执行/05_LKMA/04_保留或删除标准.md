# 保留或删除标准

本文件不是 `LKMA` 阶段里的一页经验判断，也不是“先把结果跑出来再凭最好那次决定去留”的松散备注。

它在当前目录中的唯一职责是：

> 作为 `05_LKMA` 的正式阶段裁决 protocol，把 `结直肠腺体分割_plan_优化版/01_实验执行/05_LKMA/03_实验步骤.md` 已经冻结的执行链、`结直肠腺体分割_plan_优化版/01_实验执行/05_LKMA/02_插入位置与参数范围.md` 已经冻结的搜索边界、以及 `结直肠腺体分割_plan_优化版/01_实验执行/05_LKMA/00_阶段总协议.md` 已经冻结的 `Gate_C1` 与 handoff 语义，压成唯一合法的 `keep / backup / drop` 判定规则；从现在开始，`LKMA` 是否进入后续主线、是否只能保留为备选、以及何时必须彻底回退到 `baseline`，都只能由这里写明的证据顺序、裁决等级、成本约束和交接字段决定，不能再靠单次最好值、临时多补几个变体或主观偏好改判。

从现在开始，只要讨论 `LKMA` 阶段“什么时候才允许写正式结论、哪些证据优先级最高、边界案例如何裁决、什么情况必须回退到 baseline、`06_Boundary` 到底从哪里起步”，都必须以本文件为准；若与同层文件冲突，以 `结直肠腺体分割_plan_优化版/01_实验执行/05_LKMA/00_阶段总协议.md` 管阶段身份与 `Gate_C1`，以 `结直肠腺体分割_plan_优化版/01_实验执行/05_LKMA/02_插入位置与参数范围.md` 管位置与 `kernel` 边界，以 `结直肠腺体分割_plan_优化版/01_实验执行/05_LKMA/03_实验步骤.md` 管执行顺序，而本文件负责把这些上位约束压成可直接比较、可直接验收、可直接回退、可直接交接给下游阶段的最终判定逻辑。

## 1. 文件角色与执行边界

### 1.1 当前文件负责什么

当前文件只负责冻结下面八件事：

1. `LKMA` 阶段的正式结论最早在什么资产条件下才允许书写。
2. `LKMA` 的正式证据应按什么优先级读取，避免被单次最好值带偏。
3. 什么情况下可以把 `LKMA` 判为 `keep` 并进入后续主线。
4. 什么情况下只能降级为 `backup`，而不能冒充正式保留。
5. 什么情况下必须直接 `drop`，并停止继续为 `LKMA` 扩位置或扩 `kernel`。
6. 边界案例如何按 seed 一致性、split 一致性、可视化和成本顺序复核。
7. 阶段结论必须输出哪些正式字段，才能被 `Gate_C1` 和 `06_Boundary` 稳定消费。
8. 一旦结论不成立，应该按什么规则独立回退到 `baseline` 起点，而不是让下游阶段替当前文件兜底。

### 1.2 当前文件不负责什么

当前文件不重新定义下面这些内容：

- baseline 的结构身份、encoder 初始化来源和 `Gate_B1` 通过标准。
- `train_proto_v1 / eval_proto_v1` 的训练与评估 protocol 本体。
- `LKMA` 为什么位于 baseline 之后、`Boundary` 之前的设计动机。
- `insert_position / kernel_size` 的候选集合、默认主变体和备选上限。
- `Boundary Head` 与 `Distance-aware Loss` 的结构、顺序和阶段职责。

这些职责分别以下列文件为准：

- `结直肠腺体分割_plan_优化版/01_实验执行/04_Baseline/04_阶段验收.md`
- `结直肠腺体分割_plan_优化版/01_实验执行/05_LKMA/00_阶段总协议.md`
- `结直肠腺体分割_plan_优化版/01_实验执行/05_LKMA/01_设计依据.md`
- `结直肠腺体分割_plan_优化版/01_实验执行/05_LKMA/02_插入位置与参数范围.md`
- `结直肠腺体分割_plan_优化版/01_实验执行/05_LKMA/03_实验步骤.md`
- `结直肠腺体分割_plan_优化版/01_实验执行/06_Boundary/00_阶段总协议.md`

### 1.3 为什么当前文件必须独立存在

因为 `LKMA` 阶段最容易出现的伪完成状态不是“没有写结论”，而是：

- 口头上说只看正式三 seed，实际却拿 screening run 或单个 seed 的最好结果强行保留。
- 口头上说按对象级指标决定，实际却先被局部好看的图或单一指标轻微上涨带偏。
- 口头上说主版本不过线就止损，实际却不断补额外位置和更大 `kernel`，把阶段判定偷换成持续搜索。
- 口头上说已经能交接到 `Boundary`，实际却没有把 `decision_level / decision_reason / next_stage_start_model` 写成正式字段。

如果没有一份独立判定文件，`结直肠腺体分割_plan_优化版/01_实验执行/05_LKMA/03_实验步骤.md` 会被迫替本文件承担“到底算不算保留”的职责，`06_Boundary` 会被迫自行决定该从 `baseline` 还是 `baseline + LKMA` 起步，而 `Gate_C1` 也就无法真正形成可审计的阶段闭环。

## 3. 本阶段唯一允许处理的变量

### 3.1 当前文件唯一判定对象

- 当前结论：本文件唯一判定对象是“在 baseline 协议完全不动的前提下，已经按固定执行链完成的 `LKMA main_variant = bottleneck + k15` 是否值得进入后续主线，以及可选备选版本是否只能作为辅助解释而不能取代主版本结论”
- 规则类型：`工程冻结规则`
- 适用阶段：`05_LKMA`
- 直接依据：`结直肠腺体分割_plan_优化版/01_实验执行/05_LKMA/00_阶段总协议.md`、`结直肠腺体分割_plan_优化版/01_实验执行/05_LKMA/02_插入位置与参数范围.md`、`结直肠腺体分割_plan_优化版/01_实验执行/05_LKMA/03_实验步骤.md`

- 采用原因：只有把判定对象收缩为“已完成正式主链的 LKMA 主版本”，后续保留/删除结论才具有单变量与可追溯性
- 不采用的相邻方案：不把本文件写成执行步骤文件；不把本文件写成参数边界文件；不把任何未授权扩表变体纳入正式主结论
