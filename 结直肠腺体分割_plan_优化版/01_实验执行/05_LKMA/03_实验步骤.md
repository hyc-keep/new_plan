# 实验步骤

本文件不是 `LKMA` 阶段里的一页流程提醒，也不是“先跑主版本看看，不行再自由补几个变体”的松散说明。

它在当前目录中的唯一职责是：

> 作为 `05_LKMA` 的正式执行 protocol，把 `结直肠腺体分割_plan_优化版/01_实验执行/05_LKMA/00_阶段总协议.md` 的阶段门控、`结直肠腺体分割_plan_优化版\01_实验执行\05_LKMA\01_设计依据.md` 的模块职责、`结直肠腺体分割_plan_优化版\01_实验执行\05_LKMA\02_插入位置与参数范围.md` 的参数边界，压成唯一合法的执行顺序；后续任何 `LKMA` run 是否属于正式主线、是否有资格进入阶段比较、是否足以交给 `结直肠腺体分割_plan_优化版/01_实验执行/05_LKMA/04_保留或删除标准.md` 做最终裁决，都以本文件冻结的步骤链为准。

从现在开始，只要讨论 `LKMA` 阶段“什么时候允许启动、首个 run 应该是什么、什么时候补三 seed、什么时候才允许补一个备选、什么时候必须停止扩表并进入阶段判定”，都必须以本文件为准；若与同层文件冲突，以 `结直肠腺体分割_plan_优化版/01_实验执行/05_LKMA/00_阶段总协议.md` 管阶段身份与 `Gate_C1`，以 `结直肠腺体分割_plan_优化版/01_实验执行/05_LKMA/02_插入位置与参数范围.md` 管位置和 `kernel` 边界，而本文件负责把这些上位约束压成可直接执行、可直接验收、可直接回退的运行顺序。

## 1. 文件角色与执行边界

### 1.1 当前文件负责什么

当前文件只负责冻结下面八件事：

1. `LKMA` 阶段在什么前提下才允许从 baseline handoff 正式启动。
2. 第一个 screening run 的唯一合法配置是什么。
3. screening 通过后，主版本三 seed 应该如何补齐。
4. 每个进入正式结果层的 run 必须完成哪些固定执行环节。
5. 什么时候才允许额外补一个且仅一个备选版本。
6. 阶段汇总、比较、成本统计和决策说明按什么顺序形成。
7. `Gate_C1` 的资产、比较和 handoff 条件在执行层如何落地。
8. 哪些情况必须停止扩表并回退到 `结直肠腺体分割_plan_优化版/01_实验执行/05_LKMA/04_保留或删除标准.md` 的正式裁决。

### 1.2 当前文件不负责什么

当前文件不重新定义下面这些内容：

- baseline 的结构身份、encoder 初始化来源和 `Gate_B1` 的通过标准。
- `train_proto_v1 / eval_proto_v1` 的训练与评估协议本体。
- `LKMA` 为什么位于 baseline 之后、`Boundary` 之前的设计动机。
- `insert_position / kernel_size` 的候选集合和复杂度边界。
- `keep / backup / drop` 的最终证据优先级和结论分级。

这些职责分别以下列文件为准：

- `结直肠腺体分割_plan_优化版/01_实验执行/04_Baseline/04_阶段验收.md`
- `结直肠腺体分割_plan_优化版/01_实验执行/05_LKMA/00_阶段总协议.md`
- `结直肠腺体分割_plan_优化版/01_实验执行/05_LKMA/01_设计依据.md`
- `结直肠腺体分割_plan_优化版/01_实验执行/05_LKMA/02_插入位置与参数范围.md`
- `结直肠腺体分割_plan_优化版/01_实验执行/05_LKMA/04_保留或删除标准.md`
- `结直肠腺体分割_plan_优化版/01_实验执行/06_Boundary/00_阶段总协议.md`

### 1.3 为什么当前文件必须独立存在

因为 `LKMA` 阶段最容易出现的伪完成状态不是“没有写实验步骤”，而是：

- 口头上说先做主版本，实际却把位置和 `kernel` 同时扩成搜索矩阵。
- 口头上说先 screening，实际却把 screening run 和正式三 seed run 混进同一层结论。
- 口头上说只补一个备选，实际却在没有正式理由的情况下不断追加变体。
- 口头上说阶段已经能判定，实际却没有 raw 表、聚合表、成本表和阶段说明。

如果没有一份独立执行链文件，05_LKMA/02 会被迫承担“怎么跑”的职责，05_LKMA/04 会被迫提前处理“还没跑完就开始判”的问题，而 `06_Boundary` 也无法获得唯一合法的 `current_base`。

- 核心公式或定义参考：`execution_chain = screening_main -> three_seed_main -> optional_one_backup -> stage_decision`
- 采用原因：只有把执行对象收缩为唯一主变体和唯一顺序，后续比较才能保持单变量和可解释性
- 不采用的相邻方案：不把本文件写成设计依据文档；不把本文件写成参数边界文档；不把本文件写成最终判定文档

### 3.2 当前文件直接继承哪些冻结项

本文件直接继承并不得擅自改动的上位规则如下：

- baseline 正式基座：`ResNet34-U-Net`
- 唯一合法启动前提：B1 current_standard 真实资产已冻结，且治理消费状态为 `frozen_conditional_baseline`；`original_gate_b1=false` 不得改写

- `boundary_metric_impl = project_custom_erosion_xor_3x3_ones_8conn_border0_tol3px`

- 额外备选集合：`{(decoder_stage1, 15), (bottleneck, 21)}`

### 3.3 当前文件禁止重新打开哪些问题

本文件禁止借“实验步骤”之名重新打开：

- `Boundary Head` 与 `Distance-aware Loss` 的结构、顺序和保留逻辑。

## 5. 如何判断设计有效(验收标准)

### 4.1 启动前置条件

- 当前结论：只有当 baseline 三 seed 正式结果、`mean+-std` 表、交接资产与 baseline 通过记录都已形成后，才允许启动 `LKMA`
- 规则类型：`工程冻结规则`
- 适用阶段：`LKMA 启动前`
- 直接依据：`结直肠腺体分割_plan_优化版/01_实验执行/04_Baseline/04_阶段验收.md`、`结直肠腺体分割_plan_优化版/01_实验执行/05_LKMA/00_阶段总协议.md`
- 核心公式或定义参考：`baseline_ready_for_lkma := frozen_conditional_baseline and current_standard_assets_ready and warning_carried_forward`
- 采用原因：`LKMA` 是 post-baseline 单变量模块验证，如果 baseline 本身未冻结，后续收益无法归因
- 不采用的相邻方案：不在 baseline 只跑通一次时就启动 `LKMA`；不在 baseline 资产不齐时提前开新阶段
- 代码落点：[project_root]/scripts/summarize_stage.py、[project_root]/reports/stage_reports/baseline_stage_summary.md
- 运行记录字段：`baseline_stage_passed`、`aggregation`、`effective_batch_size`、`encoder_pretrained`、`train_proto_version`、`eval_proto_version`、`eval_cast_policy`、`boundary_metric_width`、`boundary_metric_impl`、`connected_components_impl`、`connected_components_connectivity`
- 验收方式：检查进入 `LKMA` 前已存在 baseline 正式放行记录与 handoff 资产

### 4.2 第一步：主版本 screening run

- 当前结论：第一个必须运行的版本固定为 `main_variant = (insert_position = bottleneck, kernel_size = 15, seed = 3407)`
- 规则类型：`工程冻结规则`
- 适用阶段：`LKMA 第一步`
- 直接依据：`结直肠腺体分割_plan_优化版/01_实验执行/05_LKMA/02_插入位置与参数范围.md`
- 核心公式或定义参考：`screening_run = main_variant + seed3407 + screening_only`
- 采用原因：先用单个 run 验证实现、训练、评估和导出链是否正常，是当前阶段成本最低且最可控的启动方式
- 不采用的相邻方案：不一上来直接跑三个 seed；不一上来并行开 `decoder_stage1` 或 `k21`
- 代码落点：[project_root]/configs/experiment/lkma_c1_v1.yaml、`crc_gland_segmentation_project/scripts/train.py`
- 运行记录字段：`run_name`、`seed = 3407`、`insert_position = bottleneck`、`kernel_size = 15`、`screening_only = true`、`variant_rank = main`、`train_proto_version`、`eval_proto_version`、`eval_cast_policy`、`boundary_metric_width`、`boundary_metric_impl`、`connected_components_impl`、`connected_components_connectivity`
- 验收方式：检查首个正式 `LKMA` run 只出现一个 screening 主版本，且字段完整

### 4.3 Screening 通过条件

- 当前结论：只有当 screening run 满足“无实现错误、训练曲线正常、验证与双测试产物完整、对象级指标未出现明显异常退化、可视化未出现结构性崩坏”时，才允许进入三 seed 正式主版本
- 规则类型：`工程冻结规则`
- 适用阶段：`LKMA 第一步结束`
