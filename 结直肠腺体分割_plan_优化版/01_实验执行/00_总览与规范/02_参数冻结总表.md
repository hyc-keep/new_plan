# 参数冻结总表

本文件不是“超参数备忘录”，而是 `01_实验执行` 全局参数协议的正式冻结层。

它必须先把后续代码、实验、结果表和论文写作都会共同消费的关键对象压成统一规则，再回答四个问题：

1. 当前到底冻结了什么
2. 这些值属于哪类依据
3. 它们分别从哪个阶段开始不可随意改动
4. 如果后面有人想改，应该回退到哪里、改哪些字段、影响哪些下游

换句话说，本文件的作用不是“列一个值”，而是：

> 把路线层、文献层、工程层和结果记录层会反复引用的关键参数，统一写成唯一可追溯、可落代码、可做阶段裁决的正式总表。

## 1. 文件角色与执行边界

### 1.1 当前文件负责什么

当前文件负责冻结以下内容：

- benchmark、主线方法、模块顺序和主指标链的全局参数身份
- 数据、训练、评估、命名、后处理、外部对比和统计方式中的关键冻结项
- 每个关键对象的 `规则类型 / 直接依据 / 代码落点 / 冻结阶段`
- 后续阶段如何继承这些参数，而不是各自再写一套口径
- 当参数冲突、口径漂移或版本字段失配时，应该回退到哪里修

### 1.2 当前文件不负责什么

当前文件不负责替代下面这些文件的细则职责：

- 不负责重写数据阶段的来源说明、split 生成细节和交接资产细节
- 不负责重写 `UNet / Baseline / LKMA / Boundary / Distance` 各阶段的具体执行步骤
- 不负责单独定义 `run_name`、结果表 schema 和文件命名细节本体
- 不负责单独定义对象级指标公式、实现依赖和 crosscheck 流程本体

这些内容分别以下列文件为准：

- `结直肠腺体分割_plan_优化版/01_实验执行/00_总览与规范/01_工程目录框架.md`
- `结直肠腺体分割_plan_优化版/01_实验执行/00_总览与规范/03_命名与结果记录规范.md`
- `结直肠腺体分割_plan_优化版/01_实验执行/00_总览与规范/04_评估口径与官方脚本对齐.md`
- `结直肠腺体分割_plan_优化版/01_实验执行/00_总览与规范/05_代码工程映射与实现策略.md`
- `结直肠腺体分割_plan_优化版/01_实验执行/01_数据协议/00_阶段总协议.md`
- `结直肠腺体分割_plan_优化版/01_实验执行/02_UNet流程验证/00_阶段总协议.md`

### 1.3 当前文件真正要证明什么

本文件真正要证明的不是“已经列出很多参数”，而是：

1. 后续正式 run 会用到的关键参数都已经能在本文件中找到唯一归属
2. 每个关键值都能说明自己属于 `官方协议固定项 / 路线层已锁定 / 论文支持的候选范围 / 工程冻结规则` 中的哪一类
3. 后续任何阶段如果要改变这些值，都必须升级对应版本字段，并回退修订受影响消费者
4. 下游文档和代码实现不再允许各自保留一套旧口径

### 1.4 当前文件对下游的硬约束

从 `01_数据协议` 开始，到 `11_总验收与止损` 结束，下游只允许继承本文件已经冻结的对象，不允许：

- 在阶段文件里重新发明 `best checkpoint` 选择器
- 在某个模型阶段单独切换 `ImageNet mean/std`、`threshold`、`Boundary F1` 宽度或后处理版本
- 把工程起点冒充成论文唯一原值
- 把已经冻结的字段改了，但继续沿用旧的 `config_version / data_proto_version / eval_proto_version`
- 在 run_meta.yaml、结果表和阶段总结里各写各的字段名

## 3. 本阶段唯一允许处理的变量

### 3.1 当前使用规则

后面所有会进入正式 run 的数据、训练、评估、结果配置，都必须能在本文件中找到归属。

如果某个参数当前还不能写成唯一值，也必须明确写出：

- 当前最终采用值
- 候选范围
- 为什么先选这个
- 最迟在哪个阶段冻结

不允许再出现下面这种情况：

- 配置里已经写死了某个值
- 文档里却没有说明来源
- 写论文时又说不清它到底是官方规则、路线锁定、论文候选还是工程冻结

### 3.2 `官方协议固定项`

适用于：

- benchmark 角色
- 数据划分边界
- 主指标定义

典型来源：

- 01_GlaS-Challenge.md
- 04_MILD-Net.md

### 3.3 `路线层已锁定`

适用于：

- 主 baseline
- 模块顺序
- 投稿外部对比优先级
- `CRAG` 的角色

典型来源：

- `结直肠腺体分割_plan_优化版/02_路线与投稿/01_结直肠腺体分割_高性价比路线总锁定.md`
- `结直肠腺体分割_plan_优化版/02_路线与投稿/02_结直肠腺体分割_分阶段实验路线与执行标准.md`

### 3.4 `论文支持的候选范围`

适用于：

- 输入尺寸
- loss 组合
- 大核位置与 kernel
- 边界宽度与边界权重
- 距离图和 `lambda_dist`

这类内容必须写清：

- 候选来自哪些论文
- 当前先选哪一个
- 为什么不先选别的

### 3.5 `工程冻结规则`

适用于：

- optimizer / lr / scheduler / epoch
- 命名规范
- checkpoint 选择
- threshold 固定
- 后处理默认关闭

这类内容可以不是单篇论文直接给出，但必须同时满足：

- 不违背路线层和文献层
- 能服务公平比较
- 能直接指导代码实现
- 能落到版本字段和结果记录字段

## 5. 随机性控制清单

这部分后面必须写进正式代码与配置，不能只写在文档里。

| 随机性来源 | 当前要求 | 是否必须固定 | 正式实现方式 |
|------------|----------|--------------|----------|
| Python `random` | 固定 seed | 是 | `random.seed(seed)` |
| NumPy | 固定 seed | 是 | `np.random.seed(seed)` |
| PyTorch CPU | 固定 seed | 是 | `torch.manual_seed(seed)` |
| PyTorch CUDA | 固定 seed | 是 | `torch.cuda.manual_seed_all(seed)` |
| cuDNN 行为 | 强制确定性 | 是 | `deterministic=True`、`benchmark=False`；如确因算子限制无法满足，必须在 run_meta.yaml 明记并禁止与确定性 run 同表汇总 |
| DataLoader worker | 每个 worker 派生固定 seed | 是 | `worker_init_fn` |
| augmentation random state | 跟随主 seed 派生 | 是 | 用统一 seed 初始化增强库 |
| split seed | 固定 `3407` | 是 | 只在数据阶段生成一次，并把生成脚本与导出 CSV 一起归档 |
| repeat seeds | `3407 / 1234 / 2025` | 是 | 后续正式多 seed 固定只用这三组；不得临时替换成“更好看”的 seed |

理解方式固定为：

- 随机性不是坏事
- 未记录、未控制的随机性才是坏事
- 只要 seed 集、deterministic 状态或 worker 派生规则变化，就必须升级相关版本字段并停止与旧协议并表

## 6. 颜色与归一化口径

### 6.1 当前默认口径

首轮默认不做 stain normalization，统一先用：

- 标准 `RGB`
- `ImageNet mean/std` 归一化；若某外部 backbone 有明确预训练兼容要求，必须作为统一协议升级项整体切换，禁止单个模型私自改均值方差

### 6.2 为什么现在不先锁 `Macenko`

原因不是 stain normalization 没价值，而是：

- 当前主任务首先要排除数据链、模型链、评估链问题
- 如果首轮就引入 stain normalization，会增加新的变量源
- 只有当后面出现明显颜色波动、跨 split 结果异常或 `CRAG` 趋势不稳时，才更适合把它列为补充消融

### 6.3 当前阶段如何处理

- 数据阶段：先记录当前原始颜色状态，不引入额外标准化算法
- 主线阶段：统一沿用同一归一化口径
- 如后面要补 `Macenko` 或其他 stain normalization：必须作为新增变量单列，不准混进已有主表协议

## 8. 当前最容易犯的错

### 8.1 把工程起点误写成论文固定值

比如：

- `AdamW + 1e-3`
- `epoch 120`
- `early stopping 20`

这些当前都是工程起点，不是某篇腺体论文唯一原值。

### 8.2 还没过线就提前冻结

例如：

- `UNet` 还没跑通就写死最终 `batch size`
- `LKMA` 只跑了一次就写死 kernel

### 8.3 已经冻结的内容后面偷偷改

例如：

- 主线中途改 split
- 主线中途改增强包
- 外部对比时偷偷加后处理

这种行为一旦发生，后面必须从新的版本重新起协议，不能和旧主表混写。

### 8.4 改了参数口径却不改版本字段

例如：

- 改了 `threshold_source` 还沿用旧 `eval_proto_version`
- 改了损失权重或边界宽度却不升级 `config_version`
- 改了数据或标签协议却继续沿用旧 `data_proto_version`

这是当前最容易让下游结果表失真的真实风险之一。
