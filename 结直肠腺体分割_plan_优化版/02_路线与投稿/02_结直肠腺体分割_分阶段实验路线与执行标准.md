# 结直肠腺体分割分阶段实验路线与执行标准

这份文档不是投稿自查表，而是把当前路线真正压成可执行实验阶段的工作文档。

它主要解决四件事：

1. 每个阶段具体做什么
2. 每个阶段哪些变量能改、哪些不能改
3. 到了什么标准才允许进入下一阶段
4. 每个阶段哪些结果资产必须保留，才能同时服务“中文保底收口”和“结果够强时直接按英文起步”

## 文档定位

- 类型：阶段执行标准文档
- 核心作用：把路线锁定稿变成逐阶段可执行清单，并把后续中文 / 英文两个稿件出口所需的结果资产提前组织好
- 前置阅读：`01_结直肠腺体分割_高性价比路线总锁定.md`
- 下一份建议看：`03_结直肠腺体分割_投稿层级自查与止损判断.md`

## 1. 总执行原则

### 1.1 阶段顺序固定

后面只按下面顺序推进：

1. 数据固定
2. `UNet` 跑通
3. `UNet` 稳定
4. 主 baseline 稳定
5. `LKMA`
6. `Boundary Head`
7. `Distance-aware Loss`
8. 投稿版第一批外部对比
9. `CRAG`
10. 图表与稿件出口判断

### 1.2 每阶段都要回答五个问题

- 这一步的目标是什么
- 这一步允许改什么
- 这一步必须产出什么
- 什么情况下算过线，什么情况下必须回退
- 这一步产出的哪些结果必须按“可直接进主表 / 可直接进图 / 可直接进中英文写作”标准保存

### 1.3 文献已经帮你锁定的固定项

基于 `03_文献证据` 的当前结论，下面这些默认先固定：

- 主数据：`GlaS`
- 第二 benchmark：`CRAG`
- 输入尺寸：`512x512`
- 初始训练协议：`BCE + Dice + AdamW + 1e-3 + ReduceLROnPlateau`
- 主指标：`F1 / Object Dice / Object Hausdorff`
- 补充指标：`Dice / IoU / HD95 / Boundary F1`
- 主 baseline：`ResNet34-U-Net`
- 最终模型默认优先判断：`Baseline + LKMA + Boundary Head`
- `Distance-aware Loss` 默认定位：可选增强项，不预设一定保留

### 1.4 这份文档现在锁死什么

这份文档负责锁死的是“阶段顺序、默认口径、允许改动边界、过线与回退规则”。当前在这里直接锁死的主要是：

- 阶段推进顺序
- 主数据和第二 benchmark
- 主 baseline 和模块引入顺序
- 主指标和补充指标
- 投稿主表必须提前预留的 task-specific direct comparison 最小集合
- 每一阶段允许改什么、不允许改什么
- 每一阶段最低必须产出的结果类型
- 每一阶段结果资产的保留口径

### 1.5 这份文档不在此刻直接锁死什么

下面这些内容现在必须给出默认候选范围，但不应该在路线层直接写成永久最终值：

- 最终 `batch size`
- 最终数据增强强度版本
- `LKMA` 最终插入位置、核大小和结构细节
- 边界标签最终宽度、边界损失形式和权重
- 距离图最终生成方式和 `lambda_dist`
- 外部增强对比最终入表名单
- 最终到底直接按中文收口还是按英文起步

原因不是这些内容不重要，而是：

- 它们必须建立在 `UNet` 跑通和主 baseline 稳定之后
- 它们需要结合实际显存、训练稳定性和波动范围来定

### 1.6 哪些参数必须转交执行层最终锁定

- 距离约束正式配置

### 1.7 统一默认执行口径

为了避免后面执行时从零开始摇摆，当前统一默认口径如下：

- 输入尺寸默认先按 `512x512`
- 主 loss 默认先按 `BCE + Dice`
- optimizer 默认先按 `AdamW`
- 初始学习率默认先按 `1e-3`
- scheduler 默认先按 `ReduceLROnPlateau`
- threshold 默认先按 `0.5`
- `batch size` 当前不写死具体数字，但要求在单卡条件下使用“最大稳定不溢出值”，并在 baseline 稳定后冻结
- 主线默认优先保留到 `Baseline + LKMA + Boundary Head`，`Distance-aware Loss` 只有在额外证据成立时才进入最终模型
- `GlaS` 主表从路线层开始就同步整理 task-specific direct comparison，不能等到写稿时再补

### 1.8 结果资产默认按双出口组织

后面每个阶段都默认按“两种稿件出口共用一套结果资产”组织：

- 中文保底出口：要求主线闭环、图表齐、对象级与边界级证据齐
- 英文起步出口：在中文保底资产基础上，再把通用外部对比、task-specific direct comparison 和双数据集结果整理成更规范主表

所以从现在开始，每一阶段正式保留的结果都应至少满足：

- 能回溯到 seed、配置、checkpoint 和评估脚本
- 能直接转成表格或图，不依赖临时截图
- 能明确标出是 `main`、`ablation`、`external`、`direct comparison` 还是 `supplement`
- 中文和英文写作只改表达，不重新发明结果组织方式

## 2. 阶段 A：数据固定

### 2.1 目标

形成一套不再反复改动的数据协议。

### 2.2 当前固定方案

- `GlaS official train(85)` 内部拆成 `train68 / val17`
- `train68 / val17` 默认按 `benign / malignant` 标签做分层抽样，尽量保持原始 `37 / 48` 的类别比例
- `GlaS TestA(60)` 和 `TestB(20)` 只做最终评估
- `CRAG` 保持固定 train/test 结构，不参与调参
- 实例标注先转二值语义分割

### 2.3 这一步允许改什么

- CSV 划分文件
- 图像与 mask 配对逻辑
- 实例转语义脚本
- 数据统计脚本

### 2.4 这一步不允许改什么

- 一边建模一边改 split
- 在看到测试集表现后回头重划数据
- 没记录随机种子就重新拆分

### 2.5 必须产出

- `glas_train68.csv / glas_val17.csv / glas_testA60.csv / glas_testB20.csv`
- `crag_train153.csv / crag_val20.csv / crag_test40.csv`
- 样本数统计
- 前景比例统计
- 对象尺度粗统计
- 数据检查报告
- 重复风险排查记录

### 2.6 过线标准

- image / mask 配对已人工抽查
- split 固定
- 输入尺寸固定
- 你能解释前景比例和对象尺度大致分布
- 你确认过没有明显数据泄漏风险

### 2.7 回退规则

如果下面任意一条成立，不允许进 `UNet`：

- 还在反复改 split
- 还没确认 image / mask 对齐
- 还不知道样本统计
- 还没固定实例转语义逻辑

### 2.8 本阶段现在锁死与后续再锁死

本阶段现在锁死：

- `GlaS train68 / val17 / TestA / TestB` 的角色分工
- `CRAG` 只做第二 benchmark，不参与当前主调参
- 实例标注先转二值语义分割

进入执行层后再锁死：

- 正式 `csv` 文件名和存放路径
- 图像归一化实现细节
- 数据检查报告的最终模板

## 3. 阶段 B：`UNet` 跑通

### 3.1 目标

证明数据读取、训练、验证、测试、指标、可视化这条链条是通的。

### 3.2 当前默认设置

- 模型：`U-Net`
- seed：先跑 `3407`
- loss：`BCE + Dice`
- optimizer：`AdamW`
- lr：`1e-3`
- scheduler：`ReduceLROnPlateau`
- epoch：最大 `120`
- early stopping：`20`
- threshold：`0.5`
- `batch size`：按当前显存能稳定训练且不溢出的最大值先定一版，但此时只记为默认值，不记为永久正式协议

### 3.3 这一步允许改什么

- 代码 bug
- 数据读取 bug
- 指标脚本 bug
- 可视化输出 bug

### 3.4 这一步不允许改什么

- 因为一次结果不好就随意换模型
- 还没跑通就开始堆模块
- 看测试集结果后改阈值

### 3.5 必须产出

- `best checkpoint`
- `TestA / TestB` 正式结果
- 训练日志
- loss 曲线
- 一组预测样例图
- 一份错误类型简述

### 3.6 过线标准

- 没有大面积全背景预测
- 没有明显系统性错位
- 指标和可视化没有严重矛盾
- 你能说清 `UNet` 的主要错误类型

### 3.7 回退规则

如果跑不顺，排查顺序固定为：

1. 数据读取与 mask 对齐
2. 实例转语义与 resize
3. loss、阈值和指标脚本
4. 模型实现

### 3.8 本阶段现在锁死与后续再锁死

本阶段现在锁死：

- `UNet` 只负责流程排雷，不承担最终主模型角色
- 首轮只跑 `3407`
- 不允许同时改模型和训练协议来“救结果”

进入执行层后再锁死：

- 首版可复现 `batch size`
- 首版日志命名规范
- 首版可视化导出模板

## 4. 阶段 C：`UNet` 稳定

### 4.1 目标

得到最小可复现基线均值和方差。

### 4.2 当前固定要求

- seed 固定：`3407 / 1234 / 2025`
- 三次实验的训练协议保持一致
- 主表结果开始记录 `mean +- std`

### 4.3 必须产出

- `3` 个 seed 的测试结果
- `mean +- std`
- 三组预测图
- `UNet` 失败案例汇总

### 4.4 过线标准

- 三次实验都能独立跑通
- 错误类型基本一致
- `UNet` 已能作为流程基线写进文档

### 4.5 回退规则

如果三次波动失控，不允许进主 baseline，先查：

- 数据稳定性
- 训练协议是否偷偷变化
- 随机增强是否过强

---

## 5. 阶段 D：主 baseline 稳定

### 5.1 目标

拿到一个后面所有改进都要对比的正式基座。

### 5.2 当前固定方案

- 主 baseline：`ResNet34-U-Net`
- 从 `UNet` 迁移时，优先只改模型结构
- encoder 是否使用预训练权重必须全程统一；如果第一版启用预训练，后面对比不允许偷偷改成随机初始化
- `batch size` 进入 baseline 稳定阶段后必须冻结，不允许模块阶段再改

### 5.3 这一步允许改什么

- backbone 接入实现
- 编码器输出接口
- decoder 对接细节

### 5.4 这一步不允许同时改什么

- 输入尺寸
- 数据增强包
- loss
- threshold
- 数据划分
- 边界标签
- 距离图

### 5.5 必须产出

- `3` 个 seed 的 baseline 结果
- `mean +- std`
- `UNet vs baseline` 对比表
- baseline 失败案例
- 参数记录和训练日志

### 5.6 过线标准

- 平均结果不低于 `UNet`
- 三次重复方向一致
- 失败模式基本稳定
- 已能作为全文正式比较基座

### 5.7 回退规则

如果 baseline 还不稳，不允许加模块，先查：

1. backbone 对接是否正确
2. loss / lr / batch size 是否适配
3. 训练日志是否存在异常跳动

### 5.8 本阶段现在锁死与后续再锁死

本阶段现在锁死：

- 正式主 baseline 就是 `ResNet34-U-Net`
- 从 `UNet` 到 baseline 只允许先改结构，不允许顺手换协议
- baseline 稳定前，`LKMA / Boundary Head / Distance-aware Loss` 一律不进入主线

进入执行层后再锁死：

- 正式 baseline 的 `batch size`
- encoder 预训练策略
- baseline 结果记录模板

---

## 6. 阶段 E：`LKMA`

### 6.1 目标

验证形态感知大核建模是否值得保留。

### 6.2 当前文献支持

这一层主要由下面几组文献支撑：

- `Large Kernel Matters`
- `RepLKNet`
- `WTConv`
- `MILD-Net`
- `Morphology-Inspired Unsupervised Gland Segmentation`

### 6.3 当前首轮默认实现

为了防止这一层变成“到处插模块”，当前只允许下面这套首轮默认方案：

- 首轮默认位置：最深层特征，也就是 encoder 最后一级输出与 decoder 第一层衔接之间的 bottleneck 位置
- 备选位置最多只允许再试 `1` 个：decoder 第一层输入位置
- 首轮默认策略：单模块、单位置、单尺度，不做并联多分支大核设计
- 首轮比较上限：`baseline`、`baseline + LKMA(pos1)`，必要时再补 `baseline + LKMA(pos2)`
- 本阶段所有结果默认同时保留主指标、边界指标和代表性可视化，避免后面只能靠文字回忆 `LKMA` 是否有效

### 6.4 这一步现在锁死什么

- `LKMA` 首轮最多只允许比较 `2` 个位置
- 不允许同时改 backbone、loss、增强和 `LKMA`
- 不允许把大核、多尺度、attention、额外边界模块混成一次实验
- 判断是否保留时，优先看对象级与边界相关证据，不接受只凭单个像素指标决定

### 6.5 进入执行层后再锁死什么

- `LKMA` 最终插入位置
- 最终 kernel size 或等效感受野设置
- 最终模块内部归一化与激活形式
- 最终是否只保留 `1` 个位置版本进入主表

### 6.6 固定不变项

- 数据划分
- 输入尺寸
- 增强包
- loss
- threshold
- seed

### 6.7 必须产出

- `baseline + LKMA` 的 `3` 个 seed 结果
- 与 baseline 的对比表
- 腺体轮廓可视化对比
- 训练成本记录

### 6.8 过线标准

下面至少满足大部分，才比较像值得保留：

- `Object Dice` 提升
- 多个 seed 下提升方向基本一致
- 边界指标没有明显退化
- 可视化里腺体轮廓更完整

### 6.9 回退规则

如果收益接近波动范围或成本增加明显：

- 不继续扩展更多大核变体
- 可直接判为“不保留”或“保留为备选”

## 7. 阶段 F：`Boundary Head`

### 7.1 目标

验证对象黏连与边界模糊问题是否得到更直接改善。

### 7.2 当前文献支持

主要由下面几条文献链支撑：

- `DCAN`
- `Gated-SCNN`
- `TA-Net`
- `DEA-Net`
- 边界评价相关辅助文献

### 7.3 当前首轮默认实现

当前为了避免边界分支被做散，首轮默认实现直接限定为：

- 边界标签：由二值 mask 生成单像素轮廓后再做固定宽度扩张的边界带
- 首轮默认边界宽度：`3 px`
- 备选边界宽度：只允许再试 `5 px`
- 边界分支位置：只接最终输出对应的最后一级 decoder，不做多级边界分支
- 边界损失默认：`BCE`
- 边界损失权重默认：`0.3`
- 权重备选范围：`0.2 / 0.3 / 0.5`
- 这一层的结果记录默认服务后续 direct evidence 表达，不能只保留一个最终 Dice

### 7.4 这一步现在锁死什么

- 只做单边界分支
- 边界宽度首轮只允许在 `3` 和 `5` 里选
- 边界损失权重首轮只允许在 `0.2 / 0.3 / 0.5` 里选
- 不允许同时引入多条边界支路、拓扑损失和复杂后处理

### 7.5 进入执行层后再锁死什么

- 最终边界标签生成脚本
- 最终边界宽度
- 最终边界损失形式
- 最终边界损失权重
- 最终是否把边界分支保留到主模型

### 7.6 固定不变项

- 数据划分
- 输入尺寸
- 增强包
- baseline 主干
- `LKMA` 配置
- threshold
- seed

### 7.7 必须产出

- `Boundary Head` 版本 `3` 个 seed 结果
- 边界指标表
- 对象级指标表
- 边界局部放大图
- 黏连腺体分离对比图

### 7.8 过线标准

至少满足下面之一：

- `Boundary F1` 提升
- `HD95` 或 `Object Hausdorff` 改善
- 黏连样本分离效果更好

并且同时满足：

- 主指标没有明显崩
- 可视化能支撑结论

### 7.9 回退规则

如果边界指标没提升、可视化也看不出改善：

- 先简化边界分支
- 不建议直接往上叠 distance 项
- 必要时回退到只保留 `LKMA`

## 8. 阶段 G：`Distance-aware Loss`

### 8.1 目标

验证它是不是值得保留的增强项，而不是默认主贡献。

### 8.2 当前文献支持

主要来自：

- `Distance Map Loss Penalty Term`
- `Efficient Segmentation Learning Downsampling Near Semantic Boundaries`

### 8.3 当前首轮默认实现

当前这一层只允许作为补充项验证，不允许反客为主。首轮默认实现限定为：

- 距离图：先用基于二值 gland mask 的 Euclidean distance map
- 距离图范围：归一化到 `[0, 1]`
- 第一版不做 signed distance、方向场或多种距离定义混合
- `lambda_dist` 默认：`0.1`
- `lambda_dist` 备选范围：`0.05 / 0.1 / 0.2`
- 首轮只允许在已经锁定的 `Baseline + LKMA + Boundary Head` 上追加

### 8.4 这一步现在锁死什么

- 距离约束只能作为补充 loss，不允许顺手改结构
- `lambda_dist` 首轮只允许在 `0.05 / 0.1 / 0.2` 里选
- 不允许同时改边界分支配置和距离约束
- 不允许把距离约束写成默认主贡献
- 如果收益不稳定，默认优先降级为补充实验，而不是硬塞进最终模型

### 8.5 进入执行层后再锁死什么

- 最终距离图生成脚本
- 最终距离图归一化策略
- 最终 `lambda_dist`
- 最终是否进入主模型还是降级为补充实验

### 8.6 固定不变项

- 数据划分
- 输入尺寸
- baseline 主体
- `LKMA`
- `Boundary Head`
- threshold
- seed

### 8.7 必须产出

- `Distance-aware` 版本 `3` 个 seed 结果
- 和上一版的差异表
- 失败案例比较
- 一条明确结论：保留或删除

### 8.8 过线标准

更像值得保留的情况：

- 边界指标继续提升
- `Object Dice` 继续提升
- 多个 seed 方向一致

### 8.9 回退规则

如果收益接近波动范围：

- 直接降级为补充实验
- 主线回退到 `Baseline + LKMA + Boundary Head`

## 9. 阶段 H：投稿版第一批外部对比

### 9.1 目标

形成一张代表性足够、成本可控、能直接服务 `GlaS` 主表的投稿版结果表。

### 9.2 必跑名单

- `UNet++`
- `DeepLabV3+`
- `Attention U-Net`

这三类通用外部对比默认遵守同一条协议：

- 统一沿用 `ResNet34-U-Net` 阶段已经锁定的训练协议
- 默认保持 `input size / aug / optimizer / lr / scheduler / epochs / threshold` 一致
- 默认只替换网络结构，不顺手改训练口径
- 如果某个方法因架构约束确实不能完全共用协议，例如 `nnU-Net` 这类带自适应配置的方法，必须单独记录差异，并在论文表注或实验设置中明确说明

这一阶段还有一个固定目标：

- 不是只补“通用外部方法数量”，而是开始把最终英文主表所需的结果列名、表注和对照分类提前整理好

### 9.3 主表里必须明确纳入的任务内直接量化对照

这一层不是“能不能顺手跑”的问题，而是投稿主表口径必须提前锁清：

- `GlaS` 主表优先对齐：`DCAN / MILD-Net-RTS / DEA-Net / SDT`
- `CRAG` 主表优先对齐：`MILD-Net-RTS / TA-Net / DEA-Net`

执行原则：

- 如果统一协议复现条件成熟，优先把其中最关键的 `1-3` 个做成可复现实验对照
- 如果部分方法复现代价过高，也至少要作为任务内文献 direct comparison 明确进入主结果表，不能只留在 related work
- `CRAG` 上尤其不能只拿通用外部模型说话，必须有 gland-specific 强对照的量化参照

数字来源规则也在这里提前锁死：

- 任务内 direct comparison 优先采用原文在同一 benchmark、同一 split、同一指标口径下报告的数字
- 如果是你自己按统一协议复现得到的结果，在结果表里标为 `reproduced`
- 如果是直接引用原文数字，在结果表里统一用 `†` 标记，并在表注中说明“quoted from original paper”
- 如果 benchmark、split、指标口径或测试设置差异过大，就不要硬塞进主结果表，直接降级到“不可直接横比但可保留参考”的区域

### 9.4 选跑名单

从下面只选 `1-2` 个：

- `nnU-Net`
- `TransUNet`
- `Swin-Unet`
- 一个 gland-specific 近年方法

### 9.5 当前建议

优先级建议：

1. `nnU-Net`
2. `TransUNet`
3. `Swin-Unet`

### 9.6 这一步现在锁死什么

- 投稿第一版主表默认先完成 `3` 个必跑外部对比
- 投稿第一版主表必须同时明确任务内 direct comparison 的最小集合，不能只做通用外部对比
- 增强外部对比最多只补 `1-2` 个
- 第一批外部对比优先服务 `GlaS` 主表，不和 `CRAG` 混成一阶段
- 如果复现成本过高，增强外部对比优先降级，不允许拖慢主线闭环
- 主线实验默认不引入后处理，先用最直接的 `argmax / threshold` 输出完成主表
- 如果后面必须增加简单后处理，例如 `remove small objects` 或形态学清理，必须单独做一行后处理消融，并在主表或表注中明确标记
- 对比方法如果原文结果自带后处理，也必须在结果表里用 `*` 或表注显式说明，不能和“无后处理主线结果”混写成同一口径

### 9.7 进入执行层后再锁死什么

- 最终主表到底放哪 `3` 个外部方法
- 最终哪些 gland-specific 强对照做统一协议复现，哪些保留为文献 direct comparison
- 是否补 `nnU-Net / TransUNet / Swin-Unet` 中的哪一个
- 外部方法是否补 `3` seed 或只做统一协议单次复现并明确说明

### 9.8 回退规则

如果外部方法复现成本明显失控：

- 降级为引用对比
- 不要为了凑方法数拖慢主线闭环

## 10. 阶段 I：`CRAG`

### 10.1 目标

确认主线优势在第二个同任务 benchmark 上没有完全消失。

### 10.2 当前建议测试名单

`CRAG` 至少覆盖：

- `UNet`
- `ResNet34-U-Net`
- 最终模型

主结果表层面还应至少对齐：

- `MILD-Net-RTS`
- `TA-Net`
- `DEA-Net`

如果资源允许，再补：

- 第一批外部对比中在 `GlaS` 上最强的 `1` 个方法

### 10.3 必须产出

- `CRAG` 结果表
- `CRAG` 成功案例图
- `CRAG` 失败案例图
- `GlaS + CRAG` 联合总结表
- 一条明确判断：当前结果更适合“中文保底收口”还是已经具备“英文起步整理”条件

### 10.4 过线标准

- 相对 `GlaS` 没有崩溃式下降
- 最终模型相对 baseline 的优势没有完全消失
- 错误类型可解释

### 10.5 回退规则

如果 `CRAG` 明显崩掉：

- 不要再堆新模块
- 不要写“强泛化”
- 把稿件降成以 `GlaS` 为主、`CRAG` 为补充验证

## 11. 阶段 J：图表与稿件出口判断

### 11.1 必须有的表

- 主线逐步消融表
- 外部方法主结果对比表
- task-specific direct comparison 主结果表
- 对象级与边界级指标表
- `GlaS + CRAG` 联合结果表

### 11.2 必须有的图

- 数据样例图
- 主结果可视化图
- 黏连腺体分离图
- 边界局部放大图
- 失败案例图
- `CRAG` 成功/失败案例图

### 11.3 中文保底出口最低要求

至少要能写出：

- 中文摘要
- 中文引言
- 中文方法
- 中文实验
- 中文结论

并且默认同时满足：

- 主线逐步消融已经能自圆其说
- task-specific direct comparison 至少已整理好引用值和表注来源
- `CRAG` 已能作为补充验证写清“是否保住主线优势”

如果中文稿都写不顺，说明实验线还没有真正收口。

### 11.4 可以直接按英文起步整理的结果包

如果准备不经过“先写完中文再升级”的路径，而是直接按普通英文应用期刊 / `SCI` 低区起步，至少应同时具备：

- 通用外部对比表已经成型
- task-specific direct comparison 已进入主表而不是只在 related work 里出现
- `GlaS + CRAG` 联合结果已经能支持“主结论 + 第二 benchmark 验证”
- 对象级、边界级和黏连分离可视化至少各有一组能直接进正文
- 模块职责表达已经稳定为“`LKMA` 负责 morphology / context，`Boundary Head` 负责 object separation，`Distance-aware Loss` 只是可选增强”

如果这些结果资产已经齐，就允许直接按英文写作组织；如果还没齐，就先按中文保底收口，不额外制造路线焦虑。

### 11.5 最后出口判断

这一阶段结束时只允许输出下面三类结论之一：

1. `先按中文保底收口`
2. `直接按普通英文应用期刊 / SCI 低区起步`
3. `结果包还不完整，回退到前面阶段补实验`

## 12. 一句话执行原则

> 先把数据、`UNet` 和主 baseline 跑稳，再用 `LKMA -> Boundary Head -> Distance-aware Loss(可选)` 的顺序逐层验证收益；每一层都必须有对象级、边界级或可视化中的至少一类明确证据，并把结果资产按“中文可收口、英文可直起”的标准保存，否则就不保留到最终模型。
