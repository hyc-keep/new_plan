## 为什么当前是这 `50` 个

当前这批对象被纳入，是因为它们正好构成 `02_UNet流程验证` 的最小可审计闭环:

1. 训练入口、验证入口、可视化入口都在其中。
2. 配置冻结来源都在其中。
3. data-model-loss-engine-eval 主链都在其中。
4. 主 run 的 full run 正式资产都在其中。
5. 规范 smoke run 的对照资产也在其中。

尤其要强调主 run 的资产口径已经变化:

1. 旧说明文曾记录 `best_epoch=48`、`epoch_count=68`，这些属于历史旧轮次，当前不可消费；当前真实 `run_meta.yaml` 以 `best_epoch=50`、`best_metric_value=0.7515312717616618`、`epoch_count=70`、`device=cuda`、`testA_objdice=0.6949207519755126`、`testB_objdice=0.750951634413634` 为准。
2. 旧说明文中的 `train_log.csv` 和 `val_metrics.csv` 是 `1..68` 的历史长程记录，当前不可消费，不能代表当前 A1 的 `epoch_count=70`。
3. `run_summary.md` 说明文保留对象映射，但其旧 full-run 数值只能作历史追溯，不能作为当前正式摘要消费。
4. `testA_metrics.csv`、`testB_metrics.csv`、`metric_crosscheck_note.md`、`error_cases.md` 已把测试与可视化链补齐。

因此，当前主目录不再属于“最小 smoke run 资产”。

原因不是这 `50` 个对象不够关键，而是 stage02 的正式对象总量本来就大于当前锁定范围。

如果后续继续扩范围，仍然要重新完成下面几步:

1. 把新对象纳入清单裁决。
2. 补源码来源 docstring。
3. 补对象级说明文。
4. 同步更新 `README.md` 和 `implementation_status.md`。

所以当前能成立的诚实结论是:

1. 当前锁定范围内的 `50` 个 A 类对象已经纳入正式说明文覆盖。
2. 当前正式 A1 结果已与真实 run_meta.yaml 对齐；主 run 旧 full-run 说明文明确标为历史资产。
3. 这不等于 stage02 全量所有对象都已完成说明文收口。
