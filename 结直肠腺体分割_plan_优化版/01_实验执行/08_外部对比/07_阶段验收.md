## 计划 lineage 与下游 handoff

当前/未来阶段验收必须记录：`source_stage`、`source_manifest`、`source_protocol_version`、`source_run_name`、`consumer_stage`、`consumer_file`、`consumption_boundary`。缺失任一字段、当前轮 `eval_proto_version/postprocess_version/tta_version` 未经研究定标/阶段锁定/Pre-check 后 Gate 冻结、或计划文件/config/run_name/run_meta/manifest/gate/handoff 命名不同步时，验收 Gate 必须 blocked，回到治理/阶段锁定，不得编码或进入主表。

这份文件不是给 `08_外部对比` 再补一页“差不多完成了”的收尾说明，也不是把旧版硬门槛清单加长几条就算正式验收。

它在当前阶段中的唯一职责是：

> 作为 `08_外部对比` 的阶段放行协议，把 `Gate_H1` 的真正通过线、最低交付物、红线问题、独立回退条件、下游 handoff 资产和重新放行条件一次写死。自本文件起，`08_外部对比` 不再允许以“几个外部模型已经跑完”为理由提前过线；只有当统一公平协议、首批外部层、来源边界、结果表层和下游可消费资产同时闭环后，才允许进入 `09_CRAG验证` 与 `10_结果汇总`。

从现在开始，`08_外部对比` 阶段是否通过，只能按本文件裁决；若与同层执行细则或阶段总结冲突，以本文件为准回改。

## 1. 文件角色与执行边界

### 1.1 当前文件负责什么

当前文件只负责冻结下面八件事：

1. `08_外部对比` 阶段真正要证明什么，以及什么状态才算“外部比较层已经成立”。
2. `Gate_H1` 的正式通过线，也就是 `current_mainline_ready / fairness_protocol_fixed / core_external_runs_complete / source_boundary_ready / external_table_ready / handoff_ready` 六个子门的验收关系。
3. 首批外部方法 `UNet++ / DeepLabV3+ / Attention U-Net` 在阶段级必须交出的最小闭环资产。
4. `reproduced / † / *`、`single_seed / mean+-std`、`direct_comparison_eligible` 和主表资格在阶段验收中的必过边界。
5. external_main_table_draft.csv、task_specific_direct_comparison_draft.csv、plus_external_entry_decision.md 等正式产物是否已经足以支撑下一阶段。
6. 红线问题、独立 `回退条件`、固定回退顺序和回退后的重新放行条件。
7. 阶段验收在代码和结果层中的落地接口、前置断言和 handoff 入口。
8. 当前文件自身的 `冲突裁决记录`、不缩水的 `文件质量自检` 和独立 `Diagnostics 闭环`。

### 1.2 当前文件不负责什么

当前文件不重新定义下面这些内容：

- `current_mainline` 的来源；它只能由 `07_Distance` 的正式结论唯一给出。
- 统一数据、训练、评估协议本体；它们由 `结直肠腺体分割_plan_优化版/01_实验执行/08_外部对比/01_统一公平协议.md` 及其上游文件冻结。
- `UNet++`、`DeepLabV3+`、`Attention U-Net` 三个方法文件各自的结构本体与最小必要差异细则。
- `结直肠腺体分割_plan_优化版/01_实验执行/08_外部对比/05_增强对比进入条件.md` 中增强外部候选的优先级和准入细节本体。
- `结直肠腺体分割_plan_优化版/01_实验执行/08_外部对比/06_引用值与复现值标记规则.md` 中来源标记和 direct comparison 准入公式本体。
- `09_CRAG验证` 与 `10_结果汇总` 各自的阶段角色定义和结果整理细则。

### 1.3 为什么当前文件必须独立存在

`08_外部对比` 最容易出现的伪完成状态不是“完全没跑”，而是：

- 三个外部方法都跑过一次，但仍说不清是否共享同一公平协议。
- 主表草稿已经有了数字，但 `reproduced / † / *` 与 `single_seed / mean+-std` 还没有真正固定。
- `TestA / TestB`、对象级指标和 direct comparison 资格仍然混着写，却想直接进入下一阶段。
- 首批外部层还没收口，就先把增强外部扩成新的主要工作量。
- `09_CRAG验证` 和 `10_结果汇总` 想读取外部层时，只能读到分数，读不到主表资格、来源边界和 handoff 状态。

如果没有一份独立阶段验收协议，后续文件就无法知道：哪些结果已经具备正式外部层身份，哪些还只是 screening 或参考值，哪些问题必须先回退而不是继续推进。

## 2. 核心验收问题

### 3.1 当前阶段真正要验什么

`08_外部对比` 真正要验收的不是“跑了几个外部模型”，而是：

> 在 `current_mainline` 已唯一冻结、统一公平协议已固定、首批外部方法已完成最小闭环、来源边界已明确、主表外部层已形成且下游可直接消费的前提下，是否已经形成第一层可进入论文主结果体系的正式外部比较层。

### 3.2 当前阶段不允许用什么替代正式通过

下面这些状态都不能被写成“阶段通过”：

- 只有方法 run，没有正式主表草稿和 task-specific direct comparison 草稿。
- 有主表草稿，但 `reproduced / † / *`、`single_seed / mean+-std` 仍然混写。
- 有结果数字，但实现来源、适配差异、特殊测试设置和公平修正还不可追溯。
- 首批外部层尚未收口，就把增强外部扩成新的主要任务。
- `09_CRAG验证` 与 `10_结果汇总` 仍然无法直接读取外部层资格与来源边界。

### 3.3 `Gate_H1` 正式通过式

当前文件固定采用下面的阶段总门：

```text
Gate_H1 = current_mainline_ready
          AND fairness_protocol_fixed
          AND core_external_runs_complete
          AND source_boundary_ready
          AND external_table_ready
          AND handoff_ready
```

只有 `Gate_H1 = true`，本阶段才允许声明通过。

### 3.4 六个子门的阶段含义

- `current_mainline_ready`：`07_Distance` 已形成正式 `keep / downgrade / drop` 结论，且当前外部对比只使用唯一合法 `current_mainline`。
- `fairness_protocol_fixed`：统一数据、训练、评估和记录协议已经落入配置、运行和结果文件，没有方法间协议漂移。
- `core_external_runs_complete`：`UNet++ / DeepLabV3+ / Attention U-Net` 已完成最小闭环；若满足竞争条件，已补齐三 seed 聚合。
- `source_boundary_ready`：`reproduced / † / *`、`single_seed / mean+-std`、`direct_comparison_eligible` 和特殊测试设置边界已固定。
- `external_table_ready`：external_main_table_draft.csv、task_specific_direct_comparison_draft.csv 和增强外部进入结论已经成型。
- `handoff_ready`：`09_CRAG验证` 与 `10_结果汇总` 所需资产已经形成，不需要额外口头解释才能继续。

## 4. 本阶段规则卡片

下面所有核心规则都写成最终协议格式。后续阶段总结、表格整理和放行判断只允许按这些规则执行，不允许再口头放宽。

### 4.1 `Gate_H1` 总裁决规则

- 当前结论：只有 `Gate_H1 = true`，`08_外部对比` 才允许通过。
- 规则类型：`阶段放行规则`
- 适用阶段：`08_外部对比`
- 直接依据：`结直肠腺体分割_plan_优化版/01_实验执行/08_外部对比/00_阶段总协议.md`
- 核心公式或定义参考：`Gate_H1 = current_mainline_ready AND fairness_protocol_fixed AND core_external_runs_complete AND source_boundary_ready AND external_table_ready AND handoff_ready`
- 采用原因：阶段通过的含义必须是“正式外部层已成立”，而不是“若干 run 已存在”。
- 不采用的相邻方案：不采用“主表差不多有了就先过线”；不采用“等下游再整理来源边界”。
- 代码落点：crc_gland_segmentation_project/scripts/summarize_stage.py、crc_gland_segmentation_project/reports/stage_reports/external_stage_summary.md
- 运行记录字段：`gate_h1`, `stage_pass`, `pass_blocker`
- 验收方式：检查六个子门都有明确状态、依据和结论。

### 4.2 `current_mainline_ready` 规则

- 当前结论：外部对比只允许从 `07_Distance` 正式 handoff 的唯一 `current_mainline` 出发。
- 规则类型：`handoff 冻结规则`
- 适用阶段：`08_外部对比`
- 直接依据：`结直肠腺体分割_plan_优化版/01_实验执行/07_Distance/00_阶段总协议.md`、`结直肠腺体分割_plan_优化版/01_实验执行/07_Distance/04_保留或降级标准.md`
- 核心公式或定义参考：`current_mainline = (distance_decision_level == keep) ? distance_kept_base : distance_input_base`
- 采用原因：如果比较起点不唯一，外部比较、公平协议和下游表格都失去统一参照。
- 不采用的相邻方案：不采用把 `distance_kept_base` 和 `distance_input_base` 同时写进同一层正式外部比较。
- 代码落点：crc_gland_segmentation_project/scripts/run_external.py、run_meta.yaml
- 运行记录字段：`current_mainline`, `distance_decision_level`, `mainline_source_note`
- 验收方式：检查所有正式外部 run 与表格条目都引用同一个 `current_mainline`。

### 4.3 `fairness_protocol_fixed` 规则

- 当前结论：所有首批外部 run 必须共享统一数据、训练、评估和结果记录协议。
- 规则类型：`公平比较规则`
- 适用阶段：`08_外部对比`
- 直接依据：`结直肠腺体分割_plan_优化版/01_实验执行/08_外部对比/01_统一公平协议.md`
- 核心公式或定义参考：`external_run_fair = unified_data_proto + unified_train_proto + unified_eval_proto + documented_minimal_gap_only`
- 采用原因：外部层要比较的是结构路线，不是各自默认仓库训练技巧。
- 不采用的相邻方案：不允许单独改 `best_selector`、单独改 `threshold_source`、单独改后处理、单独改 TTA 或特殊测试链。

## 7. 回退条件

### 7.1 独立回退触发条件

即使结果已经跑完，只要出现下面任意一条，也必须先回退修正，而不是直接进入下游阶段：

- `current_mainline` 仍不唯一，或正式外部 run 无法追溯到同一个 `current_mainline`。
- 不同外部方法使用了不同 `best_selector`、不同 `threshold_source`、不同后处理或不同 TTA。
- `reproduced / † / *`、`single_seed / mean+-std`、`direct_comparison_eligible` 仍然没有字段级统一定义。
- external_main_table_draft.csv、task_specific_direct_comparison_draft.csv 和 plus_external_entry_decision.md 之间对同一方法的资格判断不一致。
- 首批外部层尚未形成，却已经试图以增强外部结果替代阶段说服力。
- `09_CRAG验证` 或 `10_结果汇总` 所需资产仍然缺失，只能靠口头说明继续。

### 7.2 固定回退顺序

回退时统一按下面顺序排查：

1. 先检查 `current_mainline` 与 `07_Distance` handoff 是否一致。
2. 再检查统一数据、训练、评估和记录协议是否真的落入配置、脚本和结果文件。
3. 再检查三类首批外部方法的实现来源、适配差异和正式结果文件是否齐全。
4. 再检查来源标记、聚合身份和 direct comparison 资格是否完整一致。
5. 最后再检查主表草稿、task-specific 草稿、增强外部决议和下游 handoff 是否对齐。

### 7.3 回退后的重新放行条件

发生回退后，只有同时满足下面条件，才允许重新声明本阶段通过：

- 问题来源已经按固定回退顺序定位并修复。
- 修复动作、影响范围和复验结果已写回 external_reproduction_note.md 或 external_stage_summary.md。
- external_main_table_draft.csv、task_specific_direct_comparison_draft.csv、plus_external_entry_decision.md 与文字结论重新对齐。
- 当前阶段重新满足第 `4-5` 节的规则、交付物和下游 handoff 要求。
