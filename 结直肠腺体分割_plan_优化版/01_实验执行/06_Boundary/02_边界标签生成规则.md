# 边界标签生成规则

本文件不是 `06_Boundary` 阶段里“边界图大概怎么做”的说明文，也不是训练前随手写的实现备忘录。

它在当前项目中的唯一职责是：

> 作为 `06_Boundary` 的正式边界目标 protocol，把边界标签唯一从哪里来、唯一按什么流程生成、哪些字段必须被版本化、训练/可视化/分析三条链怎样保证完全同口径，以及这些规则如何落到代码、运行记录、质检、回退和下游交接，一次写死。

从现在开始，只要讨论 `Boundary Head` 使用的监督目标来源、边界轮廓定义、宽度规则、生成时机、字段链和一致性检查，都必须以本文件为准；若与同层文件冲突，以 `结直肠腺体分割_plan_优化版/01_实验执行/06_Boundary/00_阶段总协议.md` 管阶段身份与门控，以 `结直肠腺体分割_plan_优化版/01_实验执行/06_Boundary/01_设计依据.md` 管“为什么边界目标只能这样定义”，而本文件负责把“边界标签究竟怎样唯一生成”压成可直接落代码、可直接验收、可直接交接的正式协议。

## 1. 文件角色与执行边界

### 1.1 当前文件负责什么

当前文件只负责冻结下面八件事：

1. `Boundary` 阶段监督目标唯一来自哪里，以及为什么必须固定为冻结二值 gland mask。
2. 边界标签唯一合法流程是什么，以及为什么必须固定为 `binary mask -> single-pixel contour -> fixed-width dilation -> boundary band`。
3. 轮廓定义、宽度候选、标签值定义、生成时机和 resize 口径怎样收口成唯一协议。
4. 训练、可视化、边界案例分析和边界指标准备怎样复用同一正式实现，避免三条链各写一套。
5. 哪些代码入口、配置字段和运行记录字段必须显式存在，不能再藏在默认值里。
6. 边界标签协议冻结后，最小人工抽查、一致性验证和异常排查必须怎样执行。
7. 本文件怎样作为 `结直肠腺体分割_plan_优化版/01_实验执行/06_Boundary/03_实验步骤.md`、`结直肠腺体分割_plan_优化版/01_实验执行/06_Boundary/04_保留或删除标准.md` 和 `结直肠腺体分割_plan_优化版/01_实验执行/07_Distance/00_阶段总协议.md` 的上游标签母规范。
8. 当边界标签协议漂移时，什么情况必须先回退修正，而不能继续跑实验或向下游交接。

### 1.2 当前文件不负责什么

当前文件不重新定义下面这些内容：

- `Boundary` 阶段为什么值得做，以及 `Boundary Head v1` 为什么只能是最小可解释抽象。
- `Boundary Head` 的接入位置、边界损失类型与 `lambda_boundary` 的设计裁决。
- `screening_main -> three_seed_main -> optional_one_backup -> stage_decision` 的执行顺序。
- `keep / backup / drop` 的最终证据排序和阶段裁决细则。
- 数据身份、split、输入尺寸、主 gland 标签定义、训练协议和评估协议本体。
- `Distance-aware Loss`、外部对比与 `CRAG` 阶段的目标定义。

这些职责分别以下列文件为准：

- `结直肠腺体分割_plan_优化版/01_实验执行/06_Boundary/00_阶段总协议.md`
- `结直肠腺体分割_plan_优化版/01_实验执行/06_Boundary/01_设计依据.md`
- `结直肠腺体分割_plan_优化版/01_实验执行/06_Boundary/03_实验步骤.md`
- `结直肠腺体分割_plan_优化版/01_实验执行/06_Boundary/04_保留或删除标准.md`
- `结直肠腺体分割_plan_优化版/01_实验执行/01_数据协议/00_阶段总协议.md`
- `结直肠腺体分割_plan_优化版/01_实验执行/04_Baseline/02_训练协议.md`
- `结直肠腺体分割_plan_优化版/01_实验执行/07_Distance/00_阶段总协议.md`

### 1.3 为什么当前文件必须独立存在

因为 `Boundary` 阶段最容易出现的伪完成状态不是“完全没写边界标签”，而是：

- 训练、可视化和分析都说自己在用“边界标签”，但其实轮廓定义或宽度不同。
- 主版本名写成同一个 `Boundary Head`，实际却混用了不同 `boundary_width` 或不同生成时机。
- 边界标签一边从 resize 前 mask 生成，一边又拿去和 resize 后样本训练，导致目标错位。
- 结果里看到边界收益，却说不清收益来自边界头，还是来自目标定义被偷偷换掉。
- 下游 `07_Distance` 想继承 `Boundary` 的结果时，说不清它继承的是哪一版边界目标协议。

如果没有一份独立标签协议文件，`结直肠腺体分割_plan_优化版/01_实验执行/06_Boundary/01_设计依据.md` 会被迫同时承担“为什么这样定义”和“具体怎样生成”的双重职责，`结直肠腺体分割_plan_优化版\01_实验执行\06_Boundary\03_实验步骤.md` 与 `结直肠腺体分割_plan_优化版\01_实验执行\06_Boundary\04_保留或删除标准.md` 也会被迫各自补一套边界目标定义，最终使标签口径再次分叉。

## 3. 本阶段唯一允许处理的变量

### 3.1 当前文件真正要证明什么

本文件真正要证明的不是“边界图可以这样做”，而是下面六件事已经被正式写成唯一协议：

1. 当前 `Boundary` 阶段边界监督目标只能从冻结二值 gland mask 派生，而不能临时引入新的监督源。
2. 当前合法边界标签流程已经唯一固定为 `binary mask -> single-pixel contour -> fixed-width dilation -> boundary band`。
3. 当前轮廓定义、边界宽度、标签值定义、生成时机和实现入口已经收口成一条单一字段链。
4. 当前训练、可视化、案例分析和边界指标准备必须复用同一正式实现，不允许各自拷贝不同版本。
5. 当前边界目标协议已经足以直接支撑 `Boundary Head v1` 的代码接入、运行记录、人工抽查和阶段验收。
6. 当前文件已经能为 `结直肠腺体分割_plan_优化版\01_实验执行\06_Boundary\03_实验步骤.md`、`结直肠腺体分割_plan_优化版\01_实验执行\06_Boundary\04_保留或删除标准.md` 和 `07_Distance` 提供唯一合法的边界目标来源说明。

### 3.2 为什么边界目标必须先于正式 run 被写死

因为 `Boundary` 阶段当前真正要验证的是：

> 在 `current_base` 已冻结、`Boundary Head v1` 已压成最小可解释抽象的前提下，显式边界监督本身是否值得保留。

如果边界目标协议本身还漂移，后续就无法判断：

- 收益来自 `Boundary Head` 的监督价值。
- 还是来自目标定义、宽度或生成时机被偷偷换掉。

因此，标签协议必须先于 `screening_main` 和三 seed 主版本冻结，否则整个阶段都会从单变量验证退化成目标定义漂移实验。

### 3.3 当前文件与 `结直肠腺体分割_plan_优化版\01_实验执行\06_Boundary\01_设计依据.md` 的分工

两者的固定分工是：

- `结直肠腺体分割_plan_优化版\01_实验执行\06_Boundary\01_设计依据.md` 回答“为什么边界监督值得验证，以及为什么当前只能这样压缩变量”。
- 本文件回答“边界监督标签究竟怎样唯一生成，以及如何保证三条链完全同口径”。

如果把两者混写，就会同时丢失两种清晰性：

- 设计层会说不清哪些是“为什么”，哪些是“怎样生成”。
- 实现层会说不清哪些字段是边界头设计字段，哪些字段是边界目标字段。

原因是：边界标签协议若不与正式评估实现共同写死，后续 `Boundary F1`、对象级分离证据、per-seed raw、`mean+-std`、manifest 与 handoff 就无法确认是在同一边界定义上得到的正式结果。

### 4.3 当前文件明确不允许重开的内容

当前文件不允许借机重新打开：

- 主 gland 标签定义
- `GlaS` 或 `CRAG` 的 split 规则
- 输入尺寸与主 resize 流程本体
- `Boundary Head` 的接入位置
- 边界损失类型与 `lambda_boundary` 搜索边界
- 距离图、marker 图、topology 或实例中心图等相邻目标路线
- 训练协议、评估协议、阈值来源和后处理定义

换句话说，这里只负责“边界标签怎样唯一生成”，不负责重新发明新的监督对象或新的阶段变量。

## 5. 如何判断设计有效(验收标准)

下面所有核心规则都写成正式协议格式；后续实现、实验和阶段汇总时，只允许把这些规则忠实落地，不允许再擅自补新的隐含边界目标变量。

### 5.1 边界标签唯一来源规则

- 当前结论：边界标签唯一来源固定为数据阶段冻结的二值 gland mask，且二值前景定义固定为 `M(x) = 1[mask(x) > 0]`。
- 规则类型：`工程冻结规则 + 标签定义规则`
- 适用阶段：`06_Boundary`
- 直接依据：`结直肠腺体分割_plan_优化版/01_实验执行/06_Boundary/00_阶段总协议.md`、`结直肠腺体分割_plan_优化版/01_实验执行/06_Boundary/01_设计依据.md`、03_DCAN.md
- 核心公式或定义参考：`M(x) = 1[mask(x) > 0]`
- 采用原因：只有先把监督源唯一化，边界目标才能可追溯、可复算、可跨脚本复用。
- 不采用的相邻方案：不从距离图、marker 图、中心点图或人工临时边缘图直接定义当前阶段边界标签。
- 代码落点：`crc_gland_segmentation_project/src/data/mask_ops.py`、`crc_gland_segmentation_project/src/data/boundary_targets.py`
- 运行记录字段：`boundary_target_source`、`boundary_target_version`
- 验收方式：检查所有正式 `Boundary` run 的 run_meta.yaml 都明确记录边界标签来源为冻结二值 mask。

### 5.2 边界标签唯一生成流程规则

- 当前结论：首轮正式边界标签流程固定为 `binary mask -> single-pixel contour -> fixed-width dilation -> boundary band`。
- 规则类型：`工程冻结规则`
- 适用阶段：`06_Boundary`
- 直接依据：`结直肠腺体分割_plan_优化版/01_实验执行/06_Boundary/00_阶段总协议.md`、`结直肠腺体分割_plan_优化版/01_实验执行/06_Boundary/01_设计依据.md`、03_DCAN.md、01_Gated-SCNN.md
- 核心公式或定义参考：`C = Contour(M)`；`B_w = Dilate(C, w)`
- 采用原因：这个流程足够简单、可解释、可直接实现，同时能把监督聚焦在真实腺体轮廓附近。
- 不采用的相邻方案：不采用实例级复杂外轮廓重建；不采用多脚本各自实现不同 contour 流程；不把厚边界直接当 contour 本体。
- 代码落点：`crc_gland_segmentation_project/src/data/boundary_targets.py`
- 运行记录字段：`boundary_target_version`、`boundary_generation_impl`
- 验收方式：检查训练、可视化和分析脚本都调用同一正式边界生成入口。

### 5.3 单像素轮廓定义规则

- 当前结论：轮廓对象固定为冻结二值 mask 的单像素边界，不允许在首版协议里扩展成厚边界原型、双边界或多层轮廓。
- 规则类型：`工程冻结规则`
- 适用阶段：`06_Boundary`
- 直接依据：03_DCAN.md、`结直肠腺体分割_plan_优化版/01_实验执行/06_Boundary/01_设计依据.md`
- 核心公式或定义参考：`C = Contour(M)`，其中 `C` 表示 `M` 的单像素轮廓集合。
- 采用原因：先把边界本体与“边界带宽度”解耦，避免把轮廓定义和膨胀宽度混成同一个变量。
- 不采用的相邻方案：不直接把宽边界带当 contour；不在当前协议中同时引入内边界、外边界和双边界三套定义。
- 代码落点：`crc_gland_segmentation_project/src/data/mask_ops.py`
- 运行记录字段：`contour_extract_mode`
- 验收方式：检查主实现确实先得到单像素轮廓，再进行固定宽度膨胀。

### 5.4 边界宽度候选与主版本规则

- 当前结论：边界宽度候选固定为 `3 / 5 px`，首轮主测固定为 `w = 3`，`w = 5` 只允许作为必要时的单个备选；并且训练侧 `boundary_width` 与评估侧 `boundary_metric_width` 在正式主链中必须保持 `3 px` 对齐，除非按协议显式升级版本并完整重跑。
- 规则类型：`论文支持的候选范围 + 工程冻结规则`
- 适用阶段：`06_Boundary`
- 直接依据：`结直肠腺体分割_plan_优化版/01_实验执行/00_总览与规范/02_参数冻结总表.md`、`结直肠腺体分割_plan_优化版/01_实验执行/06_Boundary/03_实验步骤.md`、03_DCAN.md
- 核心公式或定义参考：`W = {3, 5}`；`w_main = 3`
- 采用原因：`3 px` 足以给出稳定边界带监督，对小目标和薄边界的侵入性更可控；`5 px` 只保留为必要时的唯一备选。
- 不采用的相邻方案：不做 `1 / 2 / 3 / 4 / 5 / 7` 全扫；不在训练、可视化和分析中使用不同宽度；不临时改宽度却不升级记录字段；不允许训练侧 `boundary_width = 3`、评估侧 `boundary_metric_width != 3` 却仍写成同一正式版本。
- 代码落点：`crc_gland_segmentation_project/src/data/boundary_targets.py`、[project_root]/configs/model/boundary_head_v1.yaml
- 运行记录字段：`boundary_width`、`boundary_width_policy_version`、`boundary_metric_width`
- 验收方式：检查主版本统一记录为 `boundary_width = 3` 且 `boundary_metric_width = 3`；若补备选，只允许额外记录 `boundary_width = 5`，并同步说明是否进入正式评估口径。

### 5.5 标签值定义规则

- 当前结论：边界标签值固定为背景 `0`、边界带 `1`，并且边界标签始终作为独立辅助监督目标存在。
- 规则类型：`工程冻结规则`
- 适用阶段：`06_Boundary`
- 直接依据：`结直肠腺体分割_plan_优化版/01_实验执行/06_Boundary/01_设计依据.md`、`结直肠腺体分割_plan_优化版/01_实验执行/06_Boundary/00_阶段总协议.md`
- 核心公式或定义参考：`Y_boundary(x) in {0, 1}`
- 采用原因：首版协议首先追求清楚、简单和可复查，不应把边界辅助任务写成新的多类别分割任务。
- 不采用的相邻方案：不把 gland 区域和边界写进同一混合标签图；不使用 `ignore index` 风格的复杂边界编码作为首版默认。
- 代码落点：`crc_gland_segmentation_project/src/data/boundary_targets.py`、[project_root]/src/losses/boundary_losses.py
- 运行记录字段：`boundary_value_definition`
- 验收方式：检查训练入口显式接收边界标签张量，而不是隐式从多类别 mask 中拆出边界类。

### 5.6 边界标签与主 gland 标签关系规则

- 当前结论：主任务仍是 gland 区域二值分割，边界标签只作为辅助监督目标，不改变主分割标签身份。
- 规则类型：`上游继承规则 + 工程冻结规则`
- 适用阶段：`06_Boundary`
- 直接依据：`结直肠腺体分割_plan_优化版/01_实验执行/06_Boundary/01_设计依据.md`、`结直肠腺体分割_plan_优化版/01_实验执行/06_Boundary/03_实验步骤.md`
- 核心公式或定义参考：`L_total = L_seg + lambda_boundary * L_boundary`
- 采用原因：这样才能把当前阶段准确解释为“主分割 + 边界辅助监督”，而不是新任务重定义。
- 不采用的相邻方案：不把 `Boundary` 阶段改写成全新的多类别 segmentation 任务。
- 代码落点：[project_root]/src/losses/seg_losses.py、[project_root]/src/losses/boundary_losses.py
- 运行记录字段：`boundary_loss_type`、`lambda_boundary`、`loss_version`
- 验收方式：检查训练日志能区分 `loss_seg` 与 `loss_boundary`，且主 gland target 身份未被改写。

### 5.7 生成时机与 resize 一致性规则

- 当前结论：边界标签必须从当前训练、验证或测试实际使用的冻结二值 mask 口径上生成，禁止在 resize 前 mask 上生成后再与另一套 resize 后样本混用。
- 规则类型：`工程冻结规则`
- 适用阶段：`06_Boundary`
- 直接依据：`结直肠腺体分割_plan_优化版/01_实验执行/01_数据协议/00_阶段总协议.md`、`结直肠腺体分割_plan_优化版/01_实验执行/06_Boundary/03_实验步骤.md`
- 核心公式或定义参考：`B_w` 与 `M` 必须共享同一空间分辨率 `H x W`
- 采用原因：边界图最容易因为 resize 前后口径不一致而产生错位，这会直接污染训练、可视化和案例解释。
- 不采用的相邻方案：不从原尺寸 mask 生成边界带后再粗暴 resize；不在训练和测试阶段分别采用不同生成时机。
- 代码落点：`crc_gland_segmentation_project/src/data/datasets.py`、`crc_gland_segmentation_project/src/data/boundary_targets.py`
- 运行记录字段：`mask_resize_stage`、`boundary_resize_policy`
- 验收方式：抽查边界 overlay 与 gland mask 是否逐像素对齐，不出现系统性偏移。

### 5.8 训练、可视化、分析共用同一实现规则

- 当前结论：训练数据加载、验证样本可视化、边界案例分析和边界指标准备必须共用同一正式边界生成实现；正式结果还必须继续绑定冻结的 `eval_cast_policy / boundary_metric_width / boundary_metric_impl / connected_components_impl / connected_components_connectivity`，避免“标签协议一致但评估实现偷换”。
- 规则类型：`工程冻结规则`
- 适用阶段：`06_Boundary`
- 直接依据：`结直肠腺体分割_plan_优化版/01_实验执行/06_Boundary/00_阶段总协议.md`、`结直肠腺体分割_plan_优化版/01_实验执行/00_总览与规范/05_代码工程映射与实现策略.md`
- 核心公式或定义参考：`boundary_target_fn(train) == boundary_target_fn(visualize) == boundary_target_fn(analyze)`
- 采用原因：只有实现唯一，才有资格比较边界证据和阶段结论。
- 不采用的相邻方案：不在 notebook、导图脚本和训练脚本里各自复制一版边界生成逻辑；不在标签协议不变的名义下私自切换 `Boundary F1` 实现或连通域实现。
- 代码落点：`crc_gland_segmentation_project/src/data/boundary_targets.py`、[project_root]/scripts/export_visuals.py、[project_root]/scripts/summarize_stage.py
- 运行记录字段：`boundary_generation_impl`、`boundary_target_version`、`eval_cast_policy`、`boundary_metric_width`、`boundary_metric_impl`、`connected_components_impl`、`connected_components_connectivity`
- 验收方式：检查三个入口都引用同一正式函数，而不是本地私有实现；检查 run_meta.yaml、边界指标表和 compare 表能回指出同一标签协议与同一评估实现。

### 5.9 最小人工抽查规则

- 当前结论：边界标签协议冻结后，至少必须抽查 `train 3-5` 个样本、`val 2-3` 个样本、`TestA 2` 个样本、`TestB 2` 个样本。
- 规则类型：`工程冻结规则`
- 适用阶段：`06_Boundary` 主版本启动前
- 直接依据：`结直肠腺体分割_plan_优化版/01_实验执行/06_Boundary/03_实验步骤.md`、`结直肠腺体分割_plan_优化版/01_实验执行/06_Boundary/04_保留或删除标准.md`
- 核心公式或定义参考：人工检查重点固定为 `boundary_on_contour / no_shift / width_reasonable / no_region_swallowing`
- 采用原因：边界标签类错误往往在训练指标出来之前，就能从少量 overlay 样本上先被发现。
- 不采用的相邻方案：不在未经人工抽查时直接批量跑三 seed；不只看训练损失不看边界图。
- 代码落点：[project_root]/scripts/export_visuals.py、[project_root]/reports/stage_reports/boundary_visual_casebook.md
- 运行记录字段：`boundary_label_qc_note`、`boundary_label_qc_pass`
- 验收方式：检查正式阶段报告中保留抽查说明与代表性边界 overlay。

### 5.10 输出与命名规则

- 当前结论：如果边界标签需要导出图像，命名必须显式带宽度；如果运行时动态生成，也必须在配置和运行记录中明确写出宽度与版本；正式 handoff 还必须同时记录五个评估实现硬字段。
- 规则类型：`工程冻结规则`
- 适用阶段：`06_Boundary`
- 直接依据：`结直肠腺体分割_plan_优化版/01_实验执行/00_总览与规范/03_命名与结果记录规范.md`
- 核心公式或定义参考：静态产物示例 `*_boundary_w3.png`、`*_boundary_w5.png`
- 采用原因：边界监督的关键变量不是目录名，而是边界目标版本链和宽度字段；这些必须能被回查。
- 不采用的相邻方案：不导出匿名边界图；不只在代码默认值里藏 `w = 3/5`；不把边界目标版本写进图名却把正式评估实现字段遗漏在 run_meta.yaml、compare 表或 handoff 清单之外。
- 代码落点：[project_root]/scripts/export_visuals.py、[project_root]/experiments/[run_name]/config.yaml、[project_root]/experiments/[run_name]/run_meta.yaml
- 运行记录字段：`boundary_width`、`boundary_target_version`、`boundary_target_source`、`eval_cast_policy`、`boundary_metric_width`、`boundary_metric_impl`、`connected_components_impl`、`connected_components_connectivity`
- 验收方式：检查任何正式导出或运行记录都能反推出本次使用的边界标签协议，并能回指出对应的正式评估实现。

### 5.11 固定排查顺序规则

- 当前结论：边界标签异常统一按“来源口径 -> resize 对齐 -> 轮廓定义 -> 膨胀宽度 -> 三链实现一致性”顺序排查。
- 规则类型：`工程冻结规则`
- 适用阶段：`06_Boundary`
- 直接依据：`结直肠腺体分割_plan_优化版/01_实验执行/06_Boundary/03_实验步骤.md`
- 核心公式或定义参考：`error_rank = [source_mismatch, resize_mismatch, contour_error, width_error, impl_drift]`
- 采用原因：边界标签问题最常见的不是公式写错，而是口径不一致和实现分叉。
- 不采用的相邻方案：不一出问题就先怀疑主干结构；不在未确认标签协议前继续扩搜权重和损失。
- 代码落点：`crc_gland_segmentation_project/src/data/boundary_targets.py`、[project_root]/notes/debug_note.md
- 运行记录字段：`failure_type`、`fix_action`、`debug_note_version`
- 验收方式：检查异常 run 的修复记录能对应到具体边界标签问题，而不是笼统写“重新试一下”。
