# 训练步骤

这份文件不是“训练时看一眼的大概流程”，而是 `UNet` 首轮流程验证阶段的正式训练执行协议。

它在当前阶段只负责一件事：

> 把 `结直肠腺体分割_plan_优化版\01_实验执行\02_UNet流程验证\01_任务与默认配置.md` 中已经冻结的默认协议，压成一条可执行、可记录、可回退、可交接的唯一训练主链。

因此，本文件必须同时写清：

- 训练启动前必须先过哪些门控。
- smoke check 和正式 run 如何明确分开。
- 每个 epoch 内哪些动作必须发生，哪些字段必须落盘。
- `best.ckpt / last.ckpt`、scheduler、early stopping、日志和总结如何形成闭环。
- 训练结束后哪些资产必须留给 `结直肠腺体分割_plan_优化版\01_实验执行\02_UNet流程验证\03_验证测试与可视化.md`、`结直肠腺体分割_plan_优化版\01_实验执行\02_UNet流程验证\05_阶段验收.md` 和 `结直肠腺体分割_plan_优化版/01_实验执行/03_UNet稳定性/00_阶段总协议.md`。

## 1. 文件角色与执行边界

### 1.1 当前文件负责什么

当前文件负责冻结并展开下面这些执行层动作：

- 在训练启动前，怎样确认数据阶段已经放行，且正式配置字段没有漂移。
- 怎样完成 `asset_check -> preflight -> smoke_check -> formal_run` 的唯一启动顺序。
- 怎样把单个 epoch 压成“训练 -> 验证 -> 调度 -> checkpoint -> 日志”的固定闭环。
- 怎样记录训练侧最小日志、最小目录资产、最小总结资产和停止原因。
- 怎样保证训练结果能被同层评估文件和下一阶段稳定性文件直接消费。

### 1.2 当前文件不负责什么

当前文件不负责重新打开下面这些问题：

- 重新定义任务边界、split 角色、标签口径、输入尺寸和默认增强。
- 重新定义 `UNet` 结构、loss、optimizer、`best_selector`、`threshold_source` 等默认值。
- 展开 `TestA / TestB` 的正式评估、可视化导出和阶段通过线。
- 逐条展开错误排查动作和 debug 证据化写法。

对应职责分别由下列文件承担：

- `结直肠腺体分割_plan_优化版/01_实验执行/02_UNet流程验证/01_任务与默认配置.md`
- `结直肠腺体分割_plan_优化版/01_实验执行/02_UNet流程验证/03_验证测试与可视化.md`
- `结直肠腺体分割_plan_优化版/01_实验执行/02_UNet流程验证/04_错误排查.md`
- `结直肠腺体分割_plan_优化版/01_实验执行/02_UNet流程验证/05_阶段验收.md`

### 1.3 为什么现在必须完成

当前阶段训练文件不能继续停留在“怎么跑一下”的说明层，因为后续至少有三类文件直接消费它：

1. `结直肠腺体分割_plan_优化版\01_实验执行\02_UNet流程验证\03_验证测试与可视化.md` 需要消费训练阶段导出的 `best.ckpt`、`last.ckpt`、val_metrics.csv、`threshold_source` 和 run_meta.yaml。
2. `结直肠腺体分割_plan_优化版\01_实验执行\02_UNet流程验证\05_阶段验收.md` 需要消费训练阶段的正式资产，判断 `pass_train / pass_val / pass_record` 是否真的成立。
3. `结直肠腺体分割_plan_优化版/01_实验执行/03_UNet稳定性/00_阶段总协议.md` 需要把当前训练协议当成冻结输入，只改 `seed` 重跑三次。

如果训练文件还不能清楚回答下面四件事，就说明当前阶段还没有真正跑通：

- 数据阶段放行资产是否被正确消费。
- smoke check 和正式 run 是否被明确区分。
- `best checkpoint` 是否真由 `val17` 驱动。
- 训练资产是否足以支撑 `A2` 只改 `seed` 复跑。

## 3. 本阶段唯一允许处理的变量

### 3.1 当前训练启动前必须已经成立的前提

只有当下面这些前提已经成立时，才允许进入本文件定义的训练执行链：

- `dataset = GlaS`
- split 固定为 `train68 / val17 / testA60 / testB20`
- `target_type = instance_to_binary`
- `foreground_rule = mask > 0`
- `color_mode = RGB`
- `input_size = 512x512`
- `normalize_name = imagenet_mean_std`
- `train_proto_version = train_proto_v1`
- `aug_profile_name = light_aug_v1`
- `eval_aug_enable = false`
- `eval_proto_version = eval_proto_v1`
- `best_selector = val_objdice_max`
- `threshold_source = val17`
- `data_stage_pass = true`
- `handoff_ready = true`
- `preflight_pass = true`

### 3.2 当前阶段明确不做的事情

当前训练执行文件不允许借“训练步骤”之名提前打开：

- `ResNet34-U-Net`
- `LKMA`
- `Boundary Head`
- `Distance-aware Loss`
- stain normalization
- postprocess 搜索
- test-time augmentation
- 测试集反向调参
- 正式 run 中途切换 loss、optimizer、scheduler 或输入尺寸

这样规定的原因是：

- `A1` 训练文件只负责把首版默认协议跑通。
- 如果训练执行层在首轮阶段就打开多个变量，后续无法判断闭环到底是靠基础协议还是靠额外改动跑通。

## 4. 阶段门控表达式

### 4.1 训练启动门控规则

- 当前结论：只有当数据阶段放行门控和当前正式配置字段同时满足时，才允许进入训练预检查。
- 当前值或最终版本：`preflight_gate_v1`
- 规则类型：`上游阶段交接规则 + 工程冻结规则`
- 适用阶段：`02_UNet流程验证`
- 直接依据：`结直肠腺体分割_plan_优化版/01_实验执行/01_数据协议/00_阶段总协议.md`，`结直肠腺体分割_plan_优化版/01_实验执行/01_数据协议/07_数据阶段验收.md`，`结直肠腺体分割_plan_优化版/01_实验执行/02_UNet流程验证/00_阶段总协议.md`
- 核心公式或定义参考：

```text
train_launch_allowed =
  data_stage_pass
  and handoff_ready
  and preflight_pass
  and (dataset = GlaS)
  and (train_split = train68)
  and (val_split = val17)
```

- 采用原因：训练执行层不能绕过数据阶段门控直接假设数据永远正确。
- 不采用的相邻方案：不采用“先开跑，跑挂了再看数据阶段”；不采用只检查路径存在、不检查门控字段。
- 代码落点：`crc_gland_segmentation_project/scripts/train.py`，`crc_gland_segmentation_project/configs/data/glas.yaml`，run_meta.yaml
- 运行记录字段：`data_stage_pass`，`handoff_ready`，`preflight_pass`，`dataset`，`train_split`，`val_split`
- 验收方式：检查启动前日志和 run_meta.yaml 中门控字段完整；任一门控未通过时不得启动训练。

### 4.2 输入资产与 batch 预检查规则

- 当前结论：正式训练前必须完成输入资产、batch 形状、值域、同步增强和样本配对的固定预检查。
- 当前值或最终版本：`asset_check_v1`
- 规则类型：`工程冻结规则`
- 适用阶段：`A1` 首轮训练启动前
- 直接依据：`结直肠腺体分割_plan_优化版\01_实验执行\02_UNet流程验证\01_任务与默认配置.md`，`01_数据协议/*`，`结直肠腺体分割_plan_优化版/01_实验执行/00_总览与规范/05_代码工程映射与实现策略.md`
- 核心公式或定义参考：

```text
x in R^(B x 3 x 512 x 512)
y in {0,1}^(B x 1 x 512 x 512)
y(x) = 1[M_inst(x) > 0]
```

- 采用原因：训练第一步必须先证明 dataloader、resize、标签转换和 batch 组织没有漂移。
- 不采用的相邻方案：不采用仅检查 `csv` 存在；不采用不抽样可视检查就直接开跑。
- 代码落点：`crc_gland_segmentation_project/src/data/datasets.py`，src/data/transforms.py，`crc_gland_segmentation_project/src/data/mask_ops.py`，`crc_gland_segmentation_project/scripts/train.py`
- 运行记录字段：`asset_check_pass`，`sample_shape`，`mask_value_range`，`mask_binary_pass`，`pair_check_pass`
- 验收方式：抽查 `2-3` 个 batch；确认 `image/mask` 对齐、mask 为二值、增强同步、张量尺寸正确。

### 4.3 模型、loss 与单步前后向预检查规则

- 当前结论：正式 run 前必须先确认 `UNet` 单步前向、loss 计算、反向传播、optimizer 更新和 AMP 状态全部可用。
- 当前值或最终版本：`step_check_v1`
- 规则类型：`工程冻结规则`
- 适用阶段：`A1` 首轮训练启动前
- 直接依据：`结直肠腺体分割_plan_优化版\01_实验执行\02_UNet流程验证\01_任务与默认配置.md`，`结直肠腺体分割_plan_优化版/03_文献证据/01_经典基线与对比方法/02_U-Net.md`
- 核心公式或定义参考：

```text
z = f_theta(x)
p = sigmoid(z)
L_seg = L_BCE(p, y) + L_Dice(p, y)
theta_(t+1) = Optim(theta_t, grad_t)
```

- 采用原因：训练执行文件必须先排除最基础的模型和 loss 接线错误，避免把结构性报错拖进正式 run。
- 不采用的相邻方案：不采用直接开始长时间训练再看是否报错；不采用只验证前向不验证反向。
- 代码落点：src/models/unet.py，src/losses/seg_losses.py，src/engine/trainer.py
- 运行记录字段：`forward_pass`，`loss_total_ok`，`loss_bce_ok`，`loss_dice_ok`，`backward_pass`，`optimizer_step_pass`，`amp_nan_inf_pass`
- 验收方式：检查单步前后向无报错，无首轮 `NaN/Inf`，主输出通道与标签通道一致。

### 4.6 单个 epoch 训练闭环规则

- 当前结论：每个 epoch 都必须完成完整 `train68` 遍历、loss 聚合、反向传播、optimizer 更新和训练日志写入。

### 4.8 scheduler、early stopping 与 checkpoint 更新规则

- 当前结论：`ReduceLROnPlateau` 只能监控 `val_objdice`；正式 run 必须同时维护 `best.ckpt` 和 `last.ckpt`，并依据 `val_objdice` 与 early stopping 更新状态。
- 当前值或最终版本：`scheduler_checkpoint_v1`
- 规则类型：`工程冻结规则`
- 适用阶段：`A1` 正式训练
- 直接依据：`结直肠腺体分割_plan_优化版\01_实验执行\02_UNet流程验证\01_任务与默认配置.md`，`结直肠腺体分割_plan_优化版\01_实验执行\02_UNet流程验证\03_验证测试与可视化.md`，`结直肠腺体分割_plan_优化版\01_实验执行\02_UNet流程验证\05_阶段验收.md`
- 核心公式或定义参考：

```text
if no_improve(ObjDice(val17), patience):
    lr_(t+1) = factor * lr_t

```

- 采用原因：训练调度、模型选择、正式测试和阶段验收必须围绕同一个主指标闭环。

- 运行记录字段：`scheduler_monitor`，`scheduler_factor`，`scheduler_patience`，`scheduler_min_lr`，`best_selector`，`best_epoch`，`stop_epoch`，`stop_reason`
- 验收方式：检查 `scheduler_monitor = val_objdice`；检查磁盘上同时存在 `best.ckpt` 和 `last.ckpt`；检查 run_meta.yaml 记录 `best_epoch` 和 `stop_reason`。

## 5. 固定执行顺序

### 5.1 启动前固定检查顺序

正式 run 前必须严格按下面顺序执行，不能跳步：

1. 检查数据阶段放行门控和当前配置字段。
2. 检查 split 资产、路径、样本配对和 batch 内容。

5. 执行一次最小 smoke check。

smoke check 至少满足：

- `1-2` 个 epoch
- 极少 batch
- 至少完成一次 train + val
- 至少成功落盘 config.yaml 和 run_meta.yaml
- 至少成功导出一行训练日志和一行验证日志

### 5.3 正式 run 固定执行顺序

1. 加载冻结配置与正式 `run_name`

5. 按 `val_objdice` 更新 scheduler、early stopping 和 best checkpoint
6. 保存 `last.ckpt`
7. 合法停止后导出曲线和总结
8. 把训练资产交给同层评估文件进入正式测试

## 9. 回退条件

### 9.1 独立回退触发条件

只要出现下面任意一条，当前训练协议就必须优先回退检查，而不是继续推进正式结果：

- 训练启动前缺失 `data_stage_pass`、`handoff_ready` 或 `preflight_pass`。
- batch 抽查发现图像与 mask 错位、mask 非二值或 resize 口径漂移。
- 单步前后向出现 `NaN/Inf`、loss 无法计算、optimizer 无法更新。
- smoke check 与正式 run 混用同一个 `run_name`。
- 正式 run 中途更改 loss、optimizer、scheduler、输入尺寸或增强，但未升级版本号。
- `scheduler_monitor` 不是 `val_objdice`。
- `best.ckpt` 不是由 `val_objdice` 选出，或 `best.ckpt / last.ckpt` 任一缺失。
- 训练结束后资产不完整，无法交给同层评估和验收文件消费。

### 9.2 固定回退顺序

回退时统一按下面顺序排查：

1. 先查数据阶段交接门控和 split 资产。
2. 再查 dataloader、标签转换、resize 和 batch 内容。
3. 再查模型、loss、optimizer、AMP 和单步前后向。
4. 再查验证、scheduler、checkpoint 和训练日志闭环。
5. 只有前面都确认无误后，才检查 `UNet` 结构实现本身。

### 9.3 回退后的重新放行条件

只有同时满足下面条件，才允许重新声明当前训练协议有效：
