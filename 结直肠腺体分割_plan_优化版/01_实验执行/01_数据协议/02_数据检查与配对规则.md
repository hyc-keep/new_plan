# 数据检查与配对规则

本文件不是“做 split 前顺手看一眼数据”的提醒单，而是 `01_数据协议` 内负责冻结数据检查门控、配对口径、放行字段与回退条件的正式协议。  
它的职责不是补充说明，而是把 `GlaS / CRAG` 的数据检查写成后续 `CSV` 固化、训练前 preflight、阶段验收与论文写作都必须共同继承的前置 gate。

如果这一层没有先写死，最危险的问题不是“少了几个统计数字”，而是坏路径、错配标注、跨 split 泄漏、批量空 mask、系统性错位或来源不清的本地副本被直接冻结进正式 `CSV`，然后继续污染 `02_UNet流程验证` 与后续全部阶段。

---

## 1. 文件角色与执行边界

当前文件负责冻结六步检查闭环、配对口径、检查资产、放行字段和回退顺序。

当前文件不负责重开 split 定义、标签生成公式和下游模型训练协议；它只负责把进入正式 `CSV` 前必须成立的检查门槛写死。

不允许在这里重开 benchmark 身份、split 边界、标签规则和模型阶段参数。

## 3. 正式提取与冻结边界

### 3.1 路线层提取

从 `02_路线与投稿` 层提取到的当前结论是：

- 数据阶段必须先形成正式可追溯资产，后续模型阶段不得绕过
- `GlaS` 是主 benchmark，`CRAG` 是第二 benchmark，因此两者都必须在数据检查阶段留下可被论文与实验共同复用的正式证据
- 数据问题必须在 split 固化之前暴露，不允许把训练异常当成数据阶段的替代检测器

### 3.2 文献层提取

从 `GlaS-Challenge`、`MILD-Net`、`TA-Net` 提取到的当前结论是：

- `GlaS` 的官方 train / `TestA` / `TestB` 边界必须被保护，不能为了工程方便而合并测试子集
- 腺体任务的对象级评估依赖标注真实性与边界一致性，因此空 mask、错位 mask 和跨 split 泄漏会直接破坏后续指标解释
- `CRAG` 不是可随意裁切的辅助数据，而是腺体任务中有明确使用背景的正式 benchmark，因此其来源说明、路径边界和 `source_subset` 必须可追溯

### 3.3 代码映射边界

本轮必须明确区分两件事：

- `[project_root]/` 下的路径是未来正式工程必须落地的接口契约
- 当前 `plan/` 工作区中的文档并不等于这些脚本已经存在

因此，本文件后续写到的 `tools/*.py`、`scripts/*.py` 都只能被解释为正式工程落点，不得被误读为“当前工作区已经存在实现”。  
本轮复查中，当前工作区未检索到 check_dataset_pairs.py 现成实现，因此本文件必须显式写出“未来工程契约”和“当前未落地状态”的区别。

## 4. 放行链与核心定义

### 4.1 数据检查阶段总门

- 当前结论：正式 split 写入前必须完成“数量检查 -> 配对检查 -> 路径可读性检查 -> 前景有效性检查 -> 统计汇总 -> 人工抽查”六步闭环
- 规则类型：`工程冻结规则`
- 适用阶段：`01_数据协议`
- 直接依据：`结直肠腺体分割_plan_优化版\01_实验执行\01_数据协议\00_阶段总协议.md`，`结直肠腺体分割_plan_优化版\01_实验执行\01_数据协议\07_数据阶段验收.md`
- 核心公式或定义参考：`pass_data_check = pass_count and pass_pair and pass_readable and pass_foreground and pass_stats and pass_manual_audit`
- 采用原因：先把最危险的数据问题挡在 split 固化与模型训练之前，避免后续把数据问题误判成模型问题
- 不采用的相邻方案：不采用只看目录浏览；不采用只有机器检查没有人工抽查；不采用先写正式 `CSV` 再补数据检查
- 代码落点：`crc_gland_segmentation_project/tools/stage01_data_protocol/check_dataset_pairs.py`，`crc_gland_segmentation_project/tools/stage01_data_protocol/preview_dataset_samples.py`，`crc_gland_segmentation_project/tools/stage01_data_protocol/prepare_glas_split.py`，`crc_gland_segmentation_project/tools/stage01_data_protocol/prepare_crag_split.py`
- 运行记录字段：`data_check_version`，`pair_rule_version`，`manual_audit_version`，`pass_data_check`，`data_check_report_path`
- 验收方式：六步闭环每一步都必须留下正式产物；任一步缺失或失败，都不得进入正式 split 写入

### 4.2 数据检查与阶段放行的关系

当前文件不是孤立地给出一个 `pass_data_check`，而是必须把它接到阶段总门上：

```text
data_stage_pass
= pass_source
and pass_split
and pass_pair
and pass_label
and pass_check
and pass_preview
and pass_handoff

其中：
pass_check   <- pass_data_check
pass_preview <- preview assets and manual audit evidence ready
pass_handoff <- reports, split metadata and asset_manifest are traceable
```

这意味着：

- `pass_data_check = False` 时，`data_stage_pass` 不可能成立
- `pass_data_check = True` 但人工抽查证据或报告资产缺失时，`pass_preview` 与 `pass_handoff` 仍然不得成立
- `data_stage_pass = False` 时，`handoff_ready` 与 `preflight_pass` 都不得被下游训练入口强行置真

### 4.3 为什么六步顺序不能打乱

六步检查是串联门，不是平级建议：

- 数量没对齐，配对检查没有意义
- 配对不稳定，路径检查与统计汇总都会失真
- 路径不可读，训练期会随机报错
- 前景有效性不过，loss 与指标解释会被污染
- 没有统计汇总，后续无法解释 split 分布
- 没有人工抽查，系统性错位与异常标注很容易漏检

因此，当前文件只承认固定顺序，不接受并行打勾式“看起来都做过”。

## 5. 正式检查范围与正式对象

### 5.1 `GlaS` 检查范围规则

- 当前结论：`GlaS` 检查必须覆盖官方 `train85`、`TestA60`、`TestB20` 与 Grade.csv
- 规则类型：`官方协议固定项 + 工程冻结规则`
- 适用阶段：`01_数据协议`
- 直接依据：01_GlaS-Challenge.md，`结直肠腺体分割_plan_优化版\01_实验执行\01_数据协议\03_GlaS划分协议.md`
- 核心公式或定义参考：`GlaS_check_scope = {official_train85, official_testA60, official_testB20, Grade.csv}`
- 采用原因：后续正式工程使用的 `train68 / val17 / testA60 / testB20` 全部从这组官方对象衍生，漏查任何一项都会破坏正式 split 的可信度
- 不采用的相邻方案：不采用只检查 train；不采用把 Grade.csv 当作可有可无的附表；不采用等 split 生成后再回头补查
- 代码落点：`crc_gland_segmentation_project/tools/stage01_data_protocol/check_dataset_pairs.py`，`crc_gland_segmentation_project/tools/stage01_data_protocol/prepare_glas_split.py`
- 运行记录字段：`dataset_name`，`subset_name`，`grade_csv_version`，`glas_scope_version`
- 验收方式：train / `TestA` / `TestB` 都必须进入检查报告，并确认 Grade.csv 已被读取和登记

## 6. 六步检查闭环规则卡片

### 6.1 数量检查

- 当前结论：每个正式子集都必须先确认 image 数量、mask 数量以及两者是否一一对应
- 规则类型：`工程冻结规则`
- 适用阶段：`01_数据协议`
- 直接依据：`结直肠腺体分割_plan_优化版\01_实验执行\01_数据协议\07_数据阶段验收.md`
- 核心公式或定义参考：对任意子集 `S`，必须满足 `N_img(S) = N_mask(S)`
- 采用原因：数量不一致说明配对链天然不可信，后续任何 `CSV` 都不应继续生成
- 不采用的相邻方案：不采用“先写 CSV 再补缺口”；不采用“少几个样本问题不大”
- 代码落点：`crc_gland_segmentation_project/tools/stage01_data_protocol/check_dataset_pairs.py`，`crc_gland_segmentation_project/reports/data_checks/dataset_stats.csv`
- 运行记录字段：`subset_name`，`num_images`，`num_masks`，`count_match`，`count_fail_reason`
- 验收方式：所有正式子集都必须在统计表中给出 image / mask 数量和一致性状态；任一子集不一致直接阻断

### 6.2 配对检查

- 当前结论：正式 split 只允许通过确定性命名映射完成 image / mask 配对，不允许手工补例外样本
- 规则类型：`工程冻结规则`
- 适用阶段：`01_数据协议`
- 直接依据：`结直肠腺体分割_plan_优化版\01_实验执行\01_数据协议\03_GlaS划分协议.md`，`结直肠腺体分割_plan_优化版\01_实验执行\01_数据协议\04_CRAG划分协议.md`
- 核心公式或定义参考：必须存在稳定映射 `pair(image_name) -> mask_name`，并能稳定导出 `sample_id`
- 采用原因：正式主键、结果回查和数据重跑都建立在稳定配对链上
- 不采用的相邻方案：不采用运行时人工补洞；不采用多套命名规则混用却不版本化
- 代码落点：`crc_gland_segmentation_project/tools/stage01_data_protocol/check_dataset_pairs.py`，`crc_gland_segmentation_project/tools/stage01_data_protocol/prepare_glas_split.py`，`crc_gland_segmentation_project/tools/stage01_data_protocol/prepare_crag_split.py`
- 运行记录字段：`pair_rule_version`，`sample_id_rule_version`，`pair_success_count`，`pair_fail_count`，`pair_fail_samples`
- 验收方式：配对失败样本必须完整列出；正式 `CSV` 中的每个 `sample_id` 都必须能回指唯一配对规则

### 6.3 路径可读性检查

- 当前结论：正式资产中的 image 和 mask 路径都必须真实存在、可读取、非 `0` 字节且非损坏文件
- 规则类型：`工程冻结规则`
- 适用阶段：`01_数据协议`
- 直接依据：`结直肠腺体分割_plan_优化版\01_实验执行\01_数据协议\07_数据阶段验收.md`
- 核心公式或定义参考：对任意路径 `p`，必须满足 `exists(p) and readable(p) and size(p) > 0`
- 采用原因：训练与评估期的随机报错很多时候本质是坏路径、坏文件或路径漂移
- 不采用的相邻方案：不采用只在训练时暴露文件错误；不采用只抽查少量目录
- 代码落点：`crc_gland_segmentation_project/tools/stage01_data_protocol/check_dataset_pairs.py`，[project_root]/reports/data_checks/path_readability_report.csv
- 运行记录字段：`num_missing_paths`，`num_unreadable_files`，`num_zero_byte_files`，`bad_path_samples`
- 验收方式：坏路径、坏文件和 `0` 字节文件必须被逐样本列出；未修复前不得进入正式 split

### 6.4 前景有效性检查

- 当前结论：必须确认 mask 在 `mask > 0` 二值化后仍保留有效前景，且不存在批量空 mask 或异常近全白 mask
- 规则类型：`工程冻结规则`
- 适用阶段：`01_数据协议`
- 直接依据：`结直肠腺体分割_plan_优化版\01_实验执行\01_数据协议\05_标签转换与可视化规则.md`，`结直肠腺体分割_plan_优化版\01_实验执行\01_数据协议\07_数据阶段验收.md`
- 核心公式或定义参考：`y(p) = 1[m(p) > 0]`；`r_fg = sum_p y(p) / (H * W)`
- 采用原因：空 mask、异常高前景比例和二值化异常会直接污染训练损失、阈值解释和对象级指标
- 不采用的相邻方案：不采用只检查文件可读；不采用等训练出全背景预测再回头找原因
- 代码落点：`crc_gland_segmentation_project/tools/stage01_data_protocol/check_dataset_pairs.py`，`crc_gland_segmentation_project/reports/data_checks/foreground_summary.csv`
- 运行记录字段：`num_empty_masks`，`mean_foreground_ratio`，`min_foreground_ratio`，`max_foreground_ratio`，`foreground_outlier_samples`
- 验收方式：报告中必须给出空 mask 数、前景比例摘要和异常样本清单；批量异常直接阻断

### 6.5 统计汇总

- 当前结论：正式 split 写入前必须为每个 split 输出样本数、空 mask、前景比例和对象尺度统计
- 规则类型：`工程冻结规则`
- 适用阶段：`01_数据协议`
- 直接依据：`结直肠腺体分割_plan_优化版\01_实验执行\01_数据协议\06_数据产物清单.md`，`结直肠腺体分割_plan_优化版\01_实验执行\01_数据协议\07_数据阶段验收.md`
- 核心公式或定义参考：对象面积定义为 `a_k = |C_k|`，其中 `C_k` 为二值前景连通域；前景比例沿用 `r_fg`
- 采用原因：没有正式统计，后续无法解释 split 偏差、前景分布与对象尺度风险
- 不采用的相邻方案：不采用只导出样本数；不采用无版本的临时 notebook 统计
- 代码落点：`crc_gland_segmentation_project/reports/data_checks/dataset_stats.csv`，`crc_gland_segmentation_project/reports/data_checks/object_size_summary.csv`
- 运行记录字段：`dataset_stats_version`，`object_size_rule`，`foreground_rule_version`，`small_object_threshold`
- 验收方式：每个正式 split 都必须有统计记录，并且与当前 split 版本和检查版本一致

### 6.6 人工抽查

- 当前结论：正式 split 写入前必须通过人工抽查确认 image、mask 与 overlay 不存在系统性错位和明显异常标注
- 规则类型：`工程冻结规则`
- 适用阶段：`01_数据协议`
- 直接依据：`结直肠腺体分割_plan_优化版\01_实验执行\01_数据协议\05_标签转换与可视化规则.md`，`结直肠腺体分割_plan_优化版\01_实验执行\01_数据协议\07_数据阶段验收.md`
- 核心公式或定义参考：每个抽查样本至少保留 `raw / mask / overlay` 三联证据，并回指 `sample_id`
- 采用原因：系统性错位和边界异常很难只靠数量、路径和统计自动发现
- 不采用的相邻方案：不采用完全不做人检；不采用只看训练集；不采用只截几张图但不登记样本身份
- 代码落点：`crc_gland_segmentation_project/tools/stage01_data_protocol/preview_dataset_samples.py`，`[project_root]/reports/data_preview/*`，`crc_gland_segmentation_project/reports/data_checks/manual_audit_notes.md`
- 运行记录字段：`manual_audit_version`，`manual_audit_split`，`manual_audit_sample_id`，`alignment_pass`，`manual_audit_decision`
- 验收方式：抽查记录必须达到最小覆盖量，且每条记录都能回指图像、mask、overlay 与最终结论

## 9. 不通过线、红线与回退顺序

### 9.1 一般不通过线

下面任一情况成立，都不允许进入正式 split 写入：

- image / mask 数量不一致
- 命名规则无法稳定配对
- 路径存在损坏、缺失或不可读文件
- 前景有效性检查失败
- 统计汇总缺失或与当前版本不一致
- 人工抽查覆盖量不足
- Grade.csv 读取异常
- `CRAG` 的 `train / val / test` 边界无法自洽

### 9.2 数据红线

下面任一情况成立，都必须停在数据阶段，不允许以“先记着后面再说”的方式放行：

- 同一 `sample_id` 已经或即将进入多个 split
- `TestA / TestB / test40` 被误混入训练或验证池
- 批量空 mask、异常近全白 mask 或系统性错位未修复
- 正式检查结果只停留在终端，未形成正式报告资产
- 正式 `CSV` 与检查版本、来源说明和目录身份对不上

### 9.3 下游异常时的强制回退顺序

即使后续已经进入 `UNet`，只要出现大量全背景预测、可视化明显错位或指标与肉眼长期严重矛盾，也必须先按下面顺序回退：

1. `CSV` 的 `sample_id / image_relpath / mask_relpath`
2. image / mask 配对逻辑
3. `mask > 0` 二值化口径
4. 路径可读性与坏文件
5. 前景比例与异常样本
6. 人工抽查证据

这条回退链必须在当前文件先写死，不能等下游文件出问题再临时发明。
