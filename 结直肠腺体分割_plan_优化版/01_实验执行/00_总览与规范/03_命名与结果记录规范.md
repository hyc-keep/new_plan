# 命名与结果记录规范

## 1. 文件角色与执行边界

- 本文件负责统一 `run_name`、结果表字段、版本字段、实验资产命名和记录口径。
- 本文件不替代单阶段训练协议、设计依据、阶段验收或回退裁决。

## 2. 本轮直接依赖的前置文件

- 直接依赖 `结直肠腺体分割_plan_优化版/01_实验执行/00_总览与规范/00_执行导航.md`、`结直肠腺体分割_plan_优化版/01_实验执行/00_总览与规范/01_工程目录框架.md`、`结直肠腺体分割_plan_优化版/01_实验执行/00_总览与规范/02_参数冻结总表.md`。
- 同时约束 `crc_gland_segmentation_project/experiments/`、`crc_gland_segmentation_project/reports/`、`crc_gland_segmentation_project/scripts/` 的正式输出命名。

## 3. 本阶段唯一允许处理的变量

- 只允许统一命名规则、版本字段、结果记录字段和正式资产文件名，不允许在本文件里重开模型设计、训练策略或评估定义。

## 4. 阶段门控表达式

- 后续阶段必须能依据本文件稳定生成可回链的 `run_name`、`run_meta.yaml`、结果表和图像资产命名，本文件才算有效。

这份文件负责把后续代码工程里最容易失控的内容一次写死：

- `run_name` 怎么命名
- 协议版本怎么记录
- 结果目录怎么落盘
- checkpoint / log / metrics / visual 文件名怎么统一
- 哪些字段必须写进配置和结果表

它不负责给训练参数找论文依据，那部分应回到：

- `02_路线与投稿`
- `03_文献证据`
- `结直肠腺体分割_plan_优化版/01_实验执行/00_总览与规范/06_实验执行证据化写作模板.md`

这份文件的定位非常明确：

> 它属于 `工程冻结规则`，目的是保证后续代码、实验、汇总、写作用的是同一套命名与记录协议。

## 1. 使用原则

后面任何一个想进入正式结果层的 run，都必须同时满足下面三条：

1. 能从名字看出阶段、模型、数据集、协议版本、seed
2. 能在固定目录结构下找到配置、日志、权重、指标和可视化
3. 能在结果表里追溯到唯一的协议版本和唯一的 checkpoint

如果一个结果只剩一个截图、一个最好数字或一个随手起的目录名，它默认不具备进入正式汇总层的资格。

## 2. `run_name` 统一规则

### 2.1 标准格式

后面正式 run 统一使用稳定格式，协议版本不得进入 `run_name`：

```text
[stage_code]_[model_code]_[dataset_code]_[seed_tag]
```

正式稳定实验身份固定为：

- `A2_UNet_GlaS_seed3407`
- `A2_UNet_GlaS_seed1234`
- `A2_UNet_GlaS_seed2025`
- `B1_ResNet34_UNet_GlaS_seed3407`
- `B1_ResNet34_UNet_GlaS_seed1234`
- `B1_ResNet34_UNet_GlaS_seed2025`

其它阶段也沿用同一稳定规则，例如 `C1_R34UNet_LKMA_GlaS_seed3407`。协议版本只能写入配置、元数据、结果表和 manifest。

### 2.2 各字段含义

#### `stage_code`

只允许用固定阶段码：

- `A1`：`UNet` 首轮跑通
- `A2`：`UNet` 三次重复稳定
- `B1`：`ResNet34-U-Net` baseline
- `C1`：`LKMA`
- `D1`：`Boundary`
- `E1`：`Distance`
- `H1`：第一批外部对比
- `I1`：`CRAG`
- `J1`：结果汇总或补表重算

如果后面某阶段有第二版协议，升级版本号，不另外乱造阶段码。

#### `model_code`

统一采用短名，不允许同一模型多套写法混用。

固定写法：

- `UNet`
- `R34UNet`
- `R34UNet_LKMA`
- `R34UNet_LKMA_Boundary`
- `R34UNet_LKMA_Boundary_Dist`
- `UNetPP`
- `DeepLabV3P`
- `AttUNet`
- `nnUNet`
- `TransUNet`
- `SwinUNet`
- `Final`

禁止写法：

- `res34unet`
- `deeplabv3+`
- `attention_u_net`

因为后面结果汇总会出现字符串分裂问题。

#### `dataset_code`

只允许：

- `GlaS`
- `CRAG`

#### 稳定身份与协议版本

`run_name` 只表达阶段、模型、数据集和 seed，不表达协议版本。协议版本必须独立记录在：

- `config_version`
- `data_proto_version`
- `train_proto_version`
- `aug_version`
- `eval_proto_version`
- `model_version`
- `loss_version`
- `postprocess_version`
- `config.yaml`、`run_meta.yaml`、结果表和阶段 manifest

只要训练、评估、增强、后处理或模型结构中的正式协议发生变化，就升级对应字段；不要通过改 `run_name` 伪造新身份。

#### `seed_tag`

正式 seed 固定写法：

- `seed3407`
- `seed1234`
- `seed2025`

## 3. 协议版本记录规则

### 3.1 后面至少要同时记录 8 类版本

每个正式 run 的配置或结果记录里，至少应明确写出：

- `config_version`
- `data_proto_version`
- `train_proto_version`
- `aug_version`
- `eval_proto_version`
- `model_version`
- `loss_version`
- `postprocess_version`

固定示例：

```yaml
config_version: v1
data_proto_version: data_proto_v1
train_proto_version: train_proto_v1
aug_version: main_aug_v1
eval_proto_version: eval_proto_v1
model_version: r34unet_lkma_boundary_v1
loss_version: seg_loss_v1
postprocess_version: none_in_v1
```

### 3.2 什么情况下必须升级版本

只要出现下面任意一种变化，就必须升级相关版本号：

- 数据划分变了
- 标签生成逻辑变了
- 增强内容或概率变了
- checkpoint 选择规则变了
- threshold 规则变了
- 后处理从关闭改为开启
- 模型结构真正变了

不允许出现下面情况：

- 改了协议，但还沿用旧 `v1`
- 结果已经混到主表里，才发现不同 run 根本不是同一协议

### 3.3 `run_name` 与协议字段的分离

`run_name` 不再包含 `proto_version` 或 `config_version`。稳定身份只由阶段、模型、数据集和 seed 组成，例如：

- `A2_UNet_GlaS_seed3407`
- `B1_ResNet34_UNet_GlaS_seed1234`

详细版本链必须进入：

- `config.yaml`
- `run_meta.yaml`
- 结果表
- 阶段 manifest

固定理解方式：

- `run_name = B1_ResNet34_UNet_GlaS_seed3407` 是稳定实验身份；协议版本只写入 config_version、model_version、data_proto_version、train_proto_version、eval_proto_version、result_tag 或 manifest
- `config_version / data_proto_version / train_proto_version / aug_version / eval_proto_version / model_version / loss_version / postprocess_version` 是该身份在某轮执行中的协议字段
- 协议字段变化不得创建带版本号的新正式身份；需要重跑时仍使用稳定身份，并以版本字段、时间和 manifest 区分记录

## 4. 实验目录统一规则

### 4.1 正式目录结构

后面正式 run 必须统一落到：

```text
experiments/
└─ [run_name]/
   ├─ {stage}_{model}_{dataset}_{seed}.yaml
   ├─ seed_record.txt
   ├─ run_meta.yaml
   ├─ train_log.csv
   ├─ val_metrics.csv
   ├─ testA_metrics.csv        # 仅 GlaS
   ├─ testB_metrics.csv        # 仅 GlaS
   ├─ test_metrics.csv         # 仅 CRAG
   ├─ metric_crosscheck_note.md
   ├─ checkpoints/
   │  ├─ best.ckpt
   │  └─ last.ckpt
   ├─ predictions/
   │  ├─ val/
   │  ├─ testA/
   │  ├─ testB/
   │  └─ test/
   ├─ visuals/
   │  ├─ val/
   │  ├─ testA/
   │  ├─ testB/
   │  └─ test/
   ├─ curves/
   ├─ summaries/
   │  ├─ run_summary.md
   │  └─ error_cases.md
   └─ artifacts/
```

### 4.2 固定说明

- `GlaS` 必须分开保存 testA_metrics.csv 和 testB_metrics.csv
- `CRAG` 只有一个正式测试集，因此固定为 test_metrics.csv
- metric_crosscheck_note.md 固定放在 run 根目录，避免评估口径留痕和训练总结分家
- `best.ckpt` 和 `last.ckpt` 统一使用 `.ckpt`
- summary.md 统一收敛为 summaries/run_summary.md
- error_cases.md 固定放在 `summaries/`

### 4.3 为什么这样统一

因为当前文档里已经出现过这些冲突：

- `best.pth` vs `best.ckpt`
- training_log.csv vs train_log.csv
- val_log.csv vs val_metrics.csv
- test_metrics.csv vs `testA_metrics.csv / testB_metrics.csv`

后面一律按本文件统一，不再多写法并存。

## 7. run_meta.yaml 最少字段

为了让结果汇总脚本稳定复用，每个正式 run 都必须额外导出一个轻量元信息文件：

```yaml
run_name:
status:
start_time:
end_time:
git_or_snapshot_id:
config_version:
data_proto_version:
train_proto_version:
aug_version:
aug_profile_name:
eval_aug_enable:
eval_proto_version:
eval_cast_policy:
device:
num_epochs_trained:
best_epoch:
best_selector:
best_metric_value:
threshold_source:
threshold_value:
boundary_metric_width:
boundary_metric_impl:
connected_components_impl:
connected_components_connectivity:
metric_crosscheck_done:
metric_crosscheck_result:
metric_crosscheck_note_path:
seed_reporting_mode:
main_seed_set:
additional_seed_set:
combined_seed_count:
checkpoint_path:
notes:
```

说明：

- `status` 只允许：`running / done / failed / aborted`
- `best_selector` 必须写清具体规则，例如 `val_objdice_max`
- `threshold_source` 只允许来自验证集
- `boundary_metric_width` 主版本固定记录 `3`
- `metric_crosscheck_done` 不允许只写口头“已对齐”；必须能回指 `metric_crosscheck_note_path`
- `seed_reporting_mode` 默认记录冻结主协议统计口径，若进入补充 reporting，必须同步记录 `additional_seed_set / combined_seed_count`
- `checkpoint_path` 必须指向 `best.ckpt`

### 7.1 历史归档与旧 run 映射规则

- 正式工作区只使用稳定 `run_name`；历史迭代不得在正式工作区继续作为当前身份。
- 历史迭代只能放在所属 `_historical_archive/` 下，并使用 `__historical_YYYYMMDD_<reason_code>` 后缀，例如 `B1_ResNet34_UNet_GlaS_seed3407__historical_20260714_protocol_reset`。
- 每个归档目录必须包含独立的 `archive_manifest.md`。该文件不是一句备注，而是该归档对象的历史实验说明主文件，必须记录：原路径、归档路径、归档时间、`reason_code`、详细归档原因、stage/model/dataset/seed、`run_name`、`config_version`、`model_version`、`data_proto_version`、`train_proto_version`、`eval_proto_version`、关键训练参数、评估参数、checkpoint 状态与 hash（若存在）、结果资产状态、`historical_archive_only`、`valid_for_current_gate`、是否允许复用以及复用前置条件。
- `reason_code` 只允许使用受控值：`protocol_reset`、`config_changed`、`eval_protocol_changed`、`identity_migration`、`interrupted_fresh`、`invalid_concurrent_writers`、`stale_smoke`、`runtime_probe`、`contaminated`、`aborted`、`completed_replaced`。
- 归档说明必须同时记录“当时是什么”和“为什么归档”，不能只写新旧目录名；未知字段必须写 `unknown`，不得凭记忆补写。自动归档脚本必须在归档时生成或更新该中文说明文件；指标结果、失败原因和参数差异必须来自归档前后的真实资产，不能凭记忆或预期补写。
- 旧 `A2_UNet_GlaS_v1_seed*` 映射为 `A2_UNet_GlaS_seed*`；旧 `B1_ResNet34_UNet_GlaS_v1_seed*` 映射为 `B1_ResNet34_UNet_GlaS_seed*`。映射只更新身份索引和交接文本，不改结果 CSV/JSON、实验产物或历史数值。旧配置文件名作为历史/兼容资产保留，当前 manifest 不再引用它们。
- 映射后的协议字段必须从旧 `config.yaml`、`run_meta.yaml`、结果表或 manifest 回填；无法确认的字段标记为历史未知，不得补写成当前协议。
- `eval_proto_v3` 等旧评估 lineage 只能在历史说明或归档记录中保留；04 重置后的新轮必须等待研究定标/阶段锁定/Pre-check/Gate 后冻结当前 `eval_proto_version`。

## 7.2 结果作废与重跑规则

如果后面发现下面任意一种情况，该 run 默认不能继续留在正式主表：

- `best_selector` 与总览层冻结规则不一致
- `threshold` 来源不是验证集
- `config_version / eval_proto_version` 没有随协议变化升级
- 结果目录文件名不符合当前命名规范

出现这些问题时，必须执行二选一：

1. 作为旧协议 run 单独归档，不再混入当前主表
2. 按当前协议重跑并生成新的 `run_name`

### 8.1 单次 run 记录字段

后面任何汇总表，只要记录单次 run，至少要有：

| 字段 | 说明 |
|------|------|
| `run_name` | 唯一运行名 |
| `stage` | 阶段名 |
| `dataset` | `GlaS` 或 `CRAG` |

| `eval_aug_enable` | 评估侧是否允许增强，当前主线固定为 `false` |
| `eval_proto_version` | 当前评估协议版本 |
| `seed` | 当前 seed |
| `split_role` | `val / testA / testB / test` |
| `best_selector` | best model 选择规则 |

### 8.2 多 seed 汇总字段

多 seed 表至少要有：

| 字段 | 说明 |
|------|------|
| `model_name` | 模型名 |
| `config_version` | 统一协议版本 |
| `train_proto_version` | 统一训练协议版本 |
| `eval_proto_version` | 统一评估协议版本 |
| `dataset` | 数据集 |
| `split_role` | 测试角色 |
| `seeds` | 参与统计的 seeds |
| `seed_reporting_mode` | `fixed_3_seed_main / fixed_3_seed_main + additional_reporting` |
| `main_seed_set` | 主协议冻结 seed 集 |
| `additional_seed_set` | 若有补充 reporting，显式记录额外 seed 集 |
| `metric_name` | 指标名 |
| `mean` | 均值 |
| `std` | 标准差 |
| `n_runs` | 重复次数 |
| `aggregation` | 固定写 `mean+-std` |
| `result_tag` | 通常写 `reproduced` |

### 8.3 引用值、复现值与统计方式标记

后面结果表里，不允许只写一个数字而不说明它到底是什么身份。

固定标记规则：

- `reproduced`：你自己按当前统一协议跑出来的结果
- `†`：直接引用原文报告值
- `*`：原文自带后处理、特殊测试流程或其它需要额外提醒的设置

固定统计规则：

- `A1`、第一轮外部筛查等排雷用途结果，可以记为 `single_seed`
- 进入正式主线比较层的 `A2 / B1 / C1 / D1 / E1 / I1 / J1` 结果，默认应收敛为 `mean+-std`
- 如果同一张表里同时出现 `single_seed`、`mean+-std` 和 `† quoted` 值，必须额外保留 `aggregation` 与 `result_tag` 字段，或在表注中等价说明

这样做的目的不是让表更复杂，而是防止下面三种混写：

- 把单次试跑值冒充稳定结论
- 把原文引用值冒充当前统一协议复现值
- 把带特殊后处理的数字和无后处理数字写成同一口径

## 9. 代码侧必须统一的命名常量

后面代码里必须统一设一组常量，不要把文件名散落在各脚本里。

固定统一常量：

```python
BEST_CKPT_NAME = "best.ckpt"
LAST_CKPT_NAME = "last.ckpt"
TRAIN_LOG_NAME = "train_log.csv"
VAL_METRICS_NAME = "val_metrics.csv"

TEST_METRICS_NAME = "test_metrics.csv"

```

原因：

- 避免某个脚本写 `.pth`，另一个脚本写 `.ckpt`
- 避免某个阶段还在用旧命名
- 方便后续统一汇总

## 10. 当前明确禁止的做法

- 同一模型出现两种以上 `run_name` 写法
- 改了协议版本但不改 `config_version`
- checkpoint 用 `best.pth`，结果表却写 `best.ckpt`
- `GlaS` 只保留一个总测试值，不分 `TestA / TestB`
- `CRAG` 结果沿用 `testA / testB` 命名
- 同一张表里把 `single_seed`、`mean+-std` 和 `†` 引用值混成一种无标记数字
- 汇总脚本靠手工改字符串适配不同文件名
- 先看测试结果，再回头改 `best checkpoint`

## 12. 冲突裁决记录

- 冲突对象：`run_name` 中的简写版本号、`config.yaml / run_meta.yaml` 中的详细协议版本链，以及结果表中的统计身份标记
- 冲突来源：当前文件旧版虽然冻结了 `run_name`、目录树和基础文件名，但还没有把 `config_version / data_proto_version / eval_proto_version` 之外的 `train_proto_version / loss_version / postprocess_version` 纳入统一记录，也没有显式规定 `reproduced / † / *` 与 `single_seed / mean+-std` 的记录规则；这会和 `结直肠腺体分割_plan_优化版/01_实验执行/00_总览与规范/02_参数冻结总表.md`、`结直肠腺体分割_plan_优化版/01_实验执行/00_总览与规范/04_评估口径与官方脚本对齐.md`、`结直肠腺体分割_plan_优化版/01_实验执行/02_UNet流程验证/00_阶段总协议.md` 中已经出现的协议版本字段和统计口径发生分裂
- 裁决结论：`run_name` 不再包含 `[proto_version]` 或 `config_version`；正式 run 只表达稳定实验身份（阶段、模型、数据集、seed）。详细协议版本统一写入 config.yaml、run_meta.yaml、结果表和 manifest；结果表必须显式记录 `result_tag` 与 `aggregation`，区分 `reproduced / quoted / conditional` 以及 `single_seed / mean+-std`。
- 裁决理由：目录名必须稳定，不能在协议迭代时反复改写正式实验身份；训练协议、评估协议、损失版本和后处理状态仍必须完整可追溯，因此将稳定身份层与详细版本字段层严格分离，并用历史归档后缀保存每轮旧产物。
- 受影响文件：本文件、04_Baseline 五份阶段协议、05-11 中直接消费 baseline 身份和协议字段的文件；01-03 已完成阶段不重写、不重跑。
- 是否需要回流修订：本轮 04-11 命名治理已同步；后续每轮协议迭代前必须完成历史归档、manifest 和字段链复核。
- 代码实现影响：影响 config.yaml、run_meta.yaml、汇总表 schema、阶段汇总脚本字段，以及结果表对 `single_seed / mean+-std / reproduced / quoted / conditional` 记录身份的自动识别逻辑
