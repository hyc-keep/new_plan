# 实验步骤

本文件不是 `06_Boundary` 阶段里“先跑一个看看”的执行备忘录，也不是把几条 run 顺序追加到设计依据后面的口头安排。

它在当前项目中的唯一职责是：

> 作为 `06_Boundary` 的正式实验步骤 protocol，把 `screening_main -> three_seed_main -> optional_one_backup -> stage_decision` 这条唯一合法执行链写成可直接指导训练、测试、可视化、汇总、回退、门控与阶段交接的执行母流程。

从现在开始，只要讨论 `Boundary` 阶段到底按什么顺序跑、什么时候可以补第二第三个 seed、什么情况下才允许补一个备选版本、每个 run 必须交什么资产、阶段结论能否下发到 `07_Distance`，都必须以本文件为准；若与同层文件冲突，以 `结直肠腺体分割_plan_优化版/01_实验执行/06_Boundary/00_阶段总协议.md` 管阶段身份与 `Gate_D1`，以 `结直肠腺体分割_plan_优化版/01_实验执行/06_Boundary/01_设计依据.md` 管为什么只能做最小边界验证，以 `结直肠腺体分割_plan_优化版/01_实验执行/06_Boundary/04_保留或删除标准.md` 管最终 `keep / backup / drop` 裁决细则，而本文件负责把这些上游冻结内容压成唯一可执行流程。

## 1. 文件角色与执行边界

### 1.1 当前文件负责什么

当前文件只负责冻结下面八件事：

1. `Boundary` 阶段的唯一合法起点 `current_base` 如何继承上游，而不是在本阶段口头重选。
2. `Boundary Head v1` 主版本到底先跑什么配置、先跑哪个 seed、主版本筛查通过后如何补齐正式三 seed。
3. 阶段内唯一允许的总执行链只能是 `screening_main -> three_seed_main -> optional_one_backup -> stage_decision`。
4. 每一步的进入条件、停止条件、输出资产和执行状态字段如何写死。
5. 什么时候允许补一个唯一备选版本，什么时候必须停住而不能扩搜索空间。
6. 阶段比较层必须准备哪些 raw、aggregate、visuals 和 cost 资产，才能进入 `stage_decision`。
7. 发生异常时统一按什么顺序回退，什么情况下必须回到前一步重跑。
8. 阶段结束后如何把 `boundary_kept_base / boundary_input_base` 和 `current_base_next` 的唯一语义交给 `07_Distance`。

### 1.2 当前文件不负责什么

当前文件不重新定义下面这些内容：

- 数据身份、split、标签总口径、输入尺寸和归一化。
- baseline 与 `LKMA` 的结构身份、训练 protocol 和正式保留结论。
- `Boundary Head` 为什么值得验证的设计层裁决。
- 边界标签从 mask 派生的详细规则和代码实现细节。
- `keep / backup / drop` 的正式裁决阈值和证据排序本体。
- `Distance-aware Loss` 阶段的结构、损失和下游执行细则。

### 1.3 为什么当前文件必须独立存在

因为 `Boundary` 阶段最容易出现的伪完成状态，不是完全没跑，而是：

- `current_base` 还没唯一确定，就先用不同起点各跑一个版本。
- `screening` 还没排掉标签或接入 bug，就直接铺三 seed 或多个备选配置。
- 主版本还没有形成完整 raw 资产，就提前汇总 `mean+-std` 或下结论。
- 发现结果一般后，不是先回退排查，而是继续把 `w`、`lambda`、loss 或其它变量一起展开。
- `07_Distance` 想继承本阶段结论时，仍然说不清下游该从 `boundary_kept_base` 还是 `boundary_input_base` 启动。

如果没有一份独立实验步骤文件，`结直肠腺体分割_plan_优化版/01_实验执行/06_Boundary/00_阶段总协议.md` 会被迫同时承担阶段门控和执行细则，`结直肠腺体分割_plan_优化版/01_实验执行/06_Boundary/04_保留或删除标准.md` 会被迫补写前置执行链，而下游阶段也无法拿到唯一合法起点。

## 3. 本阶段唯一允许处理的变量

### 3.1 当前阶段直接继承哪些冻结项

本文件直接继承并不得擅自改动的上位规则如下：

- 主任务：`2D gland segmentation`
- 主数据集：`GlaS`
- 正式 split：`train68 / val17 / testA60 / testB20`
- 正式三 seed：`3407 / 1234 / 2025`
- baseline 结构身份：`ResNet34-U-Net`
- `current_base` 来源：仅由 `05_LKMA` 正式结论决定
- `Boundary Head v1`：单边界分支，默认固定接在 `decoder_final`
- 边界标签协议：`binary mask -> single-pixel contour -> fixed-width dilation -> boundary band`
- 主版本默认：`boundary_width = 3`、`boundary_loss_type = BCE`、`lambda_boundary = 0.3`
- `best_selector = val_objdice_max`
- `threshold_source = val17`
- `train_proto_version = train_proto_v1`
- `aug_profile_name = light_aug_v1`
- `eval_aug_enable = false`
- `eval_proto_version` 由 04 阶段当前轮 Gate 冻结
- `eval_cast_policy = logits/probabilities must be kept or cast to float32 before thresholding`
- `boundary_metric_width = 3 px`
- `boundary_metric_impl = project_custom_erosion_xor_3x3_ones_8conn_border0_tol3px`
- `connected_components_impl = scipy.ndimage.label`
- `connected_components_connectivity = 8`
- `postprocess_version = none_in_v1`

### 3.2 当前阶段唯一合法起点

本阶段不允许自由挑选基座，唯一合法定义固定为：

```text
current_base = (lkma_decision_level == keep) ? baseline_plus_lkma : baseline
```

这意味着：

- 若 `LKMA` 判定为 `keep`，则 `Boundary` 必须以 `baseline_plus_lkma` 为起点。
- 若 `LKMA` 判定为 `backup` 或 `drop`，则 `Boundary` 必须回到 `baseline`。
- 任何正式 `Boundary` run 都必须能追溯到 `结直肠腺体分割_plan_优化版/01_实验执行/05_LKMA/04_保留或删除标准.md` 的 decision note 和 handoff 资产。

### 3.3 当前阶段真正要证明什么

`06_Boundary` 当前阶段不是搜索所有边界技巧，而是只回答一个问题：

> 在 `current_base` 已冻结、训练与评估 protocol 不变的前提下，最小显式边界监督是否值得保留进主线。

因此，本文件的执行目标固定为：

1. 先确认主版本最小配置能否稳定跑通。
2. 再确认主版本三 seed 是否形成稳定证据。
3. 只有当主版本结论不清晰时，才允许补一个唯一备选版本。
4. 最后必须用完整证据链形成 `keep / backup / drop`。

### 3.4 当前阶段明确不允许重开的内容

本文件不允许借执行步骤之名重开：

- 数据协议、split、标签总口径和输入尺寸。
- baseline 与 `LKMA` 结构、训练协议和正式结论。
- 边界标签来源与实现路线。
- `best_selector`、阈值来源、后处理和 `TestA / TestB` 统计方式。
- `Distance-aware Loss`、TTA 或其它后续模块。
- 多位置、多分支、多损失、多备选同时展开。

## 4. 本阶段规则卡片

### 4.1 总执行链规则

- 当前结论：`Boundary` 阶段的正式执行链固定为 `screening_main -> three_seed_main -> optional_one_backup -> stage_decision`。
- 规则类型：`路线层已锁定 + 工程冻结规则`
- 适用阶段：`06_Boundary`
- 直接依据：`结直肠腺体分割_plan_优化版/01_实验执行/06_Boundary/00_阶段总协议.md`、`结直肠腺体分割_plan_优化版/02_路线与投稿/02_结直肠腺体分割_分阶段实验路线与执行标准.md`
- 核心公式或定义参考：`execute_d1 = [screening_main, three_seed_main, optional_one_backup, stage_decision]`
- 采用原因：保证先排雷、再补稳定性、最后才在必要时补一个小范围备选，并把搜索空间压到最小。
- 不采用的相邻方案：不先铺大矩阵；不一上来多 seed 多版本并跑；不在主版本不过线时继续扩搜索。
- 代码落点：`crc_gland_segmentation_project/scripts/train.py`、[project_root]/scripts/test.py、[project_root]/scripts/summarize_stage.py
- 运行记录字段：`stage_code`、`step_code`、`variant_rank`、`screening_only`、`decision_level`
- 验收方式：检查阶段内所有正式 run 都能映射到四步之一，不存在脱链实验。

### 4.2 主版本锚点规则

- 当前结论：主版本固定为 `current_base + BoundaryHead(decoder_final, w=3, loss=bce, lambda=0.3)`，首个 screening run 固定使用 `seed3407`。
- 规则类型：`参数冻结执行化 + 工程冻结规则`
- 适用阶段：`06_Boundary`
- 直接依据：`结直肠腺体分割_plan_优化版/01_实验执行/06_Boundary/01_设计依据.md`、`结直肠腺体分割_plan_优化版/01_实验执行/06_Boundary/02_边界标签生成规则.md`、`结直肠腺体分割_plan_优化版/01_实验执行/00_总览与规范/02_参数冻结总表.md`
- 核心公式或定义参考：`screening_run = current_base + BoundaryHead(decoder_final, w=3, lambda=0.3, seed=3407)`；`boundary_insert_position = decoder_final`
- 采用原因：必须先验证最小主版本是否跑通，再决定是否值得补齐三 seed 和进入阶段比较层。
- 不采用的相邻方案：不先测 `w = 5`；不先测 `lambda = 0.2 / 0.5`；不跳过 `seed3407` 直接做多 seed 聚合。
- 代码落点：[project_root]/configs/experiment/boundary_d1_v1.yaml、`crc_gland_segmentation_project/scripts/train.py`
- 运行记录字段：`base_variant`、`boundary_head_version`、`boundary_insert_position`、`boundary_width`、`boundary_loss_type`、`lambda_boundary`、`seed`
- 验收方式：检查第一个正式 run 必须对应主版本 screening 配置。

补充当前唯一允许作为代码起点的 `BoundaryHead.forward()` 伪代码：

```text
Input:
    F_dec in R^(B x C x H x W)            # decoder_final feature

Forward:
    seg_logits = Conv2d(C, 1, kernel_size = 1)(F_dec)
                                           # seg_logits in R^(B x 1 x H x W)
    boundary_feat = Conv2d(C, C, kernel_size = 3, padding = 1)(F_dec)
                                           # boundary_feat in R^(B x C x H x W)
    boundary_feat = ReLU(boundary_feat)
    boundary_logits = Conv2d(C, 1, kernel_size = 1)(boundary_feat)
                                           # boundary_logits in R^(B x 1 x H x W)

Output:
    {
        "seg_logits": seg_logits,
        "boundary_logits": boundary_logits
    }
```

这段伪代码在当前阶段具有唯一解释权，必须同时满足：

- `Boundary Head` 是挂在 `decoder_final` 上的并行轻量分支，不允许再引入第二套 encoder-decoder、gated stream 或 feature feedback 回灌主干。
- `seg_logits` 与 `boundary_logits` 的空间尺寸必须与 `decoder_final` 一致；若主干最终预测分辨率与 GT 不一致，统一在主干既有上采样出口对齐，不允许只给边界分支单独补私有 resize 路线。
- 边界监督只作用于 `boundary_logits`，其训练形式固定为 `BCEWithLogits(boundary_logits, boundary_target_w3)`；总损失写成 `L_total = L_seg(seg_logits, gland_mask) + lambda_boundary * L_bnd`。
- 推理与正式评估主链只消费 `seg_logits` 生成 gland mask；`boundary_logits` 在 `v1` 中只用于训练监督、可视化和误差分析，不进入后处理或官方评测前的掩码修正。

## 5. 如何判断设计有效(验收标准)

### 5.1 `screening_main`

当前阶段的首个正式动作固定为：

1. 用唯一 `current_base` 加载主版本边界配置。
2. 固定 `seed = 3407`。
3. 运行完整训练、验证、`TestA`、`TestB`、边界指标和可视化导出。
4. 在正式评估链中固定 `eval_cast_policy = float32 before thresholding`、`boundary_metric_width = 3 px`、冻结实现与 `8-connectivity`。
5. 检查标签、loss、可视化、日志和资产是否全部正常。

只有当下面条件同时成立，才允许从 `screening_main` 进入 `three_seed_main`：

- 训练无明显崩溃或 loss collapse。
- 边界标签与预测 overlay 无明显错位。
- `best.ckpt`、`last.ckpt`、`val/testA/testB` 指标和边界指标文件齐全。
- metric_crosscheck_note.md 已记录 `Boundary F1` 宽度、实现依赖和抽样核对结论。
- 主版本配置字段链完整写入 run_meta.yaml。

### 5.2 `three_seed_main`

`screening_main` 通过后，必须按正式三 seed 补齐主版本：

1. 保留 `seed3407` 的 screening 结果作为主版本第一次正式 run。
2. 继续补齐 `seed1234`。
3. 继续补齐 `seed2025`。
4. 三个 seed 全部跑完后，汇总 per-seed raw 和 `mean+-std`。

当前阶段不允许：

- 另起一个与 `screening` 不同配置的三 seed 主版本。
- 因为某个 seed 不好看而换 seed。
- 在主版本三 seed 未齐前，先把结论写成正式阶段结论。

### 5.3 `optional_one_backup`

只有当主版本三 seed 跑完后仍出现“结论不清晰而非明显失败”的情况，才允许补一个备选版本。

允许触发备选的典型情形只有：

- 主版本边界级或对象级信号略有改善，但波动较大。
- 主版本表现接近通过线，但怀疑 `w` 或 `lambda` 稍微偏弱或偏强。
- 主版本没有明显 bug，也不是明显失败，只是需要一个小范围 sanity check。

不允许触发备选的情形包括：

- `screening_main` 明确失败。
- 主版本三 seed 明显不过线却仍想靠扩搜索强行保留。
- 想同时补多个宽度、多个权重或换损失。

### 5.4 `stage_decision`

进入 `stage_decision` 前，至少要先确认以下资产齐备：

1. 主版本三 seed raw 指标。
2. 主版本 `mean+-std` 聚合结果。
3. `Boundary F1`、`HD95`、对象级指标和 split 分开报告。
4. metric_crosscheck_note.md 已固定记录 `eval_cast_policy = float32 before thresholding`、`boundary_metric_width = 3 px`、`boundary_metric_impl`、`connected_components_impl`、`connected_components_connectivity = 8` 与抽样样本。
5. 黏连案例、边界 overlay 和失败案例图册。
6. 参数量、FLOPs、延迟或其它成本记录。
7. 若存在备选版本，其 raw 和 aggregate 资产也已齐备。

随后才能把上述资产交给 `结直肠腺体分割_plan_优化版/01_实验执行/06_Boundary/04_保留或删除标准.md` 进行正式裁决。

## 8. 回退条件

### 8.1 独立回退触发条件

### 8.2 固定回退顺序

回退时统一按下面顺序排查：

1. 先检查边界标签协议是否与 `结直肠腺体分割_plan_优化版\01_实验执行\06_Boundary\02_边界标签生成规则.md` 完全一致。

3. 再检查 `lambda_boundary`、边界损失与主损失的平衡是否出现 collapse。
4. 再检查训练与评估 protocol 是否仍完整继承当前基座。
5. 最后才判断是否值得补一个唯一备选版本。

### 8.3 回退后的重新放行条件

发生回退后，只有同时满足下面条件，才允许重新声明本文件通过：

- 异常来源已经按固定顺序定位并修复。
