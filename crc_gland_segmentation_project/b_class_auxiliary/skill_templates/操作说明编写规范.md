# 阶段操作说明编写规范（Skill 模板）

版本: v1.0  
生成时间: 2026-07-11  
适用范围: 所有实验阶段（01-09）

---

## 一、强制要求

### 1.1 文件命名与存放位置

- **文件名**: `操作说明_如何运行本阶段.md`（固定命名，不得更改）
- **存放路径**: `reports/stage_reports/implementation_tracking/{stage_id}_{stage_name}/`
- **示例**:
  - `reports/stage_reports/implementation_tracking/01_数据协议/操作说明_如何运行本阶段.md`
  - `reports/stage_reports/implementation_tracking/03_UNet稳定性/操作说明_如何运行本阶段.md`

### 1.2 交付标准

每个实验阶段交付时，**必须包含操作说明文档**，否则该阶段不算完成。操作说明必须经过以下验证：

- [ ] 按照文档中的命令能从头跑通整个阶段（由非开发者验证）
- [ ] 所有参数都有明确说明（必填/默认值）
- [ ] 包含明确的成功标准（文件路径 + 指标范围 + gate 状态）
- [ ] 包含至少 2 条常见坑说明（基于真实踩坑经验）
- [ ] 诚实边界清晰（明确说明本阶段做什么、不做什么）

## 二、标准六章节模板

### 章节一：这个阶段在做什么

**内容要求**:
1. 1-2 段话说明阶段定位（本阶段在整个实验流程中的位置）
2. 核心产物清单（文件路径 + 简短说明）
3. 与前后阶段的关系（前置条件 + 后续交接内容）
4. 阶段标识符（如 A1/A2/B1，方便溯源）

**模板**:
```markdown
## 一、这个阶段在做什么

用**{关键技术点}**实现{核心目标}，验证「{核心假设}」。产出正式 run：`{output_directory}/`。

这是「{阶段定位}」（{stage_id}），前置是 {previous_stage} 阶段已通过。本阶段{与前阶段的关系说明}。

注意：{关键概念澄清或常见误解说明}。
```

**示例**（来自 03_UNet稳定性）:
```markdown
## 一、这个阶段在做什么

用**三个 seed（3407 / 1234 / 2025）**各跑一次完整 UNet 训练→测试，然后把三次结果聚合成 mean±std，验证「训练与评估链路对随机种子是稳定的」（换 seed 指标抖动小），并产出论文用的稳定性表。产出正式 run：`experiments/A2_UNet_GlaS_v1_seed{3407,1234,2025}/`。

这是「稳定性验证」（A2），前置是 02 阶段（A1，单 seed 流程验证）已通过。A2 只是把 A1 的完整链在多 seed 上复跑并聚合，**评估协议与 A1 完全一致**。

注意：本阶段主指标 F1 是 **GlaS 官方对象级检测 F1**（F1_obj = 2TP/(2TP+FP+FN)，对象匹配 overlap>0.5），不是像素 Dice。三者是不同口径，别混（详见 `实现依据记录.md` 第 13 节）。
```

### 章节二：先读哪些文件（推荐阅读顺序）

**内容要求**:
1. 至少 3 个文件的推荐阅读顺序
2. 从总览到细节（清单 → README → 当前状态 → 具体实现细节）
3. 新手友好（假设读者第一次接触本阶段）

**模板**:
```markdown
## 二、先读哪些文件（推荐阅读顺序）

1. `00_交付范围内正式对象清单.md`
2. `README.md`
3. `implementation_status.md`
4. `当前阶段为什么能pass以及下一步怎么看.md` —— 含 {关键内容说明}
5. `实现依据记录.md` —— 尤其第 X 节（{重要章节说明}）
6. 配置逐文件说明：`configs_{category}_{file}.yaml.md`
7. 脚本逐文件说明：`scripts_{script}.py.md`
```

**要点**:
- 前 4 个文件是所有阶段的标准文档，顺序固定
- 第 5-7 项根据阶段特点调整（突出本阶段最关键的文档）
- 如果某个文档包含重要章节，用 `—— 尤其第 X 节` 明确指出

### 章节三：配置引用关系 / 运行顺序总览

**内容要求**:
1. **训练类阶段**：展示配置文件的引用层级（experiment 入口 → 子配置）
2. **工具链阶段**：展示脚本的运行顺序流程图（ASCII 或 Markdown）
3. 突出关键约束（如"三 seed 必须共用同一份 eval_proto_v1.yaml"）

**模板A - 配置引用关系**（适用于训练阶段）:
```markdown
## 三、配置引用关系（重要）

{数量描述，如"三个 seed 各有一个 experiment 入口配置，除 seed 值外其余完全同构"}：
- `configs/experiment/{entry_config_1}.yaml`
- `configs/experiment/{entry_config_2}.yaml`
- ...

每个 experiment 入口都引用同一套子配置：
- `configs/model/{model_config}.yaml` —— 模型结构
- `configs/train/{train_config}.yaml` —— 训练超参（epoch、patience、lr 等）
- `configs/eval/{eval_config}.yaml` —— 评估协议（阈值来源、指标口径）
- `configs/data/{data_config}.yaml` —— 数据（继承自 XX 阶段）

{关键约束说明，如"三个 seed 必须共用同一份 eval_proto_v1.yaml，否则聚合失去可比性"}。
```

**模板B - 运行顺序流程图**（适用于工具链阶段）:
```markdown
## 三、运行顺序总览

```
step_1 / step_2   （{简短说明}）
        ↓
step_3            （{简短说明}）
        ↓
step_4 / step_5   （{简短说明}）
        ↓
step_6            （{简短说明}）
```
```

### 章节四：完整执行流程与命令

**内容要求**（最重要章节）:
1. 逐步可复现的命令清单
2. 每个命令都带完整参数说明
3. 明确区分必填参数和可选参数
4. 说明产物路径和验证方法
5. 标注命令执行前置条件（如"若 checkpoint 已存在可跳过"）

**参数说明格式规范**:
```markdown
- `--param-name`：参数说明文字，默认 `default_value`（或 **必填**）。
- `--another-param`：参数说明，默认 `None`（自动推断）；仅在 {特殊场景} 时传。
```

**模板**:
```markdown
## 四、完整执行流程与命令

> 二次重做直接照抄本节。前置：`cd /home/featurize/work/Paper/crc_gland_segmentation_project`

### 步骤 0（{条件说明，如"若 checkpoint 已存在可跳过"}）：{步骤名称}

{步骤说明段落}

```bash
{command_line}
```
- `--param1`：**必填**。{参数说明}
- `--param2`：默认 `value`（{默认值说明}）。{参数用途}
- `--param3`：默认 `None`（{自动行为说明}）；{何时需要传参}
- {特殊说明，如"早停 patience=20，best_selector=`val_objdice_max`"}
- 产物：`{output_path}`（{产物说明}）、`{another_output}`。{验证方法，如"输出应含 `key=value`"}

### 步骤 1：{步骤名称}

...
```

**关键要求**:
- ❌ 禁止："根据情况调整"、"请参考源码"
- ✅ 必须：每个参数都有明确的"必填"或"默认 `value`"标注
- ✅ 必须：每个步骤都有产物路径和验证方法（如"输出末尾应为 `stage_pass_a2=true`"）

### 章节五：跑完怎么确认成功

**内容要求**:
1. 文件存在性检查（关键产物路径）
2. 文件内容检查（关键字段值）
3. 指标范围检查（带容差说明）
4. Gate 状态检查（如 `workflow_gate_status=pass`）

**模板**:
```markdown
## 五、跑完怎么确认成功

- {关键文件路径 1}：`{key1}={expected_value1}`、`{key2}={expected_value2}`、...
- {关键文件路径 2}：有 `{key_pattern}` 的行。
- {脚本输出}末尾 `{gate_key}={pass_value}`；{报告路径}末尾 `{another_gate_key}={pass_value}`。
- 真实参考指标（{实验条件说明，如"三 seed mean±std"}，重跑因环境略有浮动属正常）：

| split | 指标1 | 指标2 | 指标3 | 指标4 |
|---|---|---|---|---|
| testA | {value1} ± {std1} | {value2} ± {std2} | ... | ... |
| testB | {value3} ± {std3} | {value4} ± {std4} | ... | ... |

判定要点：{指标解读说明，如"看的是抖动小（std < 0.02）不是指标高"}。{预期现象说明，如"对象级 F1 低于像素 Dice 是 vanilla UNet 的架构限制，属预期"}。
```

**关键要求**:
- ✅ 必须：参考指标附带"重跑因环境略有浮动属正常"免责说明
- ✅ 必须：判定要点说明"看什么"（如抖动小 vs 指标高）
- ✅ 必须：预期现象说明（避免误判为失败）

### 章节六：常见坑

**内容要求**:
1. 至少 2 条常见坑说明
2. 基于真实踩坑经验（不是臆想的"可能会遇到的问题"）
3. 每条坑都包含：错误现象 + 原因 + 正确做法

**模板**:
```markdown
## 六、常见坑

- **{坑的简短标题}**：{错误现象描述}。{原因说明}。{正确做法}。
- **{另一个坑}**：{错误现象}。{原因}。{正确做法}。
- **{阶段特有的坑}**：{特殊说明}。
- 命令必须在 `{project_root}/` 目录下跑（不是外层 `{parent_dir}/`）。
```

**示例**（来自 03_UNet稳定性）:
```markdown
## 六、常见坑

- **F1 是对象级不是像素 Dice**：主表 F1、Object Dice、像素 Dice 是三种口径，看结果别张冠李戴。历史示例数字 F1≈0.6089/0.6247、Dice≈0.8908/0.8854 仅用于说明旧口径错误，不是当前结果；当前 A2 正式指标需读取真实七项 raw/mean±std 表。
- **summarize_stage 两种模式**：03 阶段用 `--stage a2`（多 seed 聚合）；02 阶段用 `--config`（单 run）。用错模式聚合结果不对。
- 命令必须在 `crc_gland_segmentation_project/` 目录下跑（不是外层 `Paper/`）。
```
