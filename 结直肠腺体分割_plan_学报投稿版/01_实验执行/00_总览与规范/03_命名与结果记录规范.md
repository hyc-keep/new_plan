# 命名与结果记录规范

## 1. 文件角色与执行边界
本文件是 active journal 的 run identity、版本、事实记录和 lineage 协议，归属 A 类。命名表达身份，不表达成功；`status=planning_rewrite_verified_pending_diagnostics`，`formal_result=not_run`，`result_eligibility=false`。

## 2. 本轮直接依赖的前置文件
旧计划同名 03 为命名结构基线；active 00/01/02/04/05/06/07 分别约束阶段、目录、参数、评估、接口、写作和质检；路线与投稿最低证据包定义 05-11 的消费者；GlaS/DCAN/MILD-Net 和 `/home/featurize/work/Paper/结直肠腺体分割_正式参考资料/04_task_specific_benchmarks/GlandSegBenchmarks-master/DCAN/test_GlaS.py` 定义 benchmark/metric 身份；旧工程接口（`/home/featurize/work/Paper/crc_gland_segmentation_project/scripts/train.py`、`/home/featurize/work/Paper/crc_gland_segmentation_project/scripts/test.py`、`/home/featurize/work/Paper/crc_gland_segmentation_project/src/engine/trainer.py`、`/home/featurize/work/Paper/crc_gland_segmentation_project/src/eval/run_eval.py`、`/home/featurize/work/Paper/crc_gland_segmentation_project/src/eval/checkpoint_selector.py`、`/home/featurize/work/Paper/crc_gland_segmentation_project/scripts/summarize_stage.py`） 仅用于记录旧接口职责。

## 3. 本阶段唯一允许处理的变量
只处理 run identity、版本、状态、事实字段和 lineage，不改模型设计或评估定义。

## 4. 阶段门控表达式
`record_gate = unique_identity AND metadata_complete AND artifact_consistent AND blind_preserved`；否则结果不得进入汇总。

## 上游、同层与下游
上游是参数冻结和执行导航，提供 stage、方法身份与状态词典；同层工程目录和评估规范提供资产路径与指标字段；下游所有运行、汇总、Gate、handoff 和论文表格必须通过本文件追溯唯一 run。结果文件没有完整元数据时只能 `blocked/not_run`。

## 路线、文献与参考代码依据
当前 `stage_code` 只允许 `s01..s11`；N1/N2/N3 只能是 `method_id`。GlaS 的 TestA/TestB 必须分开；DCAN/MILD-Net 的对象级评价要求不能被单一 Dice 取代。旧工程提供 train -> val -> checkpoint -> test -> summary 的接口基线，但 current 实现只能引用 journal root。

## 命名规则
`run_name=<stage_code>_<method_id>_<dataset>_<protocol_version>_<seed>`，例如 `s05_n1_glas_p05_v1_s3407`。`method_id` 可用 `b1/n1/n2/n3`；数据集只允许 `glas/crag`。协议版本不得隐藏在自然语言目录中，必须在元数据中显式记录。

## 固定、可调与禁止字段
- 固定：`stage_code` 语义、dataset code、七字段 lineage、状态词典、`formal_result=not_run`。
- 可调：stage card 预注册的 `method_id/protocol_version/seed`，升级协议须新增版本并保留旧事实记录。
- 禁止：同身份多配置无 digest、把 `best` 写入未执行的 checkpoint、使用旧 run_name 伪装 current、从 test 结果回流调参、把 `quoted_from_original_paper` 当复现值。

## 版本字段与运行记录
每个 run_meta/config 至少记录：`config_version/data_proto_version/train_proto_version/aug_version/eval_proto_version/model_version/loss_version/postprocess_version`；事实字段为 `run_name,status,started_at,ended_at,git_commit,config_sha256,split_sha256,seed,device,metric_names,checkpoint,prediction_manifest,warning_codes,planned_or_actual,formal_result,result_eligibility`。规划阶段使用 `started_at=not_run`、`checkpoint=not_created`、`prediction_manifest=not_created`。

## 规则卡片：best checkpoint 身份
- 当前结论：原则上记录 `best_selector=val_objdice_max`，最终以 active 评估协议和 stage lock 的实际版本为准。
- 规则类型：工程冻结规则。
- 适用阶段：首个可运行 baseline 起至汇总。
- 直接依据：active 02/04、旧工程 `checkpoint_selector.py` 职责、GlaS/MILD-Net 对象级评价。
- 采用原因：统一跨方法选择口径且不窥视测试集。
- 不采用的相邻方案：不混用 `val_loss`、`val_dice` 或 test 最优。
- 代码落点：`crc_gland_segmentation_project_journal/src/eval/checkpoint_selector.py`。
- 运行记录字段：`best_selector/selected_epoch/selection_split/selection_metric/eval_proto_version`。
- 验收方式：selector audit 能回查 val 行和 config；未运行时只记录 planned。

## 代码落地接口
`src/utils` 或 config validator 负责校验 run_name 与版本；`scripts/train.py` 写 run_meta；`src/eval/run_eval.py` 写 per-image/aggregate metric artifact；`src/eval/checkpoint_selector.py` 记录选择；`scripts/summarize_stage.py` 读取 raw 资产生成汇总。没有实际实现的字段必须标 `planned_not_created`，不能声称已写入。

## 验收、blind 与 handoff
名称、config、目录、manifest、报告必须逐字一致；06 的 `consumption_boundary=val_only_no_test_labels`；07/08 使用 frozen-test 只读边界；handoff 七字段完整且包含 `formal_result`、`result_eligibility`。当前任何结果行必须是 `not_run/false`。

## 独立回退条件
自然语言名无法唯一定位、同 run 多配置、版本字段缺失、manifest/hash 不一致、测试结果回流、状态与资产冲突时，废止该 run 身份并回退到最近未泄漏阶段；不得修补结果表掩盖身份问题。

## 冲突裁决记录
旧格式可能只写 stage/model/dataset/seed，本轮为 journal 增加 protocol version 以支持重建审计；旧 run 不迁移。旧 `LKMA/Boundary/Distance` 阶段名仅 historical provenance，不可生成 current run。

- [x] 状态、formal result、资格和 not_run 语义明确。
- [x] 版本、事实字段、七字段 lineage、blind/handoff 完整。

## 7. 资产命名与状态转移
`planned/not_run -> running -> completed` 只能由真实运行事实推动；`warning`、`failed`、`blocked` 必须保留原因。`best` 只能在 selector audit 后出现，不能作为目录装饰词。

## 8. 结果记录验收
检查 run_name 唯一性、版本链完整性、config/hash/manifest 一致性、checkpoint 来源、指标 identity、测试盲态和 handoff 七字段。缺任何一项就留在 `blocked` 或 `not_run`。

## 9. 当前禁止动作
不得复用旧 run_name 生成 current 结果，不得手工覆盖 run_meta，不得将论文引用数字写入实验 CSV，不得因为目录已有文件就声称本轮运行完成。

## 10. 运行资产逐项核对
- 核对 run_name 可唯一定位。
- 核对 stage_code 不承载 N1/N2/N3 阶段语义。
- 核对协议版本独立记录。
- 核对 seed、device、commit 和 hash 字段。
- 核对 checkpoint 选择 split。
- 核对 prediction manifest。
- 核对 per-image 与 aggregate 的关联。
- 核对 TestA/TestB 分开。
- 核对引用值与复现值标签。
- 核对 handoff 七字段。
- 核对 warning 和 blocked 原因。
- 核对未运行资产仍为 not_created。

## 11. 当前审计结论
本文件定义记录协议，不改写历史运行事实；任何缺字段的 run 不能进入正式结果层。

## 12. 记录审计结论
运行事实、规划事实和历史事实必须分开；任何字段缺失、路径冲突或状态越级均不得进入正式结果汇总。当前只登记协议，不生成结果资产。

## 13. 记录变更回流
run identity、版本、状态或字段改变时，必须同步目录、配置、manifest、报告和 handoff；不能只改 CSV。

## 14. 记录责任
运行作者负责事实，阶段负责人负责选择，汇总作者负责聚合，治理负责人负责准入。

## 结果污染与恢复规则
- 同一 `run_name` 不得同时对应多个 seed、config digest 或 protocol version。
- `epoch_metrics.csv` 只能由训练进程按 epoch 写入；resume 前必须校验 epoch 单调、train/val 行成对、batches 为全量或显式 smoke。
- 发现 epoch 缺行、重复、越界、孤儿行或多进程并发写入时，当前 run 标记为 `invalid_contaminated`，完整归档，禁止手工修 CSV。
- `best.ckpt` 必须能回链 `best_selector`、`best_epoch`、`metric_value`、config digest、code digest 和 data digest。
- `testA_metrics.csv`、`testB_metrics.csv` 只能在 checkpoint 和阈值冻结后生成；测试文件不能回流到候选或 checkpoint 选择。

## 结果审计字段
- 每行 raw 结果至少记录 `row_type=sample`、`run_name`、`seed`、`split`、`sample_id`、`metric_set`、`checkpoint_sha256`、`eval_proto_version`。
- aggregate 必须独立文件记录 `n_runs`、seed 列表、mean、std 和统计版本；raw 文件禁止 aggregate 行。
- 每个阶段必须生成 `result_manifest.json` 或等价 manifest，记录文件 sha256、行数、schema、消费边界和 Gate 状态。

## Diagnostics 闭环
本轮需运行正式文档检查和 schema/路径检查。当前所有结果状态为 `not_run`，不写任何性能数字。复核必须包含命名唯一性、run/config 一一映射、epoch 完整性、raw/aggregate 分离、checkpoint identity 和 blind 边界；任一失败时回退并归档，不允许补改结果文件。
