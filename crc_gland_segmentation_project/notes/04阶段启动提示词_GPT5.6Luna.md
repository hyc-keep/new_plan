## 五、04-11 固定工作流

04 到 11 的每个正式阶段都必须复用同一个通用 `stage contract checker`，每个阶段只提供自己的契约输入，不复制 checker 实现。

固定顺序：

```text
当前阶段 research alignment
→ 当前阶段 stage lock
→ 当前阶段 Pre-check
→ stage contract checker
→ 最小代码实现
→ 单样本 runtime
→ 小规模 smoke
→ 正式代码/配置冻结
→ 正式训练/测试/预测/汇总
→ 独立指标复核
→ code quality
→ workflow gate
→ 最后写学习型说明文档
```

### 每一步的硬性规则

1. research alignment：锁定研究问题、文献/参考代码依据、候选方案、指标依据和不可改变的科学边界。
2. stage lock：锁定唯一目标、允许改动、禁止改动、变量隔离、正式产物、回退条件和下游消费边界。
3. Pre-check：检查计划、代码库、路径、依赖、数据入口、评估入口和工程落点。
4. stage contract checker：机器检查 `identity/schema/lineage/freshness`。
5. 最小代码实现：只实现当前阶段最小功能；先过语法、导入、静态检查。
6. 单样本 runtime：真实验证 shape、dtype、forward、loss、finite、backward、optimizer.step。
7. 小规模 smoke：用锁定配置验证数据、训练、验证、checkpoint、测试/评估入口连通。
8. 正式代码/配置冻结：smoke 通过后冻结；正式运行期间不改代码、配置、指标或评估口径。
9. 正式运行：只使用冻结代码和配置，生成正式原始结果和资产。
10. 独立指标复核：从 prediction + GT 独立重算，逐样本、逐 split 与正式 CSV 比较并记录容差。
11. code quality/workflow gate：只消费真实 runtime、独立复核、正式资产和当前 gate。
12. 最后写说明文档：只解释已经通过 gate 的真实代码和证据，不能反向修改结果或 gate。

禁止：

- 跳过最小代码实现；
- 用单样本 runtime 代替 smoke；
- 用 smoke 代替正式实验；
- 用正式指标 CSV 存在代替独立指标复核；
- 在 workflow gate 通过前写完成型学习说明文档；
- 修改代码/配置后继续沿用旧 runtime、code-quality 或 workflow 报告；
- 手工修改 metrics CSV、train_log、val_metrics、run_meta 结果字段、runtime evidence 或 gate 数字。
