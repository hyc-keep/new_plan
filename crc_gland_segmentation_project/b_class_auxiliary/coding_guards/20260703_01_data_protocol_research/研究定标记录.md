## 6. 当前阶段边界初判
- 当前阶段应该做什么: 锁定正式数据资产身份、split 协议、二值标签与派生目标规则、数据检查与预览闭环、交接资产、以及下游唯一允许消费的数据资产链
- 当前阶段不该做什么: 不开始写 `UNet`、不定义 optimizer/lr/loss、不过早做 Pre-check、不过早讨论 runtime evidence、不过早写说明文
- 为什么现在不能直接进入下一个阶段: 因为当前 `datasets` 磁盘状态与文档描述不一致，正式 `GlaS` 和 `CRAG` 数据根并未出现在当前工作区；在正式输入层没有锁死前直接进入 `UNet` 会把数据问题和训练问题混在一起
- 如果现在跳过边界锁定,最容易犯什么错: 会把“研究已经说明文档怎么写”误认为“正式数据资产已经存在”，会在没有正式 split / config / manifest 的情况下直接写训练代码，最后无法解释训练入口到底读取了哪一版数据

## 7. 对后续阶段锁定的输入
- 后续 `00_阶段实现卡.md` 必须写清的关键点: 当前唯一目标只能是“恢复并冻结 `01_数据协议` 的正式输入层”；允许改动应限定在正式数据资产、split 清单、数据配置、数据检查输出、数据预览输出和相应工具脚本；禁止改动应明确包括训练框架、模型结构、loss、评估实现和说明文
- 后续阶段锁定最容易写空的地方: 一是把“数据协议固定”写成空泛文档任务，不写磁盘真实缺口；二是不写当前 `datasets` 只剩说明文和参考工具、缺少主数据目录这一事实；三是不写下游唯一允许消费的资产链
- 后续最小运行验证计划至少要覆盖: 验证正式 GlaS 与 CRAG 数据根真实存在；验证 7 个正式 split CSV 能生成；验证 image/mask 配对和路径可读；验证 `mask > 0` 二值化与 resize 规则可复算；验证最小检查资产和预览资产可输出；验证数据配置文件能唯一指向正式 split 和正式数据根
