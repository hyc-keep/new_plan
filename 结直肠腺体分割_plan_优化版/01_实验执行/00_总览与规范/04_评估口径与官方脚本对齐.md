## 5. 阈值规则怎么定

### 5.1 当前统一规则

2. 真实正式阈值只允许在验证集上确定

- `threshold_source`
- `threshold_value`
- `best_selector`
- `eval_proto_version`
- `postprocess_version`

### 5.4 `best checkpoint` 选择器

当前统一冻结为：

- `best_selector = val_objdice_max`

原因：

- 主线任务的正式主指标本来就围绕对象级指标展开
- `Object Hausdorff` 不适合作为单一 best selector
- 继续允许 `val loss / val Dice / val F1` 混用，会直接破坏跨阶段公平性

落地要求：

- src/eval/checkpoint_selector.py 只能保留一个正式入口
- config.yaml 与 run_meta.yaml 都必须记录 `best_selector: val_objdice_max`
- run_meta.yaml 必须记录 `best_selector: val_objdice_max`
- 如果未来确实要改 selector，必须同步升级 `config_version` 与 `eval_proto_version`

### 5.5 评估执行链固定顺序

后面正式评估统一按下面顺序执行：

1. 读取模型输出 `logits`
2. 将 `logits` 保持为或显式转换为 `float32`
3. 在 `float32` 上执行 `sigmoid` 得到概率图
4. 仅允许对 `float32` 概率图应用当前冻结的 `threshold`
5. 得到二值 mask
6. 提取连通域
7. 计算对象级与像素级指标

当前 `v1` 评估协议额外固定下面三条硬规则：

- `eval_cast_policy = logits/probabilities must be kept or cast to float32 before thresholding`
- `eval_pipeline = logits(float32) -> sigmoid(float32) -> threshold -> binary mask(uint8/bool) -> connected components -> metrics(float32)`
- 如果训练或推理开启 `AMP`，混合精度只允许存在于前向计算阶段；进入正式评估链后，`threshold` 之前的 `logits / probabilities` 一律回收到 `float32`

当前 `v1` 口径下：

- 默认不加额外 postprocess
- 默认不做 TTA

如果后面引入：

- hole filling
- small object removal
- morphology opening / closing
- TTA 融合

都必须视为新的评估协议版本，不能继续沿用 `eval_proto_v1`

### 5.6 `Boundary F1` 评估宽度与唯一来源

当前全局评估层新增固定下面五条规则：

1. `Boundary F1` 的正式评估宽度固定为 `boundary_metric_width = 3 px`
2. `boundary_metric_width_source = inherit_boundary_main_width`
3. `Boundary F1` 的正式主表和补充表都只允许使用这一宽度
4. 如果未来为了敏感性分析额外报告 `5 px`，必须单列为 `boundary_metric_width = 5 px (sensitivity only)`
5. 不允许训练标签宽度、可视化宽度和评估宽度各写各的版本

这样固定的原因是：

- `结直肠腺体分割_plan_优化版/01_实验执行/06_Boundary/02_边界标签生成规则.md` 已经把主版本边界宽度冻结为 `w = 3`
- 如果评估层不显式继承这一宽度，`Boundary F1` 很容易在不同阶段被写成不同带宽
- `Boundary` 与 `Distance` 的边界证据只有在训练标签宽度和评估宽度可回查一致时才具备解释力

当前正式字段固定为：

- `boundary_metric_name = boundary_f1`
- `boundary_metric_width = 3`
- `boundary_metric_width_unit = px`
- `boundary_metric_width_source = inherit_boundary_main_width`

### 5.7 评估实现依赖与库级边界

当前 `eval_proto_v1` 额外冻结下面实现依赖，后续不得一边运行一边换库：

- `connected_components_impl = scipy.ndimage.label`
- `connected_components_connectivity = 8`
- `boundary_metric_impl = project_custom_erosion_xor_3x3_ones_8conn_border0_tol3px`
- `hausdorff_impl = symmetric discrete Hausdorff on object boundaries`
- `hd95_impl = MedPy-style discrete surface distance`
- `pixel_spacing = 1`

统一实现解释如下：

- 连通域提取统一按二维 `8-connectivity` 处理，保证 `GlaS / CRAG` 上对象拆分规则一致
- `Boundary F1` 先从 GT 与 prediction 二值 mask 提取内边界，再按 `boundary_metric_width = 3 px` 生成匹配带
- 对象级 `Hausdorff` 必须按对称距离实现，不允许把 `HD95` 或单向距离冒充对象级 `Hausdorff`
- `HD95` 只作为补充指标实现，不得替代对象级 `Hausdorff`

只要上面任一实现库、连通性或边界提取模式发生变化，就必须：

- 升级 `eval_proto_version`
- 重跑 `metric_crosscheck`
- 禁止与旧协议结果同表混写

## 6. 对象级指标必须怎么对齐

### 6.1 当前核心要求

后面对象级指标实现必须最大程度对齐 `GlaS Challenge` 官方口径。

这不是“可选优化项”，而是主结果可引用性的底线。

### 6.2 记号与连通域约定

后面对象级实现统一采用下面记号：

- `G = {G_1, ..., G_n}` 表示 GT 腺体连通域集合
- `S = {S_1, ..., S_m}` 表示预测前景连通域集合
- `|A|` 表示区域 `A` 的像素数
- `A ∩ B` 表示交集
- `HD(A, B)` 表示两个对象边界之间的 Hausdorff distance

统一前提：

- GT 与 prediction 都先二值化到 `0/1`
- 连通域提取规则在 `GlaS / CRAG` 上必须保持同一实现
- 不允许某个模型单独追加不同后处理再算对象级指标

### 6.3 匹配规则与 `TP / FP / FN` 定义

后面对象级 `F1` 的第一步不是直接算分，而是先做对象匹配。

当前统一规则：

1. 对每个预测对象 `S_j`，寻找与其重叠面积最大的 GT 对象 `G_i`
2. 若 `|S_j ∩ G_i| / |G_i| > 0.5`，则记该预测对象命中一个 GT 对象；分母使用 GT 对象面积，判定使用严格 `>`，不使用 `>=`
3. 每个 GT 对象最多只允许与一个预测对象形成主匹配，并按项目冻结的去重规则处理重复匹配
4. 成功匹配的预测对象记为 `TP`
5. 未匹配预测对象记为 `FP`；未被命中的 GT 对象计为 `FN`

对应计数公式：

- `TP = 匹配成功的对象数`
- `FP = m - TP`
- `FN = n - TP`

这里最容易漂移的不是公式本身，而是：

- overlap 是否按预测对象面积归一
- 多个小预测同时碰到同一 GT 时如何去重
- 连通域提取前是否做了额外 morphology 操作

这些细节后面必须全部固化在 src/metrics/object_metrics.py 与 `eval_proto_version` 里。

### 6.4 `Object F1` 数学定义

对象级 `F1` 统一写成：

- `F1_obj = 2TP / (2TP + FP + FN)`

解释方式：

- 它衡量的是“腺体对象有没有被识别出来”
- 不是像素重叠率
- 因此它对过分裂、漏检、黏连导致的对象识别失败很敏感

代码实现时必须确保：

- `TP / FP / FN` 来自同一套匹配规则
- 不允许训练阶段和测试阶段各写一套匹配逻辑

### 6.5 `Object Dice` 数学定义

对象级 `Dice` 统一采用对象加权版本：

- `Dice(A, B) = 2|A ∩ B| / (|A| + |B|)`
- `gamma_i = |G_i| / sum_p |G_p|`
- `sigma_j = |S_j| / sum_q |S_q|`
- `D_obj(G, S) = 1/2 * [sum_i gamma_i * max_j Dice(G_i, S_j) + sum_j sigma_j * max_i Dice(G_i, S_j)]`

统一理解：

- 前半部分从 GT 视角衡量每个真实腺体被覆盖得如何
- 后半部分从 prediction 视角衡量每个预测腺体是否对应真实目标
- 两侧一起算，才能避免只偏向大对象或只偏向预测侧

工程上必须注意：

- 如果某个对象没有任何重叠对象，相关项记为 `0`
- 不允许偷偷把对象级 `Dice` 简化成整张图的像素 `Dice`

### 6.6 `Object Hausdorff` 数学定义

对象级 `Hausdorff` 统一采用对象加权版本：

- `H_obj(G, S) = 1/2 * [sum_i gamma_i * min_j HD(G_i, S_j) + sum_j sigma_j * min_i HD(G_i, S_j)]`

统一理解：

- 它衡量的是对象边界之间最坏点偏差
- 权重仍然按对象面积分配
- GT 侧与 prediction 侧都要纳入，避免只看一边

实现约定：

- 若对象存在重叠，优先在重叠候选中选对应对象
- 若对象完全无重叠，再退化为选最近对象计算 `HD`
- 必须明确采用对称 `Hausdorff`，不要把 `HD95` 和对象级 `Hausdorff` 混用

## 7. 辅助指标怎么使用

### 7.1 `Dice / IoU`

这两项主要用于：

- 辅助解释整体分割重叠趋势
- 和语义分割口径的方法对话

它们不能替代主表里的对象级三指标。

### 7.2 `HD95 / Boundary F1`

`HD95` 正式名称固定为 `MedPy-style discrete surface distance`，`pixel spacing = 1`。`Boundary F1` 是项目自定义口径：边界提取使用 `erosion_xor`，结构元为 `3x3 ones`，二维 `8-connectivity`，`border_value = 0`，匹配容差为 `3 px`。

### 7.3 空集 totalization（项目自定义）

对 Object F1、Object Dice、Object Hausdorff、Dice、IoU、HD95 与 Boundary F1 统一 totalize：双空时相似指标返回 `1`、距离指标返回 `0`；单侧空时相似指标返回 `0`、距离指标返回 `Dmax`，其中 `Dmax = sqrt((H-1)^2 + (W-1)^2)`。这是项目自定义有限值规则，不宣称为 GlaS、DCAN 或 MedPy 官方空集行为。

这两项主要用于：

- 边界分支是否真的改善轮廓质量
- `Distance-aware Loss` 是否真的对边界附近有帮助

特别是：

- 如果你后面要保留 `Boundary Head`
- 或者要保留 `Distance-aware Loss`

那至少应有一类边界证据支持。

## 8. 引用值、复现值、不可直接横比值怎么区分

### 8.1 `reproduced`

指的是：

- 你自己按当前统一协议跑出来的结果

### 8.2 `†`

指的是：

- 直接引用原文在相同 benchmark、相近 split、相同指标口径下的数字

### 8.3 `不可直接横比`

下面这些情况不能硬塞进主结果表做一行对一行公平比较：

- `Part A/B` 和 `A/B 平均` 混写
- 对象级三指标和纯语义 `Dice / IoU` 混写
- `zero-shot` 和同协议训练测试混写
- 自建数据集或 only benign 子集混进主表

### 8.4 表注规则

后面正式主表和补充表统一采用：

- `reproduced`：你自己统一协议复现值
- `†`：原文报告值
- `*`：原文自带后处理或特殊测试流程

与 `结直肠腺体分割_plan_优化版/01_实验执行/00_总览与规范/03_命名与结果记录规范.md` 对齐后的结果表字段固定为：

- `result_tag`
- `aggregation`

如果同一张表里同时出现 `single_seed`、`mean +- std`、`reproduced` 和 `†` 数字，必须显式保留这两个字段，或在表注中做完全等价说明。

## 9. 外部对比的评估公平规则

外部方法要进入主表，至少要满足：

- 同一 `GlaS` split
- 同一输入尺寸
- 同一阈值规则
- 同一指标集合
- 明确是否有后处理
- 明确是 `single_seed` 还是 `mean +- std`
- 同一 `result_tag / aggregation` 标记规则

特别注意：

- 第一批外部方法首轮允许先用 `seed3407` 做筛查
- 如果差距 `<= 1.5` 个百分点、或者它是第二强方法，必须补三次重复

## 10. 统计显著性怎么处理

### 10.1 当前主文最低要求

当前主文最低要求仍然是：

- `mean +- std`

### 10.2 什么时候要补显著性检验

如果后面出现下面这些情况，必须在 supplementary 或 rebuttal 里补：

- 模块间差异接近 `std` 范围
- 外部方法与最终方法差距很小
- 英文稿审稿人明确质疑差异是否稳定

### 10.3 当前正式做法

统一采用：

- 配对 `t-test`
- 或 `Wilcoxon signed-rank test`

但当前不要求一开始就把显著性检验写成主线门槛。

## 11. 当前最容易犯的评估错误

### 11.1 主表只报 `Dice`

这会让腺体任务退化成一般语义分割口径，削弱任务针对性。

### 11.2 把 `TestA / TestB` 混成一个值

这会掩盖困难样本上的边界与黏连问题。

### 11.3 只保存最好一次

这会让很多“稳定性”结论失去基础。

### 11.4 不区分原文值和复现值

这会直接引来公平性质疑。

### 11.5 没做官方口径交叉验证

这会让对象级三指标的可引用性存在根本风险。
