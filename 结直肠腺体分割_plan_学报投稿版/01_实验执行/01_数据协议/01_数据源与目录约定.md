## 实施与状态转移
- `existing_unverified` 只能进入检查队列，不能进入训练或验收；`planned_not_created` 表示协议已定义但资产尚未创建。
- source manifest 创建后先校验 root、相对路径和 sha256，再将 source 状态写入交接记录；任一字段缺失仍为 blocked。
- 资产状态只能按 `planned_not_created -> existing_unverified -> checked -> accepted` 递进；失败回到最早未满足节点，不允许直接手改为 accepted。
- 下游读取必须使用 manifest 中的 `image_relpath` 和 `mask_relpath`，禁止通过目录扫描补齐缺行。
- source schema 的最小样例为 `dataset,dataset_version,dataset_root,sample_id,image_relpath,mask_relpath,source_partition,source_subset,source_sha256,blind_state`。

## 字段级验收清单
- 核对 dataset_root 是否等于 active project root 下登记的来源根，禁止旧工程根和绝对路径漂移。
- 核对 image_relpath、mask_relpath 均能回指 sample_id，越界、空字段和 digest 缺失均为 blocked。
- 核对 source_partition、source_subset、blind_state 与 GlaS/CRAG 协议一致，不能用目录名推断身份。
- 核对 source manifest 的 protocol_version、dataset_version、created_at 和 sha256 可重算。
- 核对 current consumer 只读取 manifest 绑定的 CSV，不允许训练入口自行扫描 source root。
- 核对 planned_not_created 资产没有伪造时间、数量或 hash；existing_unverified 不得进入 handoff。
- 核对回退记录包含触发字段、旧 digest、新 digest、责任人和 next_action。
- 核对来源版本变更有 old_value/new_value、reason、evidence、approver 和 affected_config。
