# 00_交付范围内正式对象清单

## 当前定位

- 当前编号阶段: `01_数据协议`
- 当前 skill 工作流阶段: `说明文档`
- 这份清单的作用: 先把本轮真正会随项目一起交付的正式对象列清楚,再决定哪些对象必须补逐文件说明文

用人话说,这份清单不是流程日志,也不是 guard 留痕。
它就是这轮说明文写作的“对象边界表”。

## 先看结论

- 本轮纳入说明文映射的对象,只看正式项目文件,不看 `.trae/`、guard、gate、diagnostics
- 当前已经有逐文件说明文的对象,已经覆盖到 `scripts/train.py`、数据加载链、标签派生链、两份正式数据配置、全部正式 split CSV,以及核心 split/验收工具链
- 当前 A 类正式对象共 `24` 个,其中逐文件说明文已完成 `24` 个,当前没有 `pending`
- 阶段级文档当前共 `3` 份: `README.md`、`implementation_status.md`、`当前阶段为什么能pass以及下一步怎么看.md`
- 本轮先把最短主干补成 `配置 -> split CSV -> sample -> mask/target 派生 -> preflight`,再补正式 split 生成工具、总校验工具和正式验收结果说明文
- 当前这一轮已经把全部交付型 `tools/*.py` 补齐；A 类正式对象说明文覆盖为 24/24

## 为什么要先列这个清单

如果不先列清单,最容易犯的错有两个:

1. 把内部流程里碰过的文件,误写成“项目交付对象”
2. 只写入口页或阶段总说明,却默认“代码说明已经覆盖”

现在先把对象列出来,后面每补一份逐文件说明文,都能回到这里核对。

- 计划未点名、仅为生成某个计划产物而临时多造的工具脚本
- 其他只服务门禁和治理的内部脚本
