# Boundary 阶段总协议

本文件不是 `06_Boundary` 目录旧稿的补充说明，也不是“先把边界头试几版再回头整理”的阶段记录。

它在当前项目中的唯一职责是：

> 作为 `06_Boundary` 阶段的正式主协议，把 `Boundary Head as the second post-baseline enhancement / Single-variable explicit boundary supervision / Frozen boundary-target protocol / Keep-backup-drop decision with boundary evidence` 一次写死；代码、实验与阶段汇总只允许忠实实现和验证这份协议，不允许一边训练一边重新决定边界头接在哪里、边界标签怎么生成、边界宽度多大、损失权重多少、什么情况下才允许保留。

从现在开始，`06_Boundary` 子目录下的其他文件都视为本文件的细化与执行展开；若与本文件冲突，以本文件为准回改。

## 计划 lineage 与下游 handoff

统一字段：`source_stage`、`source_manifest`、`source_protocol_version`、`source_run_name`、`consumer_stage`、`consumer_file`、`consumption_boundary`。当前消费边界是 LKMA 保留版本到 Boundary head、边界标签和下游 Distance handoff；缺失字段或计划/config/run_name/run_meta/manifest/gate/handoff 不同步时 Gate blocked，回到治理/阶段锁定。

## 回退条件

边界标签宽度、评估实现、loss 权重或版本字段未同步时停止训练与汇总，回退到冻结协议并重新生成真实证据。

## 代码落地接口

边界目标落点为 `src/data/boundary_targets.py`，head/loss 分别落在 `src/models/heads/boundary_head.py`、`src/losses/boundary_losses.py`，评估与 handoff 必须读取同一 manifest。

## 冲突裁决记录

边界宽度与评估实现冲突时以当前 Gate 冻结字段和全局评估规范为准，历史结果只保留历史身份。

## 文件质量自检

检查七字段、边界目标与评估实现、配置/run_meta/manifest同步、Gate阻断条件、下游资产和回退顺序。

## Diagnostics 闭环

落盘后复核标题、字段、代码路径、公式引用和旧口径残留；问题未修复前不得放行。

## 1. 文件角色与执行边界

### 1.1 当前文件负责什么

当前文件只负责冻结下面六件事：

1. `Boundary Head` 作为 `LKMA` 之后第二模块的唯一身份、阶段目标和单变量边界。
2. `current_base -> current_base + Boundary Head` 的正式比较起点、搜索边界、阶段门与交接语义。
3. `Boundary Head v1` 的模块定义、接入位置、边界标签协议、边界宽度、边界损失与成本记录规则。
4. `Boundary` 阶段何时允许 `keep`、何时降级为 `backup`、何时必须 `drop` 并回退到不带边界头的当前基座。
5. `Boundary` 阶段最低交付物、正式资产、stage manifest 和对 `07_Distance` 的 handoff 语义。
6. `Boundary` 阶段的实现入口、冲突裁决、文件质量自检和独立 `Diagnostics 闭环`。

### 1.3 为什么当前文件必须独立存在

因为 `Boundary` 阶段最容易出现的伪完成状态不是“完全没跑”，而是：

- `LKMA` 还没有形成唯一合法 `current_base`，就提前启动边界头验证。
- 一边改边界标签、一边改宽度、一边改损失和权重，最后无法确认收益是不是来自显式边界监督本身。
- screening run 已经跑通，却没有形成三 seed 正式比较、成本记录和 `keep / backup / drop` 结论。
- `07_Distance` 想继承边界阶段结果时，说不清应当继承 `boundary_kept_base` 还是 `boundary_input_base`。

如果没有一份独立阶段总协议，后续 `07_Distance`、阶段汇总脚本和论文消融表都会再次出现“Boundary 已经做过，但到底按哪套变量边界、哪套资产链、哪一版结论继续”不一致的问题。

## 3. 本阶段唯一允许处理的变量

### 3.1 当前阶段真正要证明什么

`06_Boundary` 阶段真正要证明的，不是“加了边界辅助头后局部图看起来更清楚”，而是：

> 在 baseline 与 `LKMA` 的训练、评估和比较协议全部冻结的前提下，`Boundary Head` 作为第二层 post-baseline 模块增强，是否能够以最小单变量设置提供可复查、可解释、可交接的边界分离与轮廓恢复收益，并因此获得 `keep / backup / drop` 的正式阶段结论。

因此，本阶段通过的含义同时包括：

- `Boundary Head` 已经被压缩成真实的单分支、单位置、单主宽度、单主损失的可解释比较。
- `current_base vs current_base + Boundary Head` 的差异来自显式边界监督本身，而不是训练或评估 protocol 漂移。
- `Boundary` 的正式资产、比较表、边界指标表、成本表、案例和 decision note 已经齐全到可被 `07_Distance` 直接消费。

### 3.2 当前阶段不允许用什么替代正式结论

当前明确不允许把下面这些情况写成“Boundary 已完成”：

- 只跑了一个 screening run 就宣称边界头有效。
- 同时试多个宽度、多个权重和多个损失，却仍把结论写成单变量比较。
- 只看单次高值或像素级指标，不看 `Boundary F1`、对象级主指标、可视化和成本。
- 只写“边界监督可能有用”，不形成 `keep / backup / drop` 正式结论和下游起点。

### 3.3 为什么当前阶段必须拆成多层阶段门

`Boundary` 最容易发生的错误是把“代码能跑且边界图看起来还行”误写成“模块值得进入主线”。

因此，当前阶段必须把放行条件拆成：

- 上游基座门：负责确认当前比较建立在已经正式放行的 `current_base` 上。
- 运行完整门：负责确认 screening、三 seed 主版本和最低资产齐全。
- 比较就绪门：负责确认公平比较、边界级指标、对象级指标、可视化和成本证据全部到位。
- 决策门：负责确认 `keep / backup / drop` 结论已经形成且可被下游直接消费。
- 交接门：负责确认 `07_Distance` 已经能读取唯一合法 `current_base` 语义。

## 4. 设计边界(做什么/不做什么)

### 4.1 当前阶段直接继承哪些冻结项

本文件直接继承并不得擅自改动的上位规则如下：

- 主任务：`2D gland segmentation`
- 主数据集：`GlaS`
- 正式 split：`train68 / val17 / testA60 / testB20`
- 标签口径：`mask > 0`
- 输入通道：`RGB`
- 输入尺寸：`512 x 512`
- 归一化：`ImageNet mean/std`
- 增强版本：`light_aug_v1`
- 主损失：`L_seg = L_BCE + L_Dice`
- 优化协议：`AdamW + lr 1e-3 + weight_decay 1e-4 + ReduceLROnPlateau + epoch_max 120 + early_stopping 20 + AMP on`
- 主指标优先级：`F1 / Object Dice / Object Hausdorff`
- 补充指标：`Dice / IoU / HD95 / Boundary F1`
- `best_selector = val_objdice_max`
- `threshold_source = val17`
- `train_proto_version = train_proto_v1`
- `eval_proto_version` 由 04 阶段当前轮 Gate 冻结
- `eval_cast_policy = logits/probabilities must be kept or cast to float32 before thresholding`
- `boundary_metric_width = 3 px`
- `boundary_metric_impl = skimage.segmentation.find_boundaries(mode=inner) + binary_dilation`
- `connected_components_impl = scipy.ndimage.label`
- `connected_components_connectivity = 8`
- `postprocess_version`：当前轮待由研究定标、阶段锁定、Pre-check/Gate 冻结；`none_in_v3` 仅作历史 lineage，不是当前值。
- baseline 正式基座：`ResNet34-U-Net`
- baseline encoder 初始化：`official ImageNet-1K supervised weights`
- 固定三 seed：`3407 / 1234 / 2025`

### 4.2 当前阶段唯一合法起点

当前阶段的比较基座不是重新选择，而是由 `LKMA` 阶段正式结论唯一决定：

```text
current_base = (lkma_decision_level == keep) ? baseline_plus_lkma : baseline
```

这意味着：

- 只有 `05_LKMA` 已形成正式结论并完成 handoff，`Boundary` 才允许启动。
- `backup` 与 `drop` 都不能被默认当成“继续把 `LKMA` 带进主线”。
- 任何正式 `Boundary` run 都必须能追溯到 `LKMA` 的 decision note、stage manifest 和 `current_base` 说明。

### 4.3 当前阶段唯一变量

本阶段唯一允许处理的核心变量只有“在 `current_base` 上追加一个可插拔 `Boundary Head` 及其最小显式边界监督协议”：

- `Boundary Head v1` 的模块定义
- `Boundary Head` 的接入位置
- 边界标签来源与生成实现
- 边界宽度候选
- 边界损失形式
- `lambda_boundary` 候选
- `Boundary Head` 是否带来可接受的复杂度成本
- `Boundary Head` 是否值得进入后续主线

本阶段明确不允许借机改动：

- baseline backbone
- encoder 预训练策略
- `LKMA` 配置
- decoder 主干
- 数据协议
- 输入尺寸
- 主增强包
- 主损失
- optimizer / scheduler / lr
- `best checkpoint` 选择器
- 阈值来源
- `Distance-aware Loss`
- 后处理
- 外部对比与 `CRAG` 正式验证

### 4.4 当前阶段正式阶段门

当前文件固定采用下面的阶段门语义：

```text
Gate_D1 = current_base_ready
          AND boundary_main_runs_complete
          AND boundary_compare_ready
          AND boundary_assets_ready
          AND boundary_decision_ready
          AND handoff_ready
```

其中：

- `current_base_ready`：上游 `LKMA` 已给出唯一合法 `current_base`
- `boundary_main_runs_complete`：主版本 screening 和三 seed 正式结果完整
- `boundary_compare_ready`：公平比较、边界级指标、对象级指标、可视化与成本证据齐全
- `boundary_assets_ready`：per-seed 表、`mean+-std` 表、成本表、案例和 decision note 已导出
- `boundary_decision_ready`：`keep / backup / drop` 结论已形成
- `handoff_ready`：`07_Distance` 可直接读取下一阶段合法起点

这意味着：

- `Boundary` 阶段不是“先跑完再说”，而是必须在上游、运行、比较、决策和交接同时闭环后才允许进入 `07_Distance`。
- 即使主版本三 seed 已完成，只要 `keep / backup / drop` 结论或 handoff 资产还没形成，也不得视为阶段通过。

## 5. 如何判断设计有效(验收标准)

### 5.1 第二模块身份与作用定义

- 当前结论：后 `LKMA` 的第二模块固定为 `Boundary Head`，其角色固定为 `边界分离 + 黏连缓解 + 轮廓恢复`，而不是距离约束、后处理或完整新 backbone。
- 规则类型：`路线层已锁定 + 工程冻结规则`
- 适用阶段：`06_Boundary` 启动时生效，后续是否继承取决于本阶段结论。
- 直接依据：`结直肠腺体分割_plan_优化版/02_路线与投稿/02_结直肠腺体分割_分阶段实验路线与执行标准.md`、03_DCAN.md、01_Gated-SCNN.md、`结直肠腺体分割_plan_优化版/01_实验执行/06_Boundary/01_设计依据.md`
- 核心公式或定义参考：当前任务难点固定为 `touching glands / blurred contour / object separation`，模块职责不包括形态主建模或距离监督。
- 采用原因：腺体分割在整体形态表达较稳后，下一层最需要单独验证的是显式边界监督。
- 不采用的相邻方案：不把 `Boundary Head` 写成默认必保留模块；不把它写成完整 `Gated-SCNN` 双流主干；不把它和 `Distance-aware Loss` 混成同一阶段。
- 代码落点：[project_root]/src/models/heads/boundary_head.py、[project_root]/configs/model/boundary_head_v1.yaml
- 运行记录字段：`boundary_head_version`、`boundary_role`、`base_variant`
- 验收方式：检查阶段文档、配置和实现都明确 `Boundary Head` 只负责边界辅助监督，没有顺手引入距离项、marker 链或额外多级边界塔。

### 5.2 `Boundary Head v1` 模块定义

- 当前结论：`Boundary Head v1` 固定为 `single boundary prediction branch attached near final decoder output` 的可插拔辅助头，不采用完整 `Gated-SCNN / DCAN / DEA-Net` 原版重结构。
- 规则类型：`工程冻结规则`
- 适用阶段：`06_Boundary`
- 直接依据：`结直肠腺体分割_plan_优化版/01_实验执行/00_总览与规范/05_代码工程映射与实现策略.md`、03_DCAN.md、01_Gated-SCNN.md、12_DEA-Net.md
- 核心公式或定义参考：首版训练目标固定为 `L_total = L_seg + lambda_boundary * L_boundary`
- 采用原因：当前项目需要的是可解释、可插拔、易落地的首版边界辅助头，而不是在 `Boundary` 阶段直接复现边界文献的完整重型结构。
- 不采用的相邻方案：不采用完整 shape stream；不采用双编码器边缘增强主干；不采用 topology-aware 多解码器。
- 代码落点：[project_root]/src/models/heads/boundary_head.py
- 运行记录字段：`boundary_head_version`、`boundary_head_type`、`use_boundary_head`
- 验收方式：检查边界头主体仍是单分支辅助头，且主干结构未被改写成双流或双编码器。

### 5.3 接入位置规则

- 当前结论：首轮正式主位置固定为 `decoder_final`；最多只允许再补一个备选版本，且仍保持单位置接入。
- 规则类型：`论文支持的候选范围 + 工程冻结规则`
- 适用阶段：`06_Boundary`
- 直接依据：`结直肠腺体分割_plan_优化版/01_实验执行/06_Boundary/03_实验步骤.md`、`结直肠腺体分割_plan_优化版/02_路线与投稿/02_结直肠腺体分割_分阶段实验路线与执行标准.md`、03_DCAN.md
- 核心公式或定义参考：位置候选集合固定为 `P_boundary = {decoder_final}`
- 采用原因：当前最重要的是验证显式边界监督本身，而不是多尺度边界架构搜索。
- 不采用的相邻方案：不采用多 decoder stage 同时接边界头；不采用 encoder 侧额外边界流；不采用多尺度边界塔。
- 代码落点：[project_root]/src/models/r34_unet.py、[project_root]/configs/model/boundary_head_v1.yaml
- 运行记录字段：`boundary_insert_position`、`boundary_stage_name`
- 验收方式：检查主版本只在最后一级 decoder 输出侧接单边界分支；若出现备选版本，也不得变成多位置并行接入。

### 5.4 边界标签来源与生成流程

- 当前结论：边界标签唯一来源固定为数据阶段冻结的二值 gland mask；首轮正式流程固定为 `binary mask -> contour -> fixed-width dilation -> boundary band`。
- 规则类型：`工程冻结规则 + 标签定义规则`
- 适用阶段：`06_Boundary`
- 直接依据：`结直肠腺体分割_plan_优化版/01_实验执行/06_Boundary/02_边界标签生成规则.md`、03_DCAN.md、01_Gated-SCNN.md
- 核心公式或定义参考：`M(x) = 1[mask(x) > 0]`；`B_w = Dilate(Contour(M), w)`
- 采用原因：边界监督必须来自同一套可追溯标签定义，否则训练、可视化和分析会口径漂移。
- 不采用的相邻方案：不采用从距离图、marker 图或实例中心点单独定义边界标签；不采用多个脚本各自生成边界图。
- 代码落点：`crc_gland_segmentation_project/src/data/boundary_targets.py`、`crc_gland_segmentation_project/src/data/mask_ops.py`
- 运行记录字段：`boundary_target_version`、`boundary_target_source`、`contour_extract_mode`、`boundary_width`
- 验收方式：检查训练、可视化和分析共用同一套边界生成实现；检查边界标签全部来自冻结二值 mask。

### 5.5 边界宽度候选与首轮默认值

- 当前结论：边界宽度候选固定为 `3 / 5 px`，首轮主测固定为 `w = 3`。
- 规则类型：`论文支持的候选范围 + 工程冻结规则`
- 适用阶段：`06_Boundary`
- 直接依据：`结直肠腺体分割_plan_优化版/01_实验执行/00_总览与规范/02_参数冻结总表.md`、`结直肠腺体分割_plan_优化版/01_实验执行/06_Boundary/02_边界标签生成规则.md`、03_DCAN.md
- 核心公式或定义参考：候选集合固定为 `W = {3, 5}`；主锚点固定为 `w_main = 3`
- 采用原因：`3 px` 已足以体现边界带监督，同时对小目标和细边缘的侵入性更可控；`5 px` 只作为必要时的更宽候选。
- 不采用的相邻方案：不采用 `1 / 2 / 3 / 4 / 5 / 7` 全扫；不采用没有记录的宽度切换；不采用训练和分析使用不同宽度。

### 5.6 边界损失与权重规则

- 当前结论：首轮正式边界损失固定为 `BCE`；`lambda_boundary` 候选固定为 `0.2 / 0.3 / 0.5`，首轮主测固定为 `0.3`。

- 核心公式或定义参考：`L_total = L_seg + lambda_boundary * L_boundary`；其中首轮 `L_boundary = BCE`
- 采用原因：当前项目先验证“显式边界监督是否有价值”，因此首轮不应同时引入复杂边界损失为额外变量。
- 不采用的相邻方案：不采用一上来同时扫 `Boundary Loss / DoU / SDM / clDice`；不采用同时扫多种权重和宽度；不采用没有版本号就更换权重。

- 验收方式：检查主版本固定为 `BCE + lambda 0.3`；若补第二版本，也只能改变一个小范围变量。

### 5.7 搜索边界与实验顺序

- 当前结论：`Boundary` 阶段正式实验顺序固定为 `screening_main -> three_seed_main -> optional_one_backup_variant -> stage_decision`。
- 规则类型：`工程冻结规则`
- 适用阶段：`06_Boundary`

- 不采用的相邻方案：不采用一上来做宽度 x 权重 x 损失矩阵；不采用同时补多个备选版本；不采用主版本不过线还继续扩搜索。

### 5.9 复杂度与成本记录规则

- 当前结论：`Boundary` 正式 run 必须同步记录精度收益与复杂度成本，至少包括 `params_m`，若条件允许再补 `flops_g` 与 `latency_ms`。

### 5.10 keep-backup-drop 判定规则

- 当前结论：`Boundary Head` 是否保留，必须由 `Boundary F1`、`HD95 / Object Hausdorff`、`Object Dice / F1`、黏连样本可视化和成本共同决定，并分为 `keep / backup / drop` 三档。

- 直接依据：`结直肠腺体分割_plan_优化版/01_实验执行/06_Boundary/04_保留或删除标准.md`、03_DCAN.md、01_Gated-SCNN.md、12_DEA-Net.md
- 核心公式或定义参考：`decision = f(boundary_metric_trend, object_metric_trend, seed_consistency, visual_support, cost_change)`
- 采用原因：边界监督的核心价值在于边界质量和对象分离，而不是单一像素指标的小幅变化。
- 不采用的相邻方案：不采用只看 `Dice` 决定保留；不采用只看局部样例图；不采用因为理论上合理就默认保留。
- 代码落点：[project_root]/scripts/summarize_stage.py、[project_root]/scripts/compare_runs.py、[project_root]/reports/stage_reports/boundary_decision_note.md
- 运行记录字段：`decision_level`、`decision_reason`、`boundary_visual_support`、`cost_assessment`

## 7. 回退条件

### 7.1 独立回退触发条件

- `LKMA` 实际尚未形成唯一合法 `current_base`，却已经启动正式 `Boundary` 比较。

- `Boundary` 平均边界级主指标明显低于当前基座，且可视化不支持边界增益。
- `TestB` 明显更差，或者某个 seed 波动失控且没有排查记录。
- 成本明显上升但没有形成清晰收益证据，却仍试图给出 `keep` 结论。
- 主版本不过线，却想通过扩大搜索空间强行保留模块。
- `keep / backup / drop` 结论与 `07_Distance` 的 `current_base` 说明不一致。

### 7.2 固定回退顺序

回退时统一按下面顺序排查：

1. 检查边界标签来源、轮廓提取和宽度是否正确。

3. 检查边界损失和 `lambda_boundary` 是否失衡。
4. 检查当前基座训练与评估协议是否真正继承。
