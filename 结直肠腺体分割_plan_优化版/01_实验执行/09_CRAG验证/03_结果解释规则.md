# `CRAG` 结果解释规则

本文件不是旧版“结果说明页”的局部修补，也不是等 `CRAG` 跑完之后再补一段口头结论。

它在当前项目中的正式职责是：

> 作为 `09_CRAG验证` 目录下、从属于 `结直肠腺体分割_plan_优化版\01_实验执行\09_CRAG验证\00_阶段总协议.md` 且受 `结直肠腺体分割_plan_优化版\01_实验执行\09_CRAG验证\01_任务定位.md`、`结直肠腺体分割_plan_优化版\01_实验执行\09_CRAG验证\02_训练与验证协议.md` 约束的强版结果解释协议，把 `CRAG` 阶段哪些结果可以进入正式主表、哪些结果只能作为补充观察、哪些结论允许写成 `A / B / C` 哪一档、任务内 direct comparison 应该怎么写、`GlaS vs CRAG` 差异应如何收口，以及何时必须降级结论或回退重审，一次写死；后续阶段验收、结果汇总与总验收只允许消费这里已经冻结的解释口径，不允许边写结果边改标准。

## 1. 文件角色与执行边界

### 1.1 当前文件负责什么

当前文件只负责冻结下面九件事：

1. `CRAG` 结果在本项目中的唯一解释前提，也就是“补充验证而非第二主结论来源”的固定角色。
2. `CRAG` 结果进入正式表格与正式文字前必须先经过的来源身份核对、聚合方式核对和 direct comparison 资格核对。
3. `F1 -> Object Dice -> Object Hausdorff -> Dice / IoU / HD95 / Boundary F1 -> visuals` 的固定指标优先级。
4. `UNet -> ResNet34-U-Net -> current_mainline`、任务内 direct comparison、`GlaS vs CRAG` 三段式固定比较顺序。
5. `A / B / C` 三档结论强度的唯一判级标准与允许/禁止写法。
6. `CRAG` 结果何时算支持主线、何时只能算有限支持、何时必须降级成“不支持强结论”的收口规则。
7. `CRAG` 与 `MILD-Net-RTS / TA-Net / DEA-Net` 的 direct comparison 在不同来源边界下的书写口径。
8. 结果解释阶段对 `结直肠腺体分割_plan_优化版\01_实验执行\09_CRAG验证\04_阶段验收.md`、`10_结果汇总`、`11_总验收与止损` 提供的最小 handoff 结论资产。
9. `same_metric_protocol` 在当前阶段的唯一展开方式，也就是 `eval_cast_policy / boundary_metric_width / boundary_metric_impl / connected_components_impl / connected_components_connectivity` 这五个评估实现硬字段如何进入解释链、资格链和下游 handoff。
9. 与当前解释协议直接相关的独立 `回退条件`、`代码落地接口`、`冲突裁决记录`、`文件质量自检` 和独立 `Diagnostics 闭环`。

### 1.2 当前文件不负责什么

当前文件不重新定义下面这些内容：

- `CRAG` 的阶段角色、核心模型名单、任务内 direct comparison 锚点和允许/禁止修改边界；这些内容由 `01_任务定位.md` 先锁定。
- `CRAG` 具体怎么训练、怎么选 checkpoint、怎么补三次重复、怎么导出 run 资产；这些内容由 `结直肠腺体分割_plan_优化版\01_实验执行\09_CRAG验证\02_训练与验证协议.md` 负责。
- `09_CRAG验证` 是否整体通过、是否允许进入 `10_结果汇总` 的最终放行裁决；这些内容由 `结直肠腺体分割_plan_优化版\01_实验执行\09_CRAG验证\04_阶段验收.md` 负责。
- 更早阶段对 `UNet`、`ResNet34-U-Net`、`LKMA`、`Boundary`、`Distance` 的结构选择和主线取舍；这些内容已经在上游阶段冻结。
- 论文最终成文的整章组织；当前文件只负责把“什么能说、能说到什么强度”压成唯一解释口径。

### 1.3 为什么当前文件必须独立存在

`CRAG` 最常见的失真不是“没有结果”，而是“有结果但解释漂移”：

- 把 `CRAG` 的单点高值直接包装成跨 benchmark 泛化证据。
- 只看 `Dice` 或某个漂亮 seed，就跳过对象级主指标和稳定性判断。
- 把引文值、复现值、特殊测试设置结果混成一个“direct comparison 结论”。
- 把 `GlaS` 主结论和 `CRAG` 补充验证写成同一强度，导致整体论证失真。
- `04 / 10 / 11` 下游需要正式结论时，只拿到零散数字，拿不到固定等级、固定写法和回退条件。

因此，`结直肠腺体分割_plan_优化版\01_实验执行\09_CRAG验证\00_阶段总协议.md` 负责冻结整个 `I1` 阶段门，`结直肠腺体分割_plan_优化版\01_实验执行\09_CRAG验证\01_任务定位.md` 负责先锁定角色边界，`结直肠腺体分割_plan_优化版\01_实验执行\09_CRAG验证\02_训练与验证协议.md` 负责把执行链写死，而本文件必须单独把“如何解释 `CRAG` 结果”压成唯一协议。

## 3. 本阶段唯一允许处理的变量

### 3.1 当前文件的唯一解释前提

`CRAG` 结果解释只能从下面这个唯一前提出发：

```text
CRAG_role = same_task_supplementary_benchmark
```

它的固定含义是：

- `GlaS` 仍然是主结论来源，`CRAG` 只检查冻结主线收益有没有完全消失。
- `CRAG` 的强弱只影响“补充证据强度”，不改写 `GlaS` 主线已经建立的结论来源。
- `CRAG` 表现偏弱不自动等于“方法无效”，但也不能被包装成“强泛化已被第二 benchmark 证明”。

### 3.2 当前文件必须回答的最低问题

本文件必须先锁死下面这些解释问题，后续阶段验收、结果汇总和写稿才有统一口径：

1. 当前结果是否具备进入正式主表和正式比较的来源资格。
2. 当前结果应优先看哪些指标，哪些指标只能补充说明。
3. 当前结果要按什么顺序比较，才能避免把局部数字误写成整体结论。
4. 当前结果应落入 `A / B / C` 中哪一档，以及相应允许/禁止用词。
5. 当 `CRAG` 与 `GlaS` 或任务内强对照不一致时，允许怎样解释、何时必须降级或回退。

### 3.3 当前文件明确不回答的问题

下面这些问题不在本文件中给最终裁决，但必须以本文件为解释输入继续展开：

- 是否补跑某个 seed、是否重跑某个模型、是否补充某类资产；这些由 `结直肠腺体分割_plan_优化版\01_实验执行\09_CRAG验证\02_训练与验证协议.md` 和 `结直肠腺体分割_plan_优化版\01_实验执行\09_CRAG验证\04_阶段验收.md` 共同决定。
- `09_CRAG验证` 阶段是否整体通过、是否立即放行到 `10_结果汇总`。
- 论文章节最终如何组织跨阶段叙述；本文件只负责锁定 `CRAG` 段落能写到什么强度。

### 3.4 当前文件的最小解释链

本文件固定采用下面的最小解释链：

```text
comparison_order = own_three_level_relation
                   -> task_specific_direct_comparison
                   -> glas_vs_crag_gap
```

```text
metric_priority = F1
                  > Object Dice
                  > Object Hausdorff
                  > Dice / IoU / HD95 / Boundary F1
                  > visuals_and_failure_cases
```

```text
CRAG_grade in {A, B, C}
```

这意味着：

- 不允许跳过自家三层关系，直接拿任务内文献值包装结论。
- 不允许跳过对象级主指标，只拿像素级补充指标承担主结论。
- 不允许出现“既没判级、也没写回退边界”的模糊收口。

## 4. 阶段门控表达式

### 4.1 结果来源身份必须先核对

任何 `CRAG` 结果在进入正式结论前，都必须先补齐下面的来源身份公式：

```text
result_identity = result_source_type + aggregation + direct_comparison_eligible + reference_only
```

其中：

- `result_source_type` 用于区分 `reproduced / quoted / mixed_source_prohibited`。
- `aggregation` 用于区分 `single_seed`、`mean+-std`、`best_single_seed_not_for_main_claim`。
- `direct_comparison_eligible` 用于区分当前结果是否具备正式并排比较资格。
- `reference_only` 用于标记只能参考、不能下公平结论的结果。
- `same_metric_protocol` 在当前项目中不能只停留在抽象版本号，而必须继续下沉到五个评估实现硬字段是否一致。

如果来源身份没有写清，本文件禁止对该结果给出正式 direct comparison 结论。

### 4.2 direct comparison 资格必须独立判断

`CRAG` 上的直接比较资格只能按下面这个固定公式判断：

```text
direct_comparison_eligible = same_benchmark
                             AND same_split
                             AND same_metric_protocol
                             AND no_major_test_gap
```

它的固定含义是：

- 必须同属 `CRAG` 这个 benchmark。
- 必须没有明显 split 差异或测试集角色差异。
- 必须没有对象级 / 像素级口径错位、后处理差异未标注、评估实现字段漂移或特殊测试设置未隔离。
- 只要存在 `RTS`、单独后处理、不同官方脚本、不同指标定义、`eval_cast_policy / boundary_metric_width / boundary_metric_impl / connected_components_impl / connected_components_connectivity` 任一不一致，或明显 test gap，就只能降级成 `reference_only` 或显式注明“不构成完全公平 direct comparison”。

### 4.2.1 `same_metric_protocol` 的唯一展开方式

当前阶段禁止把 `same_metric_protocol` 写成“大家都属于 `eval_proto_v1`”这种抽象短语后就停止。它在 `CRAG` 结果解释中的唯一合法展开固定为：

```text
same_metric_protocol = same_eval_proto_version
                       AND same_eval_cast_policy
                       AND same_boundary_metric_width
                       AND same_boundary_metric_impl
                       AND same_connected_components_impl
                       AND same_connected_components_connectivity
```

其中当前项目冻结为：

```text
eval_cast_policy = logits/probabilities must be kept or cast to float32 before thresholding
boundary_metric_width = 3 px
boundary_metric_impl = project_custom_erosion_xor_3x3_ones_8conn_border0_tol3px
connected_components_impl = scipy.ndimage.label
connected_components_connectivity = 8
```

这条展开式在当前文件中的作用是：

- 防止 `CRAG` 结果解释阶段退回成“版本号一致就算一致”，却把真实评估实现口径留给下游猜。
- 确保 `结直肠腺体分割_plan_优化版\01_实验执行\09_CRAG验证\04_阶段验收.md`、`10_结果汇总`、`11_总验收与止损` 读取到的不只是 `eval_proto_version`，而是完整评估实现口径。
- 保证 `08_外部对比 -> 09_CRAG验证` 的统一评估实现字段在结果解释层不断链。

### 4.2.2 当前工程冻结 split 与文献值之间的解释边界

这里必须明确补一条，避免把“当前项目已经冻结 `CRAG` split”误写成“所有文献 `CRAG` 结果都能直接并排”：

- 当前项目正式解释入口固定消费 `train153 / val20 / test40` 这一版项目冻结 split。

split_provenance_type = project_frozen_from_local_layout

### 4.2.3 结果解释阶段必须消费的评估实现字段

从本文件开始，任何进入正式解释、正式主表注释、direct comparison 注释或下游 handoff 的 `CRAG` 结果，都必须同时显式消费：

- `eval_cast_policy`
- `boundary_metric_width`
- `boundary_metric_impl`
- `connected_components_impl`
- `connected_components_connectivity`

这意味着：

- 结果解释阶段不能只检查分数和来源边界，不检查评估实现字段。
- `metric_protocol_version` 必须能回溯到上述五个字段，而不是只留一个抽象代号。
- 若文献值无法确认这五个字段中的任一项，则该条结果默认不能升级成“完全公平 direct comparison”。

### 4.3 固定指标优先级

`CRAG` 结果解释时，优先级固定为：

1. `F1`
2. `Object Dice`
3. `Object Hausdorff`
4. `Dice / IoU / HD95 / Boundary F1`
5. 可视化与失败案例

这意味着：

- 主结论优先依赖对象级指标，不允许让 `Dice` 单独承担主结论。
- `Object Hausdorff` 方向与前两项不一致时，必须明确写成“对象级证据不完全同步”，不能强行写满。
- 可视化和失败案例只承担支持或反证职责，不能反过来覆盖主指标。

### 4.4 固定比较顺序

看 `CRAG` 结果时，必须按下面顺序比较：

1. 先比我们自己的三层关系：
   - `UNet -> ResNet34-U-Net`
   - `ResNet34-U-Net -> current_mainline`
2. 再看任务内 direct comparison：
   - `MILD-Net-RTS`
   - `TA-Net`
   - `DEA-Net`
3. 最后再看 `GlaS vs CRAG`：
   - 同一模型在两个 benchmark 上的差值
   - 差值主要落在哪类指标
   - 是整体下降还是局部波动

固定原因是：

- 先回答“主线内部收益是否还保留”，才能判断 `CRAG` 是否继续支撑自己的路线。
- 再回答“在任务内文献中大致处于什么层级”，才能补上外部参照系。
- 最后才讨论 `GlaS vs CRAG` 的差异来源，避免把 benchmark gap 误写成路线失效。

### 4.5 `GlaS vs CRAG` 差异的允许解释边界

如果 `CRAG` 表现弱于 `GlaS`，允许优先从下面几个角度解释：

- 数据分布差异。
- 腺体形态复杂度差异。
- 黏连、边界模糊或目标密集度差异。
- benchmark 难度、标注风格或对象级拆分难度差异。

不允许直接使用下面这类偷懒解释：

- “随机波动所以不用管”。
- “应该是代码问题”。
- “再调一调应该就会更好”。

没有证据时，只能写成观察，不能写成因果断言。

## 6. 判级标准、写法边界与最低 handoff 结论

### 6.1 `A` 档：强支撑

满足下面条件时，记为 `A`：

- `current_mainline` 相对 `ResNet34-U-Net` 至少一项对象级主指标稳定改善。
- 主要 seed 方向一致，且 `mean+-std` 不落入明显不稳区间。
- 可视化和失败案例与对象级改善方向一致，没有出现大面积反证。
- 与任务内代表性方法比较时，至少能写成“处于同一量级”或“在部分对象级指标上接近/达到强对照层级”，且来源身份合规。

允许写成：

- 主线收益在第二个同任务 benchmark 上得到进一步支持。
- `CRAG` 为冻结主线提供了较强的补充证据。

禁止写成：

- 已证明强泛化。
- 已对所有任务内方法形成全面优势。

### 6.2 `B` 档：有限支撑

满足下面条件时，记为 `B`：

- `current_mainline` 相对 baseline 仍有一定改善趋势，但幅度有限。
- 不同对象级指标改善不完全同步，或三次重复稳定性一般。
- 可视化能提供部分支持，但失败案例仍然明显存在。
- 与任务内强对照相比，仍有差距或只能达到局部接近。

允许写成：

- `CRAG` 对主线提供了有限但方向一致的补充证据。
- 主线收益在第二 benchmark 上仍可观察到，但稳定性和强度有限。

禁止写成：

- 已稳定验证强泛化。
- 已在 `CRAG` 上全面超越代表性任务内方法。

### 6.3 `C` 档：不构成强支撑

满足下面任一情况时，记为 `C`：

- `current_mainline` 相对 baseline 的核心对象级优势消失。
- 结果方向反转，或三次重复无法给出稳定、可解释的改善方向。
- 只有补充指标好看，而对象级主指标不支持。
- 与 `GlaS` 主结论相比，`CRAG` 上的 gap 已经大到无法作为稳定补充证据书写。

只能写成：

- 当前主线结论主要建立在 `GlaS` 上，`CRAG` 仅作为补充验证。
- `CRAG` 不支持更强的跨 benchmark 泛化表述。

禁止写成：

- 虽然 `CRAG` 不稳，但整体仍证明了泛化。
- 只因个别 seed 漂亮就把结论升级成强支撑。

### 6.4 哪些情况算支持主线

下面这些情形可以视为 `CRAG` 对主线有正向支撑：

- `current_mainline` 相对 `ResNet34-U-Net` 的 `F1` 更高。
- `current_mainline` 相对 `ResNet34-U-Net` 的 `Object Dice` 更高。
- `current_mainline` 相对 `ResNet34-U-Net` 的 `Object Hausdorff` 更低。
- 指标提升与可视化中的黏连分离、边界完整性改善方向一致。
- 即使优势变小，但方向没有反转，且失败模式仍可解释。

### 6.5 哪些情况只能算有限支持

下面这些情况只能写成有限支持，不能写得过满：

- 优势只体现在补充指标，主对象级指标没有明显改善。
- `seed3407` 好看，但 `mean+-std` 接近波动范围。
- 可视化只在少数漂亮样本上成立，失败案例没有同步改善。
- 任务内对照里只能达到“接近某些指标”而非整体同级。

### 6.6 哪些情况必须判为不支持强结论

下面这些情况不允许写强结论：

- `current_mainline` 相对 baseline 的优势完全消失。
- `GlaS` 上优于 baseline，但 `CRAG` 上稳定差于 baseline。
- 结果相对 baseline 大幅下降，且失败模式无法解释。
- direct comparison 本身不具备公平资格，却仍被写成正式优势结论。

### 6.7 任务内 direct comparison 的固定写法

`CRAG` 主表解释时，只允许按下面口径表述：

- 若结果接近或超过任务内强对照，可写“在统一协议下达到与代表性任务内方法同一量级的对象级表现”。
- 若结果低于任务内最强方法，可写“相对 baseline 保持改进，但与任务内最强专用方法仍有差距”。
- 若存在特殊设置、来源身份差异或比较资格不足，必须加上“仅供参考”“不构成完全公平 direct comparison”之类的降级说明。
- 若文献值与当前项目固定的 `train153 / val20 / test40` 不同，或存在 `RTS / 后处理 / 评估协议差异`，必须进一步显式写出 `split_protocol_gap_note` 或 `paper_setting_gap`，不能只用一句“参考比较”带过。

下面这些写法全部禁止：

- 把引用值和复现值混成同一结论。
- 把带后处理的方法和无后处理主线结果写成完全公平。
- 把不同 split、不同指标口径或特殊测试设置结果硬塞进同一公平主表。
