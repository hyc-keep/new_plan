# 从头实现论文验证所需高质量参考代码与资料总表

## 1. 文档目的

这份文档不回答“我们的实验怎么设计”，而是回答另一个已经进入执行前的关键问题：

> 如果现在要按照 `01_实验执行` 这套冻结协议，从头把整篇论文验证代码真正写出来，到底需要准备哪些高质量参考代码、官方资料、核心库和实现锚点，才足以支撑一套科学、统一、公平、可复现的工程实现。

本文件的定位是：

- 给后续正式代码工程提供一份高质量参考资料总清单；
- 把“必须参考什么”和“绝对不要乱拼什么”一次写清；
- 把每一类参考资料分别对应到数据、训练、评估、模块、外部对比、结果汇总和总验收这些具体阶段；
- 防止后续因为“看到什么 repo 都想拿来拼”而破坏统一协议。

本文件不是：

- 低质量代码仓库收集目录；
- 随手记录若干 GitHub 链接的备忘录；
- 对 `01_实验执行` 正式协议的替代。


## 2. 最终结论

当前项目如果要从头写出一套科学实现，参考资料不应过少，也不应过多。

最合理的做法不是：

- 只靠 `3-4` 个参考代码仓库硬写全部工程；
- 也不是收集 `20+` 个 repo 然后拼接成一个混乱工程。

更合理的结论是：

- 需要一套覆盖整条实现链的“能力位参考体系”；
- 这些能力位大约落在 `4` 层、`12-18` 个高质量参考点；
- 其中真正必须重点准备的核心参考，大约是 `8-12` 项；
- 剩余部分只作为模块借鉴、交叉核对或后期补强资料，不应直接进入主工程。

一句话概括：

> 我们需要的不是“尽可能多的代码”，而是“足以支撑统一主框架、统一评估口径和统一公平比较的高质量参考体系”。


## 3. 使用原则

### 3.1 总原则

后续实现统一遵守下面四条：

1. 主线训练框架自己统一写。
2. 主线方法模块自己写。
3. 外部 baseline 优先接高质量实现，但必须做统一协议适配。
4. 任何第三方参考都只能服务当前协议，不能反向改写当前协议。

### 3.2 参考资料的正确用途

这些参考资料存在的作用，不是为了“照搬整个工程”，而是为了分别解决下面四类问题：

- 官方定义与标准口径从哪里来；
- 某个模块的核心实现思路从哪里借鉴；
- 某类成熟架构如何高质量接入，而不必从零重造；
- 哪些基础库最稳定、最适合做正式实现。

### 3.3 明确禁止的错误用法

后续不得：

- 从多个完整训练仓库里直接拼数据加载、训练循环、评估脚本和日志系统；
- 为了让某个外部方法更好跑，单独修改数据协议、增强协议、阈值来源或后处理；
- 把随机看到的低质量 GlaS/CRAG 仓库当作正式实现锚点；
- 把任务内 direct comparison 的论文代码、主线训练框架和外部 baseline 适配代码混成同一套训练世界观。


## 4. 质量分级标准

后续所有参考资料按下面优先级选择。

### 4.1 第一优先级：官方资料

包括：

- 官方文档；
- 官方评估脚本；
- 官方或作者仓库；
- 官方数据集说明；
- 官方 challenge 口径。

这是最重要的一层，因为它直接决定：

- benchmark 定义；
- 指标口径；
- 数据角色；
- 结果是否可引用。

### 4.2 第二优先级：高质量通用库

包括：

- `PyTorch`
- `torchvision`
- `segmentation_models_pytorch`
- `albumentations`
- `scipy`
- `scikit-image`
- `medpy`
- `pandas`

这层的价值在于：

- 实现稳定；
- 社区成熟；
- 文档完善；
- 便于后续长期维护。

### 4.3 第三优先级：模块级高质量实现

包括：

- `RepLKNet`
- `VAN`
- `Gated-SCNN`
- `DEA-Net`
- `DCAN` 相关可靠实现

这层只用于：

- 学习模块设计；
- 学习结构拆分方式；
- 学习 target 与 loss 的组织方式；
- 学习接口边界。

这层不应用于：

- 直接搬运整套训练框架；
- 替代我们的统一工程主干。

### 4.4 第四优先级：后期补强资料

包括：

- `nnU-Net`
- `TransUNet`
- `Swin-Unet`
- `MILD-Net-RTS`
- `TA-Net`
- `Distance Map Loss` 或 `SDM` 相关代码

这层主要服务：

- 增强外部对比；
- 英文稿防守；
- 写作期的 task-specific direct comparison 补强；
- 个别模块的语义交叉核对。

## 6. 最终锁定的参考资料清单

下面是建议正式准备的参考清单。每一项都写清：

- 为什么需要；
- 服务哪个阶段；
- 应该借鉴什么；
- 不应该借鉴什么。

### 6.1 A层：必须具备的官方资料与核心库

#### A1. `PyTorch` 官方文档

- 角色：全项目正式深度学习底座。
- 服务阶段：`01-11`。
- 主要借鉴：
  - 模型定义；
  - 训练循环；
  - `AMP`；
  - checkpoint；
  - deterministic / seed 控制；
  - `DataLoader` 和 tensor 计算语义。
- 不借鉴的东西：
  - 不依赖第三方训练框架替代自己的主训练入口。
- 必要性：最高。

#### A2. `torchvision` 官方模型与权重文档

- 角色：`ResNet34` encoder 的高质量官方来源。
- 服务阶段：`04_Baseline` 及其后续全部主线阶段。
- 主要借鉴：
  - `resnet34` 结构；
  - 官方 ImageNet 预训练权重加载方式；
  - encoder stage 划分方式。
- 不借鉴的东西：
  - 不把 `torchvision` 分类头直接带入分割工程。
- 必要性：最高。

#### A3. `segmentation_models_pytorch`

- 角色：通用分割架构与 encoder-decoder 组织的高质量参考。
- 服务阶段：
  - `02_UNet流程验证`
  - `04_Baseline`
  - `08_外部对比`
- 主要借鉴：
  - `Unet`
  - `UnetPlusPlus`
  - `DeepLabV3Plus`
  - encoder registry
  - decoder 连接方式
  - segmentation head 组织方式
- 不借鉴的东西：
  - 不直接沿用它自己的所有默认训练策略和结果口径；
  - 不把它当作统一评估与结果记录框架。
- 必要性：最高。
- 备注：
  - `UNet++` 和 `DeepLabV3+` 基本可以优先用它做适配；
  - `Attention U-Net` 不建议假设它一定能由 `smp` 完整覆盖，最好准备单独参考实现。

#### A4. `albumentations`

- 角色：正式数据增强库。
- 服务阶段：`02-09`。
- 主要借鉴：
  - 翻转、旋转、尺度扰动、颜色扰动等统一增强实现；
  - mask 同步变换逻辑。
- 不借鉴的东西：
  - 不引入超重增强搜索；
  - 不用它来替代数据协议本身。
- 必要性：最高。

#### A5. `SciPy`

- 角色：评估、连通域、距离图和若干几何计算的核心底座。
- 服务阶段：
  - `01_数据协议`
  - `06_Boundary`
  - `07_Distance`
  - `02-11` 评估层
- 主要借鉴：
  - `scipy.ndimage.label`
  - `scipy.ndimage.distance_transform_edt`
  - `scipy.spatial.distance` 相关函数
- 不借鉴的东西：
  - 不随意切换到别的 connected-components 库破坏口径。
- 必要性：最高。

#### A6. `scikit-image`

- 角色：边界提取、形态学处理、连通域分析和后期形态学统计的核心工具库。
- 服务阶段：
  - `01_数据协议`
  - `06_Boundary`
  - `10_结果汇总`
  - 写稿后期可选形态学补强
- 主要借鉴：
  - `find_boundaries`
  - `regionprops`
  - 基础 morphology 工具
- 不借鉴的东西：
  - 不把临时形态学后处理偷混进正式 `eval_proto_v1`。
- 必要性：最高。

#### A7. `MedPy`

- 角色：`HD95` 等补充距离指标的稳定实现来源。
- 服务阶段：`02-11`。
- 主要借鉴：
  - `medpy.metric.binary.hd95`
- 不借鉴的东西：
  - 不让 `HD95` 取代对象级 `Object Hausdorff` 的主指标地位。
- 必要性：高。
- 轻量备选：`surface-distance`（Google DeepMind 出品，`pip install surface-distance`）
  - 提供 `compute_surface_distances`、`compute_robust_hausdorff` 等函数，可直接计算 HD95。
  - 若 `MedPy` 安装出现依赖冲突（ITK 等），可切换至此包，计算口径与 MedPy 一致。

#### A8. `pandas` + `matplotlib`

- 角色：结果表、stage summary、图像导出与结果包整理的基础库。
- 服务阶段：`10-11`。
- 主要借鉴：
  - 表格拼接；
  - `csv` 读写；
  - 主表/消融表构建；
  - 曲线与对比图导出。
- 不借鉴的东西：
  - 不把 notebook 临时输出替代正式 `reports/*` 资产。
- 必要性：高。


### 6.2 B层：必须参考的官方定义与评估锚点

#### B1. `GlaS` Challenge 官方评估定义 / 官方脚本

- 角色：对象级主指标的最高优先口径来源。
- 服务阶段：`02-11`。
- 主要借鉴：
  - `F1`
  - `Object Dice`
  - `Object Hausdorff`
  - `TestA / TestB` 分开报告的 benchmark 语义
- 不借鉴的东西：
  - 不完全照搬旧 challenge 的全部训练 recipe；
  - 不把“官方挑战数据角色”误写成“当前项目训练协议”。
- 必要性：最高。
- 说明：
  - 即使后面我们自己实现对象级评估，也必须拿它做 cross-check。
- Python 实现备选说明：
  - 如果官方 MATLAB 脚本不便直接使用，可按以下逻辑自行实现 Python 版本：
    - 连通域标记：`scipy.ndimage.label`
    - 对象匹配：预测与 GT 之间按 IoU > 0.5 做匈牙利匹配（或贪心最大 IoU 匹配）
    - `F1`：基于匹配对数、FP 数和 FN 数计算 `Precision / Recall / F1`
    - `Object Dice`：对每个匹配对计算单对象 Dice，再取所有匹配对均值
    - `Object Hausdorff`：对每个匹配对计算边界点之间的有向 Hausdorff 距离，再取所有匹配对均值
  - 自行实现后，应在至少一批已知结果上与文献报告值做 cross-check，确认计算口径无偏差。

#### B2. `CRAG` 数据与 benchmark 说明

- 角色：第二 benchmark 数据角色与 split 身份依据。
- 服务阶段：`01_数据协议`、`09_CRAG验证`、`10-11`。
- 主要借鉴：
  - `train153 / val20 / test40` 的本地组织；
  - 第二 benchmark 的使用边界。
- 不借鉴的东西：
  - 不把 `CRAG` 当成独立重新调参的第二主战场。
- 必要性：高。
- 备注：
  - `CRAG` 在公开网络上常见的是二次整理版、增强版或格式转换版仓库；
  - 正式数据身份应优先锚定 `MILD-Net` 论文中的 benchmark 说明与原始数据描述，而不是某个 augmentation / COCO 转换仓库。

#### B3. `GlaS-Challenge / MILD-Net / TA-Net / DEA-Net` 的结果表与协议说明

- 角色：task-specific direct comparison 的主要写作与表格边界来源。
- 服务阶段：`08_外部对比`、`09_CRAG验证`、`10_结果汇总`、写稿期。
- 主要借鉴：
  - benchmark 身份；
  - 指标类型；
  - 任务内对照对象；
  - 是否能进入主表直比；
  - 哪些数字只能作引文对照。
- 不借鉴的东西：
  - 不把这些论文各自的训练世界观整体搬进主框架。
- 必要性：高。


### 6.3 C层：主线结构与模块实现参考

#### C1. 标准 `U-Net` 结构资料

- 角色：流程排雷和最小可复现基线的结构来源。
- 服务阶段：`02_UNet流程验证`、`03_UNet稳定性`。
- 主要借鉴：
  - encoder-decoder 对称结构；
  - skip connection；
  - 单通道 segmentation head。
- 推荐用法：
  - 可参考 `smp` 的 `Unet`；
  - 或按协议自写简洁版。
- 必要性：高。

#### C2. `ResNet34-U-Net` 参考

- 角色：全文唯一正式 baseline。
- 服务阶段：`04-09`。
- 主要借鉴：
  - `torchvision` 的 `resnet34` encoder；
  - `smp` 中 encoder-decoder glue 方式；
  - decoder 与 skip 对齐方式。
- 不借鉴的东西：
  - 不从低质量 blog repo 中抄一版不透明 `ResUnet`。
- 必要性：最高。

#### C3. `RepLKNet`

- 角色：`LKMA` 的大核卷积分解与高效实现锚点之一。
- 服务阶段：`05_LKMA`。
- 主要借鉴：
  - 大核 depthwise conv 的实现；
  - 大核结构如何保持工程可用；
  - kernel 大小与复杂度记录思路。
- 不借鉴的东西：
  - 不把完整大模型 backbone 替换进我们的主线；
  - 不把整套训练 recipe 带进来。
- 必要性：高。

#### C4. `VAN`

- 角色：`LKMA` 的 large-kernel attention / depthwise decomposition 另一条高质量参考。
- 服务阶段：`05_LKMA`。
- 主要借鉴：
  - 大核 attention block 的轻量组织；
  - `DW-Conv + dilation / projection` 类写法；
  - 模块如何插入中间特征层。
- 不借鉴的东西：
  - 不照搬整个 classification 主干。
- 必要性：高。

#### C5. `DCAN`

- 角色：腺体任务内 contour / boundary 分支的直接任务化锚点。
- 服务阶段：`06_Boundary`。
- 主要借鉴：
  - contour head 的任务意义；
  - 主分割与边界辅助监督的关系；
  - 对腺体黏连分离的写法与设计动机。
- 不借鉴的东西：
  - 不强行复刻其完整旧训练设定。
- 必要性：高。
- 备注：
  - 即使没有单一完美官方 repo，也必须把论文结构与高质量复现解释读透。

#### C6. `Gated-SCNN`

- 角色：边界监督和 boundary stream 设计的重要结构参考。
- 服务阶段：`06_Boundary`。
- 主要借鉴：
  - 边界 target 组织方式；
  - boundary branch 与主干的职责拆分；
  - 边界相关 loss 的写法边界。
- 不借鉴的东西：
  - 不引入完整双流大型工程，只借鉴边界监督思想和局部接口组织。
- 必要性：高。

#### C7. `DEA-Net`

- 角色：腺体任务近期 boundary-enhanced 直接竞品参考。
- 服务阶段：
  - `06_Boundary`
  - `08_外部对比`
  - `09_CRAG验证`
  - 写稿期
- 主要借鉴：
  - 边缘增强模块的任务化表达；
  - 边界增强如何写入 gland-specific 叙事；
  - 近期任务内强对照的比较边界。
- 不借鉴的东西：
  - 不照搬整个方法作为我们的主线实现。
- 必要性：高。
- 风险提醒：
  - `DEA-Net` 存在明显的同名歧义，公开搜索里很容易首先命中去雾等完全不同任务的官方仓库；
  - 因此是否纳入 task-specific direct comparison，必须先以“论文任务是否为 gland segmentation、benchmark 是否覆盖 `GlaS/CRAG`、指标口径是否可对齐”为准，不能只按仓库名认领。

- 主要借鉴：
  - 前景距离图定义；
  - 距离图归一化；
  - 附加损失与主损失的组合方式；
  - 首版为什么只做 `EDT`，不直接切到 `SDM`。
- 不借鉴的东西：
  - 不把 `signed distance map`、skeleton 或 topology loss 混进 `Distance v1`。
- 必要性：中高。
- 备注：
  - 这一阶段真正的实现底座仍然是 `scipy.ndimage.distance_transform_edt`；

### 6.6 可直接去找的正式来源候选

下面这一节不是“把看到的 repo 都贴进来”，而是把前面能力位进一步落到“正式该先找谁”的层面。

使用规则固定为：

- 同一能力位优先找“官方文档 / 官方仓库 / 作者仓库”；
- 如果没有官方可直接复用实现，再退到“高质量社区实现”；
- 一旦存在任务不匹配、许可不合适、训练世界观过重、同名歧义明显等问题，必须在这里先标清楚，再决定是否收。

| 能力位 | 推荐来源候选 | 优先级 | 应该借鉴什么 | 明确边界 |
|---|---|---|---|---|
| 主训练底座 | `PyTorch` 官方文档 | `P0` | `Module`、`AMP`、checkpoint、`DataLoader`、seed 控制 | 不引入额外训练框架替代主入口 |
| `ResNet34-U-Net` / `UNet++` / `DeepLabV3+` | `segmentation_models_pytorch` 官方仓库 + 文档 | `P0` | 架构定义、encoder registry、decoder 组织 | 不沿用其默认训练 recipe 和结果管理 |
| `ResNet34` encoder | `torchvision` 官方文档 | `P0` | 预训练权重、stage 划分、backbone 结构 | 不带分类头 |
| 数据增强 | `albumentations` 官方文档 | `P0` | 图像与 mask 同步变换 | 不扩成重增强搜索 |
| 对象级官方口径 | `GlaS Challenge` 论文 / 官方评估定义 | `P0` | `F1`、`Object Dice`、`Object Hausdorff`、`TestA/TestB` 语义 | 不把 challenge 训练设定混成当前训练协议 |

| `LKMA` attention 分解参考 | `VAN-Classification` 作者官方仓库 | `P1` | `LKA` 风格块、`DW-Conv + dilation + projection` 写法 | 不照搬整套 classification 工程 |
| `Boundary Head` 结构参考 | `nv-tlabs/GSCNN` 官方仓库 + 项目页 | `P1` | shape/boundary stream、边界监督接口、分支职责划分 | 许可为 `CC BY-NC-SA 4.0`，只能作研究参考，不宜直接搬代码进正式可分发工程 |
| `Boundary Head` 任务内语义参考 | `DCAN` 论文本身 + 可读复现实现 | `P1` | contour target、主分割与边界分支关系 | 官方代码锚点弱时，以论文定义优先，不把任意复现当官方 |
| `Attention U-Net` 结构参考 | 原论文 + `MONAI` `AttentionUnet` + 轻量 `2D PyTorch` 实现 | `P1` | attention gate 接口、skip gating、最小结构实现 | `MONAI` 主要是医学成像官方实现锚点，但要注意其默认是通用医学框架，不直接继承整套工作流 |
| 工程风格参考 | `MONAI` 或 `nnU-Net` 只选其一看工程思想 | `P2` | 目录组织、配置层次、可维护性思路 | 不复制其数据协议和训练世界观 |
| 距离图监督 | `SciPy EDT` + `Distance Map Loss / SDM` 论文资料 | `P1` | 距离图定义、归一化、loss 拼接 | 不因看到 `SDM` 仓库就把 `Distance v1` 改成 signed distance 路线 |
| task-specific direct comparison | `MILD-Net`、`TA-Net`、任务内 `DEA-Net` 等原文与结果表 | `P1` | direct comparison 身份、表格口径、benchmark 适配边界 | 先核对任务是否真为 gland segmentation，防止 `DEA-Net` 同名误收 |

补充说明：

- `Attention U-Net` 这一项，当前最稳妥的做法不是只找一个民间仓库，而是采用“三件套”：
  - 原论文负责语义定义；
  - `MONAI` 负责一个维护更稳定的官方医学实现锚点；
  - 轻量 `2D PyTorch` 实现负责帮助快速还原最小结构。
- `GSCNN` 的价值非常高，但它更适合作为 `Boundary Head` 的结构和监督参考，而不是直接并入我们的主工程。
- `DCAN` 的权威性主要来自论文本身，因为公开实现分散且任务数据域不完全统一，所以收集时要把“论文定义”放在“仓库实现”前面。
- `CRAG` 的网络镜像很多，真正该锁的是 benchmark 身份，不是某个整理者的目录格式。
