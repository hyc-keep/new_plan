# 错误排查

这份文件不是“出问题时顺手翻一下”的经验清单，也不是把若干常见 bug 按现象堆在一起。

它在当前阶段的唯一职责是：

> 把 `UNet` 首轮流程验证阶段的异常定位、证据采集、修复边界、回归验证和关闭标准，压成一条可执行、可留痕、可回退、可交接的正式排查协议，并确保排查动作不会反向破坏已经冻结的训练、验证和测试协议。

因此，本文件必须同时写清：

- 什么时候必须暂停当前正式 run。
- 异常统一按什么顺序排查，而不是按直觉乱试。
- 每一层至少要收集哪些证据、留下哪些字段。
- 什么属于普通异常，什么属于协议级红线。
- 修复后怎样判定“问题关闭”，以及怎样把关闭结论交给阶段验收和后续稳定性阶段。

---

## 1. 文件角色与执行边界

### 1.1 当前文件负责什么

当前文件负责冻结并展开下面这些排查层动作：

- `UNet` 首轮正式 run 何时必须暂停、阻断、回退，而不是继续把异常带到测试和验收阶段。
- 异常统一按“数据读取与配对 -> 标签转换与 resize -> loss、threshold_value 与评估口径 -> 训练循环与 checkpoint 保存 -> 模型实现”顺序排查。
- 每一层最小证据包、最小记录字段、最小回归验证方式和问题关闭条件。
- 正式 debug 记录、问题状态流转、异常 run 处置边界和协议级红线。
- 向 `结直肠腺体分割_plan_优化版\01_实验执行\02_UNet流程验证\05_阶段验收.md`、`结直肠腺体分割_plan_优化版/01_实验执行/03_UNet稳定性/00_阶段总协议.md` 交接时必须存在的异常说明与关闭留痕。

### 1.2 当前文件不负责什么

当前文件不负责重新打开下面这些问题：

- 重新定义任务边界、split 角色、标签口径、输入尺寸和默认增强。
- 重新定义 `UNet` 结构、loss、optimizer、scheduler、`best_selector` 或 `threshold_source`。
- 重新定义 `TestA / TestB` 的正式评估流程、指标集合和阶段通过线。
- 借排查名义提前引入 `ResNet34-U-Net`、`LKMA`、`Boundary Head`、`Distance-aware Loss`、`CRAG` 或外部对比。

对应职责分别由下列文件承担：

- `结直肠腺体分割_plan_优化版/01_实验执行/02_UNet流程验证/00_阶段总协议.md`
- `结直肠腺体分割_plan_优化版/01_实验执行/02_UNet流程验证/01_任务与默认配置.md`
- `结直肠腺体分割_plan_优化版/01_实验执行/02_UNet流程验证/02_训练步骤.md`
- `结直肠腺体分割_plan_优化版/01_实验执行/02_UNet流程验证/03_验证测试与可视化.md`
- `结直肠腺体分割_plan_优化版/01_实验执行/02_UNet流程验证/05_阶段验收.md`

### 1.3 当前文件真正要解决什么

这一阶段真正要解决的不是“多列几种 bug”，而是下面四件事已经有了唯一正式入口：

1. 当 `UNet` 首轮出现异常时，先查哪一层、后查哪一层已经固定，不再靠主观猜测。
2. 每次修复都能明确定位到单一主层级，而不是同时混改多个协议变量。
3. 修复动作和回归验证都有结构化留痕，后续稳定性阶段能够判断某个异常是否已经真正关闭。
4. 协议级错误会被及时阻断，而不是带着错误 checkpoint、错误阈值或错误评估口径继续推进。

## 3. 本阶段唯一允许处理的变量

### 3.1 当前阶段唯一有效的排查边界

排查时必须默认下面这些内容已经冻结，不能一出问题就重开：

- 数据 split 固定为 `train68 / val17 / testA60 / testB20`
- 标签定义固定为 `y(x) = 1[M_inst(x) > 0]`
- 输入尺寸固定为 `512 x 512`
- 输入通道固定为 `RGB`
- 归一化固定为 `ImageNet mean/std`
- 训练主损失固定为 `L_seg = L_BCE + L_Dice`
- `best_selector` 固定为 `val_objdice_max`
- `threshold_source` 固定为 `val17`
- `threshold_value` 首轮起点固定为 `0.5`
- 正式测试不允许用 `TestA / TestB` 回调阈值或 checkpoint

### 3.2 当前阶段明确不做的事情

当前排查文件不允许借“debug”之名提前打开：

- `ResNet34-U-Net`
- `LKMA`
- `Boundary Head`
- `Distance-aware Loss`
- stain normalization
- postprocess 搜索
- TTA
- 测试集反向调参
- 以更强模型替代当前 `UNet` 流程修复

这样规定的原因是：

- `A1` 排查协议的目标是确认基础流程是否可信，而不是通过更强结构掩盖流程问题。
- 如果当前阶段允许同时改协议和结构，后续就无法判断 bug 是真的修好，还是只是被新变量碰巧盖住。

## 4. 阶段门控表达式

### 4.1 异常暂停与阻断规则

- 当前结论：一旦出现协议级异常、关键资产缺失或结果与证据长期严重矛盾，当前正式 run 必须先暂停并进入正式排查，不得继续测试、汇总或阶段验收。
- 当前值或最终版本：`debug_gate_v1`
- 规则类型：`工程冻结规则`
- 适用阶段：`A1` 首轮正式训练、验证、测试与可视化全过程
- 直接依据：`结直肠腺体分割_plan_优化版/01_实验执行/02_UNet流程验证/00_阶段总协议.md`，`结直肠腺体分割_plan_优化版/01_实验执行/02_UNet流程验证/05_阶段验收.md`
- 核心公式或定义参考：

```text
debug_required =
  protocol_error
  or missing_core_asset
  or severe_metrics_visual_conflict
  or abnormal_run_status
```

- 采用原因：排查文件首先要解决“什么时候必须停”，否则任何排查顺序都可能被继续跑实验的惯性打断。
- 不采用的相邻方案：不采用“先把测试跑完再说”；不采用“只要能出一个数字就先留着”。
- 代码落点：`crc_gland_segmentation_project/scripts/train.py`，scripts/test.py，notes/debug_note.md，run_meta.yaml
- 运行记录字段：`debug_required`，`protocol_error`，`missing_core_asset`，`run_status`
- 验收方式：检查触发条件出现后是否真的阻断后续动作；检查 run_meta.yaml 和 debug 记录有对应标记。

### 4.2 固定排查主链规则

- 当前结论：`UNet` 首轮异常统一按“数据读取与配对 -> 标签转换与 resize -> loss、threshold_value 与评估口径 -> 训练循环与 checkpoint 保存 -> 模型实现”顺序排查。
- 当前值或最终版本：`debug_proto_v1`
- 规则类型：`工程冻结规则`
- 适用阶段：`A1` 首轮流程验证
- 直接依据：`结直肠腺体分割_plan_优化版/01_实验执行/02_UNet流程验证/00_阶段总协议.md`，`结直肠腺体分割_plan_优化版/01_实验执行/02_UNet流程验证/02_训练步骤.md`，`结直肠腺体分割_plan_优化版/01_实验执行/02_UNet流程验证/03_验证测试与可视化.md`
- 核心公式或定义参考：

```text
bug_source_rank = [data, target, loss_eval, engine, model]
```

- 采用原因：越靠前的错误越可能污染整条链；先查前置层，能避免把协议问题误判成模型能力问题。
- 不采用的相邻方案：不采用并行乱查；不采用一上来就换结构；不采用先看测试集结果再倒推根因。
- 代码落点：notes/debug_note.md，`src/data/*`，`src/eval/*`，`src/engine/*`，src/models/unet.py
- 运行记录字段：`debug_proto_version`，`failure_type`，`suspect_stage`，`fix_action`
- 验收方式：检查每次正式排查都能明确写出当前处于哪一层；检查没有出现同一轮同时改多层协议的混合修补。

### 4.3 单变量修复规则

- 当前结论：每次正式排查只允许锁定一个主异常、一个优先层级、一个最小修复动作，并立刻做一次回归验证。
- 当前值或最终版本：`single_variable_fix_v1`
- 规则类型：`工程冻结规则`
- 适用阶段：`A1` 首轮流程验证
- 直接依据：`结直肠腺体分割_plan_优化版/01_实验执行/02_UNet流程验证/04_错误排查.md` 当前文件，`结直肠腺体分割_plan_优化版/02_路线与投稿/02_结直肠腺体分割_分阶段实验路线与执行标准.md`
- 核心公式或定义参考：

```text
one_issue -> one_stage -> one_fix -> one_regression_check
```

- 采用原因：只有单变量修复，后续才能判断某个问题到底是如何被关闭的。
- 不采用的相邻方案：不采用同时改数据增强、threshold、loss 和模型结构；不采用“先把能想到的都改掉再看”。
- 代码落点：notes/debug_note.md，config.yaml，run_meta.yaml
- 运行记录字段：`issue_id`，`suspect_stage`，`fix_action`，`regression_check_id`
- 验收方式：检查每条关闭记录只对应一个主修复动作；检查协议变化时同步升级版本链。

### 4.4 证据优先与最小证据包规则

- 当前结论：任何正式排查都必须遵循“异常现象 -> 最小证据采集 -> 锁定层级 -> 最小修复 -> 回归验证 -> 记录关闭”的固定顺序。
- 当前值或最终版本：`evidence_first_v1`
- 规则类型：`工程冻结规则`
- 适用阶段：`A1`
- 直接依据：`结直肠腺体分割_plan_优化版/01_实验执行/00_总览与规范/06_实验执行证据化写作模板.md`，`结直肠腺体分割_plan_优化版/01_实验执行/02_UNet流程验证/05_阶段验收.md`
- 核心公式或定义参考：

```text
issue_t -> evidence_t -> action_t -> verify_t -> close_t
```

- 采用原因：如果没有证据，只靠“感觉这里可能有问题”，那还不算正式排查。
- 不采用的相邻方案：不采用终端滚屏代替证据；不采用修好后不留回归记录。
- 代码落点：notes/debug_note.md，experiments/[run_name]/summaries/error_cases.md，metric_crosscheck_note.md
- 运行记录字段：`evidence_path`，`evidence_type`，`verify_result`，`close_status`
- 验收方式：检查每个正式异常都有最小证据包；检查没有证据的修复不会被标记关闭。

### 4.5 数据读取与配对排查规则

- 当前结论：只要怀疑数据层，必须先验证 `sample_id -> image_path -> mask_path -> split_role` 映射链、自身形状一致性和 split 完整性。
- 当前值或最终版本：`data_debug_rule_v1`
- 规则类型：`工程冻结规则`
- 适用阶段：`A1`
- 直接依据：`结直肠腺体分割_plan_优化版/01_实验执行/01_数据协议/00_阶段总协议.md`，`结直肠腺体分割_plan_优化版/01_实验执行/01_数据协议/07_数据阶段验收.md`，`结直肠腺体分割_plan_优化版/01_实验执行/02_UNet流程验证/01_任务与默认配置.md`
- 核心公式或定义参考：

```text
pair(sample_i) = (x_i, y_i, split_i)
split_i in {train68, val17, testA60, testB20}
```

- 采用原因：数据层一旦出错，后面的 loss、指标、可视化和错误类型都会一起失真。
- 不采用的相邻方案：不采用先调学习率或换 optimizer；不采用跳过配对检查直接怀疑模型输出。
- 代码落点：`splits/glas/*.csv`，`crc_gland_segmentation_project/src/data/datasets.py`
- 运行记录字段：`sample_id`，`image_path`，`mask_path`，`split_role`，`pair_check_pass`
- 验收方式：抽查样本时能证明 image 与 mask 对应同一 `sample_id`；检查 train / val / test 没有串 split。

### 4.6 标签转换与 resize 排查规则

- 当前结论：标签链必须统一满足 `instance mask -> binary mask -> tensor -> resize -> train/eval` 的同一口径，且 mask resize 只能使用最近邻。
- 当前值或最终版本：`target_debug_rule_v1`
- 规则类型：`工程冻结规则`
- 适用阶段：`A1`
- 直接依据：`结直肠腺体分割_plan_优化版/01_实验执行/02_UNet流程验证/01_任务与默认配置.md`，`结直肠腺体分割_plan_优化版/01_实验执行/00_总览与规范/02_参数冻结总表.md`
- 核心公式或定义参考：

```text
y(x) = 1[M_inst(x) > 0]
Resize_image = bilinear
Resize_mask = nearest
```

- 采用原因：标签口径不一致会直接破坏 loss、threshold 和对象级评估解释。
- 不采用的相邻方案：不采用 mask 双线性插值；不采用训练和评估各自定义一套二值化规则。
- 代码落点：src/data/transforms.py，`crc_gland_segmentation_project/src/data/mask_ops.py`，`crc_gland_segmentation_project/src/data/datasets.py`
- 运行记录字段：`target_type`，`foreground_rule`，`image_interp`，`mask_interp`，`target_dtype`
- 验收方式：检查 mask 最终值域仍为二值；检查 image/mask 几何变换同步；检查训练和评估使用同一前景定义。

### 4.7 loss、`threshold_value` 与评估口径排查规则

- 当前结论：训练侧、验证侧和测试侧必须共同满足 `logits -> sigmoid -> threshold -> binary mask -> connected components -> metrics` 的统一预测链。
- 当前值或最终版本：`loss_eval_debug_rule_v1`
- 规则类型：`官方协议固定项 + 工程冻结规则`
- 适用阶段：`A1`
- 直接依据：`结直肠腺体分割_plan_优化版/01_实验执行/02_UNet流程验证/01_任务与默认配置.md`，`结直肠腺体分割_plan_优化版/01_实验执行/02_UNet流程验证/03_验证测试与可视化.md`，`结直肠腺体分割_plan_优化版/01_实验执行/00_总览与规范/04_评估口径与官方脚本对齐.md`
- 核心公式或定义参考：

```text
L_seg = L_BCE + L_Dice
p = sigmoid(z)
y_hat(x) = 1[p(x) >= threshold_value]
```

- 采用原因：很多异常不是 loss 公式本身错了，而是 logits、probability、threshold 和对象级评估链前后不一致。
- 不采用的相邻方案：不采用每个脚本各写一套 threshold；不采用测试时再手工改二值化规则；不采用对象级和像素级链条口径分裂。
- 代码落点：src/losses/seg_losses.py，src/eval/threshold.py，src/eval/run_eval.py，src/metrics/object_metrics.py，src/metrics/pixel_metrics.py
- 运行记录字段：`loss_version`，`threshold_value`，`threshold_source`，`eval_proto_version`，`metric_list`
- 验收方式：检查训练、验证、测试使用同一阈值来源；检查对象级与像素级指标输入格式一致；检查评估脚本没有跳过 sigmoid 或重复 threshold。

### 4.8 训练循环与 checkpoint 排查规则

- 当前结论：训练引擎必须保证 `train -> val -> scheduler -> checkpoint update -> log flush` 的 epoch 闭环完整执行。
- 当前值或最终版本：`engine_debug_rule_v1`
- 规则类型：`工程冻结规则`
- 适用阶段：`A1`
- 直接依据：`结直肠腺体分割_plan_优化版/01_实验执行/02_UNet流程验证/02_训练步骤.md`，`结直肠腺体分割_plan_优化版/01_实验执行/02_UNet流程验证/00_阶段总协议.md`
- 核心公式或定义参考：

```text
m_t = ObjDice(val17, epoch = t)
best_epoch = argmax_t m_t
best.ckpt = ckpt(best_epoch)
last.ckpt = ckpt(stop_epoch)
```

- 采用原因：如果训练循环本身没闭环，再好的 loss 和模型也不会留下可信结果。
- 不采用的相邻方案：不采用只看终端输出；不采用训练完成后再手工猜哪个 checkpoint 是 best；不采用日志、scheduler 和 checkpoint 各用各的监控口径。
- 代码落点：src/engine/trainer.py，src/engine/lr_scheduler.py，src/engine/early_stop.py，run_meta.yaml
- 运行记录字段：`epoch`，`lr`，`best_epoch`，`best_metric_name`，`best_metric_value`，`checkpoint_path`，`stop_reason`
- 验收方式：检查 train / val 模式切换正确；检查 `best.ckpt` 与 run_meta.yaml 对得上；检查日志轮次和 checkpoint 轮次一致。

### 4.9 模型实现排查规则

- 当前结论：模型实现只能作为最后一级怀疑对象，且必须围绕是否忠实实现标准单输出 `UNet` 来检查。
- 当前值或最终版本：`model_debug_rule_v1`
- 规则类型：`论文直接支持 + 工程冻结规则`
- 适用阶段：`A1`
- 直接依据：`结直肠腺体分割_plan_优化版/03_文献证据/01_经典基线与对比方法/02_U-Net.md`，`结直肠腺体分割_plan_优化版/01_实验执行/02_UNet流程验证/01_任务与默认配置.md`
- 核心公式或定义参考：

```text
z = f_theta(x)
decoder_l = Decode(Concat(F_l^enc, Up(F_{l+1}^dec)))
out_channels = 1
```

- 采用原因：首轮阶段最常见的根因不是“`UNet` 太弱”，而是前面协议层没有执行对。
- 不采用的相邻方案：不采用一出问题就改 backbone；不采用直接叠加 attention、边界头或距离头。
- 代码落点：src/models/unet.py
- 运行记录字段：`model_name`，`model_version`，`in_channels`，`out_channels`
- 验收方式：检查 encoder / decoder 通道匹配；检查 skip connection 正确；检查输出尺寸与标签一致；检查输出为单通道前景 logit。

### 4.10 正式 debug 记录与关闭标准规则

- 当前结论：任何进入正式修复流程的异常，都必须形成可追溯的 debug 记录，并且只有在完成单层定位、最小修复和回归验证后才允许标记关闭。
- 当前值或最终版本：`debug_note_v1`
- 规则类型：`工程冻结规则`
- 适用阶段：`A1` 以及后续复盘
- 直接依据：`结直肠腺体分割_plan_优化版/01_实验执行/02_UNet流程验证/05_阶段验收.md`，`结直肠腺体分割_plan_优化版/01_实验执行/03_UNet稳定性/00_阶段总协议.md`
- 核心公式或定义参考：

```text
close_issue =
  single_root_cause_confirmed
  and minimal_fix_applied
  and regression_check_pass
  and close_record_written
```

- 采用原因：如果不留记录，后续稳定性阶段无法判断某个 run 是否已经继承了修复后的协议。
- 不采用的相邻方案：不采用终端输出代替记录；不采用“这次看起来正常了”就算关闭。
- 代码落点：notes/debug_note.md，experiments/[run_name]/summaries/run_summary.md
- 运行记录字段：`issue_id`，`failure_type`，`suspect_stage`，`evidence_path`，`fix_action`，`verify_result`，`close_status`
- 验收方式：检查每个正式异常都有对应条目；检查修复后至少存在一次针对性回归验证结论。

### 4.11 协议级红线与阻断升级规则

- 当前结论：只要出现 split 污染、标签口径漂移、测试集回调阈值、错误 checkpoint、指标实现偏差或版本链不一致等协议级问题，当前 run 必须判定为阻断并优先回退。
- 当前值或最终版本：`protocol_error_gate_v1`
- 规则类型：`工程冻结规则`
- 适用阶段：`A1`
- 直接依据：`结直肠腺体分割_plan_优化版/01_实验执行/02_UNet流程验证/00_阶段总协议.md`，`结直肠腺体分割_plan_优化版/01_实验执行/02_UNet流程验证/05_阶段验收.md`
- 核心公式或定义参考：

```text
if protocol_error == True:
    stage_pass = False
    next_action = rollback
```

- 采用原因：协议级错误会污染后续所有重复实验和方法比较，不能当作普通 seed 波动处理。
- 不采用的相邻方案：不采用“先记一下，后面再修”；不采用“结果大概能看就继续”。
- 代码落点：notes/debug_note.md，run_meta.yaml，summaries/run_summary.md
- 运行记录字段：`protocol_error`，`rollback_reason`，`next_action`
- 验收方式：检查红线问题被明确记录并阻断后续阶段；检查没有带着协议级错误进入阶段验收。

## 5. 固定执行顺序

### 5.1 正式排查启动顺序

一旦进入正式排查，必须严格按下面顺序执行，不能跳步：

1. 先确认是否触发 `debug_required` 或 `protocol_error`。
2. 先登记问题、锁定当前异常现象和首次出现阶段。
3. 按固定主链选择当前优先排查层级。
4. 采集该层最小证据包。
5. 只执行一个最小必要修复动作。
6. 立刻做一次针对性回归验证。
7. 写入关闭结论或升级到下一层继续排查。

### 5.2 当前明确禁止的执行方式

下面这些行为当前都不允许：

- 一次同时改三四个变量再看哪个生效。
- 看完 `TestA / TestB` 结果后再回头改 `threshold_value` 或重挑 checkpoint。
- 未登记问题、未保存证据就直接改脚本。
- 把异常 run 直接覆盖或删除，不留作废说明。
- 用“后面更强模型应该会好”替代当前流程排查。

## 7. 回退条件

### 7.1 独立回退触发条件

只要出现下面任意一条，当前排查协议就必须优先回退，而不是直接继续训练、测试或阶段验收：

- `sample_id / image_path / mask_path / split_role` 映射链不自洽。
- split 边界被破坏，或 `TestA / TestB` 被误参与训练或调参。
- `target_type`、`foreground_rule`、resize 口径或 mask 值域前后不一致。
- `threshold_value` 来源不明，或不是 `val17`。
- `best.ckpt` 不是由 `val_objdice_max` 选出。
- 对象级指标、像素级指标和可视化之间长期严重矛盾且无法解释。
- 正式 run 缺失关键资产，导致无法验证修复是否生效。
- 协议字段已变化却仍沿用旧 `run_name` 或旧版本链。

### 7.2 固定回退顺序

回退时统一按下面顺序排查：

1. 先查数据阶段交接门控和 split 资产。
2. 再查 dataloader、标签转换、resize 和 batch 内容。
3. 再查 loss、`threshold_value`、对象级指标和评估链。
4. 再查训练循环、scheduler、checkpoint 和日志闭环。
5. 只有前面都确认无误后，才查 `UNet` 结构实现本身。

### 7.3 回退后的重新放行条件

只有同时满足下面条件，才允许重新声明当前排查协议有效：

- 问题来源已按固定顺序定位并修复。
- 修复动作已回写到配置字段、代码入口、结果资产或总结资产。
- config.yaml、run_meta.yaml、指标文件、可视化和 debug 记录重新对齐。
- `best_selector / threshold_source / threshold_value` 仍符合正式协议。
- 当前异常具备明确的 `close_status = closed` 或明确升级到下一层排查的结论。

## 9. 代码落地接口

### 9.1 异常登记与状态流转入口

- 代码文件：notes/debug_note.md，experiments/[run_name]/summaries/run_summary.md
- 入口类/函数：`open_debug_issue()`，`append_debug_evidence()`，`update_issue_status()`，`close_debug_issue()`
- 输入：异常现象、首次出现阶段、怀疑层级、证据路径、修复动作、回归验证结果
- 输出：结构化 debug 记录、问题状态、阶段总结中的异常条目
- `dtype`：记录资产为 `markdown/yaml/csv` 文本；关闭状态为 `bool`
- 依赖配置：`debug_proto_version`，`issue_id`，`failure_type`，`suspect_stage`
- 前置断言：任何正式排查都必须先登记问题；没有证据和回归验证的修复不得标记为关闭
- 运行产物：notes/debug_note.md，experiments/[run_name]/summaries/run_summary.md

### 9.2 数据与标签链排查入口

- 代码文件：`splits/glas/*.csv`，`crc_gland_segmentation_project/src/data/datasets.py`，src/data/transforms.py，`crc_gland_segmentation_project/src/data/mask_ops.py`
- 入口类/函数：`audit_split_pairing()`，`inspect_sample_pair()`，`verify_binary_mask_pipeline()`
- 输入：`sample_id`、`image_path`、`mask_path`、split 名、image/mask 样本对
- 输出：配对检查结论、标签与 resize 完整性证据、是否触发数据层回退
- `dtype`：image 为 `uint8/float32`；mask 为 `uint8/float32` 且应保持二值
- 依赖配置：`target_type`，`foreground_rule`，`image_interp`，`mask_interp`
- 前置断言：数据层一旦怀疑存在错配、错位或 split 污染，必须优先排查并阻断后续模型层猜测
- 运行产物：样本可视化证据、`pair_check_pass` 结论、写入 debug_note.md 的数据层条目

### 9.3 loss、`threshold_value` 与评估链排查入口

- 代码文件：src/losses/seg_losses.py，src/eval/threshold.py，src/eval/run_eval.py，src/metrics/object_metrics.py，src/metrics/pixel_metrics.py
- 入口类/函数：`compute_seg_loss()`，`freeze_threshold_from_validation()`，`run_eval_once()`，`compute_object_metrics()`
- 输入：模型 `logits`、sigmoid 概率图、冻结后的 `threshold_value`、prediction mask、GT mask
- 输出：预测链一致性检查结论、评估口径异常说明、是否触发指标实现回退
- `dtype`：`logits/probabilities` 为 `float32`；二值 mask 为 `uint8/bool`；指标为 `float32`
- 依赖配置：`loss_version`，`threshold_value`，`threshold_source`，`eval_proto_version`，`metric_list`
- 前置断言：训练、验证、测试必须共用同一预测链；`threshold_value` 只能来自 `val17`；若对象级口径不一致，必须先修评估链再继续推进
- 运行产物：评估链交叉检查记录、更新后的 debug_note.md 条目、必要时的 metric_crosscheck_note.md

### 9.4 训练引擎与 checkpoint 排查入口

- 代码文件：src/engine/trainer.py，src/engine/lr_scheduler.py，src/engine/early_stop.py，run_meta.yaml
- 入口类/函数：`inspect_epoch_loop()`，`verify_checkpoint_update()`，`inspect_scheduler_state()`
- 输入：训练日志、验证日志、checkpoint 元数据、scheduler 状态、单 batch 前后向结果
- 输出：引擎层异常定位、checkpoint 保存异常定位、是否需要升级回退
- `dtype`：训练日志与指标为 `float32`；结构摘要和状态说明为文本资产
- 依赖配置：`best_selector`，`epoch_max`，`early_stop_patience`，`scheduler_monitor`
- 前置断言：只有在数据、标签、loss 与评估链都已排除后，才允许把根因归到引擎层；不允许借排查名义切换正式训练协议
- 运行产物：引擎层检查记录、run_meta.yaml 中的异常说明、写入 debug_note.md 的关闭或升级结论

### 9.5 模型实现排查入口

- 代码文件：src/models/unet.py
- 入口类/函数：`inspect_unet_forward_shape()`，`inspect_skip_connections()`，`inspect_output_head()`
- 输入：模型结构摘要、单 batch 前向结果、encoder / decoder 通道配置
- 输出：模型实现层异常定位、结构一致性结论
- `dtype`：结构摘要为文本；中间特征与输出为 `float32`
- 依赖配置：`model_name`，`model_version`，`in_channels`，`out_channels`
- 前置断言：只有在前四层都已确认无误后，才允许把根因归到模型层；不允许在此阶段直接替换 backbone 或新增模块
- 运行产物：模型结构核对记录、写入 debug_note.md 的模型层结论
