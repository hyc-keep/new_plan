# CRAG验证阶段总协议

本文件不是 `09_CRAG验证` 目录旧稿的摘要，也不是“先把第二个数据集跑一遍，看到结果再决定怎么写”的阶段记录。

它在当前项目中的唯一职责是：

> 作为 `09_CRAG验证` 阶段的正式母协议，把 `CRAG as the second same-task benchmark / Frozen-mainline transfer validation / Task-specific direct comparison on CRAG / Honest A-B-C conclusion strength / Joint-table handoff to result aggregation` 一次写死；后续代码、实验、表格和写作只允许忠实实现并验证这份协议，不允许一边看 `CRAG` 结果一边重新决定主线是否要改、哪些数字能写成强支撑、哪些条目可以直接进入联合主表。

从现在开始，`09_CRAG验证` 子目录下的其他文件都视为本文件的细化与执行展开；若与本文件冲突，以本文件为准回改。

## 计划 lineage 与下游 handoff

## 回退条件

CRAG 数据入口、训练评估协议、来源类型或交接字段不一致时停止验证，回退到对应冻结阶段，不得用引用值替代复现证据。

数据、训练、评估和汇总入口必须读取当前 manifest 并写入七字段；CRAG 资产与结果表按 test40 边界导出。

## Diagnostics 闭环

### 1.3 为什么当前文件必须独立存在

`09_CRAG验证` 最容易出现的伪完成状态不是“完全没跑”，而是下面这些情况：

- `CRAG` 被悄悄当成第二个调参场，实际已经不再是冻结主线验证。
- `UNet / baseline / current_mainline` 三层关系没有齐，只跑最终模型就直接写结论。
- `CRAG` 主结果表只剩几个数字，却说不清哪些是统一协议复现值，哪些只是任务内引文对照。
- `CRAG` 结果一旦不够漂亮，就试图回头改结构、改阈值、开后处理或临时补更多模型。
- `10_结果汇总` 想生成 `GlaS + CRAG` 联合结果表时，说不清哪些资产是真正的 `CRAG` 阶段正式输出。

如果没有一份独立阶段总协议，后续 `10_结果汇总`、`11_总验收与止损`、联合主表脚本和论文实验章节都会再次出现“`CRAG` 跑过了，但到底按哪套协议、哪档强度、哪层主表资格继续消费”不一致的问题。

## 3. 本阶段唯一允许处理的变量

### 3.1 当前阶段真正要证明什么

`09_CRAG验证` 阶段真正要证明的，不是“第二个 benchmark 也跑了”，而是：

> 在 `GlaS` 上已经冻结的 `UNet / ResNet34-U-Net / current_mainline` 关系，在 `CRAG` 上仍能形成可追溯、可解释、不过度扩张的补充验证证据；同时 `CRAG` 的任务内强对照、联合主表输入资产和结论强度分级也已经被写成唯一协议。

因此，本阶段通过的含义同时包括：

- `CRAG` 被严格固定为 `same-task supplementary benchmark`，而不是新的主调参场。
- `UNet / baseline / current_mainline` 这三层关系在第二 benchmark 上已经形成最小而完整的比较链。
- `MILD-Net-RTS / TA-Net / DEA-Net` 已进入 `CRAG` 任务内 direct comparison 准备层，并继续遵守来源标记规则。
- `CRAG` 的最终表述强度已老实收口到 `A / B / C` 三档之一。
- `10_结果汇总` 与 `11_总验收与止损` 已能直接读取本阶段正式资产，而不是再靠口头解释。

### 3.2 当前阶段不允许用什么替代正式结论

当前明确不允许把下面这些情况写成“`CRAG` 阶段已完成”：

- 只跑 `current_mainline`，没有 `UNet / baseline` 比较链。
- 在 `CRAG` 上顺手改单独训练协议、阈值或后处理，然后把结果写成冻结主线验证。
- 只有一张主结果表，没有失败案例、任务内对照和联合交接资产。
- `CRAG` 数字一旦不够漂亮，就试图回头修改上游模块或重新定义主线。
- 只给出方向性口头判断，没有形成 `A / B / C` 正式等级。

### 3.3 为什么当前阶段必须现在收口

原因是：

- `08_外部对比` 已形成 `GlaS` 上第一层外部比较基座，本阶段必须接着回答“冻结主线在第二 benchmark 上是否仍成立”。
- 路线层已经明确 `CRAG` 是 `GlaS` 主线收口之后的第二 benchmark 验证阶段，不再是可选探索项。
- `10_结果汇总` 需要 `GlaS + CRAG` 联合表、失败案例和结论分级，本阶段若不先压成母协议，下游就没有唯一合法输入。

## 4. 设计边界(做什么/不做什么)

### 4.1 当前阶段直接继承哪些冻结项

本文件直接继承并不得擅自改动的上位规则如下：

- 主任务：`2D gland segmentation`
- 主数据集：`GlaS`
- 第二 benchmark：`CRAG`
- `CRAG` 正式 split：`train153 / val20 / test40`
- 标签口径：`mask > 0`
- 输入通道：`RGB`
- 输入尺寸：`512 x 512`
- 归一化：`ImageNet mean/std`
- 主增强包：沿用当前主线已冻结版本
- 主损失：`L_seg = L_BCE + L_Dice`
- 优化协议：`AdamW + lr 1e-3 + weight_decay 1e-4 + ReduceLROnPlateau + epoch_max 120 + early_stopping 20 + AMP on`
- 评估链：`logits -> sigmoid -> threshold -> binary mask -> connected components -> metrics`
- 主指标优先级：`F1 / Object Dice / Object Hausdorff`
- 补充指标：`Dice / IoU / HD95 / Boundary F1`
- `best_selector = val_objdice_max`

### 4.2 当前阶段唯一合法起点

`CRAG` 阶段不重新定义主模型，而是由 `Distance` 阶段正式结论唯一决定：

```text
current_mainline = (distance_decision_level == keep) ? distance_kept_base : distance_input_base
```

同时，`08_外部对比` 形成的 `H1` handoff 继续作为本阶段上游约束：

```text
CRAG_input_assets = current_mainline_note
                    + H1_external_main_table
                    + H1_source_boundary
                    + H1_task_specific_direct_note
```

这意味着：

- 只有 `Distance` 已经形成唯一 `current_mainline`，`CRAG` 才允许启动。
- `08_外部对比` 已形成的来源边界和主表分层必须被继续继承，不允许在 `CRAG` 阶段另起一套标记规则。
- `CRAG` 阶段不能因为 `08` 的结果而反向改写 `current_mainline`。

### 4.3 当前阶段固定核心集合与唯一变量

当前阶段默认只允许先完成核心三模型：

```text
CRAG_core_models = {UNet, ResNet34-U-Net, current_mainline}
```

任务内对照集合固定为：

```text
CRAG_task_specific_direct = {MILD-Net-RTS, TA-Net, DEA-Net}
```

可选补入的外部方法固定为：

```text
CRAG_optional_external = top1(H1_external_main_table)
```

但仅在核心三模型闭环完成后，才允许讨论它是否进入。

本阶段唯一允许变化的核心变量只有“冻结主线在 `CRAG` 上的迁移验证及其结论强度”：

```text
crag_run = frozen_mainline_proto + crag_data_proto_only
```

本阶段明确不允许借机改动：

- 当前主线结构与模块保留结论
- `CRAG` split
- 输入尺寸
- 主增强包
- 主损失
- optimizer / lr / scheduler
- `best checkpoint` 选择器类型
- 阈值来源规则
- 后处理默认口径
- TTA 默认口径

### 4.4 当前阶段正式阶段门

当前文件固定采用下面的阶段门语义：

```text
Gate_I1 = current_mainline_ready
          AND crag_protocol_fixed
          AND core_crag_runs_complete
          AND crag_direct_ready
          AND conclusion_grade_fixed
          AND joint_handoff_ready
```

各子门定义如下：

- `current_mainline_ready`：`Distance` 的正式 handoff 已发布，`current_mainline` 唯一合法，且本阶段所有配置都引用同一主线说明。
- `crag_protocol_fixed`：`CRAG` 数据、训练、评估和来源边界全部沿用冻结版本，没有出现第二数据集私有调参链。
- `core_crag_runs_complete`：`UNet / ResNet34-U-Net / current_mainline` 已完成最小闭环，且 `ResNet34-U-Net` 与 `current_mainline` 已按规则完成三 seed。
- `crag_direct_ready`：`MILD-Net-RTS / TA-Net / DEA-Net` 的任务内对照、来源标记和直比资格已成型。
- `conclusion_grade_fixed`：`CRAG` 正式结论已落到 `A / B / C` 三档之一，且文字表述与图表、失败案例一致。
- `joint_handoff_ready`：联合结果表、差异解释、失败案例和下游消费资产已可被 `10/11` 直接读取。

### 5.1 `CRAG` 角色固定规则

- 当前结论：`CRAG` 固定是 `same-task supplementary benchmark`，承担冻结主线补充验证角色，而不是新的主调参场

- 核心公式或定义参考：`CRAG_role = supplementary_validation NOT primary_model_selection`
- 采用原因：只有先把 `CRAG` 的角色压成补充验证，后面才不会把第二 benchmark 误写成强泛化证明

## 7. 回退条件

### 7.1 独立回退触发条件

只要出现下面任意一条，本阶段都必须先回退修正，而不是直接进入 `10_结果汇总`：

- `CRAG` 实际已经被当成第二个调参场，出现独立结构、独立训练协议或独立阈值策略
- `UNet / baseline / current_mainline` 三层比较链不完整
- `val20` 和 `test40` 被混用
- 任务内 direct comparison 缺失，或来源标记与资格边界不清
- 只有成功案例，没有失败案例，导致差异解释不成立
- 三次重复结果方向不稳，却仍试图写稳定强结论
- `CRAG` 结果、可视化和 `A / B / C` 结论等级彼此矛盾
- 联合交接资产不完整，无法支撑 `10/11` 直接消费

### 7.2 固定回退顺序

回退时统一按下面顺序排查：

1. 先检查 `CRAG` split 与数据入口是否真正固定。
2. 再检查 `val20 / test40` 的选择与使用是否严格分离。
3. 再检查当前主线模型与 baseline 是否确实沿用冻结版本。
4. 再检查指标链、阈值与来源标记是否完整一致。
5. 最后检查 `CRAG` 的结论强度是否已经过度扩张。

### 7.3 回退后的重新放行条件

发生回退后，只有同时满足下面条件，才允许重新声明 `09_CRAG验证` 阶段通过：

- 问题来源已经按固定回退顺序定位并修复
- 修复动作、影响范围和复验结果已写入 crag_stage_summary.md 或 crag_conclusion_note.md
- CRAG_main_table.md、CRAG_direct_comparison_table.md、glas_vs_crag_gap_table.csv 与文字结论重新对齐
- 当前阶段重新满足第 `5` 节全部规则卡片与第 `6` 节的最低交付物、交接资产和下游起点要求
