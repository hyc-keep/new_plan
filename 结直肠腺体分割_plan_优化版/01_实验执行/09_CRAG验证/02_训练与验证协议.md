# `CRAG` 训练与验证协议

本文件不是旧版“训练说明页”的修补，也不是把 `CRAG` 跑通之后再回头整理的一页执行备忘。

它在当前项目中的正式职责是：

> 作为 `09_CRAG验证` 目录下、从属于 `结直肠腺体分割_plan_优化版\01_实验执行\09_CRAG验证\00_阶段总协议.md` 且受 `结直肠腺体分割_plan_优化版\01_实验执行\09_CRAG验证\01_任务定位.md` 约束的强版执行协议，把 `CRAG` 上“具体怎么跑”一次写死，包括数据入口、模型来源、训练继承、seed 与执行顺序、checkpoint / threshold / test-set role、最低运行产物、阶段级汇总资产、可视化抽样、任务内对照表输入规则以及与 `03_结果解释规则.md`、`结直肠腺体分割_plan_优化版\01_实验执行\09_CRAG验证\04_阶段验收.md` 的职责边界；后续训练脚本、评估脚本、阶段汇总和下游 handoff 只允许忠实实现这里已经冻结的执行链，不允许边跑边改协议。

## 1. 文件角色与执行边界

### 1.1 当前文件负责什么

当前文件只负责冻结下面九件事：

1. `CRAG` 阶段训练与验证的唯一执行起点，也就是“只切换数据入口，继承冻结主线协议”的运行前提。
2. `UNet / ResNet34-U-Net / current_mainline` 在 `CRAG` 上的唯一合法模型来源与配置来源。
3. `CRAG train153 / val20 / test40` 的数据入口、角色分工和禁止回调边界。
4. 训练超参数、loss、optimizer、scheduler、输入尺寸、AMP、batch size 等继承规则。
5. `seed3407 / 1234 / 2025` 的正式使用方式、固定执行顺序与三次重复触发条件。
6. `best_selector = val_objdice_max`、`threshold_source = val20` 与 `test40 = report_only` 的唯一执行口径。
7. 每个正式 run 的最小落盘资产、阶段级汇总资产、可视化和失败案例抽样底线。
8. 本文件对 `结直肠腺体分割_plan_优化版\01_实验执行\09_CRAG验证\03_结果解释规则.md`、`结直肠腺体分割_plan_优化版\01_实验执行\09_CRAG验证\04_阶段验收.md`、`10_结果汇总`、`11_总验收与止损` 提供的最小运行级 handoff 资产。
9. 与当前执行协议直接相关的独立 `回退条件`、`代码落地接口`、`冲突裁决记录`、`文件质量自检` 和独立 `Diagnostics 闭环`。

### 1.2 当前文件不负责什么

当前文件不重新定义下面这些内容：

- `CRAG` 的阶段角色、核心模型名单、任务内 direct comparison 锚点和允许/禁止修改边界；这些内容由 `01_任务定位.md` 先锁定。
- `A / B / C` 三档结果强度的最终解释口径；这些内容由 `结直肠腺体分割_plan_优化版\01_实验执行\09_CRAG验证\03_结果解释规则.md` 负责。
- 阶段是否通过、何时允许进入 `10_结果汇总` 的最终裁决；这些内容由 `结直肠腺体分割_plan_优化版\01_实验执行\09_CRAG验证\04_阶段验收.md` 负责。
- `UNet`、`ResNet34-U-Net`、`LKMA`、`Boundary`、`Distance` 的结构本体和保留结论；这些内容由更早阶段 handoff 唯一给出。
- `MILD-Net-RTS / TA-Net / DEA-Net` 的文献值解释与写作强度；当前文件只负责给表格输入提供来源与产物准备。

### 1.3 为什么当前文件必须独立存在

`CRAG` 最容易出现的伪完成状态不是“完全没跑”，而是下面这些执行漂移：

- 角色上已经说 `CRAG` 是补充验证，但实际训练时又单独改结构、改 loss、改 threshold。
- 只保留一页轻量训练说明，导致后续脚本和记录字段只能靠猜。
- 只跑 `current_mainline` 的单次结果，没有形成 `UNet -> ResNet34-U-Net -> current_mainline` 的最小执行链。
- 看见 `test40` 结果后回头改 selector、改 threshold 或单独加后处理，却还把结果写成冻结迁移验证。
- 下游要做 `GlaS + CRAG` 联合表时，只拿到零散结果文件，拿不到正式 run 资产、失败案例和阶段汇总资产。

因此，`结直肠腺体分割_plan_优化版\01_实验执行\09_CRAG验证\00_阶段总协议.md` 负责冻结整个 `I1` 阶段门，`结直肠腺体分割_plan_优化版\01_实验执行\09_CRAG验证\01_任务定位.md` 负责先冻结角色与名单，而本文件必须单独把“在这个角色边界内，具体如何执行 `CRAG`”压成唯一协议。

## 3. 本阶段唯一允许处理的变量

### 3.1 当前文件的唯一执行前提

`CRAG` 的训练与验证只能从下面这个唯一前提出发：

```text
crag_run = frozen_mainline_proto + crag_data_proto_only
```

它的固定含义是：

- 允许变化的只有数据入口从 `GlaS` 切到 `CRAG`。
- 模型版本、训练协议、评估协议、结果记录方式和来源标记方式都必须继承冻结版本。
- `CRAG` 不承担重新筛结构、重新筛超参数或重新定义测试口径的职责。

### 3.2 当前文件必须回答的最低问题

本文件必须把下面这些执行问题先锁死，后续训练、评估和汇总脚本才有合法起点：

1. `CRAG` 运行时到底从哪一套数据入口、哪一套模型配置和哪一套评估口径起步。
2. 核心三模型按什么顺序推进，哪些模型必须补三次重复，哪些模型只在特定条件下补三次。
3. checkpoint、threshold 和 `test40` 的角色如何固定，避免测试集回调。
4. 每个正式 run 至少要留下哪些结果文件、记录字段和可视化资产。
5. 阶段结束时要把哪些产物交给 `03`、`04`、`10`、`11` 消费。

### 3.3 当前文件明确不回答的问题

下面这些问题不在本文件中给最终裁决，但必须以本文件为执行输入继续展开：

- `A / B / C` 三档中的哪一档成立，以及允许使用什么表述强度。
- `CRAG` 阶段是否整体通过、是否放行到 `10_结果汇总`。
- 某个任务内 direct comparison 文献值最终如何写进正文论述。

### 3.4 当前文件的最小执行链

本文件固定采用下面的最小执行链：

```text
core_crag_execution = {
  UNet(seed3407),
  ResNet34-U-Net(seed3407,1234,2025),
  current_mainline(seed3407,1234,2025)
}
```

补三次重复的正式触发规则固定为：

```text
need_three_seed = (model in {ResNet34-U-Net, current_mainline})
                  OR (model == UNet AND used_in_mean_std_main_table)
```

这意味着：

- `UNet` 默认先承担流程锚点角色，只要求 `seed3407` 跑通。
- `ResNet34-U-Net` 与 `current_mainline` 默认承担正式比较职责，必须补齐三次重复。
- 只有当 `UNet` 明确进入正式 `mean+-std` 主表时，才补 `1234 / 2025`。

## 4. 阶段门控表达式

### 4.1 数据入口唯一合法定义

`CRAG` 阶段统一只允许使用下面这套冻结数据协议：

```text
CRAG_split = {train153, val20, test40}
```

这里必须进一步写死 `train153` 的解释边界：

```text
train153 = train_sup_16 union train_unsup_137
split_provenance_type = project_frozen_from_local_layout
crag_train_pool_mode = full_supervision_engineering_pool
official_universal_split_claim = false
```

其固定含义是：

- `train153` 在当前项目里按全监督训练池使用，也就是训练阶段对这 `153` 个样本统一按正式 mask 监督消费。
- `train_sup_16 / train_unsup_137` 这组名字是当前本地整理版目录来源标记，不表示 `09_CRAG验证` 仍在做半监督 protocol。
- 当前文件继承的是项目冻结后的工程 split，用于保证后续训练与汇总链唯一；它不是对外宣称“`CRAG` 全社区唯一官方 split”。
- 后续若进入 direct comparison，仍必须继续单独检查 `same_split / same_metric_protocol / no_major_test_gap`，不能因为本项目内部已经冻结 `train153 / val20 / test40` 就自动判定外部结果完全公平可比。

并且必须完整继承：

- 图像路径和 mask 路径体系。
- split `CSV` 资产。
- `mask > 0` 的二值标签生成规则。
- resize、normalize 和默认数据读取逻辑。

当前阶段严禁：

- 为了提升 `CRAG` 结果重新切分。
- 看过 `test40` 结果后回头改 `val20`。
- 单独为某个模型建立私有 `CRAG` split。

### 4.2 模型来源唯一合法定义

`CRAG` 阶段只允许评估下面三个已经冻结的模型来源：

- `UNet`：沿用 `02_UNet流程验证 + 03_UNet稳定性` 的正式版本。
- `ResNet34-U-Net`：沿用 `04_Baseline` 阶段冻结的正式 baseline。
- `current_mainline`：沿用 `05_LKMA -> 06_Boundary -> 07_Distance -> 08_外部对比` 后最终保留的唯一主线版本。

当前阶段严禁：

- 为 `CRAG` 单独改 `LKMA` 设计。
- 为 `CRAG` 单独改 `Boundary Head`。
- 为 `CRAG` 单独改 `Distance-aware` 定义、损失权重或结构版本。

### 4.3 训练协议继承边界

`CRAG` 阶段必须继承下面这套冻结训练协议：

- 输入尺寸：`512 x 512`
- 主 loss：`BCE + Dice`
- optimizer：`AdamW`
- `lr = 1e-3`
- `weight_decay = 1e-4`
- scheduler：`ReduceLROnPlateau`
- `epoch_max = 120`
- `early_stopping = 20`
- `AMP = on`
- batch size：沿用 baseline 阶段冻结值；只有显存被动限制时才允许被动下调

当前阶段默认不触发任何例外。若因工程异常被迫修改其中任一项，必须同时满足：

1. 先明确这是工程异常，而不是针对 `CRAG` 的调参冲动。
2. 把结果登记为新的协议版本，不能与既有 `CRAG v1` 结果混写。

### 4.4 评估协议继承边界

`CRAG` 阶段统一沿用下面这套评估链：

```text
eval_cast_policy = logits/probabilities must be kept or cast to float32 before thresholding
eval_pipeline = logits(float32) -> sigmoid(float32) -> threshold -> binary mask(uint8/bool) -> connected components -> metrics(float32)
boundary_metric_width = 3 px
boundary_metric_impl = project_custom_erosion_xor_3x3_ones_8conn_border0_tol3px
connected_components_impl = scipy.ndimage.label
connected_components_connectivity = 8
```

主指标优先级固定为：

1. `F1`
2. `Object Dice`
3. `Object Hausdorff`
4. `Dice / IoU / HD95 / Boundary F1`

这意味着：

- 对象级指标承担主比较职责。
- 像素级和边界级指标只承担补充说明职责。
- 训练阶段的最优 checkpoint 选择和测试阶段汇总都必须服从这条优先级链。
- 即使训练或推理前向开启 `AMP`，正式 `CRAG` 评估链在 `threshold` 之前也必须先把 `logits / probabilities` 回收到 `float32`。
