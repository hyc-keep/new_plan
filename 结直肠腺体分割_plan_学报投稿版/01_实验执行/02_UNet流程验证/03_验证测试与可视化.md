# 验证测试与可视化

这份文件不是“训练跑完后顺手评一下”的说明，也不是给测试脚本抄几个指标名的简表。

它在当前阶段的唯一职责是：

> 把 `结直肠腺体分割_plan_优化版\01_实验执行\02_UNet流程验证\02_训练步骤.md` 已经导出的正式训练资产，升级成一条可执行、可追溯、可回退、可交接的验证、测试、评估与可视化主链，并确保 `val17 -> best.ckpt -> threshold_value -> TestA/TestB -> metrics -> visuals -> run_summary` 这一整条证据链不分裂。

因此，本文件必须同时写清：

- `val17` 只承担什么角色，不承担什么角色。
- `best checkpoint` 和 `threshold_value` 只能如何产生，不能如何产生。
- `TestA60 / TestB20` 的正式执行顺序、指标导出规则和结果目录。
- 可视化、错误类型、交叉核对和最小文字结论如何闭环。
- 什么情况下必须回退，而不是把问题带去 `结直肠腺体分割_plan_优化版\01_实验执行\02_UNet流程验证\05_阶段验收.md` 或 `结直肠腺体分割_plan_优化版/01_实验执行/03_UNet稳定性/00_阶段总协议.md`。

## 3. 本阶段唯一允许处理的变量

### 3.1 当前阶段唯一有效的评估边界

当前阶段统一只承认下面这组评估边界：

- 验证集：`val17`
- 正式测试集：`TestA60 / TestB20`
- 正式模型选择器：`best_selector = val_objdice_max`
- 正式阈值来源：`threshold_source = val17`
- 首轮阈值起点：`threshold_value = 0.5`
- 训练侧增强 profile：`aug_profile_name = light_aug_v1`
- 评估侧增强开关：`eval_aug_enable = false`
- 正式阈值前类型约束：`eval_cast_policy = logits/probabilities must be kept or cast to float32 before thresholding`
- 正式对象级指标：`F1 / Object Dice / Object Hausdorff`
- 正式像素级指标：`Dice / IoU / HD95 / Boundary F1`
- `Boundary F1` 正式宽度：`boundary_metric_width = 3 px`
- 连通域实现：`connected_components_impl = scipy.ndimage.label`
- 连通域连通性：`connected_components_connectivity = 8`
- 边界指标实现：`boundary_metric_impl = skimage.segmentation.find_boundaries(mode=inner) + binary_dilation`
- 正式评估链：`logits(float32) -> sigmoid(float32) -> threshold -> binary mask(uint8/bool) -> connected components -> metrics(float32)`

### 3.2 当前阶段明确不做的事情

当前验证测试文件不允许借“评估与可视化”之名提前打开：

- `CRAG` 正式评估
- 外部对比
- 测试集反向调阈值
- 测试集反向挑选 checkpoint
- postprocess 搜索
- TTA
- 用 `TestA + TestB` 合成单一总分替代 split 级正式记录

这样规定的原因是：

- `A1` 阶段要先证明基础评估链可信。
- 如果当前阶段同时打开多个后处理变量，后面无法判断结果变化到底来自模型、阈值、后处理还是统计方式。
- `A2` 与后续 baseline 需要继承的是这里写死的最小正式评估协议，而不是一个还在漂移的评估说明。

## 4. 阶段门控表达式

### 4.1 `val17` 角色规则

- 当前结论：`val17` 只负责训练过程观察、`best checkpoint` 选择和必要的阈值冻结，不承担正式对外测试结果角色。
- 当前值或最终版本：`val_role_v1`
- 规则类型：`官方协议固定项 + 工程冻结规则`
- 适用阶段：`A1` 开始生效，`A2` 与后续主线阶段继承
- 直接依据：`结直肠腺体分割_plan_优化版\01_实验执行\02_UNet流程验证\01_任务与默认配置.md`，`结直肠腺体分割_plan_优化版\01_实验执行\02_UNet流程验证\02_训练步骤.md`，`结直肠腺体分割_plan_优化版/01_实验执行/00_总览与规范/04_评估口径与官方脚本对齐.md`
- 核心公式或定义参考：

```text
val17_role = {
  monitor_training,
  choose_best_checkpoint,
  freeze_threshold_if_needed
}
```

- 采用原因：验证集必须承担模型选择职责，但不能和正式测试结果混用。
- 不采用的相邻方案：不采用把 `val17` 当成正式测试结果；不采用把 `val17` 和 `TestA / TestB` 混成一个评估池。
- 代码落点：src/eval/run_eval.py，val_metrics.csv，run_meta.yaml
- 运行记录字段：`split_val`，`best_selector`，`threshold_source`，`val_metric_list`
- 验收方式：检查 run_meta.yaml 中验证集角色清楚；检查正式测试结果文件不混入 `val17`。

### 4.2 `best checkpoint` 选择规则

- 当前结论：`best checkpoint` 统一按 `best_selector = val_objdice_max` 选择，且只能由 `val17` 决定。
- 当前值或最终版本：`best_selector_v1`
- 规则类型：`工程冻结规则`
- 适用阶段：`A1` 开始生效，`A2` 与后续阶段继承
- 直接依据：`结直肠腺体分割_plan_优化版/01_实验执行/00_总览与规范/02_参数冻结总表.md`，`结直肠腺体分割_plan_优化版\01_实验执行\02_UNet流程验证\01_任务与默认配置.md`，`结直肠腺体分割_plan_优化版\01_实验执行\02_UNet流程验证\02_训练步骤.md`
- 核心公式或定义参考：

```text
m_t = ObjDice(val17, epoch = t)
best_epoch = argmax_t m_t
best.ckpt = ckpt(best_epoch)
```

- 采用原因：腺体任务主结果围绕对象级指标展开，因此首轮模型选择必须先围绕对象级主指标统一。
- 不采用的相邻方案：不采用 `val_loss_min`；不采用肉眼挑 checkpoint；不采用测完测试集再改 `best.ckpt`。
- 代码落点：src/eval/checkpoint_selector.py，src/eval/run_eval.py，run_meta.yaml
- 运行记录字段：`best_selector`，`best_epoch`，`best_metric_name`，`best_metric_value`，`checkpoint_path`
- 验收方式：检查 `best_selector = val_objdice_max`；检查 `best.ckpt` 与 `best_epoch` 对应；检查测试脚本只读取正式 `best.ckpt`。

### 4.3 `threshold_value` 冻结规则

- 当前结论：首轮 `threshold_value` 起点固定为 `0.5`，若后续需要调整，只能在 `val17` 上完成并冻结；测试阶段只允许消费冻结后的阈值。
- 当前值或最终版本：`threshold_v1`
- 规则类型：`官方协议固定项 + 工程冻结规则`
- 适用阶段：`A1` 开始生效，`A2` 与后续阶段继承
- 直接依据：`结直肠腺体分割_plan_优化版/01_实验执行/00_总览与规范/02_参数冻结总表.md`，`结直肠腺体分割_plan_优化版/01_实验执行/00_总览与规范/04_评估口径与官方脚本对齐.md`，`结直肠腺体分割_plan_优化版\01_实验执行\02_UNet流程验证\01_任务与默认配置.md`
- 核心公式或定义参考：

```text
p = sigmoid(z)
y_hat(x) = 1[p(x) >= tau]
tau_init = 0.5
threshold_source = val17
```

- 采用原因：首轮阶段要先保证阈值来源唯一且可追溯，避免测试集反馈污染正式协议。
- 不采用的相邻方案：不采用 `TestA` 和 `TestB` 各自调阈值；不采用测完测试集再改阈值；不采用不同脚本各写一套 threshold。
- 代码落点：src/eval/threshold.py，src/eval/run_eval.py，configs/eval/eval_proto_v1.yaml
- 运行记录字段：`threshold_value`，`threshold_source`，`threshold_rule_version`
- 验收方式：检查 `threshold_source = val17`；检查测试阶段不重新搜索阈值；检查 run_meta.yaml 记录完整。

### 4.4 正式测试执行顺序规则

### 4.5 指标导出规则

## 7. 回退条件

### 7.1 独立回退触发条件

只要出现下面任意一条，当前评估与可视化协议就必须优先回退，而不是直接进入 `结直肠腺体分割_plan_优化版\01_实验执行\02_UNet流程验证\05_阶段验收.md`：

- `best.ckpt` 不是由 `val17` 和 `val_objdice_max` 选出。
- `threshold_value` 来源不明，或不是 `val17`。
- `TestA / TestB` 任一 split 没有正式结果文件。
- 对象级指标与像素级指标、可视化结论长期严重矛盾且无法解释。
- 指标交叉核对发现系统性口径偏差。
- 可视化缺失 `sample_id`、原图、GT、预测或 overlay 任一关键件。
- `TestA / TestB` 被混合成一个总值参与正式结论。
- run_meta.yaml、指标文件、可视化和总结之间的版本字段对不上。

### 7.2 固定回退顺序

回退时统一按下面顺序排查：

1. 先查训练资产是否正确交接，包括 `best.ckpt / val_metrics.csv / run_meta.yaml`。
2. 再查 `threshold_value`、`threshold_source` 和评估链是否一致。
3. 再查对象级与像素级指标实现、输入格式和交叉核对记录。
4. 再查可视化导出、`sample_id` 回指和错误类型记录。
5. 只有前面都确认无误后，才把异常归因到模型能力边界。

### 7.3 回退后的重新放行条件

只有同时满足下面条件，才允许重新声明当前评估协议有效：

- 问题来源已按固定顺序定位并修复。
- 修复动作已回写到配置字段、代码入口、结果资产或总结资产。
- run_meta.yaml、指标文件、可视化和总结重新对齐。
- `best_selector / threshold_source / threshold_value` 仍符合正式协议。
- 当前 run 重新满足本文件定义的最小交付物和结论要求。
