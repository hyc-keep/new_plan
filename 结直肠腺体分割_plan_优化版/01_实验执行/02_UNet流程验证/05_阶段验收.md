# 阶段验收

这份文件不是“训练跑完后最后看一眼”的说明，也不是给下一阶段写一句“可以继续”的口头放行。

它在当前阶段的唯一职责是：

> 把 `UNet` 首轮流程验证阶段已经形成的训练、验证、测试、评估、可视化、记录与 debug 结果，压成一份正式放行协议，并明确回答“当前闭环到底是否可信、能否进入 `03_UNet稳定性`、若不能该回退到哪里”。

因此，本文件必须同时写清：

- 当前阶段真正验收的是“闭环可信”还是“性能上限”。
- `结直肠腺体分割_plan_优化版\01_实验执行\02_UNet流程验证\02_训练步骤.md`、`结直肠腺体分割_plan_优化版\01_实验执行\02_UNet流程验证\03_验证测试与可视化.md`、`结直肠腺体分割_plan_优化版\01_实验执行\02_UNet流程验证\04_错误排查.md` 分别怎样被本文件消费。
- 六闭环怎样聚合成唯一的 `stage_pass`。
- 什么属于普通未达预期，什么属于协议级红线。
- 通过后到底冻结什么、交接什么、下游只能继承什么。

## 1. 文件角色与执行边界

当前文件负责冻结并展开下面这些放行层动作：

- 把训练、验证、测试、评估、可视化、记录六条链压成统一的阶段门控。
- 把协议级红线、回退触发条件、回退顺序和重新放行条件写死。
- 把 `结直肠腺体分割_plan_优化版/01_实验执行/03_UNet稳定性/00_阶段总协议.md` 只能继承的正式资产和字段写清。
- 把“允许进入 `A2`”与“必须继续留在 `A1` 修流程”的分界线写成正式规则。

### 1.2 当前文件不负责什么

当前文件不负责重新打开下面这些问题：

- 重新定义 `UNet` 结构、loss、optimizer、scheduler、`best_selector` 或 `threshold_source`。
- 重新定义 `TestA / TestB` 的正式评估顺序、指标集合和可视化导出细节。
- 逐条展开异常定位和修复动作本身。
- 借阶段验收之名提前切换 `ResNet34-U-Net`、`LKMA`、`Boundary Head`、`Distance-aware Loss` 或 `CRAG`。

- `结直肠腺体分割_plan_优化版/01_实验执行/02_UNet流程验证/02_训练步骤.md`

- `结直肠腺体分割_plan_优化版/01_实验执行/03_UNet稳定性/00_阶段总协议.md`

### 1.3 当前文件真正要证明什么

本阶段要证明的不是“`UNet` 分数已经很高”，而是下面六件事已经同时成立：

1. 数据阶段正式放行的 `GlaS train68 / val17 / testA60 / testB20` 已被训练和评估代码正确消费。
2. 首版训练协议 `train_proto_v1` 已完成正式训练闭环，并形成可回查资产。
3. 首版评估协议 `eval_proto_v1` 已完成 `val17 -> best.ckpt -> threshold_value -> TestA / TestB` 的正式结果闭环。
4. 对象级指标、像素级指标、可视化和错误类型之间没有长期无法解释的协议级矛盾。
5. 首轮异常若出现，已经经过正式排查并明确关闭或阻断，而不是靠口头说明带过。
6. `03_UNet稳定性` 可以只改变 `seed`，而不需要继续修补 `A1` 的基础协议。

### 1.4 当前文件对下游的硬约束

从 `03_UNet稳定性` 开始，下游只允许继承当前阶段已经冻结的正式协议，不允许：

- 把 `best checkpoint` 改回 `val_loss_min`、肉眼挑选或测试集回调。
- 在 `TestA / TestB` 看完结果后再改 `threshold_value`。
- 把 `A1` 阶段的异常 run、临时修补 run 或未关闭问题混入 `A2` 正式统计。
- 以“后续阶段会更强”为理由跳过当前阶段红线问题。
- 在没有完整交接包时直接进入三次重复或后续 baseline。

## 2. 核心验收问题

### 3.1 当前阶段唯一有效的验收边界

当前阶段统一只承认下面这组放行边界：

- 数据门控必须来自 `data_stage_pass = True`、`handoff_ready = True`、`preflight_pass = True`。
- 正式训练协议必须仍是 `train_proto_v1`。
- 训练侧增强 profile 必须仍是 `aug_profile_name = light_aug_v1`。
- 评估侧增强开关必须仍是 `eval_aug_enable = false`。
- 正式评估协议必须仍是 `eval_proto_v1`。
- 正式阈值前类型约束必须仍是 `eval_cast_policy = logits/probabilities must be kept or cast to float32 before thresholding`。
- `best_selector` 必须仍是 `val_objdice_max`。
- `threshold_source` 必须仍是 `val17`。
- `threshold_value` 必须可回查且只能消费冻结值。
- `Boundary F1` 必须仍按 `boundary_metric_width = 3 px`、冻结实现和 `8-connectivity` 计算。
- 正式测试必须同时覆盖 `TestA60 / TestB20`，并分开落盘。
- 放行结论必须以六闭环和协议级红线共同决定。

### 3.2 当前阶段明确不做的事情

当前阶段验收文件不允许借“是否通过”之名提前打开：

- optimizer、scheduler、loss、输入尺寸或增强重搜。
- `best checkpoint` 重选规则重写。
- 测试集回调阈值或重新挑 checkpoint。
- 以更强结构替代 `UNet` 来证明当前阶段“已经够用”。
- 直接开启 `A2` 或 baseline，再把当前阶段的缺口留到后面补。

这样规定的原因是：

- 当前文件只负责判断首版协议是否可信，不负责继续搜索更优协议。
- 如果放行文件本身还允许重开协议，后续 `A2` 就无法判断自己继承的是哪一版正式输入。

### 3.3 当前阶段通过不代表什么

当前阶段通过，不代表：

- `UNet` 已达到最终论文结果水准。
- 腺体黏连、小腺体漏检、边界模糊等问题已经全部解决。
- 后续 `Baseline / LKMA / Boundary / Distance` 不再需要做。

当前阶段通过只代表：

- 第一版训练与评估主链已经可信。
- 第一版错误类型已经可解释、可追溯。
- 下一阶段可以在固定协议上只改 `seed` 做稳定性统计。

### 4.1 六闭环聚合规则

- 规则类型：`工程冻结规则`
- 适用阶段：`A1` 阶段验收；`A2` 进入前必须先满足
- 直接依据：`结直肠腺体分割_plan_优化版/01_实验执行/02_UNet流程验证/00_阶段总协议.md`，`结直肠腺体分割_plan_优化版\01_实验执行\02_UNet流程验证\02_训练步骤.md`，`结直肠腺体分割_plan_优化版\01_实验执行\02_UNet流程验证\03_验证测试与可视化.md`
- 核心公式或定义参考：

```text
pass_stage =
  pass_train
  and pass_val
  and pass_test
  and pass_eval
  and pass_visual
  and pass_record
```

## 4. 红线条件

### 7.1 独立回退触发条件

只要出现下面任意一条，当前阶段验收就必须优先回退，而不是直接进入 `03_UNet稳定性`：

- `eval_cast_policy` 未按 `float32 before thresholding` 执行，或评估侧开启了增强。
- `TestA / TestB` 任一 split 缺失正式结果文件，或被混成一个总值参与结论。
- `Boundary F1` 宽度、实现或连通域口径无法回指到冻结主线。
- 对象级指标、像素级指标和可视化之间长期严重矛盾且无法解释。
- metric_crosscheck_note.md 指出系统性口径偏差，但当前阶段仍想继续放行。

- 正式交接包缺失关键资产，导致 `A2` 无法只改 `seed` 直接复现。

### 7.2 固定回退顺序

回退时统一按下面顺序排查：

1. 先查数据阶段放行门控和正式交接资产是否正确继承。
2. 再查训练闭环、checkpoint、日志和验证记录是否完整。
