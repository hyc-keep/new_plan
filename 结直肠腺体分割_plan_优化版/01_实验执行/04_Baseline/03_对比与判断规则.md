# 对比与判断规则

本文件不是 baseline 阶段的结果备注，也不是对 `UNet` 和 `ResNet34-U-Net` 做一段“看起来谁更强”的口头总结。

它在当前项目中的唯一职责是：

> 作为 `04_Baseline` 阶段的正式比较协议细则，把 `UNet vs ResNet34-U-Net` 的唯一正式比较对象、公平比较前提、统计身份、主指标与补充指标、通过线、阻断线、结论措辞边界，以及这些规则在结果表、阶段门控和代码接口中的落地方式一次写死。

从现在开始，只要讨论 baseline 阶段“`ResNet34-U-Net` 是否已经足够替代 `UNet` 成为后续正式基座”，都必须以本文件为准；若与同层其他文件冲突，以 `结直肠腺体分割_plan_优化版/01_实验执行/04_Baseline/00_阶段总协议.md` 为总口径，本文件负责把比较层细化到可直接落代码、可直接写入阶段报告和论文文字的程度。

### 1.3 为什么当前文件必须独立存在

因为 `Baseline` 阶段最容易发生的偷换，不是“结果没有导出来”，而是把：

- `UNet` 的稳定三 seed 资产
- `R34UNet` 的结构升级
- `train_proto_v1 / eval_proto_v1` 的协议继承
- `mean +- std` 的聚合比较

这几个本应一起锁死的比较前提，混成：

- 只看某个 seed 的最高值
- 只看 `TestA` 不看 `TestB`
- 只看像素 `Dice` 不看对象级主指标
- 训练协议已经漂移，却还写成“只差结构”

如果没有一份独立比较协议，后面的 `结直肠腺体分割_plan_优化版\01_实验执行\04_Baseline\04_阶段验收.md`、`结直肠腺体分割_plan_优化版/01_实验执行/05_LKMA/00_阶段总协议.md` 和最终论文写作都会再次出现“都叫 baseline 比较，但比较对象、统计身份和放行含义并不一致”的问题。

## 3. 本阶段唯一允许处理的变量

### 3.1 当前阶段真正要证明什么

`04_Baseline` 阶段不是继续证明流程能跑，而是要把“全文唯一正式结构 baseline 是否已经建立”压成可复查的比较协议。

当前文件真正要证明的是：

1. 在 `UNet` 三次稳定基线已经成立的前提下，baseline 阶段唯一正式比较对象固定为 `A2_UNet_GlaS_seed*` 与 `B1_ResNet34_UNet_GlaS_seed*` 的三 seed 聚合比较。
2. `UNet -> R34UNet` 这一步的差异必须被压缩为结构升级，而不是训练、评估或统计协议漂移。
3. baseline 的正式比较结论只能来自 `mean+-std` 聚合资产，而不能来自单次最好 run。
4. 后续 `LKMA / Boundary / Distance` 只能建立在这份比较已经成立的正式基座之上，而不能反向替 baseline 补结论。

### 3.2 为什么正式比较对象必须是阶段级聚合

固定为阶段级聚合比较的原因是：

- 后续模块阶段需要继承的是“正式基座”，而不是“某一次表现最好”的偶然 run。
- `GlaS` 的 `TestA / TestB` 难度不同，只看单 split 或单次峰值会直接破坏公平性。
- 对象级主指标与失败案例往往会在不同 seed 上暴露不同问题，只有聚合层才能判断方向是否稳定。
- 如果 baseline 比较对象仍可被任意切换，`Gate_B1_compare` 就失去门控意义。

### 3.3 当前不采用的相邻方案

当前明确不采用：

- `UNet` 某次最好 run 对 `R34UNet` 某次最好 run。
- 只看 `TestA` 或只看 `Dice` 决定 baseline 是否通过。
- 删掉异常 seed 后再聚合，或以“最好两个 seed”替代正式结论。
- 用协议不一致的 `UNet` 与 `R34UNet` run 强行拼比较表。

## 4. 阶段门控表达式

### 4.1 当前阶段直接继承哪些冻结项

本文件直接继承并不得擅自改动的上位规则如下：

- 主任务：`2D gland segmentation`
- 主数据集：`GlaS`
- 正式 split：`train68 / val17 / testA60 / testB20`
- 标签口径：`mask > 0`
- 输入通道：`RGB`
- 输入尺寸：`512 x 512`
- 归一化：`ImageNet mean/std`
- 增强版本：`light_aug_v1`
- 主损失：`L_seg = L_BCE + L_Dice`
- 优化协议：`AdamW + lr 1e-3 + weight_decay 1e-4 + ReduceLROnPlateau + epoch_max 120 + early_stopping 20 + AMP on`
- 主指标优先级：`F1 / Object Dice / Object Hausdorff`
- `best_selector = val_objdice_max`
- `threshold_source = val17`
- `eval_cast_policy = logits/probabilities must be kept or cast to float32 before thresholding`
- `boundary_metric_width = 3 px`
- `boundary_metric_impl = project_custom_erosion_xor_3x3_ones_8conn_border0_tol3px`
- `connected_components_impl = scipy.ndimage.label`
- `connected_components_connectivity = 8`
- `train_proto_version = train_proto_v1`
- `eval_proto_version`：新轮需在研究定标/阶段锁定后由当前 Gate 冻结；旧 `eval_proto_v3` 仅作为上一轮历史 lineage 保留
- `postprocess_version = <由当前轮 Gate 冻结>`；当前轮待冻结。
- `eval_protocol_lineage` 仅保存历史 `eval_proto_v1 -> eval_proto_v3`；当前比较只允许使用 04 阶段当前轮冻结的 `eval_proto_version` 配对结果。
- 固定三 seed：`3407 / 1234 / 2025`

### 4.1.1 当前正式指标定义与外部归因边界

- `Object Hausdorff` 是本项目的双向对象级面积加权工程适配，不是 GlaS/MedPy 官方函数逐值等价实现。
- `HD95` 是项目的 `MedPy-style discrete surface distance`，基于 erosion boundary + EDT + symmetric percentile，spacing=1；只声明 style-level 对齐。
- `Boundary F1` 是项目自定义补充指标，固定 erosion_xor、3x3、8-connectivity、border0、tolerance3 px。
- 空集规则和 `Dmax` 必须按当前轮 Gate 冻结的 `eval_proto_version` 配置执行；上一轮 `eval_proto_v3.yaml` 仅供历史对照。三 seed std 固定 `ddof=0`，稳定性 Gate 是项目验收规则，不是显著性检验。
- 新轮正式比较消费 04 阶段 Gate 冻结的 `eval_proto_version` 配对结果；上一轮 protocol_v3 只保留为 historical lineage，不得直接作为本轮当前结果。

### 4.2 当前阶段禁止重新打开：

本文件禁止借比较说明之名重新打开：

- 数据 split、标签定义、输入尺寸和归一化。
- 主增强包、主损失、optimizer、lr、scheduler、epoch 与 early stopping。
- `best_selector`、阈值来源、后处理、`threshold` 前 `float32` 边界、`Boundary F1` 宽度/实现和连通域实现/连接性定义。
- `LKMA`、边界分支、距离约束或其它后续模块。
- baseline 的结构身份和 encoder 初始化来源。

### 4.3 为什么当前阶段只能继承不能重开

因为 `B1` 比较文件要回答的不是“还有哪些比较方式也可以试”，而是：

> 在完全相同的数据、训练和评估 protocol 下，只把结构从 `UNet` 升级到 `ResNet34-U-Net`，是否已经足以建立正式 baseline。

如果这里同时夹带：

- 改 optimizer
- 改阈值规则
- 改 batch 口径
- 改后处理
- 改统计身份

后面就无法判断提升究竟来自结构升级，还是来自比较协议漂移。

- 当前结论：`ResNet34-U-Net` 只有在平均主指标不低于 `UNet`、方向稳定、`TestB` 无明显不可解释退化且可视化支持结构升级结论时，才允许作为后续正式基座。

- 运行记录字段：`gate_status`、`decision_reason`、`qualitative_support`、`failure_summary_version`
- 验收方式：平均主指标方向与定性证据按本文件其它子门核验；稳定性部分不得多数表决，六个 `{主指标 x split}` 单元必须全部通过冻结不等式，任一失败即比较门失败。

- 采用原因：baseline 如果本身不稳，后续模块即使改善，也很难证明主线结构结论成立。
- 不采用的相邻方案：不采用 baseline 不过线就靠 `LKMA` 或边界分支“硬救”；不采用删掉异常 seed 后强行过线。
- 代码落点：scripts/summarize_stage.py、notes/debug_note.md
- 运行记录字段：`blocker_type`、`rollback_required`、`repair_action`
- 验收方式：出现平均主指标明显低于 `UNet`、波动明显变大、异常 seed 无法解释、`TestB` 明显更差或可视化显示更碎更黏连等任意情况时，阶段状态必须写成阻断而非通过。

### 5.9 小幅提升措辞规则

- 当前结论：当 baseline 相对 `UNet` 的提升接近波动范围时，只允许写成“小幅但方向一致的优势”或“未形成足够稳定改进”，不得直接写“显著优于”。

- 采用原因：baseline 阶段的目标是得到可信结论，而不是夸大差异。

## 7. 当前结果解释与条件性下游使用

### 7.1 当前 B1 的科学状态

当前 `current_standard` 的三 seed 正式结果可作为真实实验资产和论文对照结果使用，但不得把 `TestB:Object Dice` 的稳定性子门未通过改写为“稳定性完全通过”。当前应同时记录：

- `main_metrics_direction = improved_or_not_worse`：主要对象级均值方向相对 A2 改善或不差；
- `metric_reproducibility_check = pass`：预测 PNG、GT PNG 与正式逐样本 CSV 已独立复核；
- `stability_warning = true`：`TestB:Object Dice` 的三 seed `std` 略高于 A2；
- `stage_gate = blocked`：原始 `Gate_B1_compare` 与 `Gate_B1` 仍按冻结规则裁决，不因小幅差值人工放宽。

这里的 `stability_warning` 是论文和后续分析的解释字段，不是对正式 Gate 的反向覆盖，也不是统计显著性检验。三个 seed 的 `std` 只描述固定三个训练随机状态下的离散程度，不证明未来所有 seed 的稳定性，也不能单独推出模型失效。

### 7.2 条件性下游实验边界

在当前治理决策下，后续阶段可以在当前 B1 三 seed 资产上开展带边界的正式研究执行和论文证据准备；这不改变 `stage_pass_b1=false`，也不把原始 Gate 改写为通过。必须遵守以下边界：

1. 后续模型继续使用同一组 seed `3407 / 1234 / 2025`，同一 GlaS split、训练协议、评估协议、阈值和统计身份。
2. 后续实验必须同时保留 B1 的 per-seed raw 结果、`mean+-std` 和 `stability_warning`，不能只比较均值。
3. 后续增益只能表述为“相对于当前 B1 对照的变化”，不能表述为“相对于稳定性完全通过的正式基座的改进”。
4. 如果后续模块的增益小于或接近 B1 的 seed 波动，或不同 seed 的方向不一致，必须停止下游主线并回退到 04 做受控稳定性迭代。
5. 条件性执行不改变 `stage_pass_b1=false`、`handoff_ready_for_c1=false`，也不允许绕过原始 Gate 生成正式放行结论。

### 7.3 论文表述边界

论文结果中应同时报告三个 seed、均值和标准差，并使用下述事实性表述：

> ResNet34-U-Net improved the mean object-level performance over the A2 reference under the same evaluation protocol. However, the TestB Object Dice variability across the three fixed seeds was slightly higher than that of A2, indicating a minor residual sensitivity to training randomness.

不得使用“所有稳定性指标均通过”“完全稳定”或“显著优于 A2”等超出证据范围的表述。该限制应放在 baseline 结果或 discussion 中，不影响继续报告后续模块是否改善均值、边界和对象级指标，但会影响对“稳定性改善”的强结论。

## 8. 回退条件

### 8.1 独立回退触发条件

只要出现下面任意一条，本文件负责的比较协议就必须先回退修正，而不是直接进入阶段验收或下游模块阶段：

- `UNet vs R34UNet` 的比较对象不是三 seed 聚合，而是单次最好 run。
- `train_proto_v1 / eval_proto_v1`、`best_selector`、`threshold_source`、`eval_cast_policy`、`boundary_metric_width`、`boundary_metric_impl`、`connected_components_impl`、`connected_components_connectivity` 或 `effective_batch_size` 实际不一致，却仍试图写成公平比较。
- raw 与 aggregate 的 `result_tag / aggregation` 字段缺失、冲突或被混写。
- 某个 seed 明显异常，却没有排查记录就被直接删掉或绕过。
- `TestB` 明显恶化或可视化不支持 baseline 作为正式基座，却仍试图强行放行。
- 比较表、阶段总结和可视化结论之间无法互相对齐。

### 7.2 固定回退顺序

回退时统一按下面顺序排查：

1. 检查 `UNet` 与 `R34UNet` 的结构、训练、评估协议字段是否真正对齐。
2. 检查 `physical_batch_size / grad_accum_steps / effective_batch_size` 与初始化来源是否一致。
3. 检查 `best_selector`、`threshold_source`、`threshold_value` 和后处理是否仍按冻结协议执行。
4. 检查 per-seed raw 表、`mean+-std` 聚合表和比较表的 `result_tag / aggregation` 是否一致。
5. 只有确认前面都无问题后，才允许把差异解释为真实结构比较结果。

### 7.3 回退后的重新放行条件
