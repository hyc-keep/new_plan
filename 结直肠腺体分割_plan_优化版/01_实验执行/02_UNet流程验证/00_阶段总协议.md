# UNet 流程验证阶段总协议

本文件不是把 `结直肠腺体分割_plan_优化版\01_实验执行\02_UNet流程验证\01_任务与默认配置.md`、`结直肠腺体分割_plan_优化版\01_实验执行\02_UNet流程验证\02_训练步骤.md`、`结直肠腺体分割_plan_优化版\01_实验执行\02_UNet流程验证\03_验证测试与可视化.md`、`结直肠腺体分割_plan_优化版\01_实验执行\02_UNet流程验证\04_错误排查.md`、`结直肠腺体分割_plan_优化版\01_实验执行\02_UNet流程验证\05_阶段验收.md` 简单拼接成一篇长说明，也不是给“先跑一个 `UNet` 看看”补一层包装。

它在当前项目中的唯一定位是：

> 作为 `02_UNet流程验证` 的正式主协议，把 `A1` 阶段必须冻结的 `UNet` 结构边界、首版训练协议、验证集模型选择、阈值来源、测试导出、错误排查顺序、阶段放行门槛和回退条件一次写死；后续代码只负责忠实实现和验证这份协议，而不是在训练过程中继续重开这些问题。

从现在开始，`02_UNet流程验证` 目录下的其他文件都视为本文件的细则补充；如果和本文件冲突，以本文件为准回改。

### 1.3 当前文件真正要证明什么

本阶段要证明的不是“已经拿到一个 `UNet` 数字”，而是下面六件事已经同时成立：

1. 数据阶段正式放行的 `GlaS train68 / val17 / testA60 / testB20` 能被训练和评估代码稳定消费。
2. 标准 `UNet` 结构在当前腺体分割口径下，能够完成一次可复现的正式训练、验证、测试闭环。
3. 首版训练协议、验证集模型选择规则和阈值来源规则已经冻结成后续阶段可继承的协议，而不是仍靠临时决定。
4. 对象级指标与像素级指标链已经真正落地，而不是只在文档中声明。
5. 首轮错误类型已经被记录到足以区分“流程问题”和“模型能力问题”的程度。
6. 本阶段结束后，`03_UNet稳定性` 可以只改变 `seed`，而不必重新修补 `A1` 协议。

### 1.4 当前文件对下游的硬约束

从 `03_UNet稳定性` 开始，下游只允许继承本文件冻结下来的正式协议，不允许：

- 绕过 `data_stage_pass / handoff_ready / preflight_pass` 直接开训
- 把 `best checkpoint` 改回 `val_loss_min`、肉眼挑选或测试集回调
- 在 `TestA / TestB` 结果出来后再改 `threshold_value`
- 把 `A1` 阶段临时排雷记录混入正式 `mean +- std` 统计
- 用后续更强模型的需要，反向重开 `UNet` 的基础协议

## 3. 本阶段唯一允许处理的变量

### 3.1 当前阶段唯一允许锁定的核心变量

本阶段唯一允许处理的核心变量只有“首版 `UNet` 正式训练与评估闭环协议”本身：

- 标准 `UNet` 如何在当前工程中实现
- 首版训练协议如何冻结为 `train_proto_v1`
- 验证集模型选择与阈值规则如何冻结为 `eval_proto_v1`
- 首轮测试与可视化资产如何导出
- 首轮错误排查与回退顺序如何固定

### 3.2 当前阶段不允许改什么

本阶段严禁借流程验证之名改动：

- 数据 split、标签规则、resize、来源说明和交接资产
- `ResNet34-U-Net` 或任何更强 encoder baseline
- `LKMA`
- `Boundary Head`
- `Distance-aware Loss`
- `CRAG` 和外部对比配置
- 测试集在模型选择或阈值搜索中的角色

### 3.3 为什么只能改这个变量

因为本阶段真正要回答的是“基础训练链和评估链是否可信”。

如果这一步同时夹带：

- 结构增强
- 更复杂的 loss
- 额外 benchmark
- 测试集回调阈值
- 多套并行调参

后面就无法判断：

- 闭环跑通是因为基础协议正确
- 还是因为更强模块碰巧掩盖了流程缺陷

因此，本阶段必须被压成“只验证标准 `UNet` 首版协议”的阶段。

## 4. 本阶段规则卡片

下面所有核心规则都写成正式协议格式；后续实现时，只允许把这些规则忠实落地，不允许擅自补新的隐含规则。

### 4.1 数据阶段放行门控继承

- 当前结论：只有当 `data_stage_pass = True`、`handoff_ready = True`、`preflight_pass = True` 同时成立时，`UNet` 正式训练入口才允许启动。
- 规则类型：`上游阶段冻结规则 + 工程阻断规则`
- 适用阶段：`UNet 流程验证阶段开始生效，后续模型阶段继承`
- 直接依据：`结直肠腺体分割_plan_优化版/01_实验执行/01_数据协议/00_阶段总协议.md`，`结直肠腺体分割_plan_优化版/01_实验执行/01_数据协议/07_数据阶段验收.md`
- 核心公式或定义参考：

```text
a1_train_entry_allowed
= data_stage_pass
and handoff_ready
and preflight_pass
```

- 采用原因：`A1` 阶段的目标是排查训练和评估链，不是继续修数据输入层。
- 不采用的相邻方案：不采用训练脚本直接扫描目录；不采用人工口头确认放行；不采用边训练边补交接资产。
- 代码落点：`crc_gland_segmentation_project/scripts/train.py`，`crc_gland_segmentation_project/configs/data/glas.yaml`，[project_root]/src/data/build_dataset.py
- 运行记录字段：`data_stage_pass`，`handoff_ready`，`preflight_pass`，`data_proto_version`
- 验收方式：检查训练入口有正式断言；检查 run_meta.yaml 记录门控状态；检查失败时会显式阻断。

### 4.2 标准 `UNet` 结构定义

- 当前结论：本阶段统一采用标准 `U-shaped encoder-decoder` 结构，保留跨层 `skip connection`，输出单通道腺体前景 `logit`。
- 规则类型：`论文直接支持 + 工程冻结规则`
- 适用阶段：`UNet 流程验证阶段开始生效，UNet 稳定性阶段继承`

### 4.4 主损失函数

- 当前结论：首版正式训练损失固定为 `L_seg = L_BCE + L_Dice`。
- 规则类型：`论文支持的候选范围 + 工程冻结规则`
- 适用阶段：`UNet 流程验证阶段开始生效，至少继承到 Baseline 阶段`
- 直接依据：`结直肠腺体分割_plan_优化版/01_实验执行/00_总览与规范/02_参数冻结总表.md`，`结直肠腺体分割_plan_优化版/03_文献证据/07_综述与写作支撑/02_Loss-Survey.md`，`结直肠腺体分割_plan_优化版/01_实验执行/02_UNet流程验证/01_任务与默认配置.md`
- 核心公式或定义参考：

```text
L_BCE = - 1/N * sum_i [y_i log(p_i) + (1-y_i) log(1-p_i)]
L_Dice = 1 - (2 * sum_i p_i y_i + eps) / (sum_i p_i + sum_i y_i + eps)
L_seg = L_BCE + L_Dice
```

- 采用原因：当前任务需要一个数值稳定、实现直接、对小样本腺体区域友好的首轮区域主损失。
- 不采用的相邻方案：不采用单独 `BCE`；不采用单独 `Dice`；不采用原始 `weighted softmax CE` 作为当前阶段唯一主损失。
- 代码落点：[project_root]/src/losses/seg_losses.py，[project_root]/configs/train/unet_flow_v1.yaml，[project_root]/configs/experiment/A1_UNet_GlaS_v1_seed3407.yaml
- 运行记录字段：`loss_name`，`loss_version`，`bce_weight`，`dice_weight`
- 验收方式：检查训练日志至少记录 `loss_total`、`loss_bce`、`loss_dice`；检查未对已 sigmoid 的概率重复计算不正确 loss。

## 6. 回退条件

### 6.1 独立回退触发条件

即使训练已经跑完，只要出现下面任意一条，本阶段都必须优先回退，而不是直接进入 `A2`：

- 大量全背景或近全背景预测
- `val17`、`TestA`、`TestB` 的指标与肉眼判断长期严重矛盾
- `TestB` 明显异常但无法由样本难度、错误类型或对象级指标差异解释
- `best.ckpt`、`last.ckpt`、`best_selector` 或 `threshold_value` 的记录存在歧义
- 训练曲线、验证曲线和导出指标之间存在无法解释的不一致
- 正式资产缺失，导致后续无法执行三次重复或阶段汇总

### 6.2 固定回退顺序

回退时统一按下面顺序排查：

1. 检查数据阶段交接资产、split 读取和 `sample_id` 对应关系是否正确继承。
2. 检查标签二值化、resize、同步增强和 target 构造是否一致。
3. 检查 `loss`、`threshold_value`、对象级指标和测试导出实现是否一致。
4. 检查训练循环、日志、checkpoint 保存与加载逻辑。
5. 只有确认前面都无问题后，才检查 `UNet` 结构实现本身。

### 6.3 回退后的重新放行条件

发生回退后，只有同时满足下面条件，才允许重新声明本阶段通过：

- 问题来源已经按固定回退顺序定位并修复
- 修复动作、影响范围和复验结果已写入 `[project_root]/summaries/error_cases.md` 或阶段总结
- config.yaml、run_meta.yaml、指标文件和可视化已重新对齐
- `best_selector / threshold_source / threshold_value` 仍与正式协议一致
- 当前 run 重新满足第 `5.2` 节的全部过线标准
