# 设计依据

本文件不是 `07_Distance` 阶段里“距离监督可能有帮助”的说明文，也不是把几篇 distance / shape / topology 论文拼在一起的讨论稿。

它在当前项目中的唯一职责是：

> 作为 `07_Distance` 的正式设计依据 protocol，把 `Distance` 为什么必须排在 `Boundary` 之后、为什么当前只能做最小可解释的 `Euclidean distance map + auxiliary distance loss`、为什么主版本固定为 `lambda_dist = 0.1`、为什么 `keep` 只能基于主版本三 seed 证据形成、为什么当前不采用 `SDM / clDice / TA-Net / SkeletonAwareDT` 等更重路线，以及这些结论如何落到代码接口、运行字段、阶段门控、回退和下游主线交接，一次写死。

从现在开始，只要讨论 `Distance-aware Loss` 的阶段身份、设计边界、文献支持、变量范围、实现落点、验收证据和下游消费方式，都必须以本文件为准；若与同层文件冲突，以 `结直肠腺体分割_plan_优化版/01_实验执行/07_Distance/00_阶段总协议.md` 管阶段身份与门控，以本文件管“为什么当前只能这样设计”，并要求 `结直肠腺体分割_plan_优化版\01_实验执行\07_Distance\02_距离图与lambda范围.md`、`结直肠腺体分割_plan_优化版\01_实验执行\07_Distance\03_实验步骤.md`、`结直肠腺体分割_plan_优化版\01_实验执行\07_Distance\04_保留或降级标准.md` 全部向这里回对齐。

## 1. 文件角色与执行边界

### 1.1 当前文件负责什么

当前文件只负责冻结下面八件事：

1. `Distance` 为什么必须排在 `Boundary` 之后，而不能提前与 `Boundary` 合并或跳到 topology 路线。
2. `Distance` 阶段真正要验证的残余误差类型是什么，为什么它们指向连续距离约束。
3. `Distance-aware Loss v1` 在本项目里应被压成什么样的最小可解释抽象，而不是新的主结构主线。
4. 距离图为什么只能从冻结二值 gland mask 派生为 `Euclidean distance map`，并统一做 `Normalize01`。
5. 本阶段为什么只能做单距离图定义、单主版本 `lambda_dist = 0.1`、极小 `lambda` 搜索边界和单变量验证。
6. 本阶段为什么必须优先看 `Boundary F1`、`HD95 / Object Hausdorff`、局部轮廓可视化、`Object Dice / F1` 和成本，而不是只看像素 `Dice`。
7. 这些设计判断如何约束代码实现、运行记录字段、阶段门控、`keep / downgrade / drop` 和下游 `current_mainline` 语义。
8. 本文件如何作为 07_Distance/02-04、`结直肠腺体分割_plan_优化版/01_实验执行/08_外部对比/00_阶段总协议.md` 和 `结直肠腺体分割_plan_优化版/01_实验执行/09_CRAG验证/00_阶段总协议.md` 的共同上游设计母规范。

## 3. 任务难点与设计动机

### 3.1 当前阶段真正要证明什么

`07_Distance` 阶段真正要证明的，不是“边界后面再加一个距离项也许会更好”，而是：

> 在 baseline、`LKMA` 和 `Boundary` 的结构、训练、评估和比较协议全部冻结的前提下，连续距离监督是否值得作为第三个 post-baseline 模块独立验证，并以最小单变量设置为边界邻域不稳定、轮廓轻度模糊和对象轮廓偏移提供可复查、可解释、可交接的收益。

因此，本文件真正要压实的是：

1. `Distance` 不是自由扩展阶段，而是 `Boundary` 之后的第三模块验证节点。
2. 当前值得单独验证的残余误差，已经可以被明确归纳为 `boundary neighborhood instability / residual contour blur / contour offset after boundary refinement`。
3. 当前最合理的实现不是大而全的距离家族拼装，而是最小可解释的 `Distance-aware Loss v1`。
4. 当前距离监督必须复用冻结 mask 派生的 `Euclidean distance map`，而不能改写为新的 shape/topology 表示主线。
5. 当前阶段结论必须优先解释其对边界级、对象级和几何级证据的影响，而不是只看像素 `Dice`。
6. 当前阶段结束后，下游主线只能从 `distance_kept_base` 或 `distance_input_base` 二者之一继续，不能再保留模糊起点。

### 3.2 为什么 `Distance` 必须排在 `Boundary` 之后

原因固定为：

- `LKMA` 先解决的是整体形态感知和有效感受野问题。
- `Boundary` 再解决显式对象分离和边界监督问题。
- `Distance` 解决的是在对象分离已经较稳后，边界邻域内部的连续位置约束是否还能带来剩余收益。
- 如果先做 `Distance` 再做 `Boundary`，距离收益会被上游边界不稳定掩盖或扭曲，无法确认它到底补的是连续距离信息，还是离散边界本身没有做好。
- 如果把 `Boundary + Distance` 同时上，后续就无法解释哪一部分增益来自哪一类监督，也无法为论文消融链保留清晰的第三节点。

### 3.3 为什么当前必须立即把设计边界写死

因为到达 `Distance` 阶段时，以下前提已经成立：

- `Boundary` 已经形成正式 `keep / backup / drop` 结论。
- `current_base` 已经可以由上游结论唯一确定。
- 后续 `08_外部对比` 和 `09_CRAG验证` 要消费的不是“某个试过距离监督的版本”，而是唯一合法的距离阶段结论。

因此，当前阶段如果还不把设计边界压实，后续 `结直肠腺体分割_plan_优化版\01_实验执行\07_Distance\02_距离图与lambda范围.md`、`结直肠腺体分割_plan_优化版\01_实验执行\07_Distance\03_实验步骤.md`、`结直肠腺体分割_plan_优化版\01_实验执行\07_Distance\04_保留或降级标准.md` 和下游主线都会失去统一的模块顺序、证据顺序和主线交接语义。

### 4.2 当前阶段唯一合法起点

当前阶段的比较基座不是重新选择，而是由 `Boundary` 阶段正式结论唯一决定：

```text
current_base = (boundary_decision_level == keep) ? boundary_kept_base : boundary_input_base
```

这意味着：

- 只有 `06_Boundary` 已形成正式结论并完成 handoff，`Distance` 才允许启动。
- `backup` 与 `drop` 都不能被默认当成“继续把边界分支带进主线”。
- 任何正式 `Distance` run 都必须能追溯到 `Boundary` 的 decision note、stage manifest 和 `current_base` 说明。
- `Boundary` 写出的 `current_base_next` 必须与这里读取的 `current_base` 完全一致；设计层不允许出现“上游 handoff 一套、Distance 起点另一套”的双口径。

### 4.3 当前阶段唯一变量

本阶段唯一允许处理的核心变量只有“在 `current_base` 上追加一个可插拔 `Distance-aware Loss` 及其最小距离图监督协议”：

- `Distance-aware Loss v1` 的模块定义
- 距离图目标来源与归一化定义
- `lambda_dist` 主版本和极小搜索边界
- 距离监督是否带来可接受的复杂度成本
- 距离监督是否值得进入后续主线

### 4.4 当前阶段明确不允许重开的内容

本阶段明确不允许借机改动：

- baseline backbone
- encoder 预训练策略
- `LKMA` 配置
- `Boundary` 标签、宽度、权重、损失或接入位置
- 数据协议
- 输入尺寸
- 主增强包
- 主损失
- optimizer / scheduler / lr
- `best checkpoint` 选择器

- `signed distance map`
- `clDice`
- `TA-Net`
- `SkeletonAwareDT`
- direction field、marker branch、`watershed` 或其他 topology-aware 后处理

换句话说，当前阶段必须回答的是“连续距离监督本身值不值得保留”，而不是继续重开主线其它变量。

### 5.1 `Distance` 必须排在 `Boundary` 之后

- 当前结论：`Distance` 固定排在 `Boundary` 之后，因为主线必须先稳定整体形态表达，再解决显式对象分离，最后才验证连续距离约束是否还能补足局部轮廓误差。

- 核心公式或定义参考：`current_base = (boundary_decision_level == keep) ? boundary_kept_base : boundary_input_base`
- 采用原因：如果上游显式边界监督尚未稳定，距离监督收益会和边界问题纠缠，单变量比较失效。

### 5.2 当前任务痛点为什么指向连续距离监督

- 当前结论：`Distance` 阶段要对准的误差类型固定为 `边界邻域概率不稳定 / 局部轮廓轻度模糊 / 黏连缓解后仍存在的 contour offset`。
- 规则类型：`论文直接支持 + 任务定义规则`
- 适用阶段：`07_Distance`
- 直接依据：02_Distance-Map-Loss.md、04_Shape-Aware-SDM.md、07_TA-Net.md
- 核心公式或定义参考：距离监督表达的不是“是否为边界”，而是像素到目标边界或内部结构的连续相对位置。
- 采用原因：在 `Boundary` 已经给出离散边界目标之后，距离图能进一步提供连续邻域信息，补足边界带的分辨率不足。
- 不采用的相邻方案：不把本阶段目标写成“整体像素全部再优化一轮”；不把它误写成 topology 主问题。
- 代码落点：[project_root]/reports/stage_reports/distance_visual_casebook.md、[project_root]/scripts/summarize_stage.py
- 运行记录字段：`error_type_tag`、`distance_visual_support`、`decision_reason`
- 验收方式：检查阶段总结是否围绕轮廓邻域细化，而不是只报告像素指标轻微波动。

### 5.3 文献链只支持“先做最小距离监督抽象”

- 当前结论：`Distance Map Loss`、`Shape-Aware SDM`、`clDice`、`TA-Net` 和 `SkeletonAwareDT` 共同支持“距离/形状/拓扑监督对复杂对象结构有价值”，但它们对当前项目首版的共同约束不是“全部采用”，而是“先从最小距离监督抽象开始”。
- 规则类型：`论文直接支持 + 论文支持的候选范围`
- 适用阶段：`07_Distance`
- 直接依据：02_Distance-Map-Loss.md、04_Shape-Aware-SDM.md、05_clDice.md、07_TA-Net.md、14_SkeletonAwareDT.md
- 核心公式或定义参考：`Distance Map Loss` 提供距离加权或距离辅助监督思路；`Shape-Aware SDM` 提供 `signed distance regression`；`clDice` 提供 skeleton / connectivity 约束；`TA-Net` 和 `SkeletonAwareDT` 提供拓扑与骨架表示路线。
- 采用原因：这条文献链说明当前阶段值得验证“distance-related supervision”，但也说明复杂度梯度很大，首轮必须控制抽象深度。
- 不采用的相邻方案：不把单篇论文完整结构直接当当前协议；不把 `SDM + skeleton + marker + watershed` 一次全开。
- 代码落点：`crc_gland_segmentation_project/src/data/distance_targets.py`、[project_root]/src/losses/distance_losses.py
- 运行记录字段：`dist_version`、`distance_type`、`distance_norm`、`dist_loss_type`
- 验收方式：检查当前正式实现只保留最小距离监督主旨，而没有扩展为 topology-aware 多分支系统。

### 5.4 `Distance-aware Loss v1` 必须是最小可解释抽象

- 当前结论：`Distance-aware Loss v1` 固定为 `single auxiliary distance loss attached to the frozen segmentation objective`，不新增第二主干、回归头或实例恢复链。
- 规则类型：`工程冻结规则 + 文献约束下的任务化抽象`
- 适用阶段：`07_Distance`
- 直接依据：`结直肠腺体分割_plan_优化版/01_实验执行/07_Distance/00_阶段总协议.md`、02_Distance-Map-Loss.md
- 核心公式或定义参考：`L_total = L_seg + lambda_dist * L_dist`
- 采用原因：首版协议必须先确认距离信号本身是否有价值，而不是先打开 shape-aware 表示家族搜索空间。
- 不采用的相邻方案：不采用 `SDM` 回归主线；不采用 skeleton-aware energy representation；不采用多输出拓扑解码器。
- 代码落点：[project_root]/src/losses/distance_losses.py、[project_root]/configs/model/distance_loss_v1.yaml
- 运行记录字段：`dist_version`、`dist_loss_type`、`lambda_dist`、`use_distance_loss`
- 验收方式：检查模型改动仅表现为单附加距离损失，而不是主干重构。

### 5.5 距离图定义必须固定为 `Euclidean + Normalize01`

- 当前结论：首轮正式距离图定义固定为从冻结二值 gland mask 派生的 `Euclidean distance map`，并统一归一化到 `[0, 1]`。
- 规则类型：`工程冻结规则 + 标签定义规则`
- 适用阶段：`07_Distance`
- 直接依据：`结直肠腺体分割_plan_优化版/01_实验执行/07_Distance/02_距离图与lambda范围.md`、02_Distance-Map-Loss.md
- 核心公式或定义参考：`M(x) = 1[mask(x) > 0]`；`D = EDT(M)`；`D_norm = Normalize01(D)`
- 采用原因：这是当前最直接、最稳、最容易实现和最容易复查的最小距离表示。
- 不采用的相邻方案：不采用 `signed distance map`；不采用 `MA distance map`；不采用 skeleton-aware `SDT`。
- 代码落点：`crc_gland_segmentation_project/src/data/distance_targets.py`、`crc_gland_segmentation_project/src/data/mask_ops.py`
- 运行记录字段：`distance_target_version`、`distance_target_source`、`distance_type`、`distance_norm`、`distance_value_range`
- 验收方式：检查训练、可视化和汇总全部复用同一 `euclidean_norm01` 定义。

### 5.6 主版本必须固定为 `lambda_dist = 0.1`

- 适用阶段：`07_Distance`

- 核心公式或定义参考：`main_distance_variant = DistanceAwareLoss(euclidean, norm01, lambda=0.1)`
- 采用原因：主版本必须是最便于解释、最接近最小验证目标且与同层执行链对齐的锚点版本。

### 5.7 搜索边界必须收口为“主版本 + 最多一个备选版本”

- 当前结论：`Distance` 阶段必须坚持“单距离图定义 + 单主版本 + 极小 `lambda_dist` 范围 + 最多一个备选版本”的最小验证策略。
