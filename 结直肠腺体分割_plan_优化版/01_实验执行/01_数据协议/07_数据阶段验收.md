# 数据阶段验收

本文件不是“最后再看一眼数据差不多就过”的说明，而是 `01_数据协议` 的正式放行协议。它负责把上游已经冻结的数据身份、split、标签、检查、预览和交接资产，统一收束成一条后续阶段必须共同服从的门控链：

`七条链验收聚合 -> 红线检测 -> 可接受风险登记 -> 最小交接放行 -> 训练前预飞检查`

如果这一层不先写死，后面最容易发生的不是“少补了一份报告”，而是：

- 数据阶段其实还没闭环，却已经启动 `UNet`
- 训练异常出现后，没人能判断该回退到 split、标签、检查还是交接层
- 目录里虽然有文件，但无法回答“下游当前到底在消费哪一版正式数据资产包”

---

## 1. 当前阶段最小交付物

- `crc_gland_segmentation_project/reports/stage_reports/data_stage_acceptance.md`
- `asset_manifest` 与 `data_stage_pass / handoff_ready / preflight_pass / next_action`
- `7` 个正式 split `CSV`
- 正式检查资产、预览资产、配置资产与来源说明资产

## 2. 核心验收问题

- 七条链是否同时通过并可回指同一套正式数据资产包。
- 红线问题是否已经被阻断，而不是被可接受风险覆盖。
- 下游训练入口是否只能在正式放行字段成立后消费当前数据资产包。

## 4. 红线条件

- 跨 split 泄漏、测试集回流训练、系统性错位、绝对路径漂移、来源说明缺失。
- 资产缺口导致 `asset_manifest`、`handoff_ready`、`data_stage_pass` 与真实目录状态不一致。
- 训练入口试图绕开正式 split、正式配置和正式放行字段。

## 5. 阶段通过声明

只有当七条链同时通过、红线为零、最小交接包齐全且训练前预飞成立时，当前阶段才允许被声明为“通过并交接到 `02_UNet流程验证`”。

### 1.3 当前文件的真实目标

当前文件真正要回答的不是“数据阶段看起来有没有做完”，而是：

> 现在这套 `GlaS / CRAG` 数据协议与正式数据资产，是否已经强到足以成为下游训练、评估、复盘和论文写作的唯一输入层。

因此，本文件的通过含义必须是：

- 协议已写死，而不是口头知道
- 资产已落盘，而不是终端里看过
- 状态已登记，而不是人工勾选
- 训练入口已能被阻断，而不是靠人自觉不出错

### 1.4 当前文件对下游的硬约束

从 `02_UNet流程验证` 开始，下游只允许消费这里放行的数据资产包，不允许：

- 配置文件绕开正式 `CSV`，直接扫描原始目录
- 因为某个资产缺失，就边训练边补
- 在训练阶段重新解释 split、标签规则、预览资产或来源说明
- 手工写 `data_stage_pass=True` 或 `handoff_ready=True` 绕过正式门控

## 5. 七条链验收定义

### 5.1 `pass_source`

`pass_source` 只有在以下条件同时满足时才允许为 `True`：

- 正式数据根已固定为 `GlaS` 与当前本地整理版 `CRAG`
- `crc_gland_segmentation_project/datasets/DATASETS_README.md` 与 `crc_gland_segmentation_project/datasets/DATASET_SOURCE_NOTES.md` 已更新
- 来源说明能够解释目录身份、整理方式和后续代码实际读取的版本
- 配置、说明和检查报告引用的是同一版正式数据根

### 5.2 `pass_split`

`pass_split` 只有在以下条件同时满足时才允许为 `True`：

- `GlaS train68 / val17 / testA60 / testB20` 已固定
- `CRAG train153 / val20 / test40` 已固定
- `7` 个正式 split `CSV` 已生成并登记版本字段
- schema 一致，且 `sample_id`、路径字段、`grade`、`source_subset` 等必要字段完整
- 测试集边界没有回流训练或调参

### 5.3 `pass_pair`

`pass_pair` 只有在以下条件同时满足时才允许为 `True`：

- 各 split 的 image / mask 数量一致
- 路径可读，且不存在损坏文件
- 文件名映射规则稳定
- 重复样本、跨 split 泄漏和系统性错位已排除

### 5.4 `pass_label`

`pass_label` 只有在以下条件同时满足时才允许为 `True`：

- 主线标签规则固定为 `mask > 0`
- image / mask resize 规则已经冻结为 `image=bilinear`、`mask=nearest`
- `input_size = 512 x 512` 等关键标签消费前提已和配置一致
- 边界图与距离图都由正式二值 mask 派生
- `mask_rule_version / boundary_target_version / distance_target_version` 已写入正式记录

### 5.5 `pass_check`

`pass_check` 只有在以下条件同时满足时才允许为 `True`：

- 数据检查总报告存在
- 样本数、前景比例、对象尺度、空 mask 等统计资产存在
- 重复检查结果存在
- 最低人工抽查已覆盖正式 split
- 检查资产能回指当前 split 与标签版本

### 5.6 `pass_preview`

`pass_preview` 只有在以下条件同时满足时才允许为 `True`：

- 各正式 split 都存在最小预览证据包
- 预览资产能回指 `dataset / split / sample_id`
- `raw / mask_bin / overlay` 三件套齐全
- 若边界图和距离图已进入正式资产链，则对应 `boundary_overlay / distance_heatmap` 也已成套

### 5.7 `pass_handoff`

`pass_handoff` 只有在以下条件同时满足时才允许为 `True`：

- `结直肠腺体分割_plan_优化版\01_实验执行\01_数据协议\06_数据产物清单.md` 定义的六类正式资产已齐全
- `asset_manifest` 已完成正式登记
- `assets_traceable = True`
- `protocol_explainable = True`
- 交接状态与实际资产一致，不存在手工硬写放行

## 6. 当前阶段通过线与最小放行含义

### 6.1 最简洁通过线

当前数据阶段通过，至少同时满足下面四条：

1. 输入层协议清楚
2. 正式资产可追溯
3. 下游代码可直接消费
4. 红线问题已被阻断

### 6.2 展开后的真实含义

它对应到可核验的话，就是：

- 你能准确说出后续训练到底读哪 `7` 个 `CSV`
- 你能解释正式标签怎么由原始标注得到
- 你能指出检查资产、预览资产、配置资产和来源说明在哪里
- 你能证明训练入口在门控失败时会被阻断

只要有任意一项仍停留在“人知道但文件没写、资产没落盘、状态没登记”，就不算真正通过。

### 6.3 通过线解释规则

## 7. 不通过线与红线问题

### 7.1 一般不通过线

下面任意一种成立，都不允许进入 `UNet`：

- 正式数据源身份仍不清楚
- `7` 个正式 split `CSV` 还未全部固定
- image / mask 配对还有疑问
- 二值化、resize、边界图或距离图规则还未写死
- 核心检查资产或核心预览资产缺失
- `asset_manifest` 未完成登记
- 训练入口还可能绕开正式配置直接扫描目录

### 7.2 红线问题

如果下面任何一条成立，必须立即暂停推进，先修数据阶段问题：

- 同一 `sample_id` 出现在多个 split
- `TestA / TestB / test40` 被误用于训练、调参或阈值搜索
- 批量空 mask、异常全白 mask 或系统性错位未修复
- 正式配置、来源说明、检查报告与 split 版本对不上
- 脚本偷偷使用绝对路径或另一套本地副本
- `CRAG` 当前本地整理版来源说不清

### 7.3 红线回退规则

- 当前结论：只要存在数据层红线问题，必须判定为“不通过并回退”，不得带着问题进入训练阶段
- 当前值或最终版本：`data_red_flag_gate_v2`
- 候选范围：`protocol-level data failure -> rollback`
- 规则类型：`工程冻结规则`
- 直接依据：`结直肠腺体分割_plan_优化版\01_实验执行\01_数据协议\02_数据检查与配对规则.md`，`结直肠腺体分割_plan_优化版\01_实验执行\01_数据协议\00_阶段总协议.md`
- 核心公式或定义参考：若 `red_flag=True` 或 `data_protocol_error=True`，则 `data_stage_pass=False` 且 `next_action=rollback`
- 采用原因：这些问题一旦存在，后面模型阶段全部比较都会失去可信度
- 不采用的相邻方案：不采用“先记录、后面再修”；不采用“模型先跑起来再说”
- 代码落点：`[project_root]/reports/data_checks/*`，`crc_gland_segmentation_project/reports/stage_reports/data_stage_acceptance.md`
- 运行记录字段：`red_flag`，`data_protocol_error`，`rollback_reason`，`next_action`
- 验收方式：检查红线问题是否被正式记录；检查出现红线时是否已阻断放行

## 8. 可接受风险与登记要求

下面这些问题不一定阻止进入 `UNet`，但必须登记在案：

- 不同 split 之间存在一定染色风格差异
- 某些样本对象极小、极密集或边界极模糊

这些问题的正确处理方式不是忽略，而是：

- 写入正式风险登记位置
- 指定后续错误分析继续跟踪
- 明确它们不能覆盖红线问题

## 10. 回退条件

### 10.1 数据阶段内回退触发条件

只要出现下面任意一条，本文件对应的正式验收协议就不得放行，必须先回退到数据阶段修正：

- 正式数据源身份、来源说明或配置指向不一致
- `7` 个正式 split `CSV` 任一缺失或 schema 失配
- 配对、路径、空 mask、重复样本或系统性错位检查未通过
- `mask > 0`、resize、边界图或距离图规则与正式配置脱节
- 某个正式 split 没有最小预览证据包
- `asset_manifest` 未登记完整，或状态字段与实际资产不一致
- 出现红线问题但仍试图继续训练

### 10.2 下游异常时的强制回退顺序

如果 `02_UNet流程验证` 出现全背景预测、可视化明显错位、对象级指标与肉眼严重矛盾、`val` 与 `test` 异常反常等问题，必须按下面顺序强制回退：

1. 检查 `CSV` 路径与 `sample_id` 对应关系
2. 检查 image / mask 配对与路径可读性
3. 检查 `mask > 0` 二值化是否被下游改写
4. 检查 image / mask 的 resize 与同步增强是否分裂
5. 检查边界图和距离图是否仍沿用正式生成函数
6. 检查检查资产与预览资产是否还能回指当前版本
7. 检查训练入口是否真的只消费正式配置与正式资产

### 10.3 回退后的重新放行条件

发生回退后，只有同时满足下面条件才允许重新把 `handoff_ready` 和 `preflight_pass` 改回 `True`：

- 缺失或错误资产已经按当前正式版本补齐
- `asset_manifest` 已重新登记并更新版本字段
- `assets_traceable=True`
- `protocol_explainable=True`
- `red_flag=False`
- `data_stage_pass=True`
- 下游训练前预飞检查重新通过
