# 投稿结果包检查表

## 1. 文件角色与执行边界

### 1.1 当前文件负责什么

这份文件不再是“投稿前随手勾一下的弱检查表”，而是 `10_结果汇总` 阶段里专门负责 `package_status_fixed` 子门的正式协议文件。

它当前只负责把下面八件事写死：

1. 什么样的结果包才算正式投稿结果包，而不是“数字很多但资产还没真正收口”的半成品集合。
2. 旧稿里的五大类资产, 也就是 协议资产 / 数值资产 / 图像资产 / 表注与来源资产 / 结论与局限资产，在本项目里的唯一合法展开方式。
3. ready / ready_with_minor_gaps / not_ready 三档状态的唯一合法判定边界、缺口登记方式和下游可消费语义。
4. 10_结果汇总/01-04 已冻结的 run、表格、图像和 summary 层，如何被重新压成结果包层的统一检查记录。
5. 最终进入 `11_总验收与止损` 之前，结果包必须具备的追溯链、公平性边界、局限性证据和 handoff 资产。
6. 当前文件自己的独立 `回退条件`、`代码落地接口`、`冲突裁决记录`、`文件质量自检` 和独立 `Diagnostics 闭环`。
7. 当前文件作为 `结直肠腺体分割_plan_优化版/01_实验执行/10_结果汇总/00_阶段总协议.md` 中 `package_status_fixed` 子门展开协议的正式职责。
8. 当前文件与 `结直肠腺体分割_plan_优化版/01_实验执行/11_总验收与止损/00_阶段总协议.md`、`结直肠腺体分割_plan_优化版/01_实验执行/11_总验收与止损/01_阶段门槛总表.md` 之间的固定交接语义。

### 1.2 当前文件不负责什么

当前文件不重新定义下面这些内容：

- 哪些 run 先算正式记录；这些由 `结直肠腺体分割_plan_优化版/01_实验执行/10_结果汇总/01_结果记录模板.md` 冻结。
- 哪些结果有资格进入主表、消融表、外部层、`CRAG` 主表和联合表；这些由 `结直肠腺体分割_plan_优化版/01_实验执行/10_结果汇总/02_主表与消融表回填规则.md` 冻结。
- 哪些图像算正式证据、失败案例和正文候选图如何分层；这些由 `结直肠腺体分割_plan_优化版/01_实验执行/10_结果汇总/03_可视化与失败案例清单.md` 冻结。
- 各阶段 summary 的字段链、handoff 结构和 `summaries_ready` 判断；这些由 `结直肠腺体分割_plan_优化版/01_实验执行/10_结果汇总/04_stage_summary模板.md` 冻结。
- `11_总验收与止损` 的 Pass / Pass with gaps / Not pass 总裁决、中文保底和英文起步判断；当前文件只负责提供结果包层输入，不代替下游最终裁决。

### 1.3 为什么当前文件必须独立存在

结果汇总阶段最容易出现的伪完成状态不是“什么都没整理”，而是下面这些情况：

- run、表格、图像和 summary 都分别存在，但没有一份文档把它们压成统一结果包状态。
- 主表已经能写，但 reproduced / † / *、single_seed / mean +- std、GlaS / CRAG、TestA / TestB / test40 的边界还没被结果包层统一检查。
- 图像很多，但失败案例、局限性文字和样本级追溯信息还不足以支撑正文和 rebuttal 风险。
- 可以大概说出“主线差不多定了”，却说不清哪些缺口只是补充层缺口，哪些已经阻断投稿结果包放行。
- `11_总验收与止损` 将要做总裁决，但还缺一份能直接回答“结果包到底是哪一档、还缺什么、是否允许进入下游”的正式输入。

如果结果包层不单独冻结，下游总验收就会再次面对“资产很多，但 package 状态没有正式成文”的灰区。

## 3. 本阶段唯一允许处理的变量

### 3.1 结果包层的统一定义

从现在开始，正式投稿结果包固定理解为：

```text
result_package
= protocol_assets
+ numeric_assets
+ figure_assets
+ source_note_assets
+ claim_boundary_assets
+ gap_register
+ downstream_handoff
```

并且：

```text
package_review_record
= package_scope
+ subgate_inputs
+ asset_status
+ gap_status
+ package_status
+ downstream_consumability
```

其中：

- `protocol_assets`：冻结协议、阶段通过状态、run 追溯链和 summary 收口层。
- `numeric_assets`：`GlaS` 主表、消融表、外部层、`CRAG` 主表、联合表和稳定性标记。
- `figure_assets`：成功案例、失败案例、比较图、局部放大图、正文候选图和追溯字段。
- `source_note_assets`：reproduced / † / *、single_seed / mean +- std、postprocess / tta、benchmark 与 split 边界说明。
- `claim_boundary_assets`：主线去留、CRAG A / B / C、任务内位置、局限性、允许和禁止写法。
- `gap_register`：阻断性缺口、轻微缺口、已知缺口补法和剩余风险。
- `downstream_handoff`：下游总验收可以直接消费的状态、缺口和资产引用。

### 3.2 当前文件在 `Gate_J1` 中的角色

`结直肠腺体分割_plan_优化版/01_实验执行/10_结果汇总/00_阶段总协议.md` 把 `package_status_fixed` 写成正式子门。

当前文件就是这个子门的展开协议：

```text
package_status_fixed
= package_status_decided
AND blocking_gaps_judged
AND minor_gaps_judged
AND missing_assets_traced
AND downstream_handoff_prepared
```

同时，当前文件还要为下游提供直接输入：

```text
package_handoff_ready
= package_status_fixed
AND summary_consistent
AND table_figure_consistent
AND final_acceptance_consumable
```

解释如下：

- `package_status_decided`：结果包已经明确落到 ready / ready_with_minor_gaps / not_ready 三档之一。
- `blocking_gaps_judged`：关键缺口是否阻断放行已经说清，不能模糊处理。
- `minor_gaps_judged`：轻微缺口是否允许随写稿同步补齐已经说清。
- `missing_assets_traced`：每个缺口都能回指具体表、图、summary、注释或说明，而不是只写“还有些东西没补”。
- `downstream_handoff_prepared`：下游总验收能直接读取当前状态和缺口，不需要再自行拼接。

### 3.3 当前文件唯一合法输入资产

当前文件只允许消费下面这些正式资产：

- `official_result_record`
- `formal_tables`
- `figure_evidence`
- `stage_summary_record`
- `stage_pass`
- package_status = ready / ready_with_minor_gaps / not_ready

当前文件明确不允许：

- 把临时筛查值、调参笔记或“看起来不错的图”直接当结果包正式资产。
- 先给结果包定级，再倒推去补表格、图像或 summary。
- 把 single_seed / mean +- std、reproduced / † / *、GlaS / CRAG、TestA / TestB / test40 的边界留给下游自己猜。
- 把“缺口还在”写成“问题不大”从而越过总验收。

## 4. 阶段门控表达式

后面统一只承认下面九条总原则：

1. 先有正式 run、正式表格、正式图像和正式 summary，再允许判定结果包状态。
2. 先查五类资产是否齐，再判断当前是 `ready`、`ready_with_minor_gaps` 还是 `not_ready`。
3. 先区分阻断性缺口和轻微缺口，再决定能否进入 `11_总验收与止损`。
4. 先区分引文值、复现值、单次结果和多次统计结果，再判断这些数值能否进入正式结果包。
5. 先让失败案例和局限性证据成文，再允许正文主张和结论边界放行。
6. 先让缺口可以回指具体资产，再允许使用“结果包基本完成”这类总结句。
7. 先保证结果包能被总验收直接消费，再考虑措辞简洁度。
8. 结果包检查只能做正式收口和放行判断，不能回头重开训练、评估或模块去留协议。
9. 任何未定级、未追溯、未说明边界的资产，都不得因为“已经有很多结果”而自动放行。

一句话理解：

- 结果包层关心的不是“资料多不多”。
- 而是“能不能直接支撑写稿、审稿追问和总验收裁决”。

## 5. 核心规则卡片

### 5.1 五类资产总检查规则

- 当前结论：结果包检查固定围绕 协议资产 / 数值资产 / 图像资产 / 表注与来源资产 / 结论与局限资产 五类资产执行，任一类存在阻断性缺口时不得判为 `ready`
- 规则类型：`工程冻结规则 + 放行规则`
- 适用阶段：`10_结果汇总` 末端
- 直接依据：`结直肠腺体分割_plan_优化版/01_实验执行/10_结果汇总/00_阶段总协议.md`，旧版 05_投稿结果包检查表.md
- 核心公式或定义参考：`package_assets = protocol + numeric + figure + source_note + claim_boundary`
- 采用原因：五类资产分别对应协议合法性、数字可写性、图像可写性、公平性可答辩性和结论可收口性
- 不采用的相邻方案：不采用只查表格不查图像；不采用只看是否有数字不看结论边界；不采用把所有缺口混成一句“后面再补”
- 代码落点：submission_package_check.md，scripts/summarize_stage.py
- 运行记录字段：`protocol_assets_status`, `numeric_assets_status`, `figure_assets_status`, `source_note_assets_status`, `claim_boundary_assets_status`
- 验收方式：检查五类资产各自都有状态、证据引用和缺口说明，且没有被合并成模糊总评

### 5.2 协议资产规则

- 当前结论：结果包必须建立在已冻结的数据协议、参数协议、评估协议、外部对比公平协议、`CRAG` 角色协议和各阶段 `stage summary` 之上
- 规则类型：`来源边界规则`
- 适用阶段：`10_结果汇总` 末端
- 直接依据：`结直肠腺体分割_plan_优化版/01_实验执行/00_总览与规范/02_参数冻结总表.md`，`结直肠腺体分割_plan_优化版/01_实验执行/00_总览与规范/04_评估口径与官方脚本对齐.md`，`结直肠腺体分割_plan_优化版/01_实验执行/10_结果汇总/04_stage_summary模板.md`
- 核心公式或定义参考：`protocol_assets = frozen_protocols + stage_pass + stage_summaries + traceability`

### 6.3 三档状态的唯一合法判定

固定判定如下：

- `ready`：五类资产全部达到正式可用状态，没有阻断性缺口，下游总验收可以直接消费。
- `ready_with_minor_gaps`：主线、主表、关键图像、关键来源边界和结论边界已齐，但仍有少量补充图、补充说明或次级索引待补，且这些缺口不会改变主裁决。
- `not_ready`：仍存在关键协议、关键表、关键图、关键来源边界或关键结论缺失，或者缺口尚未被诚实定级。

补充要求：

- `ready_with_minor_gaps` 不是“有明显大问题但先推进”的缓冲区。
- 只要缺口会改变主表、主线、`CRAG` 结论或总验收判断，就必须落到 `not_ready`。

### 6.4 六个最终自查问题的固定问法

在准备进入 `11_总验收与止损` 前，必须逐条自问并留痕：

1. 每个主结果数字都能追到具体 run 吗。
2. 每个模块保留或删除都有明确证据吗。
3. `GlaS` 和 `CRAG` 的角色有混淆吗。
4. 外部方法的公平性边界写清了吗。
5. 失败案例足够支撑局限性表述吗。
6. 有没有任何结果其实还属于内部参考值却被放进正式表。

这六条只要有一条无法明确回答，就不得把结果包升到 `ready`。

### 6.5 固定落盘位置

正式结果包检查与被引用资产一律以下面路径为准：

- submission_package_check.md
- reports/tables/
- reports/figures/
- `crc_gland_segmentation_project/reports/stage_reports/`
- notes/failure_case_notes.md

如果历史目录中还有 tables/、figures/、summaries/ 等旧路径，允许兼容保留，但正式结果包和下游总验收消费应优先映射到 reports/* 规范路径。

### 6.6 高频漏写项

结果包层最容易漏掉的不是标题，而是：

- 主表数字是否真的可回指 run 和 summary。
- single_seed / mean +- std 是否在表和文字里一致。
- 关键图是否同时具备原图、真值、预测和样本 ID。
- `CRAG` 结论到底是 A / B / C 哪一档，允许主张是什么。
- 五个评估实现硬字段是否仍可直接从结果包层追到正式值，而不是只剩抽象 `eval_proto_version`。
- 当前缺口到底是轻微补充项还是阻断性问题。
- 哪些资产可以直接给总验收读，哪些还不能直接用。

这些字段一旦漏写，下游总验收就会整体返工。

### 8.1 独立回退触发条件

4. 再检查来源边界、公平性边界和结论边界是否一致。
5. 最后检查结果包 handoff 是否足以让下游总验收直接消费。
