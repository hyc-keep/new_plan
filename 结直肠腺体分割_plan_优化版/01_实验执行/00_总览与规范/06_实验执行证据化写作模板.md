# 实验执行证据化写作模板

## 1. 文件角色与执行边界

- 本文件负责定义 `01_实验执行` 各类协议文档的证据化写法、规则卡片字段和正文落点。
- 本文件不替代任何单阶段的正式结论，也不负责替代阶段验收或总路线锁定。

## 2. 本轮直接依赖的前置文件

- 直接依赖 `结直肠腺体分割_plan_优化版/01_实验执行/00_总览与规范/00_执行导航.md`、`结直肠腺体分割_plan_优化版/01_实验执行/00_总览与规范/03_命名与结果记录规范.md`、`结直肠腺体分割_plan_优化版/01_实验执行/00_总览与规范/04_评估口径与官方脚本对齐.md`。
- 上游证据来源固定来自 `结直肠腺体分割_plan_优化版/02_路线与投稿/` 和 `结直肠腺体分割_plan_优化版/03_文献证据/`。

## 3. 本阶段唯一允许处理的变量

- 只允许规定“证据如何写进文档”，不允许在模板层直接宣告某个实验阶段已经通过。

## 4. 阶段门控表达式

- 后续阶段文档必须能依据本文件形成可回链的依据、代码落点、运行记录字段和验收方式，本模板才算有效。

这份文件的作用不是再写一版“实验报告模板”，而是把 `01_实验执行` 下面所有协议文档统一改造成：

- 可追溯到 `02_路线与投稿`
- 可追溯到 `03_文献证据`
- 能直接指导代码设计
- 能直接服务后续论文写作

的最终执行模板。

也就是说，从这份文件开始，后面任何实验执行文档都不允许只写：

- 模糊口吻
- 经验性表述
- 大概这样跑

而必须写成：

- 为什么这样定
- 依据来自哪些论文
- 哪些值是直接采用，哪些只是候选范围
- 代码里应该落到哪里
- 如果结果不对，优先怀疑什么

## 1. 这份模板解决什么问题

当前 `01_实验执行` 最容易失真的地方，不是目录不清，而是下面几类风险：

1. 有路线，没有证据链
2. 有默认值，但说不清出处
3. 有文献提取，但没有回填到代码协议
4. 有实验阶段，但没有把“允许改什么 / 不允许改什么”压成唯一规则
5. 后面能写代码，但写不出“为什么这么写”

因此，后面所有执行文档都要改成统一口径：

> 每一条实验规则，必须能回答“来源是什么、为什么采用、代码怎么落、结果怎么验、失败怎么退”。

## 2. 证据来源优先级

后面写 `01_实验执行` 时，证据来源只允许来自下面四层，而且优先级固定。

### 2.1 第一层：官方协议 / benchmark 原始规则

适用内容：

- 数据集官方划分
- 官方评价指标
- benchmark 角色
- 不可篡改的测试边界

优先来源示例：

- `结直肠腺体分割_plan_优化版/03_文献证据/05_腺体任务经典论文/01_GlaS-Challenge.md`
- `结直肠腺体分割_plan_优化版/03_文献证据/05_腺体任务经典论文/04_MILD-Net.md`

这类内容一旦确认，执行层应写成：

- `官方协议固定项`

而不是：

- `模糊口吻`

### 2.2 第二层：路线锁定层

适用内容：

- 主线方法顺序
- baseline 选择
- 模块先后关系
- 投稿主表优先级
- 止损边界

优先来源示例：

- `结直肠腺体分割_plan_优化版/02_路线与投稿/01_结直肠腺体分割_高性价比路线总锁定.md`
- `结直肠腺体分割_plan_优化版/02_路线与投稿/02_结直肠腺体分割_分阶段实验路线与执行标准.md`
- `结直肠腺体分割_plan_优化版/02_路线与投稿/03_结直肠腺体分割_投稿层级自查与止损判断.md`

这类内容在执行层应写成：

- `路线层已锁定`

### 2.3 第三层：文献参数与方法依据层

适用内容：

- 输入尺寸候选
- loss 选择依据
- 大核模块的插入位置与核大小范围
- 边界分支的结构和监督形式
- 距离图或拓扑图的生成依据
- 外部对比的公平适配范围

优先来源示例：

- `03_文献证据/01_经典基线与对比方法/*.md`
- `03_文献证据/02_边界_形状_损失支撑/*.md`
- `03_文献证据/03_大核_感受野_形态建模/*.md`
- `03_文献证据/05_腺体任务经典论文/*.md`

这类内容在执行层不能直接写成绝对真理，而应区分：

- `论文直接支持`
- `论文支持的候选范围`
- `仅作工程起点`

### 2.4 第四层：工程冻结层

适用内容：

- 命名规范
- 目录结构
- 日志落盘格式
- checkpoint 选择规则
- threshold 固定规则
- 配置文件字段
- 可视化导出模板

这一层多数情况下不是某一篇论文直接给出，而是：

- 在前面三层证据允许的范围内
- 为了保证公平比较、可复现和代码一致性
- 由项目统一冻结

这类内容必须显式标记为：

- `工程冻结规则`

不能伪装成：

- `论文原始设置`

## 3. 每条规则必须附带的 8 个字段

后面任何执行文档，只要出现“一个规则”，都必须附带下面 8 个字段。

### 3.1 规则最小卡片

```md
### 规则名

- 当前结论：
- 规则类型：
- 适用阶段：
- 直接依据：
- 采用原因：
- 不采用的相邻方案：
- 代码落点：
- 运行记录字段：
- 验收方式：
```

如果该规则涉及 `loss / metric / target / postprocess` 这类带公式或明确定义边界的对象，还必须额外补一行：

```md
- 核心公式或定义参考：
```

### 3.2 字段解释

#### `当前结论`

只写最后真正执行的规则，例如：

- `输入尺寸固定为 512x512`
- `best checkpoint 统一按 val 主指标选择`
- `主增强包固定为 main_aug_v1`

禁止写法：

- `模糊候选表述`
- `经验性描述`
- `大概先这样`

#### `规则类型`

只允许使用下面四类：

- `官方协议固定项`
- `路线层已锁定`
- `论文支持的候选范围`
- `工程冻结规则`

如果是候选范围，必须继续写清：

- 当前最终选哪一个
- 其余候选为什么没选

#### `适用阶段`

明确写清：

- 从哪个阶段开始生效
- 到哪个阶段前不得改动

例如：

- `UNet 首轮后冻结，后续 baseline 和模块阶段必须继承`

#### `直接依据`

这里必须写具体来源，优先到文件级，必要时写到论文简称级。

固定写法：

- 路线依据：`结直肠腺体分割_plan_优化版/02_路线与投稿/02_结直肠腺体分割_分阶段实验路线与执行标准.md`
- 文献依据：01_GlaS-Challenge.md, 04_MILD-Net.md
- 模块依据：01_Large-Kernel-Matters.md, 01_Gated-SCNN.md

如果某条规则没有证据来源，就不能直接进最终执行文档。

#### `采用原因`

必须说明这条规则解决什么问题，而不是只报值。

例如：

- `512x512` 的采用原因不是“常见”，而是“兼顾 gland 全局形态上下文、单卡可训练性和后续外部对比的统一输入口径”

#### `不采用的相邻方案`

这里要把最容易混淆的替代项写出来，防止后面代码漂移。

例如：

- 不采用 `best checkpoint 按 val loss 或 val Dice 二选一`
- 不采用 `每个模型自己单独调 threshold`
- 不采用 `边界分支和距离损失同时一起搜`

#### `代码落点`

这是最关键的新增字段。

必须明确这条规则最后应落到哪类代码或配置里，例如：

- `configs/train/*.yaml`
- `configs/data/*.yaml`
- src/eval/checkpoint_selector.py
- src/data/transforms.py
- src/losses/boundary_losses.py
- src/metrics/object_metrics.py

如果不能落到代码位置，这条规则大概率还不够具体。

#### `运行记录字段`

凡是会进入正式 run、正式评估或正式汇总层的规则，都必须额外写清后续记录到哪些字段，例如：

- config.yaml
- run_meta.yaml
- `val_metrics.csv / testA_metrics.csv / testB_metrics.csv / test_metrics.csv`
- `reports/tables/*.csv`

常见字段示例：

- `config_version`
- `train_proto_version`
- `eval_proto_version`
- `model_version`
- `loss_version`
- `best_selector`
- `threshold_source`
- `threshold_value`
- `result_tag`
- `aggregation`

如果一条规则进入了代码和结果层，却没有对应的运行记录字段，它通常还不算真正冻结。

#### `验收方式`

必须写清后续如何确认这条规则真正被执行，而不是只存在于文字里。

例如：

- `检查 config 导出是否记录 input_size=512x512`
- `检查所有 run 是否统一生成 best.ckpt`
- `检查 val 选择逻辑是否只有一个入口函数`
- `检查 testA/testB 是否使用同一 threshold`

## 4. 参数类规则的标准写法

后面凡是参数、结构、训练、评估、可视化相关的内容，都必须按下面模板写。

### 4.1 参数冻结模板

```md
## 参数名

- 当前结论：
- 当前值或最终版本：
- 候选范围：
- 规则类型：
- 直接依据：
- 采用原因：
- 为什么不选其它候选：
- 生效阶段：
- 冻结阶段：
- 代码落点：
- 运行记录字段：
- 验收方式：
```

### 4.2 示例一：输入尺寸

```md
## 输入尺寸

- 当前结论：正式主线统一使用 `512x512`
- 当前值或最终版本：`512x512`
- 候选范围：`416x416 / 480x480 / 512x512`
- 规则类型：`论文支持的候选范围 + 工程冻结规则`
- 直接依据：`结直肠腺体分割_plan_优化版/02_路线与投稿/02_结直肠腺体分割_分阶段实验路线与执行标准.md`，07_TA-Net.md，03_DCAN.md
- 采用原因：保证 gland 全局形态可见，同时控制显存与外部对比公平性
- 为什么不选其它候选：`416x416` 更偏特定论文设置，`480x480` 更接近早期 DCAN，不利于统一后续主线
- 生效阶段：`UNet 首轮`
- 冻结阶段：`UNet 跑通后冻结`
- 代码落点：`configs/data/*.yaml`，src/data/transforms.py
- 运行记录字段：`input_size`
- 验收方式：检查导出的 config.yaml 与推理日志是否全部一致
```

### 4.3 示例二：边界分支

```md
## 边界分支

- 当前结论：先采用单边界分支，接在最终 decoder 输出侧
- 当前值或最终版本：`single_boundary_head_v1`
- 候选范围：`单分支 / 多级分支`
- 规则类型：`论文支持的候选范围`
- 直接依据：03_DCAN.md，01_Gated-SCNN.md，12_DEA-Net.md
- 采用原因：任务痛点集中在黏连腺体分离和边界模糊，单分支更易公平验证
- 为什么不选其它候选：多级边界分支会把结构变量和监督变量混在一起
- 生效阶段：`Boundary`
- 冻结阶段：`Boundary 阶段结论形成后`
- 代码落点：src/models/heads/boundary_head.py，src/losses/boundary_losses.py
- 运行记录字段：`boundary_head_version`, `boundary_width`, `lambda_boundary`
- 验收方式：检查可视化中是否导出 boundary map，检查 loss 日志是否单独记录边界项
```

### 4.4 示例三：含数学公式的规则卡片

下面给一个可以直接照抄到后续阶段文档里的示例。重点不是公式写得多复杂，而是要把：

- 采用的对象
- 公式定义
- 对应代码位置
- 运行记录字段

一次写全。

```md
## 主 loss

- 当前结论：`L_seg = L_BCE + L_Dice`
- 当前值或最终版本：`seg_loss_v1`
- 候选范围：`BCE + Dice`、`BCE`、`Dice`、`Focal + Dice`
- 规则类型：`论文支持的候选范围 + 工程冻结规则`
- 直接依据：`结直肠腺体分割_plan_优化版/02_路线与投稿/02_结直肠腺体分割_分阶段实验路线与执行标准.md`，02_Loss-Survey.md，04_UNet++.md，10_SCAU-Net.md
- 核心公式或定义参考：`L_BCE = - 1/N * sum_i [y_i log(p_i) + (1-y_i) log(1-p_i)]`；`L_Dice = 1 - (2 * sum_i p_i y_i + eps) / (sum_i p_i + sum_i y_i + eps)`；`L_seg = L_BCE + L_Dice`
- 采用原因：首轮需要一个数值稳定、实现直接、与腺体任务文献常见组合兼容的区域主损失
- 为什么不选其它候选：单独 `BCE` 对小腺体重叠趋势不够敏感；单独 `Dice` 在早期训练更容易受概率分布波动影响；`Focal + Dice` 会引入新的聚焦系数变量
- 生效阶段：`UNet` 首轮
- 冻结阶段：`UNet` 跑通后冻结到 baseline 阶段
- 代码落点：src/losses/seg_losses.py，`configs/train/*.yaml`，`configs/experiment/*.yaml`
- 运行记录字段：`loss_name`, `loss_version`, `bce_weight`, `dice_weight`
- 验收方式：检查训练日志是否分别记录 `loss_total / loss_bce / loss_dice`，并确认推理阶段没有重复做 sigmoid 后再算训练损失
```

## 5. 阶段类文档的标准写法

后面每个阶段文档都必须回答下面 6 个问题，不能只写流程顺序。

### 5.1 阶段文档最小骨架

```md
# 阶段名

## 1. 任务定位
- 本阶段要证明什么
- 为什么此时做这一步
- 它在整条路线中的角色

## 2. 上游继承
- 继承哪些已冻结规则
- 哪些内容禁止重新打开

## 3. 本阶段唯一变量
- 这一步只允许改什么
- 为什么只改这个变量

## 4. 本阶段规则卡片
- 参数
- 结构
- loss
- 评估
- 可视化

## 5. 代码实现约束
- 需要新增哪些模块
- 需要复用哪些已有模块
- 需要记录哪些配置字段

## 6. 结果验收与回退
- 最低产物
- 过线标准
- 回退触发条件
```

### 5.2 阶段文档必须新增的“代码实现约束”

这是之前最缺的一节。

后面每个阶段文档都必须单独写：

- 本阶段新写哪些代码文件
- 哪些只能扩展，不能重写
- 哪些公共模块必须保持统一
- 哪些配置字段必须新增

固定写法：

```md
## 代码实现约束

- 本阶段新增：
  - src/models/modules/lkma.py
  - configs/model/lkma_v1.yaml
- 本阶段复用：
  - src/engine/trainer.py
  - src/metrics/object_metrics.py
- 本阶段禁止修改：
  - `splits/glas/*.csv`
  - `splits/crag/*.csv`
  - src/eval/checkpoint_selector.py
- 本阶段必须新增记录字段：
  - `module_version`
  - `kernel_size`
  - `insertion_position`
```

## 6. 命名、配置、结果记录必须脱离自然语言

后面一切会进入代码的内容，都不能只靠自然语言描述，必须进入“版本化对象”。

### 6.1 必须版本化的对象

- 增强包：`light_aug_v1`, `main_aug_v1`
- 训练协议：`train_proto_v1`
- 评估协议：`eval_proto_v1`
- 边界监督：`boundary_proto_v1`
- 距离约束：`dist_proto_v1`
- 主模型：`model_proto_v1`

### 6.2 为什么必须版本化

因为后面如果不版本化，就会出现：

- 文档里写“主增强包”
- 代码里写三套不同 transform
- 结果表里又只写 `main_aug`

最后没人知道到底跑的是哪一版。

### 6.3 执行文档里正确写法

固定写法：

```md
- 增强版本：`main_aug_v1`
- 训练协议版本：`train_proto_v1`
- 评估协议版本：`eval_proto_v1`
- 模型版本：`r34unet_lkma_boundary_v1`
```

而不是：

- `采用主增强包`
- `采用标准训练协议`

这种描述对代码没有约束力。

## 7. 后续文件改造检查清单

后面每改一份执行文档，至少逐项检查下面内容；只有大部分都打勾，这份文档才算从“说明文”升级成“执行协议”。

### 7.1 快速检查清单

```md
## 改造完成前检查

- [ ] 文档开头已写清本阶段要证明什么，而不是只写“做什么”
- [ ] 每条核心规则都已标注 `规则类型`
- [ ] 每条核心规则都已写 `直接依据`，并且至少能回指到 `02_路线与投稿` 或 `03_文献证据`
- [ ] 参数、loss、metric、target、postprocess 规则都补了 `核心公式或定义参考`
- [ ] 每条核心规则都写了明确 `代码落点`
- [ ] 每条核心规则都写了 `验收方式`
- [ ] 已区分“论文直接支持”和“工程冻结规则”，没有把工程默认值伪装成论文原值
- [ ] 已写清本阶段唯一变量，避免同时改多个因素
```

### 7.2 一票否决项

- 写了模糊口吻，却没有冻结边界
