# 03_实验步骤

本文件不是给 `Distance-aware Loss v1` 补一份“实验大概怎么跑”的说明，也不是把旧版执行链后面再追加几条注意事项。

它在当前项目中的唯一职责是：

> 作为 `07_Distance` 的正式 execution protocol，把 `current_base` 启动条件、`screening_main -> three_seed_main -> optional_one_backup -> stage_decision` 的唯一合法执行顺序、主版本优先门、run 级动作、产物字段链、异常停止与回退顺序、以及对 `结直肠腺体分割_plan_优化版\01_实验执行\07_Distance\04_保留或降级标准.md` 和 `08/09` 下游的交接前提一次写死。

从现在开始，只要讨论 `Distance` 阶段正式 run 怎样启动、怎样补三 seed、什么时候允许补一个备选 `lambda`、哪些资产必须先产出才能交给裁决文件、以及下游什么时候才允许读取 `current_mainline`，都必须以本文件为准；若与同层文件冲突，以 `结直肠腺体分割_plan_优化版/01_实验执行/07_Distance/00_阶段总协议.md` 管阶段身份与 `Gate_E1`，以 `结直肠腺体分割_plan_优化版/01_实验执行/07_Distance/01_设计依据.md` 管“为什么当前只能这样执行”，以 `结直肠腺体分割_plan_优化版/01_实验执行/07_Distance/02_距离图与lambda范围.md` 管 target 与 `lambda` 边界，而本文件负责把这些上位约束压成可直接执行的运行链。

## 1. 文件角色与执行边界

### 1.1 当前文件负责什么

当前文件只负责冻结下面九件事：

1. `Distance` 阶段从哪个已冻结基座启动，以及什么情况下根本不允许开跑。
2. 正式执行链为什么固定为 `screening_main -> three_seed_main -> optional_one_backup -> stage_decision`。
3. 每个节点的触发条件、输入前提、中止条件、标准动作和最低产物。
4. 主版本 `lambda_dist = 0.1` 为什么必须先走完整主链，备选版本为什么只能承担补充辨析职责。
5. 每个正式 run 必须完成哪些训练、验证、测试、可视化和成本动作，什么情况不计入正式 run。
6. 阶段汇总前必须准备哪些 per-seed、`mean+-std`、对照表、案例图和成本资产。
7. 执行阶段必须怎样保留字段链，才能供 `结直肠腺体分割_plan_优化版\01_实验执行\07_Distance\04_保留或降级标准.md`、`08_外部对比` 和 `09_CRAG验证` 直接消费。
8. 出现 bug、协议漂移、证据不清或收益不足时，什么情况必须先回退而不是继续扩张。
9. 本文件怎样作为 `结直肠腺体分割_plan_优化版/01_实验执行/07_Distance/04_保留或降级标准.md` 的直接执行上游，以及 `08/09` 的 handoff 前置。

### 1.3 为什么当前文件必须独立存在

因为 `Distance` 阶段最容易出现的伪完成状态不是“完全没跑”，而是：

- target protocol 明明已经冻结，执行链却仍允许一边跑一边改 `lambda_dist` 或边界阶段配置。
- screening run 已经跑通，却没有把三 seed 主版本、per-seed 表、`mean+-std`、案例图和成本记录补齐。
- 备选值单次结果看起来更好，就直接替代主版本成为阶段结论来源。
- 阶段总结想交给下游时，说不清哪些 run 是正式主版本，哪些只是 diagnostic backup。
- `08_外部对比` 和 `09_CRAG验证` 想读取 `current_mainline` 时，发现 `Distance` 只留下“做过一些实验”，却没有明确哪个执行链版本通过了阶段门。

如果没有一份独立执行协议文件，`结直肠腺体分割_plan_优化版/01_实验执行/07_Distance/00_阶段总协议.md` 会被迫同时承担“阶段身份”和“节点动作说明”，`结直肠腺体分割_plan_优化版/01_实验执行/07_Distance/04_保留或降级标准.md` 也会被迫补写执行前提，最后执行、裁决与交接三条链都会重新分叉。

## 3. 本阶段唯一允许处理的变量

### 3.1 当前文件真正要完成什么

本文件真正要完成的不是“把 `Distance` 跑一遍”，而是把下面六件事一次执行清楚：

1. 先确认 `current_base` 和所有上游冻结项都已就位，再启动正式 run。
2. 先用主版本单 seed screening 验证 target、loss、评估和资产链没有 bug。
3. screening 通过后，再把主版本三 seed 结果补齐成正式比较对象。
4. 只有在主版本三 seed 结果接近可用但证据不清时，才允许最多补一个备选 `lambda`。
5. 在进入 `结直肠腺体分割_plan_优化版\01_实验执行\07_Distance\04_保留或降级标准.md` 之前，先把 per-seed raw、`mean+-std`、对照表、案例图和成本表准备齐。
6. 若阶段执行中出现 bug、协议漂移或收益不足，先按固定顺序回退，而不是一边扩变量一边继续跑。

### 3.2 为什么执行链必须固定

`Distance` 阶段最容易发生的错误不是“实验没做”，而是：

- 主版本还没走完三 seed，就因为局部案例不错而提前写保留倾向。
- screening 已经发现 target 或 loss 有问题，却仍继续把剩余 seed 跑完。
- 主版本不过线，就马上同时补 `0.05` 和 `0.2`，把执行链变成搜索矩阵。
- 备选版本做成新的并行主实验，反过来替代主版本的统计身份。

因此，执行顺序只能固定为：

```text
screening_main -> three_seed_main -> optional_one_backup -> stage_decision
```

任何跳步、并行多备选、主版本与备选混算、或在执行过程中回改 `current_base` 的做法，均视为违反协议。

### 3.3 当前执行文件与 `01/02/04` 的分工

四份同层文件的固定分工是：

- `结直肠腺体分割_plan_优化版/01_实验执行/07_Distance/01_设计依据.md`：回答“为什么当前只能这样执行”。
- `结直肠腺体分割_plan_优化版/01_实验执行/07_Distance/02_距离图与lambda范围.md`：回答“距离图与 `lambda_dist` 究竟怎样冻结”。
- 本文件：回答“正式 run 应该按什么顺序、什么门槛、什么资产链执行”。
- `结直肠腺体分割_plan_优化版/01_实验执行/07_Distance/04_保留或降级标准.md`：回答“执行完以后怎样排序证据并形成阶段结论”。

如果把这四类职责混写，就会同时丢失两种清晰性：

- 上游文件说不清哪些是执行前提，哪些是执行动作。
- 下游文件说不清哪些结果是正式主版本资产，哪些只是 screening 或 backup diagnostic。

当前阶段的比较基座不是重新选择，而是由 `Boundary` 阶段正式结论唯一决定：

```text
current_base = (boundary_decision_level == keep) ? boundary_kept_base : boundary_input_base
```

这意味着：

- 只有 `06_Boundary` 已形成正式结论并完成 handoff，`Distance` 才允许启动。
- 任何正式 `Distance` run 都必须能追溯到 `Boundary` 的 decision note、stage manifest 和 `current_base` 说明。
- 如果 `Boundary` 仍未收口，当前文件不能“先跑 Distance 再补上游说明”。
- `Boundary` handoff 中的 `current_base_next` 必须与本文件使用的 `current_base` 一致；执行链不接受任何手工改写起点或跳过上游正式交接的 run。

### 4.3 当前执行阶段唯一允许变化的变量

当前执行阶段唯一允许变化的变量固定为：

- 在 `current_base` 上追加 `Distance-aware Loss v1`
- 其正式表达固定为：

```text
L_total = L_seg + lambda_dist * L_dist
```

- 主版本固定为：

```text
distance_type = euclidean
distance_norm = norm01
distance_value_range = [0, 1]
lambda_dist = 0.1
```

### 4.4 当前执行阶段明确不允许重开的内容

本文件不允许借执行说明之名重新打开：

- baseline backbone、`LKMA` 配置和 `Boundary` 标签协议
- 数据协议、输入尺寸、增强包、主损失、optimizer、scheduler
- checkpoint selector、阈值来源和后处理定义
- `signed distance map`
- `clDice`
- `TA-Net`
- `SkeletonAwareDT`
- `direction field`
- `marker branch`
- `watershed` 恢复链
- 多距离图定义乘多 `lambda` 的联合搜索

换句话说，这里只负责“距离项怎样按唯一变量执行”，不负责把 `Distance` 写成新的表示路线或搜索阶段。

## 6. 节点执行细则

### 6.1 节点一：`screening_main`

#### 输入前提

- `current_base` 已冻结并有正式来源说明。
- 距离图构建、归一化和 loss 接入代码已完成最小实现。
- 主版本固定为 `current_base + DistanceLoss(euclidean_norm01, lambda=0.1)`。
- 首个种子固定为 `seed3407`。

#### 标准动作

1. 训练主版本 `seed3407`。
2. 按 `val_objdice_max` 选择 `best.ckpt`。
3. 保留 `last.ckpt`。
4. 分别执行 `TestA` 与 `TestB`。
5. 导出对象级指标、边界级指标、局部轮廓可视化和成本记录。
6. 记录 `distance_type / distance_norm / lambda_dist / distance_target_version / eval_cast_policy / boundary_metric_width / boundary_metric_impl / connected_components_impl / connected_components_connectivity` 到 run_meta.yaml。

#### 节点目标

该节点只验证：

- 距离图来源、值域和 loss 组合是否正确。
- 距离项没有导致训练崩溃或显著失稳。
- 指标和可视化没有出现系统性异常退化。

#### 中止条件

出现以下任一情况，必须停止进入下一节点：

- 距离图全零、常数化、范围不在 `[0, 1]` 或前景定义错误。
- `loss_dist` 或 `loss_total` 数值异常、梯度爆炸、训练不收敛。
- `TestA / TestB` 或边界指标输出链断裂。
- 相对 `current_base` 出现明显不可接受退化，且可视化也不支持继续投入。

#### 最低产物

- `E1_[model_code]_GlaS_v1_seed3407/`
- config.yaml
- run_meta.yaml
- train_log.csv
- val_metrics.csv
- testA_metrics.csv
- testB_metrics.csv
- boundary_metrics.csv
- per-seed raw 中对齐记录的 `eval_cast_policy / boundary_metric_width / boundary_metric_impl / connected_components_impl / connected_components_connectivity`

- 单次成本记录

### 6.3 节点三：`optional_one_backup`

#### 触发条件

只有主版本三 seed 结果出现“结论接近可用但仍不清”时，才允许进入。这里的“不清”只包括：

- 改善幅度接近波动范围，无法稳定判断。
- 边界类指标与对象类指标方向不一致。
- 局部案例存在改善，但总体数值证据不足。

- `lambda_dist = 0.05`
- 或 `lambda_dist = 0.2`

选择规则固定为：

- 若怀疑主版本监督偏强，优先考虑 `0.05`
- 若怀疑主版本监督偏弱，优先考虑 `0.2`

- 不允许同时补两个备选值。
- 不允许改距离图定义后再补 `lambda`。
- 不允许改模型结构、数据增强、优化器或边界配置。
- 不允许把备选版本默认升级为新的并行主实验。

#### 最低产物

- 备选版本 `seed3407` screening 目录
- 若负责人明确要求，才允许继续补齐备选三 seed

- 阶段汇总中显式记录 `backup_reason`

## 7. 回退条件

### 7.1 独立回退触发条件

只要出现下面任意一条，本文件负责的执行协议就必须先回退修正，而不是继续推进 07_Distance/04、`08_外部对比` 或 `09_CRAG验证`：
