# 阶段验收

## 计划 lineage 与下游 handoff

当前/未来阶段验收必须记录以下字段：`source_stage`、`source_manifest`、`source_protocol_version`、`source_run_name`、`consumer_stage`、`consumer_file`、`consumption_boundary`。缺失任一字段、当前轮 `eval_proto_version/postprocess_version/tta_version` 未经研究定标/阶段锁定/Pre-check 后 Gate 冻结、或计划文件/config/run_name/run_meta/manifest/gate/handoff 命名不同步时，验收 Gate 必须 blocked，回到治理/阶段锁定，不得编码或向下游交接。

本文件不是 baseline 阶段的收尾备注，也不是对“`ResNet34-U-Net` 看起来已经能用”做一段经验性总结。

它在当前项目中的唯一职责是：

> 作为 `04_Baseline` 阶段的正式放行与交接 protocol，把 `ResNet34-U-Net` 何时才算成为全文唯一正式 baseline、何时必须阻断、通过后必须冻结什么、必须交接什么、以及这些条件如何落到阶段门控和 handoff 资产上，一次写死。

从现在开始，只要讨论 baseline 阶段“能不能进入 `05_LKMA`、后续模块应该继承什么、哪些 baseline 结果才算正式资产”，都必须以本文件为准；若与同层其他文件冲突，以 `结直肠腺体分割_plan_优化版/01_实验执行/04_Baseline/00_阶段总协议.md` 为总口径，本文件负责把阶段验收细化到可直接执行、可直接门控、可直接交接的程度。

## 1. 文件角色与执行边界

### 1.1 当前文件负责什么

当前文件只负责冻结下面六件事：

1. baseline 阶段总放行门 `Gate_B1` 的正式定义及其五个子门。
2. baseline 阶段何时通过、何时阻断、何时必须独立回退。
3. baseline 阶段最低交付物、正式资产和阶段 handoff manifest。
4. baseline 通过后必须被下游继承的冻结边界。
5. baseline 不通过时应该如何标记 blocker、作废聚合资格并阻断进入 `05_LKMA`。
6. baseline 阶段总结、代码入口和论文写作可直接消费的正式结论模板。

### 1.2 当前文件不负责什么

当前文件不重新定义下面这些内容：

- `ResNet34-U-Net` 的结构身份、encoder 初始化来源和 skip 边界。
- `train_proto_v1 / eval_proto_v1` 的完整训练与评估 protocol。
- `UNet vs R34UNet` 的公平比较前提、统计身份和 `Gate_B1_compare` 细则。
- `LKMA`、`Boundary Head`、`Distance-aware Loss` 等后续模块本身的设计。

这些职责分别以下列文件为准：

- `结直肠腺体分割_plan_优化版/01_实验执行/04_Baseline/00_阶段总协议.md`
- `结直肠腺体分割_plan_优化版/01_实验执行/04_Baseline/01_R34UNet结构与来源.md`
- `结直肠腺体分割_plan_优化版/01_实验执行/04_Baseline/02_训练协议.md`
- `结直肠腺体分割_plan_优化版/01_实验执行/04_Baseline/03_对比与判断规则.md`
- `结直肠腺体分割_plan_优化版/01_实验执行/05_LKMA/00_阶段总协议.md`
- `结直肠腺体分割_plan_优化版/01_实验执行/06_Boundary/00_阶段总协议.md`
- `结直肠腺体分割_plan_优化版/01_实验执行/07_Distance/00_阶段总协议.md`

### 1.3 为什么当前文件必须独立存在

因为 baseline 阶段最容易出现的伪完成状态不是“没有结果”，而是：

- `Gate_B1_compare` 看起来成立，但三次正式 run 资产并不完整。
- 聚合表已经导出，但冻结边界还没有写死。
- 阶段总结已经写完，但 handoff 资产和字段链还不能被下游直接消费。
- baseline 结果看起来不差，却仍说不清下游到底该继承哪一版结构、初始化、batch 策略和比较口径。

如果没有一份独立阶段验收文件，后续 `05_LKMA`、阶段汇总脚本和论文写作都会再次出现“baseline 已通过，但到底按哪套门控、哪些资产和哪些字段放行”不一致的问题。

## 2. 核心验收问题

### 3.1 当前阶段真正要验收什么

`04_Baseline` 阶段真正要验收的，不是“`ResNet34-U-Net` 接上了能跑”，也不是“某次 seed 看起来比 `UNet` 高一点”，而是：

> 在与 `UNet` 完全一致的数据、训练和评估 protocol 下，`ResNet34-U-Net` 是否已经形成稳定、可比较、可解释、可交接、足以承担后续 `LKMA / Boundary / Distance` 统一比较基座角色的正式 baseline。

因此，本阶段通过的含义同时包括：

- `ResNet34-U-Net` 已经不再只是候选结构，而是后续主线唯一正式 baseline。
- baseline 的阶段结论来自三 seed 聚合证据，而不是单次峰值。
- baseline 的正式资产、冻结字段和 handoff manifest 已经齐全到可被下游直接消费。

### 3.2 当前阶段不允许用什么替代正式验收

当前明确不允许把下面这些情况写成“baseline 已放行”：

- 只看 `Gate_B1_compare` 成立，不检查三次正式 run 和资产闭环。
- 只导出 `mean+-std` 聚合表，不保留 per-seed raw results 和异常 seed 说明。
- 只写均值高于 `UNet`，不写 `TestB`、可视化、失败案例和冻结边界。
- 只凭阶段总结文字放行，不输出交接 manifest 和路径可追溯资产。

### 3.3 为什么阶段验收必须拆成五个子门

baseline 最容易发生的错误是把“结果看起来不差”误写成“正式基座已经成立”。

因此，当前阶段必须把通过条件拆成：

- 比较门：负责确认 baseline 结论是否真实建立在公平比较之上。
- 完整运行门：负责确认三次正式 run 和最低资产是否齐全。
- 资产门：负责确认结果表、阶段总结、案例和 manifest 是否闭环。
- 冻结门：负责确认下游要继承的结构、初始化和 batch / 阈值口径已经写死。
- 交接门：负责确认 `05_LKMA` 能否直接消费 baseline 资产，而不是进入下游后再补 baseline 证据。

## 4. 红线条件

### 4.1 当前阶段直接继承哪些冻结项

本文件直接继承并不得擅自改动的上位规则如下：

- 主任务：`2D gland segmentation`
- 主数据集：`GlaS`
- 正式 split：`train68 / val17 / testA60 / testB20`
- 标签口径：`mask > 0`
- 输入通道：`RGB`
- 输入尺寸：`512 x 512`
- 归一化：`ImageNet mean/std`
- 增强版本：`light_aug_v1`
- 主损失：`L_seg = L_BCE + L_Dice`
- 优化协议：`AdamW + lr 1e-3 + weight_decay 1e-4 + ReduceLROnPlateau + epoch_max 120 + early_stopping 20 + AMP on`
- 主指标优先级：`F1 / Object Dice / Object Hausdorff`
- `best_selector = val_objdice_max`
- `threshold_source = val17`
- `eval_cast_policy = logits/probabilities must be kept or cast to float32 before thresholding`
- `boundary_metric_width = 3 px`
- `boundary_metric_impl = project_custom_erosion_xor_3x3_ones_8conn_border0_tol3px`
- `connected_components_impl = scipy.ndimage.label`
- `connected_components_connectivity = 8`
- `train_proto_version = train_proto_v1`
- `eval_proto_version`：新轮需在研究定标/阶段锁定后由当前 Gate 冻结；旧 `eval_proto_v3` 仅作为上一轮历史 lineage 保留
- `postprocess_version = <由当前轮 Gate 冻结>`；当前轮待冻结时 Gate 必须阻断。
- `eval_protocol_lineage` 仅记录历史 `eval_proto_v1 -> eval_proto_v3`；当前 Gate 只消费 04 阶段当前轮冻结的 `eval_proto_version` 配对结果。
- 固定三 seed：`3407 / 1234 / 2025`

### 4.2 当前阶段禁止重新打开哪些问题

本文件禁止借阶段验收之名重新打开：

- baseline 结构身份、encoder 初始化来源和 skip 边界。
- 数据 split、标签定义、输入尺寸、增强、loss、optimizer 和 scheduler。
- `best_selector`、阈值来源、后处理、`threshold` 前 `float32` 边界、`Boundary F1` 宽度/实现和连通域实现/连接性定义。
- `UNet vs R34UNet` 的正式比较对象和统计身份。
- `LKMA`、边界分支、距离约束或其他后续模块设计。

### 4.3 `Gate_B1_compare` 与 `Gate_B1` 的职责分层

当前文件固定采用下面的职责分层：

- `Gate_B1_compare`：只回答当前轮已冻结协议下 baseline 的正式比较是否成立；上一轮 protocol_v3 结果不得直接作为当前 Gate 输入。
- `Gate_B1`：回答 baseline 阶段整体是否允许放行给 `05_LKMA`。

更准确地说：

```text
Gate_B1 = Gate_B1_compare
          AND complete_runs
          AND baseline_assets_ready
          AND freeze_ready
          AND handoff_ready
```

这意味着：

- `Gate_B1_compare` 成立是必要条件，但不是充分条件。
- 当前文件不能绕过 `结直肠腺体分割_plan_优化版\01_实验执行\04_Baseline\03_对比与判断规则.md` 另起一套更宽松的比较门。
- 当前文件也不能只复述比较门，而不补齐运行完整性、资产闭环、冻结边界和下游 handoff 条件。

## 5. 阶段通过声明

下面所有核心规则都写成正式协议格式；后续实现与阶段汇总时，只允许把这些规则忠实落地，不允许擅自补新的隐含放行条件。

### 5.1 baseline 总放行门规则

- 当前结论：当前轮 baseline 总放行门固定为 `Gate_B1 = Gate_B1_compare AND complete_runs AND baseline_assets_ready AND freeze_ready AND handoff_ready`。历史 v1/v2/v3 结果均不得直接进入当前 Gate。
- 规则类型：`阶段门控规则 + 工程冻结规则`
- 适用阶段：`04_Baseline` 结束门控，`05_LKMA` 开始时直接消费。
- 直接依据：`结直肠腺体分割_plan_优化版/01_实验执行/04_Baseline/00_阶段总协议.md`、`结直肠腺体分割_plan_优化版/01_实验执行/04_Baseline/03_对比与判断规则.md`
- 核心公式或定义参考：`Gate_B1` 必须同时满足比较门、运行完整门、资产门、冻结门和交接门。
- 采用原因：baseline 的正式通过不只取决于比较结论，还取决于是否已经形成可交接、可冻结、可复查的正式基座。

### 7.1 独立回退触发条件

- `TestB` 明显更差，且可视化不支持 baseline 作为正式基座。
- 某个 seed 明显异常却没有排查记录，或被直接删掉后强行聚合。
- 下游需要的正式资产和字段链尚未准备好，却仍试图放行。

### 7.2 固定回退顺序

回退时统一按下面顺序排查：

1. 检查 `ResNet34` encoder 接入与预训练加载是否正确。
2. 检查 skip 抽取层级、通道匹配和输出尺寸是否正确。
3. 检查 `physical_batch_size / grad_accum_steps / effective_batch_size` 是否真实一致。
4. 检查训练协议、评估协议、`best_selector` 和阈值来源是否真正继承。
5. 检查 raw 表、聚合表、比较表、阶段总结和 manifest 的字段链是否一致。
6. 只有确认前面都无问题后，才允许把差异解释为真实结构比较结果。

### 7.3 回退后的重新放行条件

发生回退后，只有同时满足下面条件，才允许重新声明 baseline 阶段通过：

- 问题来源已经按固定回退顺序定位并修复。
- 修复动作、影响范围和复验结果已写入 notes/debug_note.md、逐 run 总结或阶段总结。
- config.yaml、run_meta.yaml、结果表、可视化和 manifest 重新对齐。
- 当前阶段重新满足第 `5` 节全部规则卡片与第 `6` 节的最低交付物、冻结边界和 handoff 要求。
