# 插入位置与参数范围

本文件不是 `LKMA` 阶段里的一页参数备忘，也不是“先跑几个位置和大 kernel 看看再说”的自由搜索说明。

它在当前目录中的唯一职责是：

> 作为 `05_LKMA` 的正式参数边界 protocol，把 `LKMA` 允许插入的位置、`kernel_size` 的首轮候选、模块内部复杂度上限、额外备选何时才允许打开、以及这些边界如何落到配置字段与正式结果链，一次写死；后续 `结直肠腺体分割_plan_优化版\01_实验执行\05_LKMA\03_实验步骤.md` 只能在这里冻结的 `position / kernel / complexity / field-chain` 范围内执行，不能再私自扩成矩阵搜索。

从现在开始，只要讨论 `LKMA` 阶段“主位置是什么、备选最多开到哪里、首版大核到底允许多大、字段怎么记、什么时候必须停止扩表”，都必须以本文件为准；若与同层文件冲突，以 `结直肠腺体分割_plan_优化版/01_实验执行/05_LKMA/00_阶段总协议.md` 管阶段身份与 `Gate_C1`，以 `结直肠腺体分割_plan_优化版/01_实验执行/05_LKMA/01_设计依据.md` 管设计动机和模块职责，而本文件负责把这些上位约束压成可直接实现、可直接执行、可直接验收的参数边界。

## 1. 文件角色与执行边界

### 1.1 当前文件负责什么

当前文件只负责冻结下面七件事：

1. `LKMA` 首轮正式主位置到底是哪一个。
2. 除主位置外，最多还能打开几个备选位置，以及这些备选何时才允许使用。
3. `kernel_size` 的首轮候选范围到底收缩到哪里。
4. 为什么首轮默认主变体必须固定为 `bottleneck + k15`。
5. `LKMA v1` 内部复杂度在当前阶段允许扩到什么程度，不允许偷换成什么。
6. 正式 run 必须记录哪些结构字段、复杂度字段和命名字段。
7. 什么时候必须停止继续扩位置或扩 `kernel`，并把阶段推进到 `保留 / 备选 / 删除` 判定。

### 1.2 当前文件不负责什么

当前文件不重新定义下面这些内容：

- baseline 的正式结构身份、encoder 初始化来源和 skip 边界。
- `train_proto_v1 / eval_proto_v1` 的完整训练与评估 protocol。
- `LKMA` 为什么位于 baseline 之后、`Boundary` 之前的设计动机。
- `LKMA` 的最终执行顺序、三 seed 调度和阶段交接门控。
- `Boundary Head`、`Distance-aware Loss` 与外部对比阶段的设计。

这些职责分别以下列文件为准：

- `结直肠腺体分割_plan_优化版/01_实验执行/04_Baseline/00_阶段总协议.md`
- `结直肠腺体分割_plan_优化版/01_实验执行/04_Baseline/02_训练协议.md`
- `结直肠腺体分割_plan_优化版/01_实验执行/04_Baseline/03_对比与判断规则.md`
- `结直肠腺体分割_plan_优化版/01_实验执行/04_Baseline/04_阶段验收.md`
- `结直肠腺体分割_plan_优化版/01_实验执行/05_LKMA/00_阶段总协议.md`
- `结直肠腺体分割_plan_优化版/01_实验执行/05_LKMA/01_设计依据.md`
- `结直肠腺体分割_plan_优化版/01_实验执行/05_LKMA/03_实验步骤.md`
- `结直肠腺体分割_plan_优化版/01_实验执行/05_LKMA/04_保留或删除标准.md`
- `结直肠腺体分割_plan_优化版/01_实验执行/06_Boundary/00_阶段总协议.md`

### 1.3 为什么当前文件必须独立存在

因为 `LKMA` 阶段最容易出现的伪完成状态不是“没有参数边界”，而是：

- 口头上说只做单变量验证，实际却把位置搜索、`kernel` 搜索和模块复杂化一起打开。
- 口头上说主版本是 `bottleneck + k15`，但配置、目录命名和结果表里没有统一字段留痕。
- 一边说 `LKMA` 只是首版可插拔大核模块，一边又把 `VAN / WTConv / UniRepLKNet` 风格复杂实现偷偷塞进同名模块。
- 主版本一旦结果不够漂亮，就继续无限扩位置和扩 `kernel`，把阶段目标从“模块是否值得保留”偷换成“如何硬找提升”。

如果没有一份独立参数边界文件，后续 `结直肠腺体分割_plan_优化版\01_实验执行\05_LKMA\03_实验步骤.md`、`结直肠腺体分割_plan_优化版\01_实验执行\05_LKMA\04_保留或删除标准.md` 和 `06_Boundary` 就会各自重写一套“到底能试什么”的规则，导致 `LKMA` 阶段不再是可解释的小边界验证。

## 3. 本阶段唯一允许处理的变量

### 3.1 当前文件唯一参数对象

- 当前结论：本文件唯一参数对象是“在已正式放行的 `ResNet34-U-Net` baseline 上，`LKMA v1` 的插入位置、`kernel_size`、复杂度边界和正式字段链应该如何冻结”
- 规则类型：`工程冻结规则`
- 适用阶段：`05_LKMA`
- 直接依据：`结直肠腺体分割_plan_优化版/01_实验执行/05_LKMA/00_阶段总协议.md`、`结直肠腺体分割_plan_优化版/01_实验执行/05_LKMA/01_设计依据.md`
- 核心公式或定义参考：`variant = {insert_position, kernel_size, lkma_impl_type, use_pwconv, use_residual}`
- 采用原因：只有把搜索边界压成明确字段，后面才能保证 `baseline vs baseline + LKMA` 真的是单变量比较
- 不采用的相邻方案：不把本文件写成训练步骤文档；不把本文件写成最终保留/删除判定文档；不把本文件写成大核方法综述
- 代码落点：[project_root]/configs/model/lkma_v1.yaml、[project_root]/configs/experiment/lkma_c1_v1.yaml
- 运行记录字段：`insert_position`、`kernel_size`、`lkma_impl_type`、`use_pwconv`、`use_residual`
- 验收方式：检查后续 `结直肠腺体分割_plan_优化版\01_实验执行\05_LKMA\03_实验步骤.md` 只在这里冻结的边界内编排实验

### 3.2 当前文件直接继承哪些冻结项

本文件直接继承并不得擅自改动的上位规则如下：

- baseline 正式基座：`ResNet34-U-Net`
- baseline 唯一合法起点：B1 current_standard 的正式真实资产，消费边界为 `frozen_baseline_with_warning`；`original_gate_b1=false` 仅作历史审计事实
- `train_proto_version = train_proto_v1`
- `eval_proto_version` 由 04 阶段当前轮 Gate 冻结
- `eval_proto_lineage = eval_proto_v1 -> eval_proto_v3`（v1 仅为历史祖先）
- `eval_cast_policy = logits/probabilities must be kept or cast to float32 before thresholding`
- `boundary_metric_width = 3 px`
- `boundary_metric_impl = project_custom_erosion_xor_3x3_ones_8conn_border0_tol3px`
- `connected_components_impl = scipy.ndimage.label`
- `connected_components_connectivity = 8`
- `effective_batch_size = physical_batch_size * grad_accum_steps`
- `best_selector = val_objdice_max`
- `threshold_source = val17`
- `postprocess_version`：当前轮待由研究定标、阶段锁定、Pre-check/Gate 冻结；`none_in_v3` 仅作历史 lineage，不是当前值。
- 对象级主指标优先：`F1 / Object Dice / Object Hausdorff`
- 固定三 seed：`3407 / 1234 / 2025`

### 3.3 当前文件禁止重新打开哪些问题

本文件禁止借参数边界之名重新打开：

- 数据 split、输入尺寸、增强包、主 loss、optimizer、scheduler、阈值来源和后处理。
- baseline 的 backbone 身份、encoder 预训练策略和 `effective_batch_size`。
- `threshold` 前 `float32`、`Boundary F1 = 3 px`、`find_boundaries(mode=inner) + binary_dilation` 与 `scipy.ndimage.label` 的评估实现口径。
- `LKMA` 的任务职责定义，即它为什么先于 `Boundary` 出现。
- `Boundary Head` 和 `Distance-aware Loss` 的结构、顺序与保留逻辑。

换句话说，本文件只冻结 `position / kernel / complexity / field-chain`，不能绕过阶段母协议重开训练协议或模块职责。

## 5. 如何判断设计有效(验收标准)

### 4.1 首轮主位置

- 当前结论：`LKMA` 首轮正式主位置固定为 `bottleneck`
- 规则类型：`论文支持的候选范围 + 工程冻结规则`
- 适用阶段：`LKMA`
- 直接依据：`结直肠腺体分割_plan_优化版/01_实验执行/05_LKMA/00_阶段总协议.md`、`结直肠腺体分割_plan_优化版/01_实验执行/05_LKMA/01_设计依据.md`、01_Large-Kernel-Matters.md、04_MILD-Net.md
- 核心公式或定义参考：`P_main = {bottleneck}`；其语义固定为 `encoder deepest output -> decoder stage1 input`
- 采用原因：这里最接近全局语义和整体形态聚合，最能体现大感受野模块对腺体对象级形态的直接影响
- 不采用的相邻方案：不把浅层 encoder 位置作为首轮默认；不把多个 decoder stage 并列为首轮主表
- 代码落点：[project_root]/src/models/r34_unet.py、[project_root]/configs/model/lkma_v1.yaml
- 运行记录字段：`insert_position = bottleneck`、`insert_stage_name = bottleneck`
- 验收方式：检查首个正式主版本只在 `bottleneck` 注入，并在 [project_root]/experiments/[run_name]/config.yaml 与 [project_root]/experiments/[run_name]/run_meta.yaml 中显式记录

### 4.2 备选位置上限

- 当前结论：除了主位置 `bottleneck` 外，最多只允许补一个高层备选位置 `decoder_stage1`
- 规则类型：`论文支持的候选范围 + 工程冻结规则`
- 适用阶段：`LKMA`
- 直接依据：`结直肠腺体分割_plan_优化版/01_实验执行/00_总览与规范/02_参数冻结总表.md`、`结直肠腺体分割_plan_优化版/01_实验执行/05_LKMA/01_设计依据.md`
- 核心公式或定义参考：`P_optional = {decoder_stage1}`；总位置集合固定为 `P = {bottleneck, decoder_stage1}`
- 采用原因：如果主位置结果接近边界但解释还不够清晰，只需要补一个更靠近重建路径的高层位点，不应把位置搜索扩成矩阵
- 不采用的相邻方案：不同时开放多个 encoder stage；不同时开放多个 decoder stage；不做全网络铺满 `LKMA`
- 代码落点：[project_root]/src/models/r34_unet.py、[project_root]/configs/model/lkma_v1.yaml
- 运行记录字段：`insert_position`、`insert_stage_name`、`variant_rank`

### 4.3 `kernel_size` 候选范围

### 4.7 模块内部复杂度边界

- 当前结论：`LKMA v1` 只允许 `single-branch large-kernel depth-wise spatial block + optional 1x1 channel mixing + optional residual`
- 规则类型：`工程冻结规则`

- 直接依据：`结直肠腺体分割_plan_优化版/01_实验执行/05_LKMA/00_阶段总协议.md`、`结直肠腺体分割_plan_优化版/01_实验执行/05_LKMA/01_设计依据.md`
- 核心公式或定义参考：`Y = X + PWConv(DWConv_k(X))`
- 采用原因：当前阶段要验证的是“是否需要一个首版大核形态建模块”，不是同时比较多分支、多尺度和注意力家族
- 不采用的相邻方案：不引入多分支并联大核；不引入 attention gating；不引入多尺度 `kernel mix`；不引入复杂重参数化
- 代码落点：[project_root]/src/models/modules/lkma.py
- 运行记录字段：`lkma_impl_type`、`use_pwconv`、`use_residual`
- 验收方式：检查 lkma.py 不膨胀成完整 `VAN / WTConv / RepLKNet / UniRepLKNet` 风格主干

补充实现级伪代码边界：

```text
input : X in R^(B x C x H x W)
spatial = DWConv_k(X)
mixed = use_pwconv ? PWConv(spatial) : spatial
output = use_residual ? (X + mixed) : mixed
```

这里的固定实现语义是：

- `DWConv_k` 负责 large-kernel spatial aggregation，不承担额外多分支或 attention gating。
- `PWConv` 只承担通道混合与输出对齐职责，不额外引入第二条并行主支路。
- residual 只允许使用同尺度、同通道的直接残差相加；若张量尺寸不一致，则必须先判为非法配置，而不是临时补额外投影分支。
- 当前阶段默认主版本按 `use_pwconv = true`、`use_residual = true` 实现，也就是 `Y = X + PWConv(DWConv_k(X))`。
- 若关闭 `use_pwconv` 或 `use_residual`，只能作为当前文件已冻结变量的显式开关，不得顺势扩成新的模块家族。

补充当前唯一允许作为代码起点的 `forward()` 伪代码：

```text
Input:
    X in R^(B x C x H x W)
    k in {15, 21}, and k must be odd

Forward:
    assert use_residual -> output shape must equal input shape
    spatial = DepthWiseConv2d(
                 in_channels = C,
                 out_channels = C,
                 kernel_size = k,
                 stride = 1,
                 padding = k // 2,
                 groups = C
              )(X)                         # spatial in R^(B x C x H x W)
    mixed = use_pwconv
            ? Conv2d(C, C, kernel_size = 1)(spatial)
            : spatial                      # mixed in R^(B x C x H x W)
    Y = use_residual ? (X + mixed) : mixed

Output:
    Y in R^(B x C x H x W)
```

这段伪代码在当前阶段具有唯一解释权，必须同时满足：

- `DWConv_k` 使用 same-padding 语义保持 `H / W` 不变，首版不允许 stride、pooling、下采样或上采样。
- `DWConv_k` 与可选 `PWConv` 都不改变通道数 `C`；若代码实现导致通道数变化，则视为偏离当前协议。
- `use_residual = true` 时，`X` 与 `mixed` 必须在 `B / C / H / W` 四个维度完全一致；否则该配置必须直接判为非法，而不是临时补 `1x1` shortcut。
- `forward()` 的返回值只能是单个特征张量 `Y`，不允许额外返回 attention map、多尺度分支结果或中间支路集合。

- 代码文件：[project_root]/configs/model/lkma_v1.yaml、[project_root]/src/models/modules/lkma.py
- 入口类/函数：`build_lkma_block()`、`validate_lkma_variant_fields()`
- 输入：`insert_position`、`kernel_size`、`lkma_impl_type`、`use_pwconv`、`use_residual`
- 输出：合法的 `LKMA` 模块配置对象
- `dtype`：配置字段为 `string / int / bool`
- 依赖配置：`insert_position`、`kernel_size`、`lkma_impl_type`、`use_pwconv`、`use_residual`、`train_proto_version`、`eval_proto_version`、`eval_cast_policy`、`boundary_metric_width`、`boundary_metric_impl`、`connected_components_impl`、`connected_components_connectivity`
- 前置断言：`insert_position` 只能取 `bottleneck / decoder_stage1`；`kernel_size` 只能取 `15 / 21`；`lkma_impl_type` 只能取首版单分支实现；`threshold` 前必须保持 `float32`，并继续沿用 `Boundary F1 = 3 px` 与 `8-connectivity` 连通域口径
- 前置断言：`forward` 只允许执行 `DWConv_k -> optional PWConv -> optional residual add`；不得插入第二主支路、attention gate、多尺度 kernel mix 或额外投影捷径
- 运行产物：[project_root]/experiments/[run_name]/config.yaml 中的模块字段、[project_root]/experiments/[run_name]/run_meta.yaml 中的结构记录；其中五个评估实现硬字段必须显式落盘

### 7.2 注入位置解析入口

- 代码文件：[project_root]/src/models/r34_unet.py
- 入口类/函数：`inject_lkma_block()`、`resolve_lkma_insert_position()`
- 输入：`R34UNet` 中间特征图 `B x C x H x W` 与 `insert_position`
- 输出：注入 `LKMA` 后的同尺寸特征图 `B x C x H x W`
- `dtype`：特征张量为 `float32`，AMP 下允许 `float16 / bfloat16`
- 依赖配置：`insert_position`、`kernel_size`、`use_residual`
- 前置断言：主版本默认只在 `bottleneck` 注入；若启用 `decoder_stage1`，必须作为独立备选版本运行
- 运行产物：模型结构摘要、调试日志、[project_root]/experiments/[run_name]/run_meta.yaml 中的位置信息

### 7.3 复杂度与变体汇总入口

- 代码文件：[project_root]/scripts/summarize_stage.py、[project_root]/reports/tables/lkma_cost_comparison.csv
- 入口类/函数：`summarize_lkma_variants()`、`estimate_lkma_complexity()`
- 输入：正式 `LKMA` run 目录、baseline 对照结果、复杂度统计结果
- 输出：per-seed 变体表、复杂度对照表、位置与 `kernel` 字段齐全的阶段比较表
- `dtype`：指标与复杂度字段为 `float`；`insert_position / kernel_size / run_name` 为字符串
- 依赖配置：`insert_position`、`kernel_size`、`params_m`、`flops_g`、`latency_ms`、`result_tag`、`aggregation`、`eval_cast_policy`、`boundary_metric_width`、`boundary_metric_impl`、`connected_components_impl`、`connected_components_connectivity`
- 前置断言：不得从目录名反解析位置与 `kernel`；必须直接读取正式字段；额外备选总数不得超过 `1`；若五个评估实现硬字段未在 [project_root]/experiments/[run_name]/run_meta.yaml 和结果表同步出现，则不得输出正式汇总
- 运行产物：[project_root]/reports/tables/lkma_per_seed_summary.csv、[project_root]/reports/tables/lkma_cost_comparison.csv，以及包含五个评估实现硬字段的正式比较输入

### 7.4 执行边界导出入口

- 代码文件：[project_root]/configs/experiment/lkma_c1_v1.yaml、`crc_gland_segmentation_project/scripts/train.py`
- 入口类/函数：`build_lkma_experiment_cfg()`、`main(cfg)`
- 输入：baseline 正式 handoff 资产、`insert_position`、`kernel_size`、seed 与冻结训练协议
- 输出：正式 `LKMA` screening run、三 seed 主版本和最多一个备选版本配置
- `dtype`：配置字段为 `string / int / bool`
- 依赖配置：`stage_code`、`run_name`、`variant_rank`、`insert_position`、`kernel_size`、`screening_only`、`train_proto_version`、`eval_proto_version`、`eval_cast_policy`、`boundary_metric_width`、`boundary_metric_impl`、`connected_components_impl`、`connected_components_connectivity`
- 前置断言：必须先有 baseline 正式起点；不得同时开放两个备选方向；不得绕开 `main_variant`；五个评估实现硬字段必须随正式版本字段一起进入 [project_root]/experiments/[run_name]/config.yaml 与 [project_root]/experiments/[run_name]/run_meta.yaml
- 运行产物：正式 [project_root]/experiments/[run_name]/config.yaml、[project_root]/experiments/[run_name]/run_meta.yaml、运行目录与阶段记录字段；其中阶段记录字段必须可被 [project_root]/reports/tables/lkma_stage_manifest.csv 与 handoff 清单继续消费
