# 实验执行证据化写作模板

## 1. 文件角色与执行边界
本文件定义 active journal 执行文档如何把规则写成可追溯、可实现、可验收和可回退正文，归属 B 类模板辅助。它不产生实验结果或阶段通过声明。`status=planning_rewrite_verified_pending_diagnostics`，`formal_result=not_run`，`result_eligibility=false`。

## 2. 本轮直接依赖的前置文件
旧计划同名 06 是模板基线；active 00/03/04/05/07 提供入口、命名、评估、代码和质检要求；路线与研究定标定义当前 05-11、N1-N3 和 blind；GlaS/DCAN/MILD-Net 定义写作可引用的任务和方法语义；`/home/featurize/work/Paper/结直肠腺体分割_正式参考资料/04_task_specific_benchmarks/GlandSegBenchmarks-master/DCAN/metrics.py`、`/home/featurize/work/Paper/结直肠腺体分割_正式参考资料/04_task_specific_benchmarks/GlandSegBenchmarks-master/DCAN/test_GlaS.py`、surface-distance、MedPy、journal/旧工程入口用于写清参考代码边界。相关 `02_路线与投稿`、`03_文献证据` 文件必须在每个实际目标文档中逐项留痕。

## 3. 本阶段唯一允许处理的变量
只处理证据写法、规则卡片、引用层级、代码落点和验收叙述，不宣告任何实验结果。

## 4. 阶段门控表达式
`writing_gate = evidence_traceable AND implementation_actionable AND acceptance_defined AND rollback_defined`。

## 上游、同层与下游
上游是路线/文献证据，提供“为什么”；同层 00-05 和 07 提供“怎样冻结、怎样实现、怎样质检”；下游所有阶段协议、设计依据、验收和结果汇总使用本模板。模板不替下游写结果，只要求结果字段显式为 `not_run/false`。

## 路线依据与文献依据
证据优先级固定为：官方 benchmark/脚本规则 > 路线层锁定 > 文献方法/参数候选 > 工程冻结规则。GlaS 的 split、TestA/TestB 与对象指标优先；DCAN/MILD-Net 的方法语义必须准确区分；参考代码只能证明接口或实现差异，不能把引用值变成复现值。

## 标准正文骨架
每个正式文件至少写出：文件角色与状态；本轮重写直接依赖的前置文件；上游/同层/下游；路线依据与文献依据；固定/可调/禁止参数；核心定义/公式；代码落地接口；运行记录字段；验收；独立回退条件；冲突裁决记录；文件质量自检；Diagnostics 闭环。涉及阶段交接时再写 blind、handoff 七字段和 `consumption_boundary`。

## 规则卡片模板
```md
### 规则名
- 当前结论：明确的最终执行规则或 planned 边界
- 规则类型：官方协议固定项 / 路线层已锁定 / 论文支持的候选范围 / 工程冻结规则
- 适用阶段：起始阶段与不可改时间点
- 直接依据：路线文件、文献文件、参考代码或配置路径
- 采用原因：研究问题与工程理由
- 不采用的相邻方案：最易混淆的替代项及排除理由
- 代码落点：active project root 的文件、函数或配置
- 运行记录字段：run_meta、manifest、metric、handoff 字段
- 验收方式：事实证据、检查脚本、通过/阻断边界
- 核心公式或定义参考：涉及 loss/metric/target/postprocess 时必填
```

## 固定、可调与禁止写法
固定项必须给唯一值和版本；候选项必须列最终选择、候选范围、选择理由和冻结时点；禁止项必须可被检查器发现。禁止“效果很好”“应当提升”“已经验证”等结果化措辞，统一使用 `planned/not_run`、`not_created`、`blocked` 等事实状态。

## 代码落地与运行记录写法
代码路径统一为 `/home/featurize/work/Paper/crc_gland_segmentation_project_journal`；历史工程路径只能在 `historical_provenance_only` 小节出现。运行记录至少含七字段 lineage、版本链、seed、split hash、checkpoint、metric identity、threshold、blind state、formal result。没有实际资产就写 `not_created`，不补造数字。

## 验收与回退写法
验收必须回答输入资产、代码身份、参数版本、运行事实、指标 identity、可视化/失败案例、handoff 是否完整；回退必须单独列触发条件、回退目标、需保留的未泄漏资产和重新验收动作。测试集泄漏、指标漂移和字段缺失均使当前结果不具备投稿资格。

## 冲突裁决记录
当路线、文献、阶段文件、配置或参考代码冲突，使用字段：`冲突对象/冲突来源/裁决结论/裁决理由/受影响文件/是否需要回流修订/代码实现影响`。优先级为官方 > 路线 > 直接文献 > 统一工程冻结。不得只在总结句中隐含裁决。

## 8. 验收与回退示例
- 规范文件：依据、接口和状态字段齐全后才进入 diagnostics。
- 计划文件：必须同时给出运行记录字段、资产清单和消费边界。
- 结果文件：没有 checkpoint、manifest 和评估 identity 时只能 not_run。
- 任何 test leakage、版本漂移或引用值误写均回退到最近 Gate。

## 9. 审计回填字段
每次重写回填文件绝对路径、旧基线、active 依赖、正文落点、自检落点、命令、退出码、问题、修复和复跑时间。记录中的 `formal_result` 和 `result_eligibility` 不得由模板自动改为通过。

## 11. 当前审计结论
模板已提供正式正文所需的最小字段与审计动作；目标文档仍须逐文件按本模板落地，不能以模板存在替代下游重写。

## 12. 模板使用结论
本模板保证正文字段不被压缩，但不替代目标文件的角色判断、上游阅读、差异化遍历和真实 diagnostics。所有结果仍保持 not_run/false。

## 最低证据包字段
每个训练/评估阶段的正式记录至少要能回填：
- `source_stage`、`source_manifest`、`source_protocol_version`、`source_run_name`。
- `consumer_stage`、`consumer_file`、`consumption_boundary`。
- config/code/data/environment digest。
- run name、seed、split、sample count、batch count、epoch count。
- checkpoint path、selector、metric value、epoch、sha256。
- raw CSV schema、行数、aggregate 策略和独立复核结果。
- failure/回退状态和下一步准入条件。

## 写作失败处理
- 如果事实证据不存在，只能写 `not_run`、`blocked` 或 `not_applicable`，不能用预期结果填空。
- 如果记录字段和真实文件冲突，先停止下游写作，回到代码/结果审计，不修改数字迎合模板。
- 如果同一事实在多个文档中不一致，以正式脚本产物和对应 Gate 为准，再更新说明文。
- 如果当前阶段发生迭代，旧记录先归档，当前正文不能直接继承旧 Gate 或旧结果。

## 逐文件回链格式
每个被修改、创建或消费的正式文件必须独立登记，至少包含：
- 文件绝对路径、A/B/C 归属、当前轮 sha256。
- 对应计划章节和协议条款。
- 对应论文、文献证据、官方脚本或参考代码的具体文件/函数/章节。
- 冻结参数、变量隔离、允许搜索范围和禁止反馈 split。
- 输入文件、输出文件、schema、行数、digest 和下游消费者。
- 验收命令、真实退出码、报告路径和未解决 blocker。

## 不同文件角色的最低证据
- 数据协议文件必须写 data identity、split、标签规则、资产 manifest 和 handoff。
- 模型/训练文件必须写结构、loss、optimizer、scheduler、checkpoint、resume 和 full/smoke 边界。
- 评估/结果文件必须写 metric identity、threshold、postprocess、raw/aggregate、独立复核和 blind。
- Gate/投稿文件必须写准入表达式、拒绝条件、回退、Plan B 和下游消费边界。

## Diagnostics 闭环
本模板文件不生成实验数字。使用前应运行文档检查；若模板字段被删除或文档状态与真实 gate 不一致，拒绝使用并回退修订。检查时逐项确认字段、来源、结果文件、Gate 和 handoff 是否能互相回链；没有真实产物时必须保持 `not_run`。
