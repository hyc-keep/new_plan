# 数据产物清单

本文件不是“把目录里要留什么文件列一下”的备忘，而是 `01_数据协议` 中负责冻结正式数据交接包的执行协议。它要写死的不是“有哪些文件可能有用”，而是：

- 数据阶段结束时哪些资产必须正式存在
- 每类资产的职责、最小字段和目录身份
- 哪些资产缺失会直接阻断进入 `02_UNet流程验证`
- `asset_manifest / assets_traceable / protocol_explainable / handoff_ready / data_stage_pass` 应该如何从资产清单落成放行门控

如果这一层不先压成正式协议，后面最容易出现的不是“少一张图”，而是训练入口、检查报告、预览资产、配置文件和来源说明各讲各的话，最终导致下游虽然开始训练，却无法回答“当前到底在消费哪一版正式数据资产包”。

---

## 1. 文件角色与执行边界

### 1.1 当前文件负责什么

当前文件负责冻结以下内容：

- 数据阶段正式交付资产的完整组成
- 每类正式资产的最小内容、最小目录落点和最小版本留痕
- 资产登记到 `asset_manifest` 的统一规则
- 进入 `02_UNet流程验证` 前的最小交接包与放行门控
- 数据阶段发现资产缺口时的回退触发条件

### 1.2 当前文件不负责什么

当前文件不负责重开以下内容：

- `GlaS / CRAG` 的 split 定义本身
- image / mask 配对与六步检查细节
- 二值标签、边界图、距离图的生成公式本身
- 模型结构、loss 组合、训练超参数与评估实现

对应职责分别由下列文件承担：

- `结直肠腺体分割_plan_优化版\01_实验执行\01_数据协议\02_数据检查与配对规则.md`
- `结直肠腺体分割_plan_优化版\01_实验执行\01_数据协议\03_GlaS划分协议.md`
- `结直肠腺体分割_plan_优化版\01_实验执行\01_数据协议\04_CRAG划分协议.md`
- `结直肠腺体分割_plan_优化版\01_实验执行\01_数据协议\05_标签转换与可视化规则.md`
- `结直肠腺体分割_plan_优化版\01_实验执行\01_数据协议\07_数据阶段验收.md`
- `结直肠腺体分割_plan_优化版/01_实验执行/02_UNet流程验证/00_阶段总协议.md`

### 1.3 当前文件的真实目标

当前文件真正要解决的问题不是“目录里最后留下点什么”，而是把数据阶段结束后的交付对象固定成一套可被下游直接消费、可被上游反查来源、可被论文写作复述的正式资产包：

> 后续训练、评估、复盘和写作都只能从同一套正式数据资产包出发，而不允许在模型阶段重新口头解释“split 是哪版、标签怎么转、预览图对应哪批样本、配置到底指向哪里”。

### 1.4 当前文件对下游的硬约束

从 `02_UNet流程验证` 开始，下游只允许消费这里锁定并已经放行的数据资产包，不允许：

- 只拿 `CSV` 就开训，剩余资产边跑边补
- 配置文件绕开正式 split，直接扫描原始目录
- 检查报告、预览图和来源说明各自使用不同版本字段
- 在训练阶段重新解释 `asset_manifest` 该怎么登记

## 4. 正式数据资产包总规则

### 4.1 数据交接主规则

- 当前结论：数据阶段正式交付物必须同时覆盖 `split assets + check assets + preview assets + config assets + source assets + manifest assets`
- 当前值或最终版本：`data_asset_bundle_v2`
- 候选范围：`formal data handoff package`
- 规则类型：`工程冻结规则`
- 直接依据：`结直肠腺体分割_plan_优化版\01_实验执行\01_数据协议\00_阶段总协议.md`，`结直肠腺体分割_plan_优化版\01_实验执行\01_数据协议\05_标签转换与可视化规则.md`，`结直肠腺体分割_plan_优化版\01_实验执行\01_数据协议\07_数据阶段验收.md`
- 核心公式或定义参考：`formal_data_package = split_assets + check_assets + preview_assets + config_assets + source_assets + manifest_assets`
- 采用原因：数据阶段交出去的必须是一套“能被代码读、能被文档解释、能被检查回指、能被下游放行”的完整输入层
- 不采用的相邻方案：不采用只交 `CSV`；不采用只留检查报告；不采用把 `asset_manifest` 视为可选附属文件
- 生效阶段：`01_数据协议`
- 冻结阶段：进入 `02_UNet流程验证` 前
- 代码落点：`crc_gland_segmentation_project/splits/`，`crc_gland_segmentation_project/reports/`，`crc_gland_segmentation_project/configs/data/`，`crc_gland_segmentation_project/datasets/`，`crc_gland_segmentation_project/reports/stage_reports/`
- 运行记录字段：`data_protocol_package_version`，`asset_manifest_version`
- 验收方式：逐项核对六类资产是否齐全、可读、可回指；任一大类缺失都不得放行

### 4.2 当前总原则

数据阶段正式资产包必须同时满足三类要求：

1. 代码可直接消费：下游训练入口无需再猜 split、路径和标签口径
2. 协议可直接解释：后续能准确回答“当时正式数据输入层到底是什么”
3. 资产可直接追溯：每一类结果都能回指到同一版 split、标签、配置和来源说明

## 5. 六类正式资产与门控关系

### 5.1 六类正式资产

当前正式数据资产包固定包含以下六类：

- `split assets`：正式 split `CSV` 与 schema 留痕
- `check assets`：检查报告、统计表、重复检查和人工抽查结果
- `preview assets`：按 `dataset / split / sample_id` 可回指的最小视觉证据包
- `config assets`：`configs/data/*.yaml` 等训练入口直接读取的配置资产
- `source assets`：数据来源、整理方式和目录身份说明
- `manifest assets`：`asset_manifest`、放行字段、交接状态与缺失项登记

### 5.2 门控绑定公式

本文件与 `结直肠腺体分割_plan_优化版\01_实验执行\01_数据协议\00_阶段总协议.md`、`结直肠腺体分割_plan_优化版\01_实验执行\01_数据协议\07_数据阶段验收.md`、`结直肠腺体分割_plan_优化版/01_实验执行/02_UNet流程验证/00_阶段总协议.md` 的绑定固定为：

```text
assets_traceable
= pass_split_traceability
and pass_check_traceability
and pass_preview_traceability
and pass_config_traceability
and pass_source_traceability
```

```text
protocol_explainable
= pass_split_docs
```


旧稿已经把 split、检查、预览、配置和来源说明列为正式交付物，但在新版强约束下仅有“五类文件”还不够，因为下游真正需要的是“可执行交接状态”，而不是“目录里有这些文件就算完成”。

- `crc_gland_segmentation_project/splits/crag/crag_test40.csv`

### 6.2 split 资产规则

- 当前结论：`7` 个正式 split `CSV` 是后续训练、验证、评估和结果导出的唯一默认入口
- 当前值或最终版本：`split_asset_v2`
- 候选范围：`official split csv package`
- 规则类型：`工程冻结规则`
- 直接依据：`结直肠腺体分割_plan_优化版\01_实验执行\01_数据协议\03_GlaS划分协议.md`，`结直肠腺体分割_plan_优化版\01_实验执行\01_数据协议\04_CRAG划分协议.md`
- 核心公式或定义参考：`split_assets = {glas_train68, glas_val17, glas_testA60, glas_testB20, crag_train153, crag_val20, crag_test40}`
- 采用原因：保证后续所有阶段在同一版 split 上运行，防止模型阶段再复制一份私有 split
- 不采用的相邻方案：不采用 dataloader 直接扫描原始目录；不采用每个模型再维护自己的一套 split 副本
- 代码落点：`[project_root]/splits/glas/*.csv`，`[project_root]/splits/crag/*.csv`
- 运行记录字段：`glas_split_version`，`crag_split_version`，`split_csv_schema_version`
- 验收方式：检查 `7` 个文件全部存在、schema 一致、配置文件只引用这些正式 split

### 6.3 每个 `CSV` 的最小字段

每个正式 `CSV` 至少包含：

- `sample_id`
- `image_relpath`
- `mask_relpath`
- `dataset`
- `split`

附加固定字段：

- `GlaS` 必须包含 `grade`
- `CRAG` 必须包含 `source_subset`

允许的统一附加字段包括：

- `height`
- `width`
- `foreground_ratio`
- `is_empty_mask`

但这些附加字段一旦启用，必须全项目统一，不允许某个 split 单独扩列又不记版本。

## 12. 回退条件

### 12.1 数据阶段内回退触发条件

只要出现以下任一情况，本文件对应的正式交接协议就不得放行，必须先回退补齐资产，而不是把问题带进 `UNet`：

- `7` 个正式 split `CSV` 任一缺失
- 核心检查资产任一缺失或无法回指当前 split 版本
- 某个正式 split 没有最小预览证据包
- `configs/data/*.yaml` 仍指向临时路径或旧版 split
- 来源说明文件无法解释当前目录身份
- `asset_manifest` 未登记完整五类基础资产和放行字段

### 12.2 下游异常时的强制回退顺序

如果 `02_UNet流程验证` 出现“训练入口找不到 split”“标签口径与配置不一致”“预览可视化和训练样本不匹配”“数据阶段状态字段缺失”等异常，必须按以下顺序回退：

1. 检查 `configs/data/*.yaml` 是否只引用正式 split
2. 检查 `asset_manifest` 是否登记了当前版本的 split、check、preview、config、source 资产
3. 检查预览资产是否还能回指 `dataset / split / sample_id`
4. 检查检查报告是否仍能回指当前 split 与标签版本
5. 检查来源说明和目录身份是否与配置、CSV 一致
6. 检查 `handoff_ready / data_stage_pass / next_action` 是否由正式校验入口生成

### 12.3 回退后的重新放行条件

发生回退后，只有同时满足以下条件才允许重新把 `handoff_ready` 改回 `True`：

- 缺失资产已经按当前正式版本补齐
- `asset_manifest` 已重新登记并更新版本字段
- `assets_traceable=True`
- `protocol_explainable=True`
- `red_flag=False`
- `data_stage_pass=True`
