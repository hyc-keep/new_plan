# `CRAG` 阶段验收

这份文件不是给 `09_CRAG验证` 再补一页“差不多完成了”的收尾说明，也不是把旧版 `A / B / C` 分级清单加长几条就算正式验收。

它在当前阶段中的唯一职责是：

> 作为 `09_CRAG验证` 的阶段放行协议，把 `Gate_I1` 的真正通过线、最低交付物、红线问题、独立回退条件、审计对表、下游 handoff 资产和重新放行条件一次写死。自本文件起，`CRAG` 阶段不再允许以“第二个 benchmark 已经跑过”为理由提前过线；只有当当前主线来源唯一、冻结协议一致、核心比较链完整、任务内 direct comparison 已成形、图表与案例资产闭环、结论强度已收口且下游可直接消费时，才允许进入 `10_结果汇总`。

从现在开始，`09_CRAG验证` 阶段是否通过，只能按本文件裁决；若与同层执行细则、阶段总结或下游汇总口径冲突，以本文件为准回改。

## 1. 文件角色与执行边界

### 1.1 当前文件负责什么

当前文件只负责冻结下面八件事：

1. `09_CRAG验证` 阶段真正要证明什么，以及什么状态才算“`CRAG` 已形成正式补充验证层”。
2. `Gate_I1` 的正式通过线，也就是 `current_mainline_ready / crag_protocol_fixed / core_crag_runs_complete / crag_direct_ready / conclusion_grade_fixed / joint_handoff_ready` 六个子门的验收关系。
3. `UNet / ResNet34-U-Net / current_mainline` 在 `CRAG` 上的最小正式比较链，以及三 seed 补证要求何时必须成立。
4. `MILD-Net-RTS / TA-Net / DEA-Net` 作为任务内 direct comparison 锚点，在阶段验收中的必过边界与来源标记要求。
5. CRAG_main_table.md、CRAG_direct_comparison_table.md、CRAG_per_seed_summary.csv、CRAG_mean_std_summary.csv、glas_vs_crag_gap_table.csv、成功/失败案例与阶段总结是否已经足以支撑下游消费。
6. 红线问题、独立 `回退条件`、固定回退顺序和回退后的重新放行条件。
7. 阶段验收在代码和结果层中的落地接口、前置断言和 handoff 入口。
8. 当前文件自身的 `冲突裁决记录`、不缩水的 `文件质量自检`、独立 `审计对表` 和独立 `Diagnostics 闭环`。

### 1.2 当前文件不负责什么

当前文件不重新定义下面这些内容：

- `CRAG` 的阶段角色、核心模型名单、允许/禁止修改边界本体；这些内容以 `结直肠腺体分割_plan_优化版/01_实验执行/09_CRAG验证/00_阶段总协议.md`、`结直肠腺体分割_plan_优化版\01_实验执行\09_CRAG验证\01_任务定位.md` 为准。
- `CRAG` 具体怎么训练、怎么选 checkpoint、怎么补三次重复、怎么导出 run 资产；这些内容以 `结直肠腺体分割_plan_优化版/01_实验执行/09_CRAG验证/02_训练与验证协议.md` 为准。
- `A / B / C` 三档结果强度的解释细则和允许/禁止写法本体；这些内容以 `结直肠腺体分割_plan_优化版/01_实验执行/09_CRAG验证/03_结果解释规则.md` 为准。
- `10_结果汇总` 和 `11_总验收与止损` 的最终表格回填、写稿进入与项目总裁决细则；当前文件只负责把 `CRAG` 阶段放行到足够可交接的状态。

### 1.3 为什么当前文件必须独立存在

`CRAG` 最容易出现的伪完成状态不是“完全没跑”，而是：

- 只跑了几个 `CRAG` run，就把第二 benchmark 写成“阶段完成”。
- `UNet / baseline / current_mainline` 三层关系没有齐，只拿最终模型单点结果写结论。
- 主表里已经有数字，但 `reproduced / † / *`、`single_seed / mean+-std` 和 `direct_comparison_eligible` 还没有真正固定。
- 结果不理想时回头改单独训练协议、阈值、后处理或结构，却仍把结果写成冻结主线验证。
- `10_结果汇总` 想读取 `CRAG` 资产时，只能看到零散数字，读不到阶段通过状态、来源边界和 handoff 结论。

如果没有一份独立阶段验收协议，后续 `10_结果汇总`、`11_总验收与止损` 和论文写作都会再次面对“`CRAG` 跑过了，但到底按什么门、哪些资产、哪些强度继续消费”的灰区。

## 2. 核心验收问题

### 3.1 当前阶段真正要验收什么

`09_CRAG验证` 真正要验收的不是“第二个 benchmark 也跑了”，而是：

> 在 `GlaS` 上已经冻结的 `UNet / ResNet34-U-Net / current_mainline` 关系，在 `CRAG` 上已经形成可追溯、可比较、可解释、不过度扩张且可直接交接给下游的正式补充验证证据层。

因此，本阶段通过同时意味着：

- `CRAG` 仍然是 `same-task supplementary benchmark`，不是第二个主调参场。
- 核心三模型比较链已经完整，不是只拿最终模型单点结果说话。
- 任务内强对照 `MILD-Net-RTS / TA-Net / DEA-Net` 已进入正式 `CRAG` direct comparison 草稿层。
- 主表、补充表、成功案例、失败案例、阶段总结和联合 handoff 资产已经成型。
- 结论已老实收口到 `A / B / C` 三档之一，下游无需再二次猜测解释强度。

### 3.2 当前阶段不允许用什么替代正式通过

下面这些状态都不能被写成“阶段通过”：

- 只有 `current_mainline` 或只有单 seed 结果，没有 `UNet -> baseline -> current_mainline` 的最小比较链。
- `baseline` 和 `current_mainline` 没有补齐三次重复，却直接写稳定结论。
- 主表里有数字，但 `reproduced / † / *`、`single_seed / mean+-std` 和 `direct_comparison_eligible` 仍然混写。
- 只有成功案例，没有失败案例，或失败案例无法支撑 gap 解释。
- 结果不理想时回头改单独训练协议、selector、threshold 或后处理，却仍想按冻结主线验证放行。
- `10_结果汇总` 仍然无法直接读取 `CRAG` 的正式资产与结论等级，只能靠口头说明继续。

### 3.3 `Gate_I1` 正式通过式

当前文件固定采用下面的阶段总门：

```text
Gate_I1 = current_mainline_ready
          AND crag_protocol_fixed
          AND core_crag_runs_complete
          AND crag_direct_ready
          AND conclusion_grade_fixed
          AND joint_handoff_ready
```

只有 `Gate_I1 = true`，本阶段才允许声明通过。

### 3.4 六个子门的阶段含义

- `current_mainline_ready`：`Distance` 的正式 handoff 已发布，`current_mainline` 唯一合法，且本阶段所有配置、结果和总结都引用同一主线来源。
- `crag_protocol_fixed`：`CRAG` 的数据、训练、评估和记录协议都沿用冻结版本，没有把第二 benchmark 改成第二轮调参链。
- `core_crag_runs_complete`：`UNet(seed3407)`、`ResNet34-U-Net(3407/1234/2025)`、`current_mainline(3407/1234/2025)` 的正式 run 与最小资产已经齐全；若 `UNet` 进入正式 `mean+-std` 主表，则其三次重复也已补齐。
- `crag_direct_ready`：`MILD-Net-RTS / TA-Net / DEA-Net` 的 direct comparison 草稿、来源标记和资格边界已成型。
- `conclusion_grade_fixed`：`A / B / C` 等级已经固定，且允许/禁止写法与图表、案例和 gap 解释一致。
- `joint_handoff_ready`：主表、per-seed 汇总、mean+-std 汇总、成功/失败案例、`GlaS vs CRAG` gap 表、阶段总结与下游消费字段都已闭环，`10_结果汇总` 与 `11_总验收与止损` 无需额外口头解释即可继续。

- 不采用的相邻方案：不允许看 `CRAG` 首轮结果后临时换回 `distance_input_base`；不允许不同 run 混用不同主线来源。
- 代码落点：crc_gland_segmentation_project/configs/experiments/crag/*.yaml、run_meta.yaml、crc_gland_segmentation_project/reports/stage_reports/crag_stage_summary.md

- 验收方式：检查所有正式 `CRAG` run、表格与阶段总结都只引用同一个 `current_mainline`。

### 4.3 `crag_protocol_fixed` 规则

- 当前结论：`CRAG` 阶段只允许切换数据入口，训练协议、评估协议和结果记录协议必须沿用冻结版本。
- 规则类型：`公平比较规则 + 工程冻结规则`
- 适用阶段：`09_CRAG验证`

- 核心公式或定义参考：`crag_run = frozen_mainline_proto + crag_data_proto_only`
- 采用原因：`CRAG` 的价值来自迁移验证，不来自第二次个性化调参。
- 不采用的相邻方案：不允许单独改 `best_selector`、单独改 `threshold_source`、单独加后处理、单独加 TTA 或单独改 loss / optimizer / scheduler。
- 代码落点：`crc_gland_segmentation_project/configs/data/crag.yaml`、crc_gland_segmentation_project/configs/train/*.yaml、crc_gland_segmentation_project/src/eval/run_eval.py

- 验收方式：检查所有正式 `CRAG` run 共享同一协议字段链，且 `test40` 没有回调痕迹。

### 4.4 `core_crag_runs_complete` 规则

- 当前结论：核心三模型必须完成最小闭环；`ResNet34-U-Net` 与 `current_mainline` 必须补齐三次重复。
- 规则类型：`最低交付物规则`
- 适用阶段：`09_CRAG验证`
- 直接依据：`结直肠腺体分割_plan_优化版/01_实验执行/09_CRAG验证/02_训练与验证协议.md`、`结直肠腺体分割_plan_优化版/01_实验执行/09_CRAG验证/01_任务定位.md`
- 核心公式或定义参考：`need_three_seed = (model in {ResNet34-U-Net, current_mainline}) OR (model == UNet AND used_in_mean_std_main_table)`
- 采用原因：只有保留流程锚点、正式 baseline 和冻结主线的完整执行链，`CRAG` 才能形成最小而可信的验证关系。
- 不采用的相邻方案：不允许只跑最终模型；不允许 baseline 或最终模型停在单 seed；不允许 `UNet` 既进入正式 `mean+-std` 主表又不补 seed。
- 代码落点：`crc_gland_segmentation_project/scripts/train.py`、crc_gland_segmentation_project/scripts/compare_runs.py
- 运行记录字段：`run_name`、`model_code`、`train_seed`、`need_three_seed_followup`、`aggregation`
- 验收方式：检查正式 run 目录、per-seed 记录和三 seed 聚合状态均与阶段角色一致。

### 4.5 `crag_direct_ready` 规则

- 当前结论：`MILD-Net-RTS / TA-Net / DEA-Net` 的 `CRAG` direct comparison 草稿必须成型，且来源标记与比较资格完整。

- 直接依据：`结直肠腺体分割_plan_优化版/01_实验执行/08_外部对比/06_引用值与复现值标记规则.md`、`结直肠腺体分割_plan_优化版/03_文献证据/05_GlaS_CRAG_主结果数值速查表.md`、`结直肠腺体分割_plan_优化版/01_实验执行/09_CRAG验证/03_结果解释规则.md`

- 采用原因：`CRAG` 不能只拿通用外部模型说话，必须先锁定腺体任务内的强对照参照系。

### 4.6 `conclusion_grade_fixed` 规则

- 当前结论：`CRAG` 结论必须唯一落到 `A / B / C` 三档之一，并同步固定允许/禁止写法。

- 适用阶段：`09_CRAG验证`

- 核心公式或定义参考：`CRAG_grade in {A, B, C}`
- 采用原因：如果不先判级，后续最容易出现“证据偏弱但措辞偏强”的越界。
- 不采用的相邻方案：不采用“基本支持”“总体不错”这类无等级口径；不让不同文档各写各的强度。
- 代码落点：crc_gland_segmentation_project/reports/stage_reports/crag_stage_summary.md、crag_conclusion_note.md

- 验收方式：检查下游所需的表格、案例、等级、局限性边界和 stop-loss 语气均已形成正式文件。

### 4.8 进入 `10_结果汇总` 的固定规则

- 当前结论：`CRAG` 不要求必须是 `A` 档才允许进入 `10_结果汇总`，但必须角色清楚、边界清楚、资产齐全。

## 7. 回退条件

### 7.1 独立回退触发条件

即使结果已经跑完，只要出现下面任意一条，也必须先回退修正，而不是直接进入 `10_结果汇总`：

- `current_mainline` 不唯一，或正式 `CRAG` run 无法追溯到同一个冻结主线来源。
- 不同 run 使用了不同 `best_selector`、不同 `threshold_source`、不同后处理或不同 TTA。
- `ResNet34-U-Net` 与 `current_mainline` 的三次重复未齐，或某个 seed 资产缺失仍被强行纳入聚合。
- `reproduced / † / *`、`single_seed / mean+-std`、`direct_comparison_eligible` 仍然没有字段级统一定义。
- CRAG_main_table.md、CRAG_direct_comparison_table.md、glas_vs_crag_gap_table.csv 和 crag_stage_summary.md 之间对同一方法或同一等级的资格判断不一致。
- `CRAG` 结果方向偏弱，却继续使用强泛化、强支撑或全面领先式表述。
- `10_结果汇总` 或 `11_总验收与止损` 所需资产仍然缺失，只能靠口头说明继续。

### 7.2 固定回退顺序

回退时统一按下面顺序排查：

1. 先检查 `current_mainline` 与上游 handoff 是否一致。
2. 再检查统一数据、训练、评估和记录协议是否真的落入配置、脚本和结果文件。
3. 再检查核心三模型的正式 run 目录、seed 状态和最低资产是否齐全。
4. 再检查来源标记、聚合身份和 direct comparison 资格是否完整一致。
5. 再检查主表、direct comparison 表、gap 表、案例包和阶段总结是否对齐。
6. 最后才判断问题是 benchmark gap 观察，还是协议 / 资产 / 解释层面的真实缺口。

### 7.3 回退后的重新放行条件

发生回退后，只有同时满足下面条件，才允许重新声明本阶段通过：

- 问题来源已经按固定回退顺序定位并修复。
- 修复动作、影响范围和复验结果已写回 crag_stage_summary.md 或等价正式记录。
- CRAG_main_table.md、CRAG_direct_comparison_table.md、glas_vs_crag_gap_table.csv 和结论等级重新对齐。
- 当前阶段重新满足第 `4-5` 节的规则、交付物和下游 handoff 要求。
