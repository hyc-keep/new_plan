## 1. 文件角色与执行边界

### 1.1 当前文件负责什么

当前文件只负责冻结下面六件事：

当前阶段要证明的不是“`UNet` 已经够强”，而是下面六件事已经同时成立：

1. `A1` 阶段冻结的训练与评估协议在只改变 `seed` 的前提下仍能稳定复现。
2. 三次正式 run 都完整、可回查、可单独解释，而不是只留下三个数字。
3. raw per-seed 表和 `mean +- std` 聚合表已经形成统一 schema，而不是临时人工汇总。
4. 主要错误模式与波动范围已经可解释，而不是某个 seed 崩掉却说不清原因。
5. 没有任何协议漂移、字段漂移或异常 run 混算污染当前统计结论。
6. `04_Baseline` 可以在不回头修 `A2` 的前提下，只改结构继续推进。

### 1.4 当前文件对下游的硬约束

从 `04_Baseline` 开始，下游只允许继承当前文件放行后的稳定基线，不允许：

- 把 `A2` 三次 run 重新替换成新的 seed 集合再声称仍是同一版 `UNet` 基线。
- 只引用某个最好 seed，而不引用 raw per-seed 与 `mean +- std` 正式结果。
- 在看完 `TestA / TestB` 后反向改 `threshold_value`、`best_selector` 或 `eval_proto_version`。
- 把协议不一致 run、异常补跑 run 或未裁决异常直接混进 baseline 比较起点。
- 让 `04_Baseline` 反向替当前文件补 schema、补交接清单或补异常裁决。

## 2. 核心验收问题

### 3.1 当前阶段唯一合法的验收对象

当前阶段只验收下面这个对象：

> 在完全相同的正式协议下，由当前 `A2_UNet_GlaS_seed3407`、`A2_UNet_GlaS_seed1234`、`A2_UNet_GlaS_seed2025` 三次正式 run 形成的、可回查、可聚合、可解释的 `UNet` 稳定基线证据包。

这意味着当前阶段不验收：

- 单个 seed 的偶然高值。
- 临时补跑但未完成裁决的异常结果。
- 只存在 markdown 结论、没有 csv 和 run 资产的“口头稳定性”。
- 混入协议漂移 run 后再计算出的伪 `mean +- std`。

### 3.2 当前阶段唯一正式门控定义

当前阶段统一采用下面这组门控：

```text
Gate_A2 =
  complete_runs
  and proto_consistent
  and raw_results_ready
  and failure_summary_ready
  and blockers_resolved

handoff_ready_for_b1 =
  Gate_A2
  and baseline_manifest_ready

meanstd_export_ready =
  raw_results_ready
  and seven_metrics_complete
  and statistics_identity_complete
```

其中：

- `complete_runs`：三个正式 seed run 均存在，且逐 run 最小资产齐全。
- `proto_consistent`：三次 run 的正式协议字段完全一致，唯一合法差异只有 `train_seed / run_name / time-related fields`。
- `proto_consistent` 额外要求：`eval_cast_policy`、`boundary_metric_width`、`boundary_metric_impl`、`connected_components_impl`、`connected_components_connectivity` 必须与 `A1/A2` 冻结版本完全一致，并显式出现在 run_meta.yaml 和 raw 结果字段中；统计派生表若生成则同步携带身份字段。
- `raw_results_ready`：三个正式 run 的 sample-only TestA/TestB CSV、同空间评估资产和 raw per-seed 结果完整，覆盖七项指标，当前 raw 维度为 42；正式 per-run CSV 不要求 aggregate 行。
- `meanstd_export_ready`：如执行统计导出，则 mean±std 覆盖两个 split 的七项指标，目标维度为 14；该派生状态不反向决定已经完成的 A2 运行证据。
- `failure_summary_ready`：稳定性文字结论、主要错误模式、异常 run 裁决和允许保留问题已写清。
- `blockers_resolved`：不存在未关闭协议级红线。
- `baseline_manifest_ready`：交给 `04_Baseline` 的 manifest 和限制条件已齐全。

### 3.3 为什么 `A2` 的通过线不能退化成“数值看起来还行”

原因固定为：

- `A2` 是第一正式统计阶段，当前最重要的不是单个分数，而是结果身份和统计口径是否成立。
- 如果只看均值高低，就无法区分“真实结构能力”与“协议漂移、异常 run 或字段混乱”。
- 后续 baseline 与模块阶段需要一个最小但可信的参照，而不是一个随时会被回滚的临时数字。

## 4. 本阶段规则卡片

### 4.1 放行目标固定为“稳定基线成立”

- 当前结论：`A2` 阶段的通过线固定为“稳定 `UNet` 基线成立”，而不是“某个 seed 拿到更高数值”；运行证据采用 sample-only raw，统计派生状态与 per-run CSV 行类型分开记录。
- 规则类型：`路线层已锁定 + 工程冻结规则`
- 适用阶段：`03_UNet稳定性`
- 直接依据：`结直肠腺体分割_plan_优化版/01_实验执行/03_UNet稳定性/00_阶段总协议.md`，`结直肠腺体分割_plan_优化版/02_路线与投稿/02_结直肠腺体分割_分阶段实验路线与执行标准.md`
- 核心公式或定义参考：`stage_pass_a2 = Gate_A2`
- 采用原因：`A2` 的职责是建立可交接的最小稳定基线，而不是进行方法优化竞赛。
- 不采用的相邻方案：不采用“均值更高就通过”；不采用“只要三个 run 都跑完就通过”；不采用“文字看起来差不多就放行”。
- 代码落点：reports/stage_reports/unet_stability_stage_summary.md，scripts/summarize_stage.py
- 运行记录字段：`stage_pass_a2`，`gate_a2`，`handoff_ready_for_b1`，`workflow_gate_status`（跨阶段工作流层，独立记录）
- 验收方式：检查阶段总结是否围绕 `Gate_A2` 给出 A2 内部唯一通过或阻断结论，并单独核对 `workflow_gate_status`，不得将两者混写。

### 4.2 三次正式 run 必须先成立

- 当前结论：只有当前 `A2_UNet_GlaS_seed3407 / seed1234 / seed2025` 三次正式 run 全部完整存在，当前阶段才有资格进入放行判定。
- 规则类型：`工程冻结规则`
- 适用阶段：`03_UNet稳定性`
- 直接依据：`结直肠腺体分割_plan_优化版/01_实验执行/03_UNet稳定性/01_三次重复设计.md`
- 核心公式或定义参考：`complete_runs = all(run_k is formal and assets_k complete for k in {3407,1234,2025})`
- 采用原因：没有逐 run 完整资产，就无法判断波动是否真实，也无法给下游交付可追溯基线。
- 不采用的相邻方案：不采用缺一个 seed 仍先做均值；不采用只保留结果表不保留 run 目录；不采用把异常补跑直接顶替原 run 而不留痕。
- 代码落点：`experiments/A2_UNet_GlaS_seed*/`，`validate_seed_run_manifest()`
- 运行记录字段：`run_name`，`status`，`checkpoint_path`，`asset_manifest`，`handoff_ready`
- 验收方式：逐 run 核对正式目录、checkpoint、日志、指标、可视化和错误记录是否齐全。

### 4.3 三次 run 必须共享同一协议

- 当前结论：除 `train_seed / run_name / time-related fields` 外，三次正式 run 的协议字段必须完全一致，任一漂移都应阻断放行。
- 规则类型：`上游继承规则 + 工程冻结规则`
- 适用阶段：`03_UNet稳定性`
- 直接依据：`结直肠腺体分割_plan_优化版/01_实验执行/03_UNet稳定性/00_阶段总协议.md`，`结直肠腺体分割_plan_优化版/01_实验执行/03_UNet稳定性/01_三次重复设计.md`
- 核心公式或定义参考：`proto(r_i) = proto(r_j)`，且唯一合法差异是 `train_seed(r_i) != train_seed(r_j)`
- 采用原因：稳定性统计只允许测随机性，不允许混入协议漂移。
- 不采用的相邻方案：不采用某个 seed 单独改阈值、增强、batch size、后处理、`threshold` 前 `dtype`、`Boundary F1` 宽度/实现或连通域实现/连接性；不采用漂移后仍并入正式聚合。

- 没有哪个 seed 的结果彻底失序却解释不清。

- 三次正式 run 中任意一次缺少关键资产。

### 5.4 当前阶段允许存在但必须写入结论的问题
