## 2. 统一数据、评估和信息泄漏协议

1. `GlaS` screening 只允许消费 `train68 / val17`；所有模型选择、阈值、轮廓宽度、marker 参数和机制门均只看 `val17`。
2. `GlaS TestA / TestB` 与 `CRAG test` 为盲态测试集：在最终配置、checkpoint、阈值和后处理锁定前不得读取、比较、调参或回退。
3. 正式结果必须分开报告 TestA、TestB，不得合并成一个隐藏平均值；CRAG test 单独报告。
4. 训练/验证输入默认 `512×512`，预处理、增强、loss、optimizer、scheduler、threshold 和 batch size 必须写入运行元数据；若执行层已有冻结协议，优先继承，不得为救结果临时改动多个变量。
5. 指标至少包括 `F1 / Object Dice / Object Hausdorff`；补充 `Dice / IoU / HD95 / Boundary F1`。唯一 Boundary F1 身份固定为：`project_custom_erosion_xor_3x3_ones_8conn_border0_tol3px`。任何同名但实现身份不同的指标不得混入同一列。
6. 原 `04 std gate` 仅是项目内部稳定性参考，不是投稿门槛；不得把它写成录用条件、硬阈值或效果承诺。

## 3. 路线依据与文献依据

### 3.1 路线依据

救援路线先冻结可审计的 B1 资产，再一次只引入一个与实例分离直接相关的机制：N1 产生实例感知轮廓监督，N2 在区域与轮廓之间做轻量融合，N3 将轮廓概率转为 marker-controlled watershed 的实例恢复。这样可把“分割错误”“边界监督错误”“实例后处理错误”分层定位，避免旧 LKMA/Boundary/Distance 链继续扩大变量面。

Plan B 采用 `DCAN-inspired object-contour joint inference adaptation` 作为机制失败时的独立回退候选，仅在 N3 validation 机制门明确显示轮廓引导实例恢复不成立时触发。它不是 strict replication：必须逐项登记相对 DCAN 原文的结构、target、loss、推理和后处理差异；也不是 N3 的隐含改版或可先跑的第四个主线模型。

### 3.2 文献依据

- **DCAN**：支持对象分支与轮廓分支联合建模、利用轮廓分离相邻腺体的机制依据；本项目 Plan B 必须严格保持 object-contour joint inference 的语义，不得只借名改成普通 boundary head。
- **边界/轮廓监督路线**（如 Gated-SCNN、TA-Net、DEA-Net 等）：支持将边界作为显式监督信号，但不预设本项目一定获得同方向收益。
- **实例分离与 marker-controlled watershed**：支持以前景/轮廓生成内部 marker，再以轮廓或代价图控制分水岭实例恢复；该后处理必须只在验证集锁参后进入盲态测试。
- **GlaS/CRAG 任务文献**：用于任务内 direct comparison、指标定义和写作定位。原文数字只能以 `quoted from original paper` 标记，不能伪装成统一协议复现。

## 4. 规则卡片：当前主线

### B1 历史只读

- **当前结论**：`valid_with_stability_warning`；只读、可引用、不可重跑改写。
- **规则类型**：历史 baseline 证据源。
- **适用阶段**：N1-N3 的共同对照。
- **直接依据**：历史运行资产、checkpoint、run_meta、评估记录及 lineage。
- **采用原因**：保留真实历史结果，避免把旧路线重新包装成当前主线。
- **不采用的相邻方案**：不恢复 `LKMA→Boundary→Distance`，不把历史 C1/D1/D2 作为新实验。
- **代码落点**：`planned_not_created`：B1 artifact loader、只读 manifest 校验接口。
- **运行记录字段**：`status=historical_readonly`、`validity=valid_with_stability_warning`、`source_*` lineage、checkpoint digest、评估脚本身份。
- **验收方式**：资产可追溯、只读校验通过、测试结果与历史记录一致；不以旧 std gate 作为投稿门槛。

### N1 实例感知轮廓监督

- **当前结论**：`planned/not_run`。
- **规则类型**：单变量主线阶段 1。
- **适用阶段**：B1 冻结后、N2 前。
- **直接依据**：DCAN 的对象/轮廓解耦思想与边界监督文献。
- **采用原因**：先验证显式轮廓监督是否改善相邻对象可分性。
- **不采用的相邻方案**：不加入 LKMA、distance loss、复杂多级边界分支或后处理。
- **代码落点**：`planned_not_created`：`contour_target(project_custom_erosion_xor_3x3_ones_8conn_border0_tol3px)`、`object_head`、`contour_head`、联合 loss 接口。
- **运行记录字段**：`run_name`、`status`、`seed`、`source_*`、target identity、head/loss config、checkpoint、val metrics、artifact manifest。
- **验收方式**：val 上对象级/边界级证据与错误案例可回查；不使用 TestA/TestB/CRAG test 作选择。
- **核心公式或定义参考**：`C = erosion(M, 3×3, ones, 8-connectivity, border=0) XOR M`，再按固定容差评估；联合目标至少显式记录 object loss 与 contour loss 的权重，具体冻结值由执行前 pre-check 锁定。

### N2 区域/轮廓轻量融合

- **当前结论**：`planned/not_run`。
- **规则类型**：单变量主线阶段 2。
- **适用阶段**：N1 验收通过后。
- **直接依据**：区域语义与轮廓特征互补的任务文献和 N1 的验证资产。
- **采用原因**：验证轻量融合是否改善区域完整性而不吞没轮廓信号。
- **不采用的相邻方案**：不堆叠大核、多尺度 attention、distance-aware loss 或 watershed。
- **代码落点**：`planned_not_created`：`region_contour_fusion` 模块接口，输入/输出通道和融合位置由 stage card 冻结。
- **运行记录字段**：继承 N1 全部字段，增加 `fusion_location`、`fusion_type`、`param_count`、`flops_or_latency`。
- **验收方式**：只与同协议 N1 比较，要求记录区域、轮廓、对象和成本证据；未通过则独立回退 N1。

### N3 轮廓引导 marker-controlled watershed

- **当前结论**：`planned/not_run`；正式 seed 仅 `3407/1234/2025`。
- **规则类型**：主线阶段 3 / 机制门候选。
- **适用阶段**：N2 验收通过后。
- **直接依据**：轮廓引导实例恢复与 marker-controlled watershed 文献原理。
- **采用原因**：把轮廓概率转为可解释的实例分离步骤，专门处理黏连腺体。
- **不采用的相邻方案**：不将普通 argmax、任意连通域清理或未注册形态学后处理冒充 N3；不预先运行 Plan B。
- **代码落点**：`planned_not_created`：`marker_generator`、`marker_controlled_watershed`、`postprocess_config`、实例级评估接口。
- **运行记录字段**：继承 N2 字段，增加 `marker_source`、`contour_threshold`、`min_distance`、`watershed_connectivity`、`compactness`、`postprocess_version`、`val_lock_digest`。
- **验收方式**：先仅在 val 锁定机制参数，再以锁定配置运行盲态测试；三 seed 分别保存，TestA/TestB/CRAG test 不反馈。
- **核心公式或定义参考**：以前景概率产生内部 marker，以轮廓概率/代价图构造阻隔，再执行 marker-controlled watershed；所有阈值和连通性必须显式记录，禁止运行后补写。

## 5. Plan B 机制门与独立回退

Plan B 是 `DCAN-inspired object-contour joint inference adaptation`，状态固定为 `planned_only`。只有当 N3 在 `val17` 出现以下机制门证据之一，才允许创建并运行 Plan B：轮廓监督输出不具备可分离结构、marker 在多个 seed 上系统性失效、或 watershed 对黏连对象产生不可解释的过分割/欠分割，且已排除 target、评估和实现错误。

机制门必须由 `val_mechanism_gate.json` 记录：触发原因、排除项、对照图、N3 run IDs、批准人/时间和允许的 Plan B 配置。Plan B 仍不得读取盲态测试结果后触发。若 Plan B 运行，必须沿用相同 split、盲态边界、指标身份和 lineage，并与 N3 独立验收；Plan B 不通过则回到 N2 或 B1，不得再开第三条路线。

## 8. 阶段验收与写作交接

每阶段验收必须同时检查：实现身份、数据边界、训练可复现性、val 机制证据、对象级指标、边界指标、可视化、失败案例、成本和 lineage。任何新结果都只能在实际运行记录、checkpoint 和 manifest 三者齐全后写入结果汇总。

投稿写作只消费 `accepted` 的结果资产：主线消融表（B1/N1/N2/N3）、GlaS TestA/TestB 分开表、N3 三 seed 的 mean±std、唯一 Boundary F1 身份、CRAG test 盲态结果、失败案例、复杂度/统计语义和 direct comparison 来源标记。结果未运行时，论文只能写 planned protocol，不能写“提升”“优于”“达到”。

## 9. 独立回退条件

- B1 资产、split、指标或接口不一致：停止当前阶段，回退到 pre-check/协议修复，不训练。
- N1 不通过：保持 B1，修复 contour target 或实现；不得进入 N2。
- N2 不通过：删除融合，独立回到 N1；不得借 N3 后处理掩盖问题。
- N3 机制门不通过且非可修复实现错误：按记录触发 Plan B；Plan B 未获门批准不得运行。
- N3 与 Plan B 均不通过：回到 N2；若 N2 也不成立，回到 B1，形成中文学报的诚实 baseline/方法失败报告，不继续扩模块。
- 盲态泄漏、测试集调参、指标身份漂移或 lineage 缺失：结果作废，回退到最近一个未泄漏的阶段并重建证据包。
