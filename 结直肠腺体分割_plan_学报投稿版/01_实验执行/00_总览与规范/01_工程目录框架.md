## 4. 阶段门控表达式
目录只有在 active root 可定位、资产有 manifest、测试标签隔离且 handoff 七字段齐全时才允许被下游消费。

## 上游、同层与下游
上游是导航和参数/评估协议，决定目录必须支持的对象；同层命名和代码映射文件规定资产身份与入口；下游数据、方法、验证、对比、汇总和验收阶段消费目录中的配置、manifest、运行证据和报告。`external/` 只接参考实现，不能绕过 active `src/` 评估链。

## 路线依据与文献依据
目录服从 `01 -> 11` 当前编号路线，N1/N2/N3 只能是 `05_方法协议` 下的方法身份。GlaS 的 split 和 TestA/TestB 分离、DCAN 的 object/contour 证据、MILD-Net 的同任务比较需求，决定数据、目标、评估和结果资产必须分层保存。旧工程目录可说明接口演进，但不提供 current 结果。

## 正式目录与资产边界
```text
crc_gland_segmentation_project_journal/
  configs/{base,data,stages,models,train,eval}/
  datasets/{raw,interim,processed}/
  splits/{glas,crag}/
```

`configs` 只存协议；`datasets` 存来源与数据；`splits` 存清单及哈希；`src` 存唯一正式实现；`experiments` 存事实资产；`reports` 存 Gate/handoff；`external` 存只读适配来源；历史归档不得被 current Gate 扫描。

## 固定、可调与禁止参数
- 固定：active project root、目录职责、GlaS/CRAG 分目录、预测与标签隔离、报告与运行资产分离、七字段 handoff。
- 可调：阶段内新增模块的文件名和配置字段，仅在 stage card、研究定标和冲突裁决后登记。
- 禁止：把训练结果放 `src/`、把 split 放 `reports/`、从历史工程复制结果、外部实现自行重划数据、在目录名中把 N1/N2/N3 当阶段号。

## 7. 目录验收分层
- 路径层：active root、configs、datasets、splits、src、scripts、experiments、reports 可定位。
- 资产层：配置、manifest、run_meta、checkpoint、预测和指标分目录保存。
- 代码层：训练、推理、评估和汇总入口职责不互相替代。
- 证据层：每个阶段报告能回链运行目录和 protocol version。
- 治理层：Gate 与 handoff 只读事实资产，不从历史归档生成。

## 8. 目录消费边界
数据协议只生产清单和哈希；训练只消费 train/val；评估只消费冻结预测和对应标签；汇总只消费通过验收的 raw 资产。`external/` 只能作为适配来源，不能写入正式主指标。

## 9. 当前禁止动作
不在本目录协议中创建实验资产，不把目录存在误写成代码已实现，不把旧工程复制物作为 current，不启动训练或测试。planned 接口只能在 stage lock 后创建。

## 10. 目录交接记录
目录变更必须同步 manifest、config、run_meta、报告和 handoff；目录存在不等于接口已实现，接口存在不等于运行已完成。历史归档始终只读，current Gate 只能扫描 active root。

## 11. 路径核对结果
本轮正文中的正式路径均指向 active journal 或只读参考资料；发现无法解析的占位路径时，改为 `planned_not_created` 并记录原因。

## 工程异常与恢复规则
- 目录创建失败、权限不足或磁盘不足时，训练入口必须在写入前失败，不得写入半成品 manifest。
- 发现 active 目录混入旧 root、历史 checkpoint 或 runtime probe 时，先停止运行，按 A/B/C 归档后重新生成当前目录。
- 同一 run 目录不得由两个 Python 训练进程同时写入；检测到并发 PID 时，当前 run 判为污染，不得继续消费。
- 每个输出目录必须记录 `created_at_utc`、`project_root`、`run_name`、`protocol_version` 和文件清单摘要。

## 目录验收字段
- `directory_manifest.json`：记录目录角色、路径、文件类别、sha256 和消费边界。
- `run_meta`：记录配置、代码、数据、评估和 checkpoint identity。
- `archive_manifest.md`：记录所有被归档目录的原路径、归档路径、原因和是否允许 current 消费。
- 只有 manifest、run_meta、Gate 和 handoff 的路径一致时，目录才可进入下游。

## 工程落点的可执行解释
- “当前已创建”只允许指向 active journal 中实际存在的文件；“planned interface”只表示接口名称和预期职责，不表示代码已经存在。
- N1 当前可以作为方法实现对象，但其正式训练/评估资格仍受全局阶段 Gate、真实 GlaS preflight 和正式评估入口约束。
- N2 fusion 与 N3 postprocess 当前只登记接口，不得在研究、代码、结果或投稿文档中写成已实现模块。
- 历史工程路径只写在来源和 provenance 表中，不进入当前 active 目录表，不参与当前 Gate。
- `src/` 只放当前实现代码；`scripts/` 只放当前可执行入口；`configs/` 只放冻结配置；`experiments/` 只放当前轮产生的 A 类资产；`reports/` 只放当前轮可回查报告；`b_class_auxiliary/` 只放 B 类辅助门禁和审计证据。
- 任何 Markdown 计划正文不得承载实验数字，任何 metrics/CSV/JSON 结果不得由人工说明文替代。
- 代码、配置、结果和 handoff 使用相同的 `protocol_version` 与 `run_name`；发现字段不一致时先阻断下游，不允许靠目录名称猜身份。
- 阶段切换时，上一轮 active 资产进入对应 `_historical_archive`，当前目录只保留当前轮对象；归档必须有 manifest，且明确 `historical_archive_only` 和 `valid_for_current_gate`。
- 配置文件必须记录其上游计划文件、数据文件、模型文件、评估协议和 seed 绑定；缺任一绑定不得启动正式 run。
- 报告文件必须记录生成脚本、命令、退出码、生成时间和输入 digest；手工创建的摘要不能作为 Gate 证据。
- 目录表中的“存在”与“可消费”分开记录：文件存在但未通过 runtime、独立复核或阶段 Gate 时，只能标记为开发/历史资产。
- 任何跨阶段 handoff 必须携带文件清单、sha256、producer、consumer 和消费边界，缺字段时回到当前阶段，不得继续下游。

## Diagnostics 闭环
本轮运行正式文档检查和路径检查；路径不存在时记录 `planned_not_created`，不得伪装为已创建。当前不启动实验。检查后还需逐项确认 active root、planned interface、归档边界、run 目录覆盖策略和并发写入规则；任一项不一致则回写本文件并重新运行 diagnostics。
