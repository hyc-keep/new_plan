# Distance 阶段总协议

本文件不是 `07_Distance` 目录旧稿的补充说明，也不是“先把距离约束试几版再回头整理”的阶段记录。

它在当前项目中的唯一职责是：

> 作为 `07_Distance` 阶段的正式主协议，把 `Distance-aware Loss as the third post-baseline optional enhancement / Single-variable distance supervision / Frozen euclidean target protocol / Keep-downgrade-drop decision with distance evidence` 一次写死；代码、实验、阶段汇总和后续写作只允许忠实实现与验证这份协议，不允许一边训练一边重新决定距离图怎么生成、是否切到 `signed distance map`、`lambda_dist` 扫多大、什么情况下还能算值得进入主线。

从现在开始，`07_Distance` 子目录下的其他文件都视为本文件的细化与执行展开；若与本文件冲突，以本文件为准回改。

## 计划 lineage 与下游 handoff

## Diagnostics 闭环

## 1. 文件角色与执行边界

### 1.1 当前文件负责什么

当前文件只负责冻结下面七件事：

1. `Distance-aware Loss` 作为 `Boundary` 之后第三模块的唯一身份、阶段目标和单变量边界。
2. `current_base -> current_base + Distance-aware Loss` 的正式比较起点、阶段门、搜索边界和交接语义。
3. `Distance-aware Loss v1` 的模块定义、距离图来源、距离图定义、归一化规则、`lambda_dist` 候选与主版本优先级。
4. `Distance` 阶段何时允许 `keep`、何时只能 `downgrade`、何时必须 `drop` 并回退到不带距离项的 `current_base`。
5. `Distance` 阶段最低交付物、正式资产、handoff 语义以及对 `08_外部对比` 与 `09_CRAG验证` 的唯一主线传递方式。

### 1.3 为什么当前文件必须独立存在

`Distance` 阶段最容易出现的伪完成状态不是“完全没跑”，而是：

- `Boundary` 还没有形成唯一合法 `current_base`，就提前启动距离监督验证。
- 一边改距离图定义、一边改 `lambda_dist`、一边回改边界配置，最后无法确认收益是不是来自距离监督本身。
- screening run 已经跑通，却没有形成三 seed 主版本、成本记录、主版本优先门和 `keep / downgrade / drop` 正式结论。
- `08_外部对比` 和 `09_CRAG验证` 想继承主线时，说不清应该读 `distance_kept_base` 还是 `distance_input_base`。

如果没有一份独立阶段总协议，后续 `08_外部对比`、`09_CRAG验证`、阶段汇总脚本和论文消融表都会再次出现“Distance 做过了，但到底按哪套变量边界、哪套资产链、哪一版结论继续”不一致的问题。

## 3. 本阶段唯一允许处理的变量

### 3.1 当前阶段真正要证明什么

`07_Distance` 阶段真正要证明的，不是“在当前主版本后面再加一个 loss 项，也许还能涨一点”，而是：

> 在 baseline、`LKMA` 和 `Boundary` 的结构、训练、评估与比较协议全部冻结的前提下，`Distance-aware Loss` 作为第三层 post-baseline 可选增强，是否能够以最小单变量设置提供可复查、可解释、可交接的边界邻域连续监督收益，并因此获得 `keep / downgrade / drop` 的正式阶段结论。

因此，本阶段通过的含义同时包括：

- `Distance-aware Loss` 已经被压缩成真实的单距离图定义、单归一化规则、单主版本 `lambda`、单附加损失的可解释比较。
- `current_base vs current_base + Distance-aware Loss` 的差异来自距离监督本身，而不是训练、评估或边界协议漂移。
- `Distance` 的正式资产、比较表、边界指标表、成本表、局部案例和 decision note 已经齐全到可被 `08_外部对比` 与 `09_CRAG验证` 直接消费。

### 3.2 当前阶段不允许用什么替代正式结论

当前明确不允许把下面这些情况写成“Distance 已完成”：

- 只跑了一个 screening run 就宣称距离监督有效。
- 同时试多个距离图定义、多个 `lambda_dist`、多个相邻损失，却仍把结论写成单变量比较。
- 只看单次高值或像素级指标，不看 `Boundary F1`、`HD95 / Object Hausdorff`、对象级主指标、可视化和成本。
- 只写“距离监督可能有用”，不形成 `keep / downgrade / drop` 正式结论和下游起点。
- 主版本不过线，却靠扩大搜索空间、切到 `SDM` 或骨架路线继续强找保留理由。

### 3.3 为什么当前阶段必须拆成多层阶段门

`Distance` 最容易发生的错误是把“代码能跑、局部图看起来更平滑”误写成“模块值得进入主线”。

因此，当前阶段必须把放行条件拆成：

- 上游基座门：负责确认当前比较建立在已经正式放行的 `current_base` 上。
- 运行完整门：负责确认 screening、三 seed 主版本和最低资产齐全。
- 比较就绪门：负责确认公平比较、边界级指标、对象级指标、可视化和成本证据全部到位。
- 决策门：负责确认 `keep / downgrade / drop` 结论已经形成，且主版本优先门已经执行。
- 交接门：负责确认 `08_外部对比` 与 `09_CRAG验证` 已能读取唯一合法 `current_mainline` 语义。

## 4. 设计边界(做什么/不做什么)

### 4.1 当前阶段直接继承哪些冻结项

本文件直接继承并不得擅自改动的上位规则如下：

- 主任务：`2D gland segmentation`
- 主数据集：`GlaS`
- 正式 split：`train68 / val17 / testA60 / testB20`
- 标签口径：`mask > 0`
- 输入通道：`RGB`
- 输入尺寸：`512 x 512`
- 归一化：`ImageNet mean/std`
- 增强版本：沿用当前主线已冻结版本
- 主损失：`L_seg = L_BCE + L_Dice`
- 优化协议：`AdamW + lr 1e-3 + weight_decay 1e-4 + ReduceLROnPlateau + epoch_max 120 + early_stopping 20 + AMP on`
- 主指标优先级：`F1 / Object Dice / Object Hausdorff`
- 补充指标：`Dice / IoU / HD95 / Boundary F1`
- `best_selector = val_objdice_max`
- `threshold_source = val17`
- `train_proto_version = train_proto_v1`
- `eval_proto_version` 由 04 阶段当前轮 Gate 冻结
- `postprocess_version`：当前轮待由研究定标、阶段锁定、Pre-check/Gate 冻结；`none_in_v3` 仅作历史 lineage，不是当前值。
- `eval_cast_policy = float32 before thresholding`
- `boundary_metric_width = 3 px`
- `boundary_metric_impl = project_custom_erosion_xor_3x3_ones_8conn_border0_tol3px`
- `connected_components_impl = scipy.ndimage.label`
- `connected_components_connectivity = 8-connectivity`
- baseline 正式基座：`ResNet34-U-Net`
- baseline encoder 初始化：`official ImageNet-1K supervised weights`
- 固定三 seed：`3407 / 1234 / 2025`
- `LKMA` 与 `Boundary` 的正式结论、字段链和 handoff 资产必须完整继承

### 4.2 当前阶段唯一合法起点

当前阶段的比较基座不是重新选择，而是由 `Boundary` 阶段正式结论唯一决定：

```text
current_base = (boundary_decision_level == keep) ? boundary_kept_base : boundary_input_base
```

这意味着：

- 只有 `06_Boundary` 已形成正式结论并完成 handoff，`Distance` 才允许启动。
- `backup` 与 `drop` 都不能被默认当成“继续把 `Boundary Head` 带进主线”。
- 任何正式 `Distance` run 都必须能追溯到 `Boundary` 的 decision note、stage manifest 和 `current_base` 说明。
- `Boundary` handoff 中写出的 `current_base_next` 必须与本阶段读取的 `current_base` 完全一致；`07_Distance` 只允许消费 `current_base_next = (boundary_decision_level == keep) ? boundary_kept_base : boundary_input_base` 这一唯一交接语义。

### 4.3 当前阶段唯一变量

本阶段唯一允许处理的核心变量只有“在 `current_base` 上追加一个可插拔 `Distance-aware Loss` 及其最小距离监督协议”：

- `Distance-aware Loss v1` 的模块定义
- 距离图来源与前景定义
- 距离图正式定义与归一化
- `lambda_dist` 候选集合与主版本
- 距离监督是否带来可接受的复杂度成本
- `Distance-aware Loss` 是否值得进入后续主线

本阶段明确不允许借机改动：

- baseline backbone
- encoder 预训练策略
- `LKMA` 配置
- `Boundary` 标签、宽度、损失和接入位置
- decoder 主干
- 数据协议
- 输入尺寸
- 主增强包
- 主损失
- optimizer / scheduler / lr
- `best checkpoint` 选择器
- 阈值来源
- 后处理
- `08_外部对比` 与 `09_CRAG验证`
- `signed distance map`、`direction field`、`marker branch`、`watershed` 和 skeleton/topology 表示

### 4.4 当前阶段正式阶段门

当前文件固定采用下面的阶段门语义：

```text
Gate_E1 = current_base_ready
          AND distance_main_runs_complete
          AND distance_compare_ready
          AND distance_assets_ready
          AND distance_decision_ready
          AND handoff_ready
```

其中：

- `current_base_ready`：上游 `Boundary` 已给出唯一合法 `current_base`
- `distance_main_runs_complete`：主版本 screening 和三 seed 正式结果完整
- `distance_compare_ready`：公平比较、边界级指标、对象级指标、可视化与成本证据齐全
- `distance_assets_ready`：per-seed 表、`mean+-std` 表、成本表、案例和 decision note 已导出，且 `eval_cast_policy / boundary_metric_width / boundary_metric_impl / connected_components_impl / connected_components_connectivity` 已同步写入 config.yaml、run_meta.yaml、per-seed raw、`mean+-std`、compare 表、distance_stage_manifest.csv 与 handoff 资产
- `distance_decision_ready`：`keep / downgrade / drop` 结论已形成，且主版本优先门已执行
- `handoff_ready`：`08_外部对比` 与 `09_CRAG验证` 可直接读取下一阶段合法主线

这意味着：

- `Distance` 阶段不是“先跑完再说”，而是必须在上游、运行、比较、决策和交接同时闭环后才允许进入 `08_外部对比`。
- 即使主版本三 seed 已完成，只要 `keep / downgrade / drop` 结论、主版本优先门或 handoff 资产还没形成，也不得视为阶段通过。

## 5. 如何判断设计有效(验收标准)

下面所有核心规则都写成正式协议格式；后续实现、实验和阶段汇总时，只允许把这些规则忠实落地，不允许擅自补新的隐含边界。

### 5.1 第三模块身份与作用定义

- 当前结论：后 `Boundary` 的第三模块固定为 `Distance-aware Loss`，其角色固定为 `边界邻域连续监督 + 局部轮廓细化 + 残余模糊抑制`，而不是新的主结构模块或 topology 主线。
- 规则类型：`路线层已锁定 + 工程冻结规则`
- 适用阶段：`07_Distance` 启动时生效，后续是否继承取决于本阶段结论。
- 直接依据：`结直肠腺体分割_plan_优化版/02_路线与投稿/02_结直肠腺体分割_分阶段实验路线与执行标准.md`、02_Distance-Map-Loss.md、`结直肠腺体分割_plan_优化版/01_实验执行/07_Distance/01_设计依据.md`
- 核心公式或定义参考：当前任务残余误差固定为 `boundary neighborhood instability / residual contour blur / post-boundary local refinement`，模块职责不包括主结构替换或多拓扑分支。
- 采用原因：腺体分割在整体形态与显式边界监督较稳后，下一层最值得单独验证的是边界邻域的连续距离约束。
- 不采用的相邻方案：不把 `Distance-aware Loss` 写成默认必保留模块；不把它写成完整新 backbone；不把它和骨架拓扑表示混成同一阶段。
- 代码落点：[project_root]/src/losses/distance_losses.py、[project_root]/configs/experiment/distance_e1_v1.yaml
- 运行记录字段：`dist_version`、`dist_role`、`distance_type`、`base_variant`
- 验收方式：检查阶段文档、配置和实现都明确 `Distance-aware Loss` 只负责距离监督增强，没有顺手引入 `signed distance`、skeleton 或 topology loss 搜索。

### 5.2 `Distance-aware Loss v1` 模块定义

- 当前结论：`Distance-aware Loss v1` 固定为 `single auxiliary distance loss attached to the frozen segmentation objective` 的可插拔附加项，不采用完整 `SDM / clDice / topology-aware` 原版扩展路线。
- 规则类型：`工程冻结规则 + 文献约束下的任务化抽象`
- 适用阶段：`07_Distance`
- 直接依据：`结直肠腺体分割_plan_优化版/01_实验执行/00_总览与规范/05_代码工程映射与实现策略.md`、02_Distance-Map-Loss.md、04_Shape-Aware-SDM.md、05_clDice.md、14_SkeletonAwareDT.md
- 核心公式或定义参考：首版训练目标固定为 `L_total = L_seg + lambda_dist * L_dist`
- 采用原因：当前项目需要的是可解释、可插拔、易复查的首版距离监督，而不是在 `Distance` 阶段直接重开新的损失家族路线。
- 不采用的相邻方案：不采用 `signed distance regression` 主线；不采用 topology-preserving skeleton loss；不采用多输出形状分支联合搜索。
- 代码落点：[project_root]/src/losses/distance_losses.py
- 运行记录字段：`dist_version`、`dist_loss_type`、`use_distance_loss`
- 验收方式：检查距离约束主体仍是单附加 loss，且主分割目标未被替换或弱化。

### 5.3 距离图来源规则

- 当前结论：距离图唯一正式来源固定为数据阶段冻结的二值 gland mask；不允许从预测结果、后处理结果或分析脚本反推。
- 规则类型：`工程冻结规则 + 标签定义规则`
- 适用阶段：`07_Distance`
- 直接依据：`结直肠腺体分割_plan_优化版/01_实验执行/07_Distance/02_距离图与lambda范围.md`、`结直肠腺体分割_plan_优化版/01_实验执行/00_总览与规范/02_参数冻结总表.md`
- 核心公式或定义参考：`M(x) = 1[mask(x) > 0]`
- 采用原因：距离监督必须来自同一套可追溯标签定义，否则训练、可视化和分析会口径漂移。
- 不采用的相邻方案：不采用从预测 mask 反推距离图参与训练；不采用训练脚本和分析脚本各写一套来源定义。

- 验收方式：检查训练、可视化和分析共用同一套距离图来源定义；检查距离图全部来自冻结二值 mask。

### 5.4 距离图定义与归一化规则

- 当前结论：首轮唯一允许的正式距离图定义固定为 `Euclidean distance map`，并统一归一化到 `[0, 1]`。
- 规则类型：`论文支持的候选范围 + 工程冻结规则`
- 适用阶段：`07_Distance`
- 直接依据：`结直肠腺体分割_plan_优化版/01_实验执行/00_总览与规范/02_参数冻结总表.md`、`结直肠腺体分割_plan_优化版/01_实验执行/07_Distance/02_距离图与lambda范围.md`、02_Distance-Map-Loss.md
- 核心公式或定义参考：`D = EDT(M)`；`D_norm = Normalize01(D)`
- 采用原因：`Euclidean distance map` 是当前最小、最稳、最容易实现和最容易复查的距离表示，足以验证距离约束是否有补充价值。
- 不采用的相邻方案：不采用 `signed distance map`；不采用 `direction field`；不采用 `MA distance map`；不采用骨架距离或复杂拓扑场。

### 5.5 `lambda_dist` 候选与首轮主版本

- 当前结论：`lambda_dist` 候选固定为 `0.05 / 0.1 / 0.2`，首轮主版本固定为 `0.1`。
- 规则类型：`论文支持的候选范围 + 工程冻结规则`

- 采用原因：`0.1` 足以让距离项产生可见影响，同时仍维持“主分割目标主导、距离项辅助”的角色关系；`0.05` 与 `0.2` 只承担必要时的弱/强补点职责。

### 5.6 搜索边界与正式执行链

- 当前结论：`Distance-aware Loss` 是否保留，必须按 `Boundary F1 -> HD95 / Object Hausdorff -> 局部轮廓可视化 -> Object Dice / F1 -> 成本变化` 的固定顺序组织证据，并分为 `keep / downgrade / drop` 三档；其中正式证据只承认 `eval_cast_policy / boundary_metric_width / boundary_metric_impl / connected_components_impl / connected_components_connectivity` 已冻结且全链留痕后的结果。

- 运行记录字段：`decision_level`、`decision_reason`、`distance_visual_support`、`cost_assessment`

### 5.9 主版本优先门规则

## 7. 回退条件

### 7.1 独立回退触发条件

- `Boundary` 实际尚未形成唯一合法 `current_base`，却已经启动正式 `Distance` 比较。

- `Distance` 平均边界级主指标明显低于当前基座，且可视化不支持距离增益。
- `TestB` 明显更差，或者某个 seed 波动失控且没有排查记录。
- 成本明显上升但没有形成清晰收益证据，却仍试图给出 `keep` 结论。
- 主版本不过线，却想通过扩大搜索空间、切换距离图定义或引入相邻路线强行保留模块。
- `keep / downgrade / drop` 结论与 `08_外部对比`、`09_CRAG验证` 的 `current_mainline` 说明不一致。

### 7.2 固定回退顺序

回退时统一按下面顺序排查：

1. 检查距离图来源、前景定义和生成脚本是否正确。
2. 检查距离值归一化范围、数值分布和可视化对齐是否正确。
3. 检查距离损失和 `lambda_dist` 是否失衡。
4. 检查当前基座训练与评估协议是否真正继承。
5. 只有前面都确认无误后，才允许决定是否补唯一一个备选版本，还是直接降级为 `downgrade / drop`。

- 前置断言：输入 mask 必须来自冻结二值 gland 标签；训练、可视化和分析必须复用同一套距离图生成实现。
- 运行产物：训练时的距离监督张量、局部 heatmap、run_meta.yaml 中的距离图字段。

### 8.2 `Distance` 模块注入与单 run 执行入口

- 代码文件：[project_root]/src/losses/distance_losses.py、[project_root]/src/models/r34_unet.py、`crc_gland_segmentation_project/scripts/train.py`、[project_root]/scripts/test.py
- 入口类/函数：`compute_distance_loss()`、`attach_distance_loss()`、`main(cfg)`、`run_eval()`
- 输入：主分割 logits、距离图张量与组合后的 `data/train/model/eval` 配置
