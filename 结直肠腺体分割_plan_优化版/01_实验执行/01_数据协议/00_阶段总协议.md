## 1. 任务定位

### 1.1 本阶段要完成的不是“整理数据”，而是“冻结数据协议”

本阶段通过的含义不是数据已经下载、解压、能打开，而是下面八件事已经同时固定并可被下游继承：

1. 正式数据资产到底是哪一版 `GlaS` 和哪一版 `CRAG`
2. 正式 `train / val / test` 边界是什么，后续不得重开
3. 原始标注如何转成主线统一消费的正式标签
4. 边界图、距离图等派生监督目标如何从正式标签生成
5. 数据检查、统计和人工抽查的最低闭环是什么
6. 数据阶段正式交接物是什么，放在哪些固定目录
7. 数据阶段通过状态如何写成 `data_stage_pass / handoff_ready / asset_manifest`
8. 什么情况下允许进入 `02_UNet流程验证`，什么情况下必须回退

### 1.2 本阶段在整条实验路线中的角色

本阶段对应论文中的四块核心内容：

- `Datasets`
- `Train / Validation / Test Protocol`
- `Label Preprocessing`
- `Data Quality Control`
- `Data Handoff and Preflight Gate`

如果这里没有先冻结，后面的 `UNet`、`Baseline`、`LKMA`、`Boundary`、`Distance` 都可能只是在吃不同数据口径，而不是在比较方法差异。

### 1.3 本阶段为什么必须排在第一位

原因不是流程习惯，而是实验可解释性要求：

- 主线 benchmark 角色已经在路线层锁定
- 对象级主指标要求 split 边界和标签口径稳定
- 训练、验证、测试、可视化和论文复盘都要共享同一套数据资产包

因此，数据阶段不是准备动作，而是整个实验系统的第一道正式门。

## 4. 阶段门控与唯一变量

### 4.1 本阶段唯一允许处理的变量

本阶段只允许处理数据协议本身：

- 数据源、来源说明与正式目录约定
- `GlaS / CRAG` 的正式 split 生成
- 二值标签、resize、边界图和距离图规则
- 数据检查、统计汇总和人工抽查
- 数据阶段交接资产与放行门槛

### 4.2 本阶段主门控表达式

数据阶段是否通过，统一按下式判断：

```text
pass_data_stage = pass_source and pass_split and pass_pair and pass_label and pass_check and pass_preview and pass_handoff
```

同时还要满足三条总体约束：

```text
assets_traceable = True
protocol_explainable = True
red_flag = False
```

进入 `02_UNet流程验证` 前，还必须把数据阶段通过状态继续压成可被训练入口读取的交接字段：

```text
handoff_ready = True
preflight_pass = True
```

这里的 `handoff_ready` 表示正式数据资产包已经齐全，`preflight_pass` 表示下游 `UNet` 训练入口已经确认自己只会读取本阶段冻结下来的正式数据资产，而不会回退到临时目录或重新扫描原始数据。

### 4.3 为什么必须压成单变量阶段

如果数据阶段同时夹带模型结构、训练策略或损失设计，后面任何指标变化都无法判断到底来自：

- 数据资产变化
- 标签生成变化
- 还是模型设计变化

所以本阶段必须严格保持“只冻结数据协议，不顺手改别的变量”。

### 5.5 二值标签规则

- 核心公式或定义参考：对任意像素 `p`，当 `m(p) > 0` 时 `y(p) = 1`，否则 `y(p) = 0`
- 采用原因：先保证主线是稳定的语义分割闭环，同时保留从实例标注派生对象级评估的可追溯性

### 5.6 resize 与插值规则

- 当前结论：图像 resize 使用双线性插值，mask resize 使用最近邻插值，默认输入尺寸固定为 `512x512`
- 规则类型：`工程冻结规则`
- 适用阶段：`01_数据协议` 定义，`02_UNet流程验证` 起执行，后续继承

- 核心公式或定义参考：`x' = BilinearResize(x, 512, 512)`；`y' = NearestResize(y, 512, 512)`

- 不采用的相邻方案：不采用图像和 mask 同用双线性；不采用 resize 后再靠阈值补救
- 代码落点：[project_root]/src/data/transforms.py，`configs/data/*.yaml`

### 5.7 归一化与 stain normalization 规则

- 当前结论：首轮正式主线统一使用 `ImageNet mean/std`，且 `stain_normalization = off`
- 规则类型：`工程冻结规则`
- 适用阶段：`01_数据协议` 定义，至少继承到 `04_Baseline`
- 直接依据：`结直肠腺体分割_plan_优化版/01_实验执行/00_总览与规范/02_参数冻结总表.md`，`结直肠腺体分割_plan_优化版\01_实验执行\01_数据协议\02_数据检查与配对规则.md`，`结直肠腺体分割_plan_优化版\01_实验执行\01_数据协议\05_标签转换与可视化规则.md`

### 5.8 边界图规则

- 当前结论：边界图从正式二值 mask 派生，首轮默认边界宽度 `w = 3 px`，唯一备选为 `5 px`
- 规则类型：`论文支持的候选范围 + 工程冻结规则`
- 适用阶段：`01_数据协议` 先定义，`06_Boundary` 正式冻结
- 直接依据：`结直肠腺体分割_plan_优化版\01_实验执行\01_数据协议\05_标签转换与可视化规则.md`，04_MILD-Net.md，`结直肠腺体分割_plan_优化版/01_实验执行/00_总览与规范/02_参数冻结总表.md`
- 核心公式或定义参考：`b_w = 1[(Dilate(y, K_w) - Erode(y, K_w)) > 0]`
- 采用原因：为后续 `Boundary` 分支提供统一、可复算、可可视化核验的派生监督
- 不采用的相邻方案：不采用多脚本各自定义边界；不采用首轮多级边界标签
- 代码落点：`crc_gland_segmentation_project/tools/stage01_data_protocol/build_boundary_targets.py`，`crc_gland_segmentation_project/src/data/boundary_targets.py`

- 验收方式：抽查 boundary overlay 与正式二值 mask 对齐，且 `w=3 / 5` 只对应一套实现

### 5.9 距离图规则

- 当前结论：距离图由正式二值 mask 生成 Euclidean distance transform，并归一化到 `[0, 1]`
- 规则类型：`论文支持的候选范围 + 工程冻结规则`
- 适用阶段：`01_数据协议` 先定义，`07_Distance` 正式冻结
- 直接依据：`结直肠腺体分割_plan_优化版\01_实验执行\01_数据协议\05_标签转换与可视化规则.md`，04_MILD-Net.md，`结直肠腺体分割_plan_优化版/01_实验执行/00_总览与规范/02_参数冻结总表.md`
- 核心公式或定义参考：`d(p) = min_{q in partial Y} ||p - q||_2`；`d_hat(p) = d(p) / (max(d) + eps)`
- 采用原因：为后续距离感知监督提供统一标签，同时保持第一版实现简单、可解释、可回放
- 不采用的相邻方案：不采用 signed distance map；不采用方向场或混合定义
- 代码落点：`crc_gland_segmentation_project/tools/stage01_data_protocol/build_distance_targets.py`，`crc_gland_segmentation_project/src/data/distance_targets.py`
- 运行记录字段：`distance_target_version`, `distance_type`, `distance_norm`
- 验收方式：抽查距离图数值范围、热力图稳定性，以及训练与可视化共用同一生成函数

### 8.2.1 下游消费约束

进入 `02_UNet流程验证` 后，下游训练、验证、测试和可视化入口只允许通过下面这条固定链消费数据阶段产物：

```text
configs/data/*.yaml
-> splits/glas/*.csv or splits/crag/*.csv
-> dataset_root + image_relpath/mask_relpath
-> 正式二值标签与派生标签
```

这里明确禁止：

- 在 `crc_gland_segmentation_project/scripts/train.py` 或任何下游脚本里重新扫描原始目录
- 因为缺某个中间资产就临时改读另一份本地副本
- 训练入口未检查 `data_stage_pass / handoff_ready / preflight_pass` 就直接开跑

这条约束必须写在本阶段总协议里，而不能只散落在下游 `UNet` 文件中。原因是：

- 数据阶段才是 split、标签和来源说明的唯一冻结层
- 如果下游还能绕开这里的资产链，数据阶段就不是真正的 gate
- 一旦消费入口不唯一，后面所有 baseline 和模块对比都会失去“同一输入层”的前提

### 8.3 红线问题

下面任一条成立，都必须停在 `01_数据协议`：

- 同一 `sample_id` 进入多个 split
- `TestA / TestB / test40` 被用于训练或调参
- 批量空 mask、异常全白 mask 或系统性错位未修复
- split 版本、来源说明、配置文件和检查报告对不上
- 训练将读取的数据路径不是正式 `CSV` 所指向的数据资产

### 8.4 当前阶段通过后的最小放行语义

当前文件通过后，正式放行的真实含义固定为：

- 数据阶段已经形成唯一合法输入层，下游不能再解释 split、标签、来源说明和预览资产的语义
- 训练入口只能从 `configs/data/*.yaml -> splits/*.csv -> dataset_root + relpath` 这条固定链启动
- 任一正式 run 都必须继续携带 `data_stage_pass / handoff_ready / preflight_pass / data_proto_version`
- 若后续阶段发现输入层异常，必须先回退当前文件定义的门控链，而不是直接在模型阶段局部打补丁

### 8.5 下游异常时的强制回退触发

即使后续已经进入 `UNet`，一旦出现以下现象，也必须先回退本阶段排查：

- 训练早期大量全背景预测
- overlay 明显错位
- 指标与肉眼判断长期严重矛盾
- `val` 与 `test` 出现不可解释的异常分歧
- 边界图或距离图与原始 mask 明显不一致
