# 训练协议

本文件不是给 `ResNet34-U-Net` baseline 补一段“训练大概怎么跑”的说明，也不是对 `UNet` 已有训练流程做口头沿用。

它在当前项目中的唯一职责是：

> 作为 `04_Baseline` 阶段的训练协议细则，把 `UNet -> ResNet34-U-Net` 这一轮比较中哪些训练与评估规则必须完整继承、哪些工程适配仍算单变量结构升级、三次正式 run 必须怎样执行与记录、什么情况必须升级协议版本、以及这些规则如何在代码与结果字段中落地，一次写死。

从现在开始，只要讨论 baseline 阶段的训练继承边界、batch 政策、checkpoint/threshold 规则、三 seed 运行闭环和训练层交接字段，都以本文件为准；若与同层其他文件冲突，以 `结直肠腺体分割_plan_优化版/01_实验执行/04_Baseline/00_阶段总协议.md` 为总口径，本文件负责把训练层细化到可直接落代码、可直接验收的程度。

## 1. 文件角色与执行边界

### 1.1 当前文件负责什么

当前文件只负责冻结下面六件事：

1. `Baseline` 阶段在训练与评估 protocol 上必须完整继承 `UNet` 的范围。
2. `UNet -> ResNet34-U-Net` 比较中唯一允许出现的训练侧附属改动及其边界。
3. baseline 三次正式 run 的执行闭环、命名、资产和记录字段。
4. `effective_batch_size`、`best_selector`、`threshold_source`、encoder 初始化一致性等训练层冻结项。
5. 训练协议升级、回退、重新放行和向同层比较/验收文件交接的条件。
6. 后续 `05_LKMA / 06_Boundary` 只能继承、不能反向重开的训练基线。

### 1.2 当前文件不负责什么

当前文件不重新定义下面这些内容：

- 数据身份、split、标签口径、输入尺寸和归一化。
- `ResNet34-U-Net` 的结构身份、skip 边界和结构字段链。
- `UNet vs R34UNet` 的正式比较对象、`Gate_B1_compare` 和阶段总放行门。
- `LKMA`、`Boundary Head`、`Distance-aware Loss` 等后续模块训练策略。

这些职责分别以下列文件为准：

- `结直肠腺体分割_plan_优化版/01_实验执行/01_数据协议/00_阶段总协议.md`
- `结直肠腺体分割_plan_优化版/01_实验执行/03_UNet稳定性/00_阶段总协议.md`
- `结直肠腺体分割_plan_优化版/01_实验执行/04_Baseline/00_阶段总协议.md`
- `结直肠腺体分割_plan_优化版/01_实验执行/04_Baseline/01_R34UNet结构与来源.md`
- `结直肠腺体分割_plan_优化版/01_实验执行/04_Baseline/03_对比与判断规则.md`
- `结直肠腺体分割_plan_优化版/01_实验执行/04_Baseline/04_阶段验收.md`

- `结直肠腺体分割_plan_优化版/01_实验执行/06_Boundary/00_阶段总协议.md`

### 1.3 为什么当前文件必须独立存在

因为 `Baseline` 阶段最容易发生的偷换，不是模型搭不起来，而是把：

- `train_proto_v1 / eval_proto_v1` 的完整继承
- `effective_batch_size` 的显存适配
- `best_selector = val_objdice_max`
- `threshold_source = val17`

这几个本应冻结的训练层规则，混成：

- baseline 结果不好就顺手换 optimizer
- 某个 seed 差就单独改 batch
- 某个 split 难看就反调 threshold
- 训练协议变了却还写成“只差结构”

如果没有一份独立训练协议，后面的 `结直肠腺体分割_plan_优化版\01_实验执行\04_Baseline\03_对比与判断规则.md` 和 `结直肠腺体分割_plan_优化版\01_实验执行\04_Baseline\04_阶段验收.md` 就无法判断当前 baseline 是否真的是公平的单变量结构比较。

## 3. 本阶段唯一允许处理的变量

### 3.1 当前阶段真正要证明什么

`04_Baseline` 阶段不是继续搜索训练配方，而是要把“全文唯一正式结构 baseline”的训练 protocol 压实。

当前文件真正要证明的是：

1. 在 `UNet` 三次稳定基线已经成立的前提下，baseline 训练协议继续固定为 `train_proto_v1 / eval_proto_v1`，而不是重新搜索新配方。
2. `UNet -> R34UNet` 这一步唯一允许变化的主变量仍然是结构升级，训练侧只允许最小必要工程适配。
3. baseline 的 batch 政策、checkpoint 选择、阈值来源、encoder 初始化一致性和正式记录字段已经冻结到可直接指导实现。
4. 后续 `LKMA / Boundary` 只能建立在这份训练协议上做单变量追加，而不能反向重开 baseline 训练口径。

### 3.2 为什么 baseline 训练协议必须以“完整继承”为主

固定为“完整继承 `UNet` protocol”的原因是：

- `UNet` 已完成流程验证和三次稳定，当前最需要的是确认结构升级本身是否足以建立正式基座。
- 如果 baseline 结果不理想就顺手改 optimizer、loss、scheduler 或 threshold，后面就无法判断提升到底来自 encoder 升级还是训练漂移。
- 统一的 `train_proto_v1 / eval_proto_v1` 更利于三 seed 聚合、同层比较和下游模块继承。

### 3.3 当前不采用的相邻方案

当前明确不采用：

- 为 `R34UNet` 单独重新搜索 optimizer、loss、scheduler。
- 因为显存占用变化就顺手改输入尺寸、增强或训练协议。
- 为某个 seed、某个 split 或某个 checkpoint 单独调 threshold。
- baseline 还没形成正式训练基线就直接交给 `LKMA / Boundary` 继续补救。

## 4. 阶段门控表达式

### 4.1 当前阶段直接继承哪些冻结项

本文件直接继承并不得擅自改动的上位规则如下：

- 主任务：`2D gland segmentation`
- 主数据集：`GlaS`
- 正式 split：`train68 / val17 / testA60 / testB20`
- 标签口径：`mask > 0`
- 输入通道：`RGB`
- 输入尺寸：`512 x 512`
- 归一化：`ImageNet mean/std`
- 增强版本：`light_aug_v1`
- 主损失：`L_seg = L_BCE + L_Dice`
- 优化协议：`AdamW + lr 1e-3 + weight_decay 1e-4 + ReduceLROnPlateau + epoch_max 120 + early_stopping 20 + AMP on`
- 主指标优先级：`F1 / Object Dice / Object Hausdorff`
- `best_selector = val_objdice_max`
- `threshold_source = val17`
- `eval_cast_policy = logits/probabilities must be kept or cast to float32 before thresholding`
- `boundary_metric_width = 3 px`
- `boundary_metric_impl = project_custom_erosion_xor_3x3_ones_8conn_border0_tol3px`
- `connected_components_impl = scipy.ndimage.label`
- `connected_components_connectivity = 8`
- `train_proto_version = train_proto_v1`
- `eval_proto_version`：新轮需在研究定标/阶段锁定后由当前 Gate 冻结；旧 `eval_proto_v3` 仅作为上一轮历史 lineage 保留
- `postprocess_version = <由当前轮 Gate 冻结>`；当前轮待冻结。
- `eval_protocol_lineage` 仅记录历史 `eval_proto_v1 -> eval_proto_v3`；训练控制变量仍继承 `train_proto_v1`，该字段不是 `run_name` 版本。
- 固定三 seed：`3407 / 1234 / 2025`

### 4.2 当前阶段禁止重新打开哪些问题

本文件禁止借训练说明之名重新打开：

- 数据 split、标签定义、输入尺寸和归一化。
- 主增强包、主损失、optimizer、lr、scheduler、epoch 和 early stopping。
- `best_selector`、阈值来源、后处理、`threshold` 前 `float32` 边界、`Boundary F1` 宽度/实现和连通域实现/连接性定义。
- `LKMA`、边界分支、距离约束或其它后续模块。

### 4.3 为什么当前阶段只能继承不能重开

因为 `B1` 训练文件要回答的不是“还有哪些训练策略也可以试”，而是：

> 在完全相同的数据、训练和评估 protocol 下，只把结构从 `UNet` 升级到 `ResNet34-U-Net`，是否足以建立正式 baseline。

如果这里同时夹带：

- 改 optimizer
- 改 batch 口径
- 改阈值规则
- 改后处理
- 改 loss 或增强

后面就无法判断提升究竟来自结构升级，还是来自训练协议漂移。

### 5.6 正式训练闭环规则

- 当前结论：每个 baseline seed 都必须按“训练 -> 每 epoch 验证 -> 固定规则保存 `best.ckpt` -> 用 `best.ckpt` 做 `TestA/TestB` -> 导出指标与可视化”的闭环执行。
- 规则类型：`工程冻结规则`
- 适用阶段：`04_Baseline`

- 采用原因：正式实验必须以统一 best model 选择逻辑闭环，而不是靠最后一个 epoch 或人工挑选。

### 5.7 batch size 与显存适配规则

- 当前结论：baseline 优先保持与 `UNet` 一致的 `effective_batch_size`；若 `R34UNet` 显存占用更高，先通过 `grad_accum_steps` 保持有效 batch 不变。
- 规则类型：`工程冻结规则`
- 适用阶段：`04_Baseline` 开始生效，后续主线阶段继承。
- 直接依据：`结直肠腺体分割_plan_优化版/01_实验执行/00_总览与规范/02_参数冻结总表.md`、`结直肠腺体分割_plan_优化版/01_实验执行/04_Baseline/00_阶段总协议.md`
- 核心公式或定义参考：`effective_batch_size = physical_batch_size * grad_accum_steps`
- 采用原因：如果不显式区分物理 batch 和有效 batch，显存适配很容易被误写成结构优势或训练协议漂移。

- 代码落点：configs/train/unet_flow_v1.yaml、`crc_gland_segmentation_project/scripts/train.py`、src/engine/trainer.py

### 5.9 `best selector` 与阈值继承规则

- 当前结论：baseline 阶段继续固定 `best_selector = val_objdice_max`；阈值如需调整只能在 `val17` 上完成一次并冻结到测试阶段。

### 5.10 统计身份与阶段交接规则

### 7.1 独立回退触发条件

只要出现下面任意一条，本文件负责的训练协议就必须先回退修正，而不是直接进入比较、验收或下游模块阶段：

- `train_proto_v1 / eval_proto_v1` 未被真实继承，却仍试图写成纯结构升级。
- `effective_batch_size` 实际变化，但未显式记录或未升级协议版本。
- 某个 seed 单独修改 `best_selector`、`threshold_value`、batch 政策或初始化策略。
- `ResNet34` encoder 预训练权重未正确加载或来源不可追溯。
- 三次正式 run 资产不完整，导致无法形成正式 per-seed 和聚合结果。
- run_meta.yaml 中训练字段缺失，或 `eval_cast_policy / boundary_metric_width / boundary_metric_impl / connected_components_impl / connected_components_connectivity / result_tag / aggregation` 没有继续落盘，导致后续无法确认 baseline 继承的是哪一版 protocol 和哪一版正式结果身份。

### 7.2 固定回退顺序

回退时统一按下面顺序排查：

1. 检查 `ResNet34` encoder 接入与预训练加载是否正确。
2. 检查 `physical_batch_size / grad_accum_steps / effective_batch_size` 是否真实一致。
3. 检查 `train_proto_v1 / eval_proto_v1` 是否真正继承，是否存在 optimizer、loss、scheduler 或阈值漂移。
4. 检查 `best_selector`、`threshold_source` 和测试入口是否仍按正式协议执行。
5. 只有确认前面都无问题后，才允许把差异解释为真实结构比较结果。

### 7.3 回退后的重新放行条件

发生回退后，只有同时满足下面条件，才允许重新声明本文件对应的训练协议已恢复可用：

- 训练问题来源已经按固定顺序定位并修复。
- 修复动作、影响范围和复验结果已写入 notes/debug_note.md、逐 run 总结或阶段总结。
- config.yaml、run_meta.yaml、指标文件和阶段汇总字段重新对齐。
- 当前训练重新满足第 `5` 节全部规则卡片与第 `6` 节代码实现约束。
