# 设计依据

本文件不是 `06_Boundary` 阶段里“为什么边界可能有帮助”的说明文，也不是把几篇边界论文和项目直觉拼在一起的讨论稿。

它在当前项目中的唯一职责是：

> 作为 `06_Boundary` 的正式设计依据 protocol，把 `Boundary` 为什么必须排在 `LKMA` 之后、为什么当前只能做最小显式边界监督验证、为什么边界标签只能来自冻结二值 mask、为什么本阶段证据必须优先看边界级与对象级、以及这些结论分别如何落到代码、运行字段、阶段门控、回退和下游交接，一次写死。

从现在开始，只要讨论 `Boundary Head` 的阶段身份、设计边界、文献支持、变量边界、实现落点、验收证据和下游消费方式，都必须以本文件为准；若与同层文件冲突，以 `结直肠腺体分割_plan_优化版/01_实验执行/06_Boundary/00_阶段总协议.md` 管阶段身份与门控，以本文件管“为什么只能这样设计”，并要求 `结直肠腺体分割_plan_优化版\01_实验执行\06_Boundary\02_边界标签生成规则.md`、`结直肠腺体分割_plan_优化版\01_实验执行\06_Boundary\03_实验步骤.md`、`结直肠腺体分割_plan_优化版\01_实验执行\06_Boundary\04_保留或删除标准.md` 全部向这里回对齐。

## 1. 文件角色与执行边界

### 1.1 当前文件负责什么

当前文件只负责冻结下面八件事：

1. `Boundary` 为什么必须排在 `LKMA` 之后，而不能提前与 baseline 或 `Distance` 混写。
2. `Boundary` 阶段真正要验证的残余误差类型是什么，为什么它们指向显式边界监督。
3. `Boundary Head` 在本项目里应被压成什么样的最小可解释抽象，而不是变成完整新主干。
4. 边界标签为什么只能从冻结二值 gland mask 派生，不能临时引入新的监督源。
5. 本阶段为什么只能做单分支、单主宽度、单主损失、单主权重的最小单变量验证。
6. 本阶段为什么必须优先看 `Boundary F1`、`HD95 / Object Hausdorff`、`Object Dice / F1` 和黏连案例，而不是只看像素 `Dice`。
7. 这些设计判断如何约束代码实现、运行记录字段、阶段门控、回退和下一阶段 `current_base` 语义。
8. 本文件如何作为 06_Boundary/02-04 与 `结直肠腺体分割_plan_优化版/01_实验执行/07_Distance/00_阶段总协议.md` 的共同上游设计母规范。

### 1.3 为什么当前文件必须独立存在

因为 `Boundary` 阶段最容易出现的伪完成状态不是“完全没做”，而是：

- `LKMA` 还没有形成唯一合法 `current_base`，就提前启动边界头验证。
- 一边改边界标签，一边改宽度、权重、损失和接入位置，最后仍把结果写成单变量比较。
- 文献层明明只支持“边界信息有价值”，却在首版阶段直接照搬 `Gated-SCNN`、`TA-Net` 或 `DEA-Net` 的完整重结构。
- 可视化看起来更清楚，就直接默认保留，而没有对象级和边界级正式证据支撑。
- `07_Distance` 想继承边界阶段结果时，说不清下一阶段应该继承 `boundary_kept_base` 还是 `boundary_input_base`。

如果没有一份独立设计依据文件，`结直肠腺体分割_plan_优化版/01_实验执行/06_Boundary/00_阶段总协议.md` 会被迫同时承担“阶段身份”和“设计裁决来源”的双重职责，`02-04` 三份同层文件就会重新各自解释边界设计边界，`07_Distance` 也会失去唯一合法的上游设计起点。

## 3. 任务难点与设计动机

### 3.1 当前阶段真正要证明什么

`06_Boundary` 阶段真正要证明的，不是“边界信息理论上可能有帮助”，而是：

> 在 baseline 与 `LKMA` 的结构、训练、评估和比较协议全部冻结的前提下，显式边界监督是否值得作为第二层 post-baseline 模块独立验证，并以最小单变量设置为腺体黏连分离、模糊轮廓恢复和对象级边界修正提供可复查、可解释、可交接的收益。

因此，本文件真正要压实的是：

1. `Boundary` 不是自由扩展阶段，而是 `LKMA` 之后的第二模块验证节点。
2. 当前值得单独验证的残余误差，已经可以被明确归纳为 `touching glands / blurred contour / object separation failure`。
3. 当前最合理的实现不是大而全的边界技巧拼装，而是最小可解释的 `Boundary Head v1`。
4. 当前边界监督必须复用冻结 mask 派生标签，而不能新增新的监督来源。
5. 当前阶段证据必须优先看边界级与对象级，而不是只看像素 `Dice`。
6. 当前阶段结束后，下一阶段只能从 `boundary_kept_base` 或 `boundary_input_base` 二者之一继续，不能再保留模糊起点。

### 3.2 为什么 `Boundary` 必须排在 `LKMA` 之后

原因固定为：

- `LKMA` 先解决的是整体形态感知、有效感受野与上下文结构表达问题。
- `Boundary` 解决的是在整体表达较稳后，边界邻域、黏连分离和局部轮廓恢复是否还需要显式监督的问题。
- 如果先做 `Boundary` 再做 `LKMA`，边界收益会被上游表征不足掩盖或扭曲，无法确认它到底补的是边界本身，还是上游表示缺陷。
- 如果把 `LKMA + Boundary` 同时上，后续就无法解释哪一部分增益来自哪个模块，也无法为论文消融链保留清晰的第二节点。

### 3.3 为什么当前必须立即把设计边界写死

因为到达 `Boundary` 阶段时，以下前提已经成立：

- `LKMA` 已经形成正式 `keep / backup / drop` 结论。
- `current_base` 已经可以由上游结论唯一确定。
- 后续 `Distance` 阶段要消费的不是“也许带边界的某个版本”，而是唯一合法的边界阶段结论。

因此，当前阶段如果还不把边界设计压实，后续 `Distance`、外部对比和论文主线都会失去清晰的模块顺序和比较基座。

## 4. 设计边界(做什么/不做什么)

### 4.1 当前阶段直接继承哪些冻结项

本文件直接继承并不得擅自改动的上位规则如下：

- 主任务：`2D gland segmentation`
- 主数据集：`GlaS`
- 正式 split：`train68 / val17 / testA60 / testB20`
- 标签口径：`mask > 0`
- 输入通道：`RGB`
- 输入尺寸：`512 x 512`
- 归一化：`ImageNet mean/std`
- 增强版本：`light_aug_v1`
- 主损失：`L_seg = L_BCE + L_Dice`
- 优化协议：`AdamW + lr 1e-3 + weight_decay 1e-4 + ReduceLROnPlateau + epoch_max 120 + early_stopping 20 + AMP on`
- 主指标优先级：`F1 / Object Dice / Object Hausdorff`
- 补充指标：`Dice / IoU / HD95 / Boundary F1`
- `best_selector = val_objdice_max`
- `threshold_source = val17`
- `train_proto_version = train_proto_v1`
- `eval_proto_version` 由 04 阶段当前轮 Gate 冻结
- `eval_cast_policy = logits/probabilities must be kept or cast to float32 before thresholding`
- `boundary_metric_width = 3 px`
- `boundary_metric_impl = skimage.segmentation.find_boundaries(mode=inner) + binary_dilation`
- `connected_components_impl = scipy.ndimage.label`
- `connected_components_connectivity = 8`
- `postprocess_version`：当前轮待由研究定标、阶段锁定、Pre-check/Gate 冻结；`none_in_v3` 仅作历史 lineage，不是当前值。
- baseline 正式基座：`ResNet34-U-Net`
- baseline encoder 初始化：`official ImageNet-1K supervised weights`
- 固定三 seed：`3407 / 1234 / 2025`

### 4.2 当前阶段唯一合法起点

当前阶段的比较基座不是重新选择，而是由 `LKMA` 阶段正式结论唯一决定：

```text
current_base = (lkma_decision_level == keep) ? baseline_plus_lkma : baseline
```

这意味着：

- 只有 `05_LKMA` 已形成正式结论并完成 handoff，`Boundary` 才允许启动。
- `backup` 与 `drop` 都不能被默认当成“继续把 LKMA 带进主线”。
- 任何正式 `Boundary` run 都必须能追溯到 `LKMA` 的 decision note、stage manifest 和 `current_base` 说明。

### 4.3 当前阶段唯一变量

本阶段唯一允许处理的核心变量只有“在 `current_base` 上追加一个可插拔 `Boundary Head` 及其最小显式边界监督协议”：

- `Boundary Head v1` 的模块定义
- 边界头接入位置
- 边界标签来源与生成实现
- 边界宽度候选
- 边界损失形式
- `lambda_boundary` 候选
- `Boundary Head` 是否带来可接受的复杂度成本
- `Boundary Head` 是否值得进入后续主线

### 4.4 当前阶段明确不允许重开的内容

本阶段明确不允许借机改动：

- baseline backbone
- encoder 预训练策略
- `LKMA` 配置
- decoder 主干
- 数据协议
- 输入尺寸
- 主增强包
- 主损失
- optimizer / scheduler / lr
- `best checkpoint` 选择器
- 阈值来源
- `Distance-aware Loss`
- 后处理
- 外部对比与 `CRAG` 正式验证

换句话说，当前阶段必须回答的是“显式边界监督本身值不值得保留”，而不是继续重开主线其它变量。

## 5. 本阶段规则卡片

下面所有核心规则都写成正式协议格式；后续实现、实验和阶段汇总时，只允许把这些规则忠实落地，不允许再擅自补新的隐含边界。

### 5.1 `Boundary` 必须排在 `LKMA` 之后

- 当前结论：`Boundary` 固定排在 `LKMA` 之后，因为主线必须先稳定整体形态表达，再验证显式边界信号是否还能补足对象分离与轮廓恢复。
- 规则类型：`路线层已锁定 + 工程冻结规则`
- 适用阶段：`06_Boundary`
- 直接依据：`结直肠腺体分割_plan_优化版/02_路线与投稿/02_结直肠腺体分割_分阶段实验路线与执行标准.md`、`结直肠腺体分割_plan_优化版/01_实验执行/05_LKMA/00_阶段总协议.md`、`结直肠腺体分割_plan_优化版/01_实验执行/06_Boundary/00_阶段总协议.md`
- 核心公式或定义参考：`current_base = (lkma_decision_level == keep) ? baseline_plus_lkma : baseline`
- 采用原因：如果整体形态表达尚不稳定，边界监督收益会与上游表征不足纠缠，单变量比较失效。
- 不采用的相邻方案：不在 baseline 后直接同时上 `LKMA + Boundary`；不跳过 `LKMA` 直接进入复杂边界技巧。
- 代码落点：[project_root]/configs/experiment/boundary_d1_v1.yaml、[project_root]/experiments/[run_name]/run_meta.yaml
- 运行记录字段：`base_variant`、`base_decision_source`
- 验收方式：检查所有正式 `Boundary` run 都能追溯到 `LKMA` 阶段门控结论，而不是手工口头指定起点。

### 5.2 当前任务痛点为什么指向显式边界监督

- 当前结论：`Boundary` 阶段要对准的误差类型固定为 `相邻腺体黏连 / 恶性区域边界模糊 / 前景正确但对象分离不足`。
- 规则类型：`论文直接支持 + 任务定义规则`
- 适用阶段：`06_Boundary`
- 直接依据：03_DCAN.md、04_MILD-Net.md、07_TA-Net.md、12_DEA-Net.md
- 核心公式或定义参考：对象级失败并不等价于像素重叠失败，边界局部误差可以在像素指标变化不大时仍破坏 `Object Dice / F1 / Object Hausdorff`。
- 采用原因：腺体任务的关键难点不只是前景是否找出来，更关键的是对象边界是否被真正分开。
- 不采用的相邻方案：不把本阶段目标写成“让所有像素都更准”；不把它降格成普通辅助头试验。
- 代码落点：[project_root]/reports/stage_reports/boundary_visual_casebook.md、[project_root]/scripts/summarize_stage.py
- 运行记录字段：`error_type_tag`、`boundary_visual_support`、`decision_reason`
- 验收方式：检查最终案例与总结是否围绕黏连分离和轮廓恢复，而不是只报告像素指标小幅波动。

### 5.3 文献链对本阶段的共同支持结论

- 当前结论：`DCAN`、`Gated-SCNN`、`TA-Net`、`DEA-Net` 共同支持“边界信息对结构化分割任务重要，显式边界监督能为对象分离提供额外约束”，但它们不要求当前项目首版必须完整复刻原始重结构。
- 规则类型：`论文直接支持 + 论文支持的候选范围`
- 适用阶段：`06_Boundary`
- 直接依据：03_DCAN.md、01_Gated-SCNN.md、07_TA-Net.md、12_DEA-Net.md
- 核心公式或定义参考：`L_total = L_seg + lambda_boundary * L_boundary`
- 采用原因：这些文献给出的是“边界 supervision is useful”的共同方向，但当前项目要做的是可比较、可交接的任务化抽象，而不是多论文组件自由拼装。
- 不采用的相邻方案：不把单篇论文完整结构直接当当前协议；不把多篇论文的结构部件无约束叠加。
- 代码落点：[project_root]/src/models/heads/boundary_head.py、[project_root]/src/losses/boundary_losses.py
- 运行记录字段：`boundary_head_version`、`boundary_loss_type`、`lambda_boundary`
- 验收方式：检查实现保留了显式边界监督主旨，但未超出首版单分支边界。

### 5.4 `Boundary Head v1` 必须是最小可解释抽象

- 当前结论：`Boundary Head v1` 固定为 `single boundary prediction branch attached near final decoder output` 的可插拔辅助头，不采用完整 `Gated-SCNN / DCAN / DEA-Net` 原版重结构。
- 规则类型：`工程冻结规则 + 文献约束下的任务化抽象`
- 适用阶段：`06_Boundary`
- 直接依据：`结直肠腺体分割_plan_优化版/01_实验执行/06_Boundary/00_阶段总协议.md`、`结直肠腺体分割_plan_优化版/01_实验执行/06_Boundary/01_设计依据.md`、01_Gated-SCNN.md、03_DCAN.md、12_DEA-Net.md
- 核心公式或定义参考：`Y_boundary = Head(F_decoder_final)`；`L_total = L_seg + lambda_boundary * L_boundary`
- 采用原因：首版协议必须先确认边界信号本身是否有价值，而不是先打开大结构搜索空间。
- 不采用的相邻方案：不采用完整 shape stream；不采用双编码器边缘增强主干；不采用多级边界塔。
- 代码落点：[project_root]/src/models/heads/boundary_head.py、[project_root]/src/models/r34_unet.py
- 运行记录字段：`boundary_head_type`、`boundary_insert_position`、`use_boundary_head`
- 验收方式：检查模型改动仅表现为单边界辅助分支，而不是主干重构。

### 5.5 边界标签必须从冻结二值 mask 派生

- 当前结论：当前阶段的显式边界监督只能来自冻结二值 gland mask 的派生边界带标签，而不能临时引入新的监督源。
- 规则类型：`工程冻结规则 + 标签定义规则`
- 适用阶段：`06_Boundary`
- 直接依据：`结直肠腺体分割_plan_优化版/01_实验执行/06_Boundary/00_阶段总协议.md`、`结直肠腺体分割_plan_优化版/01_实验执行/06_Boundary/02_边界标签生成规则.md`、03_DCAN.md
- 核心公式或定义参考：`M(x) = 1[mask(x) > 0]`；`C = Contour(M)`；`B_w = Dilate(C, w)`
- 采用原因：若边界标签来源不唯一，训练、可视化和比较层都会出现口径漂移。

### 5.6 当前不能把 `Boundary` 做成复杂混合实验

- 当前结论：`Boundary` 阶段必须坚持“单分支 + 单标签定义 + 小范围宽度/权重候选 + 简单损失”的最小验证策略。
- 规则类型：`工程冻结规则`
- 适用阶段：`06_Boundary`
- 直接依据：`结直肠腺体分割_plan_优化版/02_路线与投稿/02_结直肠腺体分割_分阶段实验路线与执行标准.md`、`结直肠腺体分割_plan_优化版/01_实验执行/06_Boundary/03_实验步骤.md`

### 5.7 首轮主版本为什么固定为 `decoder_final + w=3 + BCE + lambda=0.3`

- 当前结论：首轮正式主版本固定为 `decoder_final + boundary_width = 3 + boundary_loss_type = BCE + lambda_boundary = 0.3`。

## 7. 回退条件

### 7.1 独立回退触发条件

只要出现下面任意一条，本文件负责的设计协议就必须先回退修正，而不是继续推进 06_Boundary/02-04 或 `07_Distance`：

### 7.2 固定回退顺序

回退时统一按下面顺序排查：

1. 先检查文件角色是否仍是“设计依据 protocol”，没有越界成执行步骤文件或验收文件。
