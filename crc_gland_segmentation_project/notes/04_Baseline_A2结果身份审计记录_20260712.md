## 4. A2 测试是否确实可绑定 best.ckpt

### 已确认的支持证据

1. 当前 `scripts/test.py` 默认 checkpoint 路径是 `<run_dir>/checkpoints/best.ckpt`。
2. 当前加载函数要求 checkpoint 含 `model_state_dict`，并能记录 checkpoint path、SHA256、epoch、metric；说明当前代码具备强绑定能力。
3. 三个历史 run 的 `run_meta.yaml` 均明确写入各自 `best_checkpoint_path=.../best.ckpt`、`best_checkpoint_epoch`，并与实际 best checkpoint 的 epoch 和 metric_value 一致。
4. 三组测试 CSV 行数、aggregate 及 `metric_crosscheck_note.md` 均自洽，run_summary 也报告相同 best epoch/指标。

### 不能确认的部分

三组历史 `testA_metrics.csv`/`testB_metrics.csv` 的实际 header 均为：

`row_type, sample_id, split_role, sample_count, image_path, mask_path, pred_path, loss, loss_bce, loss_dice, objdice, dice, iou, f1, boundary_f1, hd95, object_hausdorff`

它们不含 `run_name`、`seed`、`model_identity`、`checkpoint_relative_path`、`checkpoint_epoch`、`checkpoint_metric_value`、`checkpoint_sha256`、`config_version`。因此：

- 可以确认：三组测试资产与各自 run 目录、run_meta 和 best.ckpt 在外部目录结构上相互一致。
- 不能严格确认：历史 CSV 的逐样本预测是在该 SHA256 的 best.ckpt 上生成，而不是同目录内其他时点/其他 checkpoint 生成。
- 结论：A2 测试结果**可作间接绑定，不能作严格闭合绑定**。不能把三个 seed 判为 provenance 完整通过，也不能据此判历史数字错误。
