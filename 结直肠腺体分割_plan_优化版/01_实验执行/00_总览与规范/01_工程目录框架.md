# 工程目录框架

这份文件只负责一件事：

> 把后面真正要写的训练代码、评估代码、实验输出、结果汇总和文档记录，统一映射到一个长期可维护的工程目录里。

它不是代码实现说明书，也不是某个单独模型的训练手册。

它的作用是提前回答：

- 哪些目录属于正式代码
- 哪些目录属于正式数据资产
- 哪些目录属于实验输出
- 哪些目录属于结果汇总与投稿材料
- 后面外部方法接入时应该放在哪里

- 做什么：把 `crc_gland_segmentation_project/configs/`、`crc_gland_segmentation_project/datasets/`、`crc_gland_segmentation_project/splits/`、`crc_gland_segmentation_project/src/`、`crc_gland_segmentation_project/scripts/`、`crc_gland_segmentation_project/reports/`、`crc_gland_segmentation_project/tools/`、`crc_gland_segmentation_project/external/` 的职责边界写成后续阶段必须继承的正式工程约束。

## 5. 如何判断设计有效(验收标准)

- 后续阶段能把正式对象稳定映射到唯一目录职责，而不是同一类对象散落多处。
- 数据协议、训练入口、评估脚本、结果汇总和投稿材料能够沿同一目录语义回查，不需要临时口头解释。
- `crc_gland_segmentation_project/tools/`、`crc_gland_segmentation_project/b_class_auxiliary/`、`crc_gland_segmentation_project/reports/` 的职责不互相混写，A/B/C 分层与阶段化目录规则保持一致。

## 1. 总原则

当前工程目录必须满足下面四条：

1. 数据协议与训练代码分离
2. 正式实验输出与临时调试输出分离
3. 自己的主线实现与外部方法适配分离
4. 结果记录与论文图表汇总分离

这样设计的原因是：

- 防止后面训练脚本直接写死本地绝对路径
- 防止主表结果和临时试跑结果混在一起
- 防止外部 baseline 的适配逻辑污染主线代码
- 防止跑完实验以后找不到正式结果资产

```text

├─ experiments/
├─ reports/
```

这里的 `[project_root]/` 指的是后面真正放代码的工程根目录，不等于当前这个 `plan` 文档目录。

- `[project_root]/` 负责代码与运行

## 3. 各目录放什么

### 3.1 `crc_gland_segmentation_project/configs/`

放所有正式配置文件，不放临时测试草稿。

正式最小分层固定为：

```text
configs/
├─ data/
├─ train/
├─ model/
├─ eval/
└─ experiment/
```

分别承担：

- `data/`：数据路径、CSV、输入尺寸、归一化、增强包
- `train/`：optimizer、lr、scheduler、epoch、early stopping、AMP
- `model/`：`UNet / R34UNet / LKMA / Boundary / Distance` 相关结构配置
- `eval/`：threshold、metrics、后处理、可视化导出开关
- `experiment/`：某一正式 run 的组合配置

### 3.2 `crc_gland_segmentation_project/datasets/`

只放正式数据和来源说明，不放训练代码。

正式最小目录固定为：

```text
datasets/
├─ 01_GlaS_official_raw/
├─ 02_CRAG_reorganized_local_copy/
├─ DATASETS_README.md
└─ DATASET_SOURCE_NOTES.md
```

### 3.3 `crc_gland_segmentation_project/splits/`

只放正式 split `CSV`，不再混放数据检查报告、统计表和预览图。

正式目录结构固定为：

```text
splits/
├─ glas/
│  ├─ glas_train68.csv
│  ├─ glas_val17.csv
│  ├─ glas_testA60.csv
│  └─ glas_testB20.csv
└─ crag/
   ├─ crag_train153.csv
   ├─ crag_val20.csv
   └─ crag_test40.csv
```

这里的固定规则必须和 `结直肠腺体分割_plan_优化版/01_实验执行/01_数据协议/00_阶段总协议.md`、`结直肠腺体分割_plan_优化版/01_实验执行/01_数据协议/06_数据产物清单.md` 保持一致：

- `crc_gland_segmentation_project/splits/` 只承担“后续代码唯一默认读取入口”的职责
- `GlaS` 与 `CRAG` 必须分目录管理，避免后续 dataloader、检查脚本和结果核对把不同数据集资产混写
- data_check_report.md、dataset_stats.csv、duplicate_check_report.md、foreground_summary.csv、object_size_summary.csv 不再放在 `crc_gland_segmentation_project/splits/`，统一回收到 `crc_gland_segmentation_project/reports/data_checks/`
- `crc_gland_segmentation_project/reports/data_preview/` 也不再挂在 `crc_gland_segmentation_project/splits/` 下，避免把“正式 split 协议”和“人工核查证据”混成一个目录身份

### 3.4 `crc_gland_segmentation_project/src/`

只放正式 Python 源码，不放运行结果。

正式源码结构固定为：

```text
src/
├─ data/
├─ models/
├─ losses/
├─ metrics/
├─ engine/
├─ eval/
├─ utils/
└─ registry/
```

分别承担：

- `data/`：dataset、dataloader、augmentation、mask transform
- `models/`：`UNet`、`R34UNet`、`LKMA`、`Boundary Head`
- `losses/`：`BCE + Dice`、boundary loss、distance-aware loss
- `metrics/`：`Dice / IoU / HD95 / Boundary F1 / Object-level metrics`
- `engine/`：train、validate、test、checkpoint、logging
- `eval/`：主指标计算、导出预测图、汇总表生成
- `utils/`：seed、path、config、logger、timer
- `registry/`：模型、损失、增强、指标注册接口

### 3.5 `crc_gland_segmentation_project/scripts/`

放正式运行入口，不放核心实现。

正式最小入口脚本集合固定为：

```text
scripts/
├─ prepare_data.py
├─ train.py
├─ validate.py
├─ test.py
├─ export_visuals.py
├─ summarize_stage.py
└─ compare_runs.py
```

### 3.6 `crc_gland_segmentation_project/experiments/`

只放每一个正式 run 的运行产物。

正式运行产物结构固定为：

```text
experiments/
└─ [run_name]/
   ├─ config.yaml
   ├─ run_meta.yaml
   ├─ seed_record.txt
   ├─ train_log.csv
   ├─ val_metrics.csv
   ├─ testA_metrics.csv        # 仅 GlaS
   ├─ testB_metrics.csv        # 仅 GlaS
   ├─ test_metrics.csv         # 仅 CRAG
   ├─ checkpoints/
   │  ├─ best.ckpt
   │  └─ last.ckpt
   ├─ predictions/
   │  ├─ val/
   │  ├─ testA/
   │  ├─ testB/
   │  └─ test/
   ├─ visuals/
   │  ├─ val/
   │  ├─ testA/
   │  ├─ testB/
   │  └─ test/
   ├─ curves/
   ├─ summaries/
   │  ├─ run_summary.md
   │  └─ error_cases.md
   └─ artifacts/
```

这里必须与 `结直肠腺体分割_plan_优化版/01_实验执行/00_总览与规范/03_命名与结果记录规范.md` 完全一致。
如果后面两份文件出现冲突，以 `结直肠腺体分割_plan_优化版/01_实验执行/00_总览与规范/03_命名与结果记录规范.md` 为准，不允许工程目录层再保留旧写法。

### 3.7 `crc_gland_segmentation_project/reports/`

放阶段级和论文级结果汇总，也放数据阶段的正式检查与预览资产；不放原始模型权重。

正式目录结构固定为：

```text
reports/
├─ data_checks/
├─ data_preview/
├─ stage_reports/
├─ tables/
├─ figures/
├─ appendix/
└─ rebuttal_materials/
```

职责固定如下：

- `data_checks/`：数据阶段检查报告、统计表、重复检查和人工核查记录
- `data_preview/`：按 `dataset/split` 组织的最小预览证据
- `stage_reports/`：阶段级总结、验收和回退记录
- `tables/`、`figures/`：论文主表、补表和图像资产

这里不采用“把检查报告继续留在 `crc_gland_segmentation_project/splits/`”的旧写法，因为：

- `crc_gland_segmentation_project/splits/` 的身份是正式输入协议，不是报告归档区
- 数据检查和人工预览本质上属于 `crc_gland_segmentation_project/reports/` 资产，更适合长期复查和写作引用
- 把 `CSV` 与报告混在一起，会导致后续脚本读取入口与人工检查证据边界不清

### 3.8 `checkpoints/`

如果后面你想统一集中缓存正式权重，也可以保留这个目录。

但更稳的做法仍然是：

- 每个正式 run 的最佳权重放在对应 experiments/[run_name]/checkpoints/

因此这个全局 `checkpoints/` 更适合放：

- 长期保留的冻结版权重
- 论文最终版模型
- 需要共享或归档的模型

### 3.9 `crc_gland_segmentation_project/tools/`

放不直接参与训练、但对协议有帮助的工具脚本。

当前最小正式集合固定为：

```text
tools/
├─ prepare_glas_split.py
├─ prepare_crag_split.py
├─ convert_masks.py
├─ build_boundary_targets.py
├─ build_distance_targets.py
├─ check_dataset_pairs.py
├─ preview_dataset_samples.py
└─ metric_crosscheck.py
```

分别承担：

- prepare_glas_split.py / prepare_crag_split.py：生成正式 `CSV`
- convert_masks.py：统一实例标注到主线二值标签
- build_boundary_targets.py / build_distance_targets.py：生成派生监督目标
- check_dataset_pairs.py：输出 `reports/data_checks/*`
- preview_dataset_samples.py：输出 `reports/data_preview/*`
- metric_crosscheck.py：对齐官方口径或参考实现做指标交叉验证

### 3.10 `crc_gland_segmentation_project/external/`

放外部 baseline 的第三方实现或适配层。

正式目录结构固定为：

```text
external/
├─ unetpp/
├─ deeplabv3p/
├─ attention_unet/
├─ nnunet/
└─ transunet/
```

这里的原则是：

- 自己的主线代码不放到 `crc_gland_segmentation_project/external/`
- 第三方代码和自己写的主线实现隔离
- 所有适配说明都要同步写进 external_reproduction_note.md

补充约定：

- 第三方原始仓或整理后的外部实现放 `crc_gland_segmentation_project/external/`
- 调用这些外部实现的正式协议、比较脚本和结果汇总逻辑仍放 `crc_gland_segmentation_project/scripts/`、`crc_gland_segmentation_project/reports/`
- 不再使用 `src/external/*` 作为正式主落点，避免把第三方源码和主线源码混在一起

### 4.5 目录级代码落地接口

这份文件虽然主要负责“目录该怎么分”，但它不能只停留在树状图层面，还必须能约束后面脚本入口、输入输出身份和运行产物。

#### 数据协议入口

- 代码文件：`crc_gland_segmentation_project/tools/stage01_data_protocol/prepare_glas_split.py`、`crc_gland_segmentation_project/tools/stage01_data_protocol/prepare_crag_split.py`
- 入口类/函数：`build_glas_split()`、`build_crag_split()`
- 输入：原始图像目录、原始 mask 目录、元信息表；对象类型以路径字符串和表格字段为主，不涉及固定张量 `shape`
- 输出：`splits/glas/*.csv`、`splits/crag/*.csv`
- `dtype`：路径字段为 `string`，样本数与索引字段为 `int`，统计字段为 `float`
- 依赖配置：`crc_gland_segmentation_project/configs/data/glas.yaml`、`crc_gland_segmentation_project/configs/data/crag.yaml`、`split_seed`
- 前置断言：`sample_id` 全局唯一，image/mask 成对可读，测试集不得回流训练
- 运行产物：正式 split `CSV`、`reports/data_checks/*`、`reports/data_preview/*`

#### 训练入口

- 代码文件：`crc_gland_segmentation_project/scripts/train.py`
- 入口类/函数：`main(cfg)` 或同等单一正式入口
- 输入：图像张量 `B x 3 x 512 x 512`、mask 张量 `B x 1 x 512 x 512`
- 输出：experiments/[run_name]/
- `dtype`：图像为 `float32` 或 AMP 下的 `float16/bfloat16`，mask 为 `float32`
- 依赖配置：`configs/experiment/*.yaml`、`configs/data/*.yaml`、`configs/train/*.yaml`、`configs/model/*.yaml`、`configs/eval/*.yaml`
- 前置断言：正式 `CSV`、`best_selector`、`threshold_source` 和 `run_name` 已冻结
- 运行产物：config.yaml、run_meta.yaml、train_log.csv、`checkpoints/*`、`val/test metrics`

#### 评估与导出入口

- 代码文件：[project_root]/scripts/test.py、[project_root]/scripts/export_visuals.py、[project_root]/src/eval/run_eval.py
- 入口类/函数：`main(cfg)`、`run_eval()`、`export_visuals()`
- 输入：模型 `logits`、冻结阈值、预测与 GT 的二值 mask
- 输出：`predictions/`、`visuals/`、`summaries/` 与阶段级汇总表
- `dtype`：预测概率为 `float32`，二值 mask 为 `uint8` 或 `bool`
- 依赖配置：`eval_proto_version`、`threshold_value`、`best_selector`
- 前置断言：`best.ckpt` 存在，阈值只来自验证集，评估协议版本明确
- 运行产物：val_metrics.csv、testA_metrics.csv、testB_metrics.csv、test_metrics.csv、`[project_root]/reports/tables/*`

## 5. 正式代码与临时实验必须分开

后面最容易出问题的一点就是把临时调试代码、一次性 notebook、试跑脚本混进正式工程。

因此当前正式要求明确区分：

- 正式代码：`crc_gland_segmentation_project/src/`、`crc_gland_segmentation_project/scripts/`
- 正式实验：`crc_gland_segmentation_project/experiments/`
- 正式汇总：`crc_gland_segmentation_project/reports/`
- 外部依赖：`crc_gland_segmentation_project/external/`
- 临时探索：统一单独放到 `sandbox/` 或 `_tmp/`

只要某个东西不准备长期复查、长期复现、长期写进论文，就不要混进正式目录。
