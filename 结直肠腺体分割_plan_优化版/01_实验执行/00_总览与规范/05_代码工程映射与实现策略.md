# 代码工程映射与实现策略

## 1. 文件角色与执行边界

- 本文件负责把文档协议、方法模块、训练入口、评估入口和外部对比接入策略映射到正式工程目录。
- 本文件不替代具体阶段的设计依据、训练协议、阶段验收和结果汇总结论。

## 2. 本轮直接依赖的前置文件

- 直接依赖 `结直肠腺体分割_plan_优化版/01_实验执行/00_总览与规范/00_执行导航.md`、`结直肠腺体分割_plan_优化版/01_实验执行/00_总览与规范/01_工程目录框架.md`、`结直肠腺体分割_plan_优化版/01_实验执行/00_总览与规范/04_评估口径与官方脚本对齐.md`。
- 工程对象落在 `crc_gland_segmentation_project/src/`、`crc_gland_segmentation_project/scripts/`、`crc_gland_segmentation_project/external/`。

## 3. 本阶段唯一允许处理的变量

- 只允许决定“哪类职责应该落在哪类代码对象”，不允许在本文件里跳过阶段门控直接锁定某个实验结论。

## 4. 阶段门控表达式

- 只有当后续阶段能依据本文件把协议要求稳定映射到正式目录与入口脚本，本文件才算生效。

这份文件回答的不是“某个模型怎么写”，而是：

> 后面到底哪些代码自己写，哪些代码优先接成熟开源实现，哪些模块需要严格统一接口，才能让当前实验 protocol 真正变成可落地工程。

它的作用是防止后面出现两类问题：

- 一边写代码一边重新想工程结构
- 为了省事直接把多个来源的代码拼在一起，最后很难保证公平比较

从现在开始，这份文件再额外负责一件事：

- 把“文献证据里提到的模块职责”明确映射到“后面应该落到哪类代码文件”

---

## 1. 当前总策略

后面代码实现统一按下面三条原则：

1. 主线训练框架自己统一写
2. 自己的方法模块自己写
3. 外部 baseline 优先接高质量开源实现，但必须统一协议适配

换句话说：

- 当前项目不是“全部手写”
- 也不是“全部外包给第三方仓库”
- 而是：**主线自己掌控，外部对比有选择地接成熟实现**

---

## 1.5 证据到代码的映射原则

后面代码层不能只写“有这个模块”，而必须能回答：

1. 这个模块的直接方法依据是哪几篇
2. 它在当前项目里承担什么职责
3. 它应该落到哪个代码目录
4. 它对应哪些配置字段和结果字段

如果一个模块在代码里存在，但文档里说不清：

- 为什么需要它
- 它解决什么任务痛点
- 它和哪篇论文最接近

那它就还不是“实验执行层允许出现的模块”。

## 2. 哪些代码应该自己写

### 2.1 必须自己掌控的部分

下面这些代码统一由我们自己的工程主框架负责：

- 数据读取与 CSV 协议
- `GlaS / CRAG` 的训练、验证、测试循环
- 配置系统
- 日志系统
- checkpoint 保存
- 可视化导出
- 结果表生成
- 主线评估接口
- `UNet`
- `ResNet34-U-Net`
- `LKMA`
- `Boundary Head`
- `Distance-aware Loss`

原因：

- 这些内容决定当前实验链条是否统一
- 它们直接关系到后面主线消融是否公平
- 它们也是你后续最可能持续修改、持续调试的部分

补充要求：

- 自己写并不等于“脱离文献自己发明”
- 主线代码的每个新增模块都必须能回指到 `03_文献证据` 的正式深提取稿
- 如果某个模块只是工程占位，还没有对应证据，就不能直接进入正式主线

### 2.2 为什么主线自己写更稳

因为当前项目最重要的不是“某一个 fancy 模型能不能跑”，而是：

- 数据协议要统一
- 训练协议要统一
- 结果落盘要统一
- 消融逻辑要统一

如果连主线都完全依赖外部仓库，后面你很难做到单变量验证。

## 3. 哪些代码适合接开源实现

### 3.1 第一批外部对比

下面这些方法更适合优先接成熟实现，再做统一协议适配：

- `UNet++`
- `DeepLabV3+`
- `Attention U-Net`

这些方法的第三方源码或整理版本统一放：

- `crc_gland_segmentation_project/external/`

### 3.2 第二批增强对比

下面这些方法统一优先参考或接官方/高质量开源实现：

- `nnU-Net`
- `TransUNet`

同样遵守：

- 原始外部实现放 `crc_gland_segmentation_project/external/`
- 公平比较脚本和汇总逻辑仍放 `crc_gland_segmentation_project/scripts/`、`crc_gland_segmentation_project/reports/`

### 3.3 为什么不把这些都改成手写

原因不是写不出来，而是：

- 会显著增加额外 bug 风险
- 会拖慢主线闭环
- 会把精力从“验证你自己的主线”转移到“重造别人的实现”

因此更稳的原则是：

- 主线自己写
- 外部对比优先适配高质量实现

## 4. 外部方法接入时怎么保持公平

后面接第三方实现时，必须统一遵守下面规则：

### 4.1 数据协议统一

- 同一 `splits/glas/*.csv` 与 `splits/crag/*.csv`
- 同一 `GlaS` 与 `CRAG` 协议
- 不允许某个外部方法单独重划数据

这条规则在代码层的直接含义固定为：

- `crc_gland_segmentation_project/src/data/csv_loader.py` 只能从正式 split 目录读取，不再兼容平铺 `splits/*.csv` 旧写法
- `configs/data/*.yaml` 必须显式记录 `dataset_code`、`split_dir` 与对应 `csv` 文件名
- 外部方法适配层即使来自第三方仓库，也只能消费当前正式 split，而不能额外生成“自己的 train/val/test”

### 4.2 训练协议统一

- 同一输入尺寸
- 同一增强包
- 同一 optimizer / lr / scheduler / epoch 上限
- 同一 early stopping 规则
- 同一 threshold 规则

### 4.3 评估协议统一

- 同一指标集合
- 同一对象级指标实现
- 同一后处理默认口径
- 同一 `single_seed / mean+-std` 规则

与 `结直肠腺体分割_plan_优化版/01_实验执行/00_总览与规范/03_命名与结果记录规范.md`、`结直肠腺体分割_plan_优化版/01_实验执行/00_总览与规范/04_评估口径与官方脚本对齐.md` 对齐后，后面正式代码与汇总脚本至少统一记录：

- `best_selector = val_objdice_max`
- `threshold_source`
- `threshold_value`
- `eval_proto_version`
- `postprocess_version`
- `result_tag`
- `aggregation`

### 4.4 文档协议统一

每接一个外部方法，都必须同步写：

- `implementation_source`
- `adaptation_note`
- `paper_setting_gap`
- `fairness_adjustment`
- `result_tag`
- `aggregation`

这部分后面统一落到：

- `08_外部对比/*`
- external_reproduction_note.md
- `crc_gland_segmentation_project/reports/stage_reports/`

## 5. 当前正式代码模块映射

### 5.1 `crc_gland_segmentation_project/src/data/`

正式最小拆分固定为：

- datasets.py
- transforms.py
- csv_loader.py
- mask_ops.py

分别负责：

- 从 `splits/glas/*.csv` 或 `splits/crag/*.csv` 读取图像与 mask
- 统一训练增强与验证变换
- 统一标签转换与 resize
- 边界图、距离图等派生标签接口

### 5.2 src/models/

正式最小拆分固定为：

- unet.py
- r34_unet.py
- modules/lkma.py
- heads/boundary_head.py
- encoders/resnet34.py
- decoders/unet_decoder.py

这样做的好处是：

- baseline 和模块解耦
- 方便后面做单变量插拔

补充映射：

- unet.py 对应 02_U-Net.md
- r34_unet.py 对应 05_ResNet.md
- modules/lkma.py 对应 01_Large-Kernel-Matters.md
- heads/boundary_head.py 对应 03_DCAN.md、01_Gated-SCNN.md、12_DEA-Net.md

### 5.3 src/losses/

正式最小文件集合固定为：

- seg_losses.py
- boundary_losses.py
- distance_losses.py
- loss_registry.py

这样后面：

- `BCE + Dice`
- 边界分支损失
- 距离图损失

就不会混在一个文件里。

补充映射：

- seg_losses.py 负责区域主损失，不负责实例后处理
- boundary_losses.py 对应 `Boundary Head` 阶段证据链
- distance_losses.py 对应 02_Distance-Map-Loss.md 等距离监督文献

### 5.4 src/metrics/

正式最小文件集合固定为：

- pixel_metrics.py
- boundary_metrics.py
- object_metrics.py
- metric_registry.py

这是当前项目里非常关键的一层，因为：

- 主指标是对象级三指标
- 边界指标后面直接关系到 `Boundary Head` 和 `Distance-aware Loss`

额外要求：

- object_metrics.py 必须单独维护，不允许混在训练脚本里临时实现
- boundary_metrics.py 必须和 pixel_metrics.py 分开，避免后面边界证据口径漂移
- 所有指标实现都要对应 `eval_proto_version`

### 5.5 src/engine/

正式最小文件集合固定为：

- trainer.py
- validator.py
- tester.py
- checkpoint.py
- logger.py

### 5.6 src/eval/

正式最小文件集合固定为：

- run_eval.py
- export_visuals.py

这一层只保留评估执行与可视化导出逻辑，不再承载阶段汇总和跨 run 比较脚本。

与 `结直肠腺体分割_plan_优化版/01_实验执行/00_总览与规范/01_工程目录框架.md` 对齐后的正式脚本落点固定为：

- scripts/summarize_stage.py
- scripts/compare_runs.py

对应关系固定为：

- src/eval/run_eval.py：执行单个 run 的验证/测试评估
- src/eval/export_visuals.py：导出预测、overlay 与错误案例可视化
- scripts/summarize_stage.py：汇总阶段内多个 run，输出 `reports/stage_reports/*`
- scripts/compare_runs.py：跨方法汇总主表，输出 `reports/tables/*`

## 8. 当前明确禁止的实现偏差

### 8.1 禁止一开始就全接外部仓库

因为会导致：

- 主线逻辑难统一
- 结果落盘规则不一致
- debug 成本升高

### 8.2 禁止一开始就把所有模型都自己写

因为会导致：

- 时间被大量消耗在重造已有方法
- 主线推进明显变慢

### 8.3 禁止把评估逻辑分散在各模型仓里

当前正式做法是：

- 评估逻辑统一放在你自己的 src/metrics/ 与 src/eval/

这样才能保证：

- 所有方法共享同一评估口径

## 9. 当前正式代码质量底线

后面写代码时，至少满足下面要求：

- 配置驱动，不写死路径
- 数据读取与训练引擎分离
- 模型结构与损失函数分离
- 每个正式 run 自动落盘到固定目录
- 评估结果可复算
- 可视化能自动导出
- 外部方法适配差异有记录

如果后面代码只是“能跑一个结果”，但不满足上面这些，说明它还不是正式工程代码。

## 10. 一句话版本

> 当前最稳的实现策略是：训练框架和主线方法自己统一写，外部 baseline 优先接成熟开源实现但必须做统一协议适配；代码工程的核心目标不是“尽快堆出更多模型”，而是让每一轮实验都能在统一数据、统一训练、统一评估和统一记录规则下被复查。

## 11. 冲突裁决记录

- 冲突对象：代码工程映射中的 split 路径、阶段汇总脚本落点，以及评估/结果字段与外部适配记录
- 冲突来源：当前文件旧版仍保留 `splits/*.csv` 平铺写法，并把 `summarize_stage.py / compare_runs.py` 放在 src/eval/；同时表里的“必须记录字段”还没有完全对齐 `结直肠腺体分割_plan_优化版/01_实验执行/00_总览与规范/03_命名与结果记录规范.md` 和 `结直肠腺体分割_plan_优化版/01_实验执行/00_总览与规范/04_评估口径与官方脚本对齐.md` 中已经冻结的 `best_selector / threshold_source / threshold_value / result_tag / aggregation`
- 裁决结论：正式 split 路径统一收口为 `splits/glas/*.csv` 与 `splits/crag/*.csv`；评估执行逻辑保留在 src/eval/，阶段汇总与跨 run 比较脚本统一回到 `crc_gland_segmentation_project/scripts/`；对象级指标和外部对比适配必须显式记录 `best_selector / threshold_source / threshold_value / eval_proto_version / postprocess_version / result_tag / aggregation`
