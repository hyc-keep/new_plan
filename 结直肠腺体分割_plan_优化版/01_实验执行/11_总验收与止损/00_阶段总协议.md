# 总验收与止损阶段总协议

本文件不是 `11_总验收与止损` 目录旧稿的摘要，也不是“等 `08-10` 全部做完后再凭感觉写几条总结”的收尾说明。

它在当前项目中的唯一职责是：

> 作为 `11_总验收与止损` 阶段的正式母协议，把 Stage-by-stage final gate mapping / Project-level readiness judgement / Global stop-loss and rollback boundary / Chinese-manuscript floor / English-manuscript starting threshold / Writing-entry gatekeeping 一次写死；后续所有总裁决文档、写稿进入判断和是否继续扩实验的决定，都只能忠实继承这份协议，不允许再因为沉没成本、主观乐观或“结果看上去差不多”而临时改口。

从现在开始，`11_总验收与止损` 子目录下的其他细则文件都视为本文件的细化与执行展开；若与本文件冲突，以本文件为准回改。

## 1. 文件角色与执行边界

### 1.1 当前文件负责什么

当前文件只负责冻结下面九件事：

1. `11_总验收与止损` 作为 `10_结果汇总` 之后的唯一最终裁决阶段的身份、阶段目标和放行边界。
2. `01-10` 各阶段正式通过状态在总验收层的唯一映射方式，也就是 threshold_passed / handoff_assets_ready / next_stage_ready / stage_pass 的统一矩阵逻辑。
3. 项目总状态只允许落到 Pass / Pass with gaps / Not pass 三档之一的定级规则，以及缺口说明边界。
4. 全项目止损与回退动作的唯一裁决边界，确保争议对象只能落到 retain / downgrade / remove / rollback 四档之一。
5. 中文保底起步和英文起步的唯一裁决边界，确保写稿判断只依据正式结果包和正式结论，不依据主观感觉。
6. 是否允许正式进入写稿阶段，以及优先 cn_first / cn_first_then_en / en_direct / not_enter_yet 的唯一门控逻辑。
7. 本阶段最低交付物、正式阶段门、独立 `回退条件`、固定回退顺序和重新放行条件。
8. 主 `3-seed` 冻结结论与 supplementary `additional_reporting` 的唯一统计 reporting 边界，确保后续总状态、中文、英文和写稿裁决都继续以主 `3-seed` 作为唯一主结论依据。
9. 本阶段直接面向的下游裁决文档、字段链、实现接口、冲突裁决记录、`文件质量自检` 和独立 `Diagnostics 闭环`。
10. 本阶段与正式写稿阶段之间的唯一交接语义，防止“结果很多但总裁决还没成文”时提前进入写稿。

### 1.3 为什么当前文件必须独立存在

`11_总验收与止损` 最容易出现的伪完成状态不是“完全没做总验收”，而是下面这些情况：

- `08-10` 已经有很多表格、图像和 summary，但 `01-10` 是否真正全部过线仍没有统一矩阵。
- 模块去留、外部对比边界和 `CRAG` 结论都已经出现，却没有被压成统一止损动作。
- 结果包看上去已经够写中文或英文，但中文保底、英文起步和写稿进入条件仍互相打架。
- 已经准备写稿，却仍把“后面再补一点关键实验”当成默认前提。
- final_readiness_judgement.md、stop_loss_decision.md、cn_entry_readiness.md、en_entry_readiness.md、writing_entry_decision.md 这些下游裁决文件还没有共同的上位母协议。

如果没有一份独立阶段总协议，后续总裁决文档和写稿阶段都会再次面对“结果已经有了，但到底够不够写、该降到什么层级、哪些内容必须止损”的灰区。

> 在数据协议、训练协议、评估协议、来源标记、主线结论、外部对比边界、`CRAG` 结论强度和结果包状态都已经冻结的前提下，项目已经被诚实地压成唯一的阶段矩阵、唯一的总状态、唯一的止损动作、唯一的中文起步结论、唯一的英文起步结论和唯一的写稿进入判断。

- `01-10` 各阶段都能被映射到明确的 `stage_pass` 状态，而不是仍靠口头印象判断。

- 是否进入写稿、优先走哪条轨道、还剩哪些只是小缺口，都已经成文。

### 3.2 当前阶段不允许用什么替代正式结论

当前明确不允许把下面这些情况写成“总验收已完成”：

- 只因为 `10_结果汇总` 已经产出很多表格和图像，就默认项目已经 `Pass`。
- 只因为中文稿大概能写，就默认英文稿也可以起步。
- 只因为 `CRAG` 跑过了，就默认可以写成强泛化或稳定跨 benchmark 优势。
- 只因为某个模块已经投入很多，就回避它应当降级、删除或回退的事实。
- 只把总验收写成一套模板，却提前填写最终状态、最终写稿结论和真实证据来源。

### 3.3 为什么当前阶段必须采用“两层填写”

本阶段所有裁决文档都必须统一按“两层填写”理解：

- 第一层现在就要固定：裁决框架、允许档位、字段链、检查表、证据映射表和回填规则。
- 第二层必须等 `08-10` 的真实结果与资产正式收口后再回填：真实状态、真实证据来源、真实缺口和真实写稿进入结论。

对应固定理解为：

```text
final_decision_doc = frozen_schema_now + evidence_backfill_after_08_10_closed
```

这样规定的原因是：

- `11` 的任务是固定总裁决协议，不是提前伪造最终结论。
- 如果在 `08-10` 尚未彻底收口时就提前写死最终状态，后面很容易出现“正文结论比正式资产先到位”的证据链倒挂。
- 总验收阶段的诚实性来自“框架先冻结、证据后回填”，而不是“现在先拍一个最终结论，后面再想办法补证据”。

- 六类总裁决文档对同一对象的结论不一致。

### 7.2 固定回退顺序

回退时统一按下面顺序排查：

1. 先检查 `01-10` 各阶段是否真的收口。
2. 再检查 `10_结果汇总` 的 `package_status` 是否真的达到 `ready` 或至少 `ready_with_minor_gaps`。
3. 再检查需止损对象是否都已诚实降级、删除或回退。
4. 再判断中文起步是否成立。
5. 最后判断英文起步是否成立。
6. 只有前面都成立后，才允许判断是否正式进入写稿。

### 7.3 回退后的重新放行条件

发生回退后，只有同时满足下面条件，才允许重新声明 `11_总验收与止损` 阶段通过：

- 问题来源已经按固定回退顺序定位并修复。

- stage_gate_matrix.md、final_readiness_judgement.md、stop_loss_decision.md、cn_entry_readiness.md、en_entry_readiness.md、writing_entry_decision.md 已重新对齐。
- 当前阶段重新满足第 `5` 节全部规则卡片与第 `6` 节的最低交付物、交接资产和写稿起点要求。
