- 本文件负责定义 `01_实验执行` 全局重修顺序、互校范围、最小质检动作和统一通过线。
- 本文件不替代单个阶段的正式协议，也不替代 `06_实验执行证据化写作模板.md` 的正文写法定义。

## 2. 本轮直接依赖的前置文件

- 直接依赖 `结直肠腺体分割_plan_优化版/01_实验执行/00_总览与规范/00_执行导航.md`、`结直肠腺体分割_plan_优化版/01_实验执行/00_总览与规范/05_代码工程映射与实现策略.md`、`结直肠腺体分割_plan_优化版/01_实验执行/00_总览与规范/06_实验执行证据化写作模板.md`。
- 本文件同时约束 `结直肠腺体分割_plan_优化版/01_实验执行/` 全树文档的重写与质检动作。

## 3. 本阶段唯一允许处理的变量

- 只允许定义重修流程、互校顺序和质检要求，不允许在本文件里直接替代具体阶段的实验结论。

## 4. 阶段门控表达式

- 只有当每个目标文档都能依据本文件完成重读、回写、自检和 diagnostics 闭环时，这份总规程才算被真正执行。

这份文件不是某个单独实验阶段的细则，而是从现在开始重修整个 `01_实验执行` 时必须共同遵守的总规范。

它只解决下面四件事：

1. 后面到底按什么标准重修 `01_实验执行`
2. 每修改一个 `md` 文件时，必须同时检查哪些上游、同层和下游内容
3. 每类文件最低必须写清哪些内容，才算能直接指导代码和论文
4. 每个文件改完后，应该怎样做质量检查，避免反复返工

换句话说，这份文件的角色不是补充说明，而是后面整个 `01_实验执行` 重修工作的总工作规程。

本文件与 `结直肠腺体分割_plan_优化版/01_实验执行/00_总览与规范/06_实验执行证据化写作模板.md` 的分工必须明确：

- `结直肠腺体分割_plan_优化版/01_实验执行/00_总览与规范/06_实验执行证据化写作模板.md` 负责定义“每个文件最后应该写成什么样”，也就是格式标准、规则卡片格式、阶段骨架、命名版本化要求和改造完成检查清单
- 本文件负责定义“每个文件应该怎么改、按什么顺序改、改完后如何质检”，也就是过程标准、证据来源层级、全局互校原则和通过线

理解方式固定为：

- `06` 回答“写成什么样”
- `07` 回答“怎么写、按什么顺序写、检查什么”

如果两份文件出现交叉约束：

- 以 `06` 作为格式标准
- 以 `07` 作为流程与质检标准

## 4. 当前唯一有效的证据来源层级

后面每修改一个文件，允许使用的依据只来自下面四层，而且优先级固定。

这里要先区分两件事：

- `00_总览与规范` 负责“进入修改之前必须先走的总入口与总规程”
- 下面四层负责“进入修改后，当前文件的内容依据到底来自哪里”

也就是说，不能把“证据来源层级”误解成可以跳过 `00_总览与规范` 的理由。

### 4.1 第一层：`02_路线与投稿`

作用：

- 锁定阶段顺序
- 锁定主线方向
- 锁定 baseline 和模块顺序
- 锁定投稿层级和止损边界

这里回答的是：

- 为什么整体路线这样走
- 哪些内容已经在路线层锁定
- 哪些内容还不能在当前阶段提前写死

### 4.2 第二层：`03_文献证据`

作用：

- 提供文献出处
- 提供结构来源
- 提供公式定义
- 提供参数候选范围
- 提供模块作用和相邻方案对比

这里回答的是：

- 文献支持到了哪一步
- 哪些是论文原值
- 哪些只是论文支持的候选范围
- 哪些原理和公式可以直接引用

### 4.3 第三层：`01_实验执行` 全目录内部交叉校正

作用：

- 用前面已经写好的内容校正当前文件
- 用后面已经写过的理论内容回头修正当前文件
- 检查术语、参数、规则、结论是否前后一致

这里必须明确：

- 既要看当前文件前面的内容
- 也要看当前文件后面的内容
- 但不能把后面的内容全部默认成正确答案
- 后面的内容如果和路线、文献或全局逻辑冲突，必须被修正，而不是被继承

### 4.4 第四层：工程冻结与执行规则

作用：

- 形成统一命名
- 形成统一目录
- 形成统一日志与结果记录
- 形成统一配置与代码落点

这一层不是“拍脑袋随便定”，而是：

- 在路线和文献允许的边界内
- 为了可复现、公平比较、结果可追溯
- 必须显式标成 `工程冻结规则`

### 4.5 发现冲突时的固定裁决方式

后面重修时，不允许只在脑子里判断“这个地方应该听谁的”，而必须把冲突裁决显式写出来。

凡是出现下面任意一种情况，都必须新增一条标准化裁决记录：

- 路线层结论和阶段细则冲突
- 文献原式与当前执行写法冲突
- 前面文件与后面文件的术语、参数、公式或阶段边界冲突
- 工程冻结规则与论文支持候选范围冲突
- 同一个对象在文档、配置字段和代码命名中的含义不一致

固定记录格式如下：

```md
## 冲突裁决记录

- 冲突对象：
- 冲突来源：
- 裁决结论：
- 裁决理由：
- 受影响文件：
- 是否需要回流修订：
- 代码实现影响：
```

裁决原则固定为：

1. 官方协议固定项优先于阶段经验写法
2. 路线层锁定优先于局部阶段自行扩写
3. 文献直接支持优先于无出处工程习惯
4. 全局一致的工程冻结规则优先于单文件临时口径
5. 如果裁决改变了前面已经写好的文件，必须回流修订受影响文件，不能只在当前文件偷偷改口

### 5.2 第二步：完整读取当前文件

- 当前文件缺了什么
- 当前文件有哪些地方写得太软、太笼统、太工程化或太局部化

### 5.3 第三步：遍历 `02_路线与投稿`

### 5.4 第四步：遍历 `03_文献证据`

- 当前文件前面的相关文件
- 当前文件后面的相关文件
- 同阶段其它细则文件
- 后续阶段对当前文件提出的前置要求

检查目标：

- 有没有前后矛盾
- 有没有后面的理论约束没有回流到前面
- 有没有后面写错了、但如果不甄别就会继续污染前面

这一步的核心原则是：

> 全局参考，但不盲从；前后互校，但不机械继承。

如果这一步发现当前修改会影响其它文件，必须额外列出：

- 受影响的上游文件
- 受影响的同阶段文件
- 受影响的下游文件
- 每个文件需要同步回改的字段、公式、命名、版本名或验收口径

### 5.6 第六步：重写当前文件

重写时必须明确区分：

- `官方协议固定项`
- `路线层已锁定`
- `论文直接支持`
- `论文支持的候选范围`
- `工程冻结规则`
- `待确认项`

并且要把“是什么、为什么、依据是什么、代码怎么落、怎么验收”写出来。

当前文件完成正文修订后，还必须补两类标准收尾：

- `## 冲突裁决记录`
- `## 文件质量自检`

其中：

- 只要出现跨文件冲突、旧口径回收、字段改名、路径改名、评估规则改口，就必须补 `冲突裁决记录`
- 即使本轮没有新增理论内容，只要改了正式规则，也仍然必须补 `文件质量自检`

### 5.6.1 代码落地最小接口模板

如果某条规则最终要进入代码实现，只写一个粗粒度文件路径还不够，至少还要写清下面这些内容：

- 入口类名或函数名
- 输入输出张量 `shape`
- 输入输出 `dtype`
- 依赖的配置字段名
- 关键前置断言或边界条件
- 会产出的日志、可视化或中间结果

固定写法如下：

```md
## 代码落地接口

- 代码文件：
- 入口类/函数：
- 输入：
- 输出：
- dtype：
- 依赖配置：
- 前置断言：
- 运行产物：
```

如果一条规则无法写到这个颗粒度，通常说明它还不能直接指导实现。

适用性说明：

- 如果当前阶段的代码框架已确定，例如 `baseline` 之后的模块阶段，上述字段应尽量全部写齐
- 如果当前阶段属于首次搭建框架，例如 `UNet` 首轮排雷阶段，允许 `入口类/函数` 暂时写成“预定义名”，但 `shape`、`dtype` 和依赖配置字段必须先确定
- 对于核心创新模块，例如 `LKMA / Boundary Head / Distance-aware Loss`，除上述字段外，还建议额外补一段 `PyTorch` 风格的 `forward` 计算流伪代码，确保“文档到代码”的翻译路径无歧义

### 5.7 第七步：补写当前文件的质量自检并运行 diagnostics

每个文件末尾必须增加：

- `## 文件质量自检`

这一节不是装饰，而是当前文件是否允许进入下一份文件的通过线。

如果质检未通过，当前文件不允许算完成。

并且这里的质检不是只看本文件自己的清单，而是必须同时执行两套检查：

1. 本文件 `§9` 的通用质量自检清单，用来检查全局互校、来源分层、理论解释、代码落点和通过线
2. `结直肠腺体分割_plan_优化版/01_实验执行/00_总览与规范/06_实验执行证据化写作模板.md` `§7` 的改造完成检查清单和一票否决项，用来检查规则卡片、格式强度、证据化写法和代码约束是否齐全

两套检查都通过后，还必须再执行一轮 diagnostics。

固定要求：

- 每个文件改完都必须跑 diagnostics
- 如果 diagnostics 未运行，当前文件不能算完成
- 如果 diagnostics 发现问题，必须先修问题，再重新检查 `冲突裁决记录` 与 `文件质量自检` 是否仍然成立

## 7. 当前统一写作边界

### 7.1 必须写清的内容

- 来源
- 作用
- 原理
- 公式或定义
- 参数来源类型
- 代码落点
- 运行记录字段
- 验收方式
- 回退条件

### 7.2 必须避免的内容

- 只有参数值，没有来源
- 只有结论，没有原因
- 只有流程，没有理论
- 只有文献名字，没有提取到可执行结论
- 只有工程口径，没有说明它不是论文原值
- 只有“建议/推荐”，没有冻结边界
- 只引用后续文件，不甄别它是否正确

### 7.3 必须显式标注的内容

- 论文原值
- 工程默认值
- 论文支持候选范围
- 路线锁定项
- 待确认项

### 7.4 理论内容写作深度标准

后面阶段文档中的公式、原理和设计依据，不是“写到提了一句就行”，而必须同时满足下面三个最低标准：

1. 能直接翻译为代码：看完文档后，不需要再回头查原论文主体，就能写出对应的 `PyTorch` 或数据处理实现
2. 能直接写进论文：公式符号、变量命名和文字解释达到可直接整理进 `Method / Experiment Setup` 的程度
3. 能解释为什么：不仅列出定义，还要说清这个公式或操作解决什么问题、为什么选它而不是相邻方案

不同对象的最低深度要求如下：

- `loss` 函数：必须写出完整数学定义、符号说明、权重组合方式，并标注 `eps / smooth / class weight / lambda` 等稳定项如何处理
- 模型模块：必须写出核心操作链、输入输出张量维度变化、插入位置，以及必要时的参数量或计算量估算
- 数据变换：例如边界图、距离图、拓扑先验，必须写出像素级定义、结构元素或邻域定义、边界条件和异常样本处理方式
- 评估指标：必须写出对象匹配规则、计算公式和与官方脚本或任务论文的一致性边界；如果已在总览文件固定，也要明确回指
- 训练策略：例如 optimizer、scheduler、early stopping、best checkpoint selector，必须写出更新或选择规则的数学或逻辑形式，并标注哪些属于工程冻结值

判断是否达标时，不看“有没有提到公式”，而看是否达到：

- 看完就能写代码
- 看完就能写论文
- 看完就知道为什么这样定

公式来源标注规则：

后面每出现一个公式、定义或组合规则，必须同时标注它属于下面哪一类：

1. `原文直接采用`：完全照搬某篇论文的公式或定义，只需标注来源并保持口径一致
2. `原文式 + 本文适配`：基于某篇论文的核心思想，但在结构、参数、组合方式或适用场景上做了当前项目的适配改写。此时必须同时写清“原式是什么”和“本文改了哪里、为什么改”
3. `本文自定义`：当前项目的工程设计、组合规则或实验协议，没有单一文献原式直接对应。此时必须写清设计逻辑、选择原因和不采用相邻方案的理由

这样做的目的固定为：

- 论文写作时，能直接区分 `citation` 和 `contribution`
- 代码实现时，知道哪些地方必须严格对齐原文，哪些地方允许按当前项目设计实现
- 审稿回复时，能直接回答“这是已有方法、受启发改写，还是当前工作的自定义贡献”

### 7.5 全局符号与术语一致性规则

后面所有阶段文档中的公式、变量、张量名、指标缩写、版本名和代码字段名，必须遵守同一套口径，不能每个阶段各写一套。

最低要求如下：

- 对已经在前面文件定义过的核心符号，后面只能复用，不能改写含义
- 如果必须新增符号，必须在当前文件显式给出符号说明，且说明它与已有符号的关系
- 文档中的符号名、配置字段名、日志字段名和代码变量语义必须一一对应
- 同一个对象不能在不同文件中同时使用多套名称，除非明确写出“主名 / 别名 / 历史名”

建议采用的最小记录表如下：

```md
## 符号与术语表

- 符号或术语：
- 数学定义或语义：
- 张量形状或对象类型：
- 对应代码字段名：
- 首次定义文件：
```

判断是否通过，不看“有没有解释过”，而看：

- 论文公式能否直接映射到代码变量
- 配置字段能否直接映射到文档对象
- 后续文件是否还能无歧义复用

### 7.6 `待确认项` 的关闭机制

`待确认项` 只能作为临时状态，不能作为长期停留的写法。

后面任何文件只要出现 `待确认项`，都必须同时写清：

- 提出阶段
- 关闭截止阶段
- 关闭所需证据
- 未关闭前禁止进行的动作
- 关闭后必须回改的文件

固定写法如下：

```md
## 待确认项

- 当前问题：
- 提出阶段：
- 关闭截止阶段：
- 需要补充的证据：
- 未关闭前禁止动作：
- 关闭后必须回改文件：
```

关闭原则固定为：

1. 能在当前阶段关闭的，不允许拖到后面
2. 影响训练协议、评估口径、标签定义或主结果统计的 `待确认项`，未关闭前不得进入依赖它的下一阶段
3. `待确认项` 一旦关闭，必须回流修订所有受影响文件，不能只更新最新文件
