### 1.3 当前阶段真正要证明什么

本阶段要证明的不是“`UNet` 又多跑了两次”，而是下面六件事已经同时成立：

1. `A1` 阶段冻结的 `UNet` 训练与评估协议不是依赖单次偶然 run 才成立。
2. 在只改变 `seed` 的前提下，`UNet` 能稳定完成训练、验证、`TestA`、`TestB` 与可视化闭环。
3. 三次正式重复已经足以形成第一版正式 `mean +- std` 稳定性证据。
4. 主要失败模式在不同 seed 下具有可复查的一致性，而不是每次都完全漂移。
5. `UNet` 已经可以作为后续 `ResNet34-U-Net` 与模块阶段的最小稳定对照起点。
6. 下游阶段比较到的差异可以优先解释为结构或模块变化，而不是 `UNet` 自身的随机漂移。

### 1.4 当前文件对下游的硬约束

从 `04_Baseline` 开始，下游只允许继承本文件冻结下来的稳定性基线，不允许：

- 用新的 seed 集合替换 `3407 / 1234 / 2025` 再重写 `UNet` 稳定性结论。
- 把协议不一致的 run 混入 `mean +- std` 结果后再作为 baseline 参照。
- 在 `TestA / TestB` 结果出来后反向修改 `threshold_value`、`best_selector` 或汇总 schema。
- 只引用均值不引用 per-seed raw results，导致下游无法判断波动是否真实。
- 把异常 run 的 bug 或协议漂移，伪装成“真实 seed 波动”交给后续阶段继续比较。

## 3. 本阶段唯一允许处理的变量

### 3.1 当前阶段直接继承哪些冻结项

本阶段直接继承并不得擅自改动的上位规则如下：

- 数据协议：`GlaS train68 / val17 / testA60 / testB20`
- 标签口径：`mask > 0`
- 输入通道：`RGB`
- 输入尺寸：`512 x 512`
- 归一化：`ImageNet mean/std`
- 增强版本：`light_aug_v1`
- 主损失：`L_seg = L_BCE + L_Dice`
- 优化协议：`AdamW + lr 1e-3 + weight_decay 1e-4 + ReduceLROnPlateau + epoch_max 120 + early_stopping 20 + AMP on`
- `best_selector = val_objdice_max`
- `threshold_source = val17`
- `eval_cast_policy = logits/probabilities must be kept or cast to float32 before thresholding`
- 评估链：`logits -> sigmoid -> threshold -> binary mask -> connected components -> metrics`
- `boundary_metric_width = 3 px`
- `boundary_metric_impl = np.logical_xor(mask, ndimage.binary_erosion(mask, structure, border_value=0)) + ndimage.binary_dilation`
- `connected_components_impl = scipy.ndimage.label`
- `connected_components_connectivity = 8`
- `eval_proto_version = eval_proto_v1`
- `postprocess_version = none_in_v1`

### 3.1 当前评估协议与历史追溯边界

当前正式结果统一使用 `eval_proto_v1`。`protocol_v3` 仅用于历史追溯、历史 lineage 或归档映射，不能进入当前 `A2` 汇总、`Gate_A2`、`stage_pass_a2`、`workflow_gate_status` 或任何当前交付资产。`stage_pass_a2/Gate_A2` 是 A2 阶段内部的实验放行层级；`workflow_gate_status` 是跨阶段工作流状态层级，二者必须分别记录、分别判定，不能混写。

### 3.2 当前阶段禁止重新打开哪些问题

本阶段禁止重新打开：

- 数据 split、标签规则、输入尺寸和归一化。
- `UNet` 的结构定义和 `A1` 阶段已冻结的训练协议。
- `best_selector`、`threshold_source`、`threshold_value` 和 `eval_cast_policy` 的来源原则。
- `TestA / TestB` 分开报告的正式评估口径，以及 `Boundary F1 = 3 px`、连通域实现与连接性定义。
- 后处理、TTA、额外 benchmark 和更强 baseline。

### 3.3 当前阶段为什么只能继承不能重开

因为本阶段真正要回答的是“同一协议下的随机波动范围”，而不是再次调参。

如果这一阶段同时夹带：

- 改增强
- 改 optimizer
- 改阈值
- 改 batch size
- 改后处理

后面就无法判断：

- 波动来自 `seed`
- 还是来自协议漂移

因此，`A2` 必须被压成“只改变 seed，其余全部固定”的阶段。

## 4. 阶段门控表达式

### 4.1 当前阶段唯一允许处理的核心变量

本阶段唯一允许变化的核心变量只有：

- `train_seed`

更准确地说，只允许在完全相同的正式协议下切换：

- `3407`
- `1234`
- `2025`

### 4.2 当前阶段不允许改什么

本阶段严禁借重复实验之名改动：

- 数据划分
- 输入尺寸
- 增强版本
- loss 形式与权重
- optimizer / lr / scheduler
- `epoch_max`
- `early_stop_patience`
- `batch_size`
- `best_selector`
- `threshold_source`
- `threshold_value`
- `postprocess_version`
- 可视化模板

### 4.3 为什么唯一变量只能是 `train_seed`

因为本阶段要建立的是第一版正式稳定基线。

只有把差异压缩到：

```text
proto(r_i) = proto(r_j)
train_seed(r_i) != train_seed(r_j)
```

后续 `UNet vs Baseline` 和模块增益比较才有解释力。

### 5.4 `mean +- std` 汇总规则

- 当前结论：`A2` 阶段正式主结果必须从单次值升级为按同一协议、同一 split、同一指标、三个固定 seed 计算的 `mean +- std`。
- 规则类型：`路线层已锁定 + 工程冻结规则`
- 适用阶段：`03_UNet稳定性` 开始生效；后续 `Baseline` 与模块阶段默认继承。
- 直接依据：`结直肠腺体分割_plan_优化版/02_路线与投稿/02_结直肠腺体分割_分阶段实验路线与执行标准.md`，`结直肠腺体分割_plan_优化版/01_实验执行/03_UNet稳定性/02_mean_std汇总规则.md`，`结直肠腺体分割_plan_优化版/01_实验执行/00_总览与规范/03_命名与结果记录规范.md`
- 核心公式或定义参考：

```text
mean(m, s) = (1/3) * sum_k x_k
std(m, s) = sqrt((1/3) * sum_k (x_k - mean)^2)
```

其中 `x_k` 来自三个正式 seed run 的 sample-only 结果，且 `TestA / TestB` 必须分开统计；当前正式七项指标为 `F1/Object Dice/Object Hausdorff/Dice/IoU/HD95/Boundary F1`，对应 raw=42、mean±std 目标维度=14。

- 采用原因：后续所有结构升级和模块增益都需要和稳定基线比较，而不是和单次偶然高值比较。
- 不采用的相邻方案：不采用只汇报最好 seed；不采用混算 `TestA / TestB`；不采用把异常重跑和正式 run 混入同一统计组。
- 代码落点：[project_root]/scripts/summarize_stage.py；历史计划曾提到 compare_runs.py，但当前不新增该文件，所有统计派生统一由 summarize_stage.py 负责，[project_root]/reports/tables/unet_seed_results.csv，[project_root]/reports/tables/unet_mean_std_summary.csv
- 运行记录字段：`model_name`，`dataset`，`split_role`，`metric_name`，`seeds`，`mean`，`std`，`n_runs`，`aggregation`，`result_tag`，`config_version`，`eval_proto_version`
- 验收方式：检查三个正式 run 的 sample-only raw 结果覆盖七项指标、raw=42、协议身份和错误模式；如执行统计导出，再检查 mean±std=14、`aggregation = mean+-std` 和 `n_runs = 3`。aggregate 行不是当前 per-run CSV 的放行条件。

### 5.5 错误模式一致性记录

### 5.6 异常 run 处理边界

## 7. 回退条件

### 7.1 独立回退触发条件

即使三次训练已经跑完，只要出现下面任意一条，本阶段都必须优先回退，而不是直接进入 `Baseline`：

- 某个 seed 的结果显著异常，且没有排查记录。
- 三次 run 的停止 epoch、验证趋势或错误类型完全失序，无法由随机波动解释。
- 指标与可视化长期严重矛盾。
- run_meta.yaml 中的 `best_selector`、`threshold_source`、`threshold_value` 与正式协议不一致。
- 某次 run 的正式资产缺失，导致不能独立复核训练、测试或错误模式。
- 汇总结论与 raw per-seed 表无法相互对齐。

### 7.2 固定回退顺序

回退时统一按下面顺序排查：

1. 检查是否误改了 `A1` 已冻结协议。
2. 检查 seed 控制、worker 随机性与配置分发是否一致。
3. 检查训练、验证、测试导出资产是否齐全且可相互对齐。
4. 检查异常 run 是否属于流程 bug 或协议漂移。
5. 只有确认前面都无问题后，才把差异解释为真实波动。

### 7.3 回退后的重新放行条件

发生回退后，只有同时满足下面条件，才允许重新声明本阶段通过：

- 问题来源已经按固定回退顺序定位并修复。
- 修复动作、影响范围和复验结果已写入 [project_root]/notes/debug_note.md、逐 run 总结或阶段总结。
- config.yaml、run_meta.yaml、指标文件和可视化已重新对齐。
- `best_selector / threshold_source / threshold_value` 仍与正式协议一致。
- 当前 run 重新满足第 `6.2` 节的全部过线标准。
