> 审查日期：2026-07-13
>
> 审查对象：`04_Baseline`
>
> 审查范围：项目工程、阶段计划、项目级 `.trae`、工程级 `.trae`、正式参考资料、运行产物、门禁报告和说明文档。

## 1. 最终裁决摘要

### 1.1 04 阶段当前不能判定为完全通过

当前不能把 04 阶段称为：

- 高质量完美闭合；
- 全部科学证据已经完成；
- 所有门禁一致通过；
- 已正式 handoff 给 05_LKMA。

当前正式裁决为：

```text
04_Baseline experiment/code/workflow gate = pass
04_Baseline learning-doc manual review = conditional_partial
05_LKMA research/stage-lock entry = allowed
05_LKMA formal coding/training = must pass its own gates
```

原因不是已经发现 B1 v3 指标必然错误，而是以下证据链尚未统一：

1. 阶段实现卡、Pre-check、Post-QC、diagnostics、learning document gate、workflow gate 和 v3 summary 的状态互相冲突。
2. Object F1、Object Hausdorff、HD95、Boundary F1 的文档、实现和参考资料归因尚未达到完全一致。
3. 测试入口对 checkpoint、模型、seed、配置和评估协议的身份强校验仍不足。
4. 当前 `metric_crosscheck` 主要是同一 CSV 的再聚合，不是从预测图像重新计算的独立指标验证。
5. 参考文献、参考代码、版本、commit、权重和本项目改动点还没有逐文件完全闭合。
6. 计划中的路径和当前工程实际路径仍存在旧接口与 v3 接口并存的问题。

### 1.2 不需要全部删除 04，也不应立即全部重训

当前不建议删除：

- A2/B1 历史实验目录；
- v1/v2/v3 checkpoint；
- 训练日志和测试 CSV；
- predictions、visuals、error cases；
- runtime、code quality、聚合和审查证据。

当前更合理的处理方式是：

```text
保留真实资产
-> 明确历史轮次与当前轮次
-> 修复协议和治理冲突
-> 补强身份校验
-> 独立重算指标
-> 依据真实结果决定是否重评估或重训
```

只有在以下情况成立时，才需要重新训练：

- 独立指标重算证明正式指标 CSV 与预测结果不一致；
- checkpoint 与 run_meta、seed、模型或配置不能绑定；
- 训练协议、数据 split、模型身份或评估协议实际不符合冻结规则；
- 指标定义被判定为实质性错误，且错误已经影响正式结果；
- A2 无法获得足够身份追溯，无法作为严格比较基线。

如果身份和指标核验均通过，则优先只重新测试、聚合和生成门禁，不因为当前审查本身盲目重训。

## 2. 审查证据边界与轮次区分

### 2.1 历史轮次与当前 v3 轮次不能混写

项目现有记录中至少存在以下语义不同的轮次：

| 轮次 | 语义 | 可否作为当前 04 最终证据 |
|---|---|---:|
| A2 v1 / 历史 A2 | UNet 对照和历史基线 | 只能作为配对基线，身份强度需复核 |
| B1 v2 | 已记录过的历史重跑，曾触发稳定性阻断 | 只能作为历史 blocked 证据 |
| A2/B1 protocol_v3 | 当前评估协议下的正式结果轮次 | 只有在身份、指标和门禁重新闭合后才能作为最终证据 |
| runtime probe | 最小运行验证 | 不能替代正式训练结果 |
| smoke / CPU / backup | 辅助验证或历史资产 | 不能进入正式聚合 |

历史轮次必须保留，但所有汇总、说明文档和阶段 Gate 必须显式记录：

```text
run_version
result_tag
run_name
protocol_version
```

不得把 B1 v2 的 blocked 结论、B1 v3 的业务 pass 表述和旧学习文档中的 pass 文字拼成一个无轮次结论。

### 2.2 当前已确认的 v3 运行资产

现场核对确认 B1 protocol_v3 三个正式 run 目录实际存在以下类型资产：

- `best.ckpt`；
- `last.ckpt`；
- `train_log.csv`；
- `val_metrics.csv`；
- `testA_metrics.csv`；
- `testB_metrics.csv`；
- `config.yaml`；
- `run_meta.yaml`；
- TestA/TestB predictions；
- TestA/TestB visuals；
- run summary；
- error cases；
- train/validation curves；
- metric crosscheck note。

因此，早期关于 B1 v3 缺失 checkpoint、visuals 或 error cases 的判断不成立，不能继续作为阻断理由。

但目录存在不等于资产已经完成身份闭合。仍需验证：

- checkpoint 是否就是测试入口实际加载的 checkpoint；
- `run_meta.yaml` 与测试 CSV 是否使用同一 run identity；
- config、评估协议和模型模块是否与结果一致；
- predictions 是否由当前 checkpoint 生成；
- manifest 是否真的检查资产内容而不是只检查目录存在。

## 3. 固定执行链逐项审查

### 3.1 治理

**结论：blocked。**

根本问题是多个文件同时充当状态源，但没有统一的机器可验证裁决关系。

当前现场状态出现过以下组合：

```text
baseline_v3_stage_summary: stage_pass_b1=true
workflow_gate_report: pass
diagnostics_result: pass
阶段实现卡: business Stage Gate=blocked
Pre-check Guard: Gate_B1=blocked
Post-QC Guard: Final Status=partial
learning document report: 顶层 pass，正文仍含 partial/fail
```

这不能作为正式闭合状态。必须明确不同层级：

```text
business_stage_result
workflow_gate_status
learning_doc_automatic_status
learning_doc_manual_status
post_qc_status
handoff_status
```

并明确哪个字段最终决定 05 是否可进入。

### 3.2 研究

**结论：研究方向基本成立，但引用闭合为 partial。**

已有研究依据能够支撑：

- U-Net 作为 A2 对照；
- ResNet34 encoder + U-Net decoder 作为 B1 baseline package；
- GlaS 作为数据与任务背景；
- DCAN/GlaS 方向作为对象级指标参考；
- MedPy/surface-distance 作为距离指标实现参考。

但以下内容尚未完全达到可审计级别：

- 每个实现文件对应的具体参考文件、函数和版本；
- 外部仓库 commit 或可信本地快照 hash；
- torchvision 版本和 ResNet34 权重来源；
- 权重实体的 hash；
- 参考实现与本项目修改点；
- 哪些指标是官方实现，哪些是项目工程适配，哪些是项目自定义。

TCGA 外部资料不可访问时，可以使用 01/02/03 阶段说明文档作为内部写作模板，但必须保留：

```text
external_tcga_reference_status=unavailable
learning_doc_gate_mode=internal_stage_templates
```

内部模板不能被写成 TCGA 原始数据证据、外部指标证据或原始代码归因。

### 3.3 阶段锁定

**结论：当前锁定状态未统一。**

阶段实现卡仍然写着旧的 `blocked` 和“不授权直接编码、训练或进入 05”，而 v3 summary 已写成业务 pass。这说明 v3 结果生成后没有完成治理文件同步收口。

阶段锁定必须重新写明：

- 当前正式轮次；
- 历史轮次；
- 冻结协议；
- 允许执行的动作；
- 当前是否允许进入 Pre-check；
- 当前是否允许进入 05。

### 3.4 Pre-check

**结论：文档存在，但不能判定为最终通过。**

Pre-check 必须新增或重新验证：

- 新旧 run 物理隔离；
- checkpoint schema；
- runtime model identity；
- CUDA/AMP 证据；
- run_meta 与 test CSV 绑定；
- checkpoint hash、epoch、metric；
- 结果资产覆盖策略；
- 评估协议版本；

- 同一 run_name 并发执行拒绝或加锁。

### 3.5 编码

**结论：代码主体可行，但仍有两个正式收口缺口。**

已存在的有效工程能力包括：

- PyTorch 训练链；
- ResNet34-U-Net 模型；
- BCE + Dice；
- AdamW；

- 预测、visuals、summary 和 error cases 生成。

仍需修复：

1. `scripts/test.py` 对 checkpoint 和模型身份的强制校验不完整。
2. `summarize_stage.py` 中协议漂移 run 的 invalid 标记逻辑需要确认并真正执行。

### 3.6 运行证据

**结论：基本通过，但 runtime probe 不能替代正式训练证据。**

正式 runtime 证据至少需要能够追溯：

- 实际模型模块和类；
- input、target、output shape/dtype；
- finite loss；
- backward；
- optimizer.step；
- device；
- AMP/GradScaler；
- probe run_name 与正式 run_name 隔离。

当前已有 runtime/code quality 通过证据，但最终报告必须区分：

```text
runtime_probe_pass
formal_training_evidence_pass
```

不能把最小 probe 直接解释为三 seed 正式训练结果已经科学有效。

### 3.7 代码质量

**结论：基本可行，尚未达到最终可复现闭合。**

必须补强测试入口身份校验，并增加从预测 PNG 和 GT 独立重算指标的验证。当前同一 CSV 再聚合只能证明聚合函数内部一致，不能证明指标生成过程正确。

### 3.8 总放行

**结论：当前 `workflow gate=pass` 不能作为 04 完成依据。**

workflow gate 目前主要消费 v3 summary 中的：

```text
stage_pass_b1
handoff_ready_for_c1
```

但没有充分校验：

- 阶段实现卡；
- Pre-check；
- Post-QC；
- diagnostics；
- learning document gate；
- 参考资料闭合；
- 运行资产完整性；
- 当前正式轮次与历史轮次隔离。

应当修改为：workflow pass 只能表示其职责范围内通过，不得被解释成“04 全部完成”。如果项目最终要求 workflow gate 代表总放行，则它必须消费所有前置硬门禁并在不一致时阻断。

### 3.9 说明文档

**结论：自动 fallback 可通过，人工和整体文档状态仍 partial。**

TCGA fallback 的正确状态应当拆开写：

```text
automatic_learning_doc_gate=pass

manual_learning_doc_review=partial
overall_learning_doc_status=partial 或 conditional_pass
```

不能在报告正文中同时保留“不能返回 pass”的旧裁决和顶层 `pass`，也不能让 `diagnostics_result=pass` 覆盖人工审查 partial。

### 3.10 交付前清理

**结论：未完成。**

当前未达到交付前清理条件。需要在 04 真正完成后再执行：

- 正式资产与历史资产分层；
- runtime probe、smoke、backup、debug 和旧报告归档；
- 清理陈旧 pass 文档；
- 清理 nested duplicate project 风险；
- 保留不可替代的原始历史证据；
- 生成 A/B/C 交付清单。

## 4. 实验指标是否科学可行

### 4.1 总体判断

当前指标体系**可以实现 04 阶段的高质量规划**，但必须先完成协议统一和独立验证。指标本身不是“完全不可用”，也不能直接说“已经官方等价、已经无条件科学闭合”。

准确表述是：

```text
指标体系具有科学可行性和工程可实现性；
当前非空样例和内部聚合结果基本支持实现没有明显主路径错误；
但正式口径、空集规则、参考实现归因、独立重算和身份追溯尚未完全闭合；
因此当前只能判定为可修复、可完成，不判定为最终论文级通过。
```

### 4.2 Dice 与 IoU

Dice、IoU 属于常规像素级重叠指标，作为 GlaS 腺体分割的基础结果是科学合理的。

必须继续保证：

- 每张图独立计算；
- TestA/TestB 分开报告；
- 双空、单侧空规则显式冻结；
- sample mean 先于 seed mean/std；
- 不把 aggregate 行重复纳入 seed 聚合。

### 4.3 Object F1

当前代码采用的口径是：

```text
每个预测对象选择最大重叠 GT
overlap / GT_area > 0.5
不使用 Hungarian matching
```

该口径可以科学使用，也与当前构造性非空样例和本地 DCAN/GlaS 方向参考相容。但计划正文曾出现：

```text
overlap / prediction_area >= 0.5
一对一去重
```

这两个定义不是同一个指标。必须只保留一个冻结定义，并同步修改：

- 计划正文；
- `seg_metrics.py`；
- eval config；
- 实现依据记录；
- 构造性测试；
- 论文方法段；
- 汇总表表注。

在统一之前，Object F1 只能判为：

```text
科学可行：是
协议闭合：否
```

### 4.4 Object Dice

当前双向对象面积加权最大重叠规则具有工程可实现性，非空构造样例与本地参考方向一致。

必须明确：

- 是否允许多个 source object 复用同一 target object；
- 双向聚合的权重定义；
- 空对象集合处理；
- 指标方向为越高越好。

只要这几点写入冻结协议并通过测试，Object Dice 可以作为正式主指标。

### 4.5 Object Hausdorff

当前实现并不是简单的标准 Hausdorff 距离，而是：

- 对象级匹配；
- 有重叠时选最大重叠对象；
- 无重叠时选 Hausdorff 最近对象；
- 双向面积加权；
- 单侧空使用 `Dmax`。

该指标可以作为项目自定义对象级距离指标使用，但必须改名或加限定词，例如：

```text
project-adapted bidirectional area-weighted object Hausdorff
```

不能无条件写成 GlaS 官方 Object Hausdorff 完全等价。其方向为越低越好。

### 4.6 HD95

当前实现是 EDT、边界提取和 95 分位数的项目实现，配置名称为 MedPy-style。它具有科学可行性，但不能未经逐值验证声称与 MedPy 官方函数完全等价。

必须写清：

- pixel spacing = 1；
- discrete surface distance 的具体定义；
- 边界提取方式；
- `np.percentile` 方式；
- 双空和单侧空行为；
- 与 MedPy、surface-distance 的差异。

### 4.7 Boundary F1

当前实现采用：

```text
erosion_xor
3x3 ones
8-connectivity
border_value=0
tolerance=3 px
```

该指标作为项目自定义补充指标是可行的，但不能伪装成唯一的官方 Boundary F1。必须统一历史记录中 `find_boundaries(inner)` 与当前 `erosion_xor` 的命名，并补齐空边界和边缘对象测试。

### 4.8 三 seed mean/std

使用：

```python
np.std(values, ddof=0)
```

作为项目预注册的三 seed population-style 描述统计是可以接受的，前提是：

- A2/B1 完全使用同一口径；
- 表注明确 `ddof=0`；
- 不把三 seed 当作统计显著性检验；
- 不将三 seed std 误写成总体方差估计的强推断；
- TestA/TestB 分开报告。

### 4.9 稳定性 Gate

当前 Gate 要求：

```text
{F1, Object Dice, Object Hausdorff} x {TestA, TestB}
六个单元全部满足：
sigma_B1 <= sigma_A2 + 1e-12 * max(1, sigma_A2)
```

这是一条可执行、保守、可复现的工程验收规则，但不是论文领域普遍公认的统计显著性标准。它可以作为 04 阶段项目 Gate，不能称为统计学证明。

必须提前冻结：

- 主指标集合；
- 指标方向；
- `ddof=0`；
- 浮点容差；
- 是否六单元全部不增；
- 是否采用 per-seed 配对 delta；
- TestB 小样本的解释边界；
- 失败后的回退动作。

不得根据当前结果改变 Gate。

## 5. 当前结果的科学解释

### 5.1 已有结果可以支持什么

当前独立统计复核支持以下表述：

- A2 与 B1 使用相同的主要训练控制变量；
- A2/B1 的 TestA、TestB 样本数量和聚合结构基本正确；
- B1 的主要均值指标方向优于 A2；
- B1 TestA/TestB 的 F1、Object Dice 和 Object Hausdorff 方向改善；
- 六个稳定性单元在当前 v3 数字下数学上通过；
- 当前 B1 TestB Object Dice 的低 std 可以由三个真实 seed 数值接近解释；
- 暂未发现聚合行混入、复制字段、NaN、越界或明显手工修改证据。

### 5.2 当前结果不能支持什么

不能据此直接写：

- B1 已被统计显著证明优于 A2；
- B1 已证明泛化能力更强；
- 指标与所有官方脚本逐值等价；
- TCGA 外部标杆已完成人工对照；
- 04 阶段所有门禁已完成；
- B1 可以无条件进入 05。

推荐的科学表述：

> 在当前冻结的项目评估实现和三 seed 描述统计下，B1 在 TestA/TestB 的主要对象级指标上相对 A2 呈现方向一致的改善，且稳定性规则未观察到 B1 波动劣于 A2。该结果支持 B1 作为后续模块实验的候选 baseline，但不构成统计显著性优越证明；正式 handoff 仍需完成指标独立复核、身份链和治理门禁闭合。

## 6. 后续整改路线

### 阶段 A：先统一状态，不改实验数字

1. 明确 `v3` 为当前审查轮次，v1/v2 为历史轮次。
2. 统一阶段实现卡、Pre-check、Stage Gate、Post-QC、diagnostics、learning gate、workflow gate 和 v3 summary 的字段语义。
3. 将自动说明文档 Gate、人工审查、TCGA 外部可用性和业务 Gate 分开记录。
4. 删除或改写所有引用旧轮次的“已完成”“可进入 05”文字，但不删除历史文件。
5. 明确唯一的 04 handoff 机器判断入口。

### 阶段 B：冻结指标协议

1. 统一 Object F1 公式、阈值严格性和匹配复用规则。
2. 将 Object Hausdorff 明确标记为项目适配指标。
3. 将 HD95 明确标记为 MedPy-style 项目实现，补充与官方库的差异。
4. 将 Boundary F1 明确标记为项目自定义实现。
5. 冻结空集规则、`Dmax`、边界提取、连通性、tolerance 和 `ddof=0`。
6. 将稳定性 Gate 写成独立、可执行、不可后验放宽的规则。
