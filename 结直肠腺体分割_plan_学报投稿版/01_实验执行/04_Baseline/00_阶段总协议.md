### 1.3 当前阶段真正要证明什么

本阶段要证明的不是“`ResNet34-U-Net` 跑出了一个更高数值”，而是下面六件事已经同时成立：

1. 在数据协议、训练协议和评估协议都已冻结的前提下，`UNet -> ResNet34-U-Net` 的差异被压缩为结构升级，而不是协议漂移。
2. `ResNet34-U-Net` 能在 `3407 / 1234 / 2025` 三个固定 seed 下完成正式训练、验证、`TestA`、`TestB` 与可视化闭环。
3. `UNet vs baseline` 的比较已经可以按 per-seed raw results 与 `mean +- std` 双层证据正式成立，而不是只靠单次最好值。
4. encoder 初始化、skip 对接、`effective_batch_size` 和 checkpoint/threshold 继承规则已经被写成可直接指导代码实现的冻结协议。
5. baseline 阶段通过后，后续模块只允许在这份唯一正式基座上追加单变量，而不能再回头改比较起点。
6. 当前 baseline 尚未解决的腺体形态建模、边界模糊和黏连分离难点已经被明确写出，后续模块目标不再模糊。

### 1.4 当前文件对下游的硬约束

从 `05_LKMA` 开始，下游只允许继承本文件冻结下来的 baseline，不允许：

- 用新的 seed 集合替换 `3407 / 1234 / 2025` 再重写 baseline 结论。
- 把协议不一致的 `ResNet34-U-Net` run 混入正式 `mean +- std` 后继续当基座。
- 在 `TestA / TestB` 结果出来后反向修改 `best_selector`、`threshold_source` 或 `threshold_value`。
- 在 baseline 尚未放行时，就把 `LKMA / Boundary / Distance` 当作“顺手补 baseline”的手段。
- 只引用 baseline 均值，不引用 per-seed raw results、冻结字段和失败模式，导致下游无法判断增益是否真实。

### 1.5 正式 handoff 与条件性研究执行的边界

当前 `Gate_B1=false` 必须继续保留为原始阶段 Gate 与原始 handoff 的历史审计事实；不得把 `05_LKMA` 或后续阶段写成建立在“稳定性完全通过”的 baseline 上。与此同时，当前 B1 的正式预测、GT、逐样本指标、三 seed 聚合和独立复核均有效时，项目治理允许后续阶段在 warning 边界下正式执行研究、训练、测试、Gate 和 handoff，但必须显式标记：

- `baseline_status = valid_with_stability_warning`；
- `stability_warning = true`，仅针对 `TestB:Object Dice` 的三 seed 波动略高于 A2；
- `formal_handoff = conditional_handoff`，不覆盖原始 `Gate_B1`；
- 后续实验沿用 `3407 / 1234 / 2025`、当前 split、当前训练/评估协议、阈值和统计身份；
- 后续结果解释为相对于带 warning 的冻结 B1 对照，不得声称相对于稳定性完全通过的正式基座；
- 若后续模块增益接近 B1 波动、跨 seed 方向不一致、明显退化或发现真实工程错误，必须停止后续主线，按问题登记回退条件回退到 04；当前 warning 本身不触发自动回退。

该边界不是事后把 `false` 改成 `true`，也不是用后续模块补救 baseline；它把“04 原始 Gate/handoff 的历史状态”和“后续阶段对冻结 B1 的条件性正式消费”分层记录。后续阶段的正式执行只表示该阶段自身可以在 warning 边界下运行，不表示 B1 的原始稳定性 Gate 已通过。

## 3. 路线依据与文献依据

### 3.1 路线依据

当前路线已经把 baseline 的角色锁定为：

- `UNet` 负责建立最小稳定基线。
- `ResNet34-U-Net` 负责建立正式结构 baseline。
- `LKMA / Boundary / Distance` 只允许在 baseline 上做单变量追加。
- `外部对比` 和 `结果汇总` 只消费已经放行的 baseline，不负责替它补定义。

因此，本阶段不是“看看换 encoder 有没有提升”，而是要把全文后续实验都要继承的结构基座正式冻结下来。

### 3.2 文献依据

当前文献证据共同支持的结论是：

- `U-Net` 提供最小可解释的 encoder-decoder + skip 主干。
- `ResNet34` 提供成熟稳定的 residual encoder 方案，适合作为结构升级而不是方法创新本体。
- `GlaS` 和腺体任务文献强调对象级指标、`TestA / TestB` 分开报告、边界模糊和黏连分离难点。
- baseline 若不能先稳定回应这些难点的统计身份，后续模块就无法判断是在解决任务难点，还是在补 baseline 的缺口。

## 3. 本阶段唯一允许处理的变量

### 4.1 当前阶段直接继承哪些冻结项

本阶段直接继承并不得擅自改动的上位规则如下：

- 主任务：`2D gland segmentation`
- 主数据集：`GlaS`
- 正式 split：`train68 / val17 / testA60 / testB20`
- 标签口径：`mask > 0`
- 输入通道：`RGB`
- 输入尺寸：`512 x 512`
- 归一化：`ImageNet mean/std`
- 增强版本：`light_aug_v1`
- 主损失：`L_seg = L_BCE + L_Dice`
- 优化协议：`AdamW + lr 1e-3 + weight_decay 1e-4 + ReduceLROnPlateau + epoch_max 120 + early_stopping 20 + AMP on`
- 评估链：`logits -> sigmoid -> threshold -> binary mask -> connected components -> metrics`
- 主指标优先级：`F1 / Object Dice / Object Hausdorff`
- `best_selector = val_objdice_max`
- `threshold_source = val17`
- `eval_cast_policy = logits/probabilities must be kept or cast to float32 before thresholding`
- `boundary_metric_width = 3 px`
- `boundary_metric_impl = project_custom_erosion_xor_3x3_ones_8conn_border0_tol3px`
- `connected_components_impl = scipy.ndimage.label`
- `connected_components_connectivity = 8`
- `train_proto_version = train_proto_v1`
- `eval_proto_version`：新轮需在研究定标/阶段锁定后由当前 Gate 冻结；旧 `eval_proto_v3` 仅作为上一轮历史 lineage 保留
- `postprocess_version = <由当前轮 Gate 冻结>`；当前轮待冻结前不得进入正式 run。
- `eval_protocol_lineage` 只记录历史 `eval_proto_v1 -> eval_proto_v3`；当前 run/gate/main table 只消费当前轮 Gate 冻结的 `eval_proto_version`，不消费历史 lineage。

### 4.1.1 protocol_v3 指标定义与归因边界

当前指标定义必须来自当前轮 Gate 冻结的 `eval_proto_version` 配置与 `src/metrics/seg_metrics.py`；上一轮 `eval_proto_v3` 仅供历史对照，不得作为当前来源。以下定义必须在代码、配置、阶段报告和论文方法中保持一致：

- `Object F1`：每个预测对象选择最大重叠 GT，使用 `overlap / GT_area > 0.5`；不使用 Hungarian；这是当前项目冻结的对象检测口径。
- `Object Dice`：预测到 GT、GT 到预测双向最大重叠对象匹配，按对象面积加权后双向平均；允许的对象复用规则必须按当前代码执行。
- `Object Hausdorff`：项目适配的双向对象级面积加权 Hausdorff 距离；有重叠时选择最大重叠对象，无重叠时选择 Hausdorff 最近对象；双空为 0，单侧空使用图像对角线 `Dmax`。该名称不得解释为 GlaS/MedPy 官方函数的逐值等价实现。
- `HD95`：基于二值腐蚀边界、EDT 和双向 95 分位数的 `MedPy-style discrete surface distance`，pixel spacing 固定为 1；当前只声明 style-level 对齐，不声明与 MedPy 官方函数逐值等价。
- `Boundary F1`：项目自定义补充指标，固定 `erosion_xor`、3x3 ones、8-connectivity、border_value=0、tolerance=3 px；不冒充唯一官方 Boundary F1。
- 空集规则：重叠指标双空为 1、单侧空为 0；距离指标双空为 0、单侧空为 `Dmax`。
- 三 seed 统计使用 `np.std(values, ddof=0)`；六单元稳定性 Gate 是项目验收规则，不是统计显著性检验。

### 4.2 当前阶段禁止重新打开哪些问题

本阶段禁止重新打开：

- 数据 split、标签规则、输入尺寸和归一化。
- `UNet` 阶段已经冻结的 loss、optimizer、scheduler 和 checkpoint 选择逻辑。
- 主指标体系、对象级匹配规则、`threshold` 前 `float32` 边界，以及 `TestA / TestB` 分开报告口径。
- `Boundary F1 = 3 px`、边界实现、连通域实现与连接性定义。
- 后处理、TTA、外部对比方法、`CRAG` 正式验证和更强 backbone。
- `ResNet34-U-Net` 作为唯一正式 baseline 的路线角色。

### 4.3 当前阶段为什么只能继承不能重开

因为本阶段真正要回答的是“在同一 protocol 下，residual encoder 升级是否足以建立正式 baseline”，而不是再次调参。

如果这一阶段同时夹带：

- 改增强
- 改 optimizer
- 改阈值搜索策略
- 改 loss
- 改后处理
- 改外部 benchmark

后面就无法判断：

- 提升来自 residual encoder
- 还是来自额外协议漂移

因此，`B1` 必须被压成“只冻结结构基座，不重开训练和评估协议”的阶段。

## 4. 设计边界(做什么/不做什么)

### 5.1 当前阶段唯一允许处理的核心变量

本阶段唯一允许变化的核心变量只有：

- `encoder: plain -> ResNet34`
- 与该 encoder 升级直接绑定的初始化、skip 层级对接和最小必要显存适配记录

更准确地说，本阶段允许的变化应满足：

```text
proto(B1) = proto(A2)
diff(B1, A2) in {encoder_upgrade, encoder_init, shape_alignment, minimal_batch_adaptation_if_forced}
```

### 5.2 当前阶段不允许改什么

本阶段严禁借 baseline 之名改动：

- 数据划分
- 输入尺寸
- 增强版本
- loss 形式与权重
- optimizer / lr / scheduler
- `epoch_max`
- `early_stop_patience`
- `best_selector`
- `threshold_source`
- `postprocess_version`
- `LKMA / Boundary / Distance / Attention / multi-scale` 额外模块
- `CRAG` 或外部方法正式比较

### 5.3 为什么唯一变量只能是结构升级

因为 baseline 阶段的目标是建立全文唯一正式比较起点。

只有把差异压缩到：

```text
structure(B1) != structure(A2)
train_proto(B1) = train_proto(A2)
eval_proto(B1) = eval_proto(A2)
```

后续模块增益才具有解释力。

### 6.1 正式 baseline 身份与结构定义

- 当前结论：本阶段正式主 baseline 固定为 `ResNet34-U-Net`，其结构定义固定为 `encoder = ResNet34`、`decoder = U-Net style decoder`、输出为单通道腺体前景 `logit`。
- 规则类型：`论文直接支持 + 路线层已锁定 + 工程冻结规则`
- 适用阶段：`04_Baseline` 开始生效，后续主线阶段全部继承。

- 核心公式或定义参考：残差 encoder 的基本块满足 `y = F(x, {W_i}) + x`；整体结构固定为 `logits = Dec_theta_d(Enc_theta_e(x))`。
- 采用原因：`ResNet34` 作为经典 residual encoder，结构成熟、优化稳定、复杂度和表达能力平衡，适合作为 `UNet` 之后、模块改进之前的正式结构基座。
- 不采用的相邻方案：不采用 plain encoder 继续充当全文主 baseline；不采用 `ResNet50/101` 直接替代；不采用 attention、多头或多尺度增强结构充当 baseline 首版。

### 6.2 结构继承边界

- 当前结论：baseline 只允许把 `ResNet34` 作为 encoder 升级，把 `U-Net` 作为 decoder 组织原则；不得把 baseline 混成“残差 encoder + 额外模块集合体”。

- 采用原因：只有把差异压缩为 encoder 升级，`UNet vs R34UNet` 才能写成真正的公平结构对比。

## 8. 回退条件

### 8.1 独立回退触发条件

即使三次训练都已经跑完，只要出现下面任意一条，本阶段都必须优先回退，而不是直接进入下一阶段：

- baseline 平均主指标明显低于 `UNet`，且无法解释为协议一致前提下的正常波动。
- 某个 seed 长期异常或波动失控，但没有完整排查记录。
- `TestB` 明显更差，且可视化也不支持 baseline 作为正式主基座。
- encoder 预训练权重未正确加载或来源不可追溯。
- `effective_batch_size` 改了，却仍把结果写成“纯结构对比”。
- run_meta.yaml 中的 `best_selector`、`threshold_source`、`threshold_value`、`eval_cast_policy`、`boundary_metric_width` 或连通域实现字段与正式协议不一致。
- 汇总结论与 per-seed raw table、阶段 manifest 或可视化证据无法相互对齐。

### 8.2 固定回退顺序

回退时统一按下面顺序排查：

1. 检查 `ResNet34` encoder 接入与预训练加载是否正确。
2. 检查 skip 抽取层级、通道匹配和输出尺寸是否正确。
3. 检查 `physical_batch_size / grad_accum_steps / effective_batch_size` 是否真实一致。
4. 检查训练协议、评估协议、`best_selector` 与阈值来源是否真正继承。
5. 只有确认前面都无问题后，才把差异解释为真实结构比较结果。

### 8.3 回退后的重新放行条件

发生回退后，只有同时满足下面条件，才允许重新声明本阶段通过：

- 问题来源已经按固定回退顺序定位并修复。
- 修复动作、影响范围和复验结果已写入 [project_root]/notes/debug_note.md、逐 run 总结或阶段总结。
- config.yaml、run_meta.yaml、指标文件、可视化和阶段 manifest 已重新对齐。
- `best_selector / threshold_source / threshold_value` 仍与正式协议一致。
- 当前 run 与聚合结果重新满足第 `7.2` 节的全部过线标准。
