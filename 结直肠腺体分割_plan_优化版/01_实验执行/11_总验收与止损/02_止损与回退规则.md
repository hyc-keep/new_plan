# 止损与回退规则

本文件不是经验提醒，也不是“等最后写稿时再看一眼”的备忘录。

它在 `11_总验收与止损` 阶段中的唯一职责是：

> 把所有争议对象统一压成正式止损协议底表，确保模块去留、外部对比边界、`CRAG` 结论强度、写作降调和工程回退都只依据冻结协议、正式结果、来源边界和公平性边界做裁决；后续任何总状态判断、中文保底、英文起步和写稿进入结论，都只能忠实继承本文件，不允许再因为沉没成本、实验投入或主观乐观而临时改口。

## 1. 文件角色与执行边界

### 1.1 当前文件负责什么

当前文件只负责冻结下面八件事：

1. 全项目统一的止损触发公式，也就是哪些情形一旦成立，争议对象就必须进入正式止损判断。
2. 全项目统一的四档动作边界，也就是 retain / downgrade / remove / rollback 的唯一含义和适用边界。
3. 主线模块层、外部对比层、`CRAG` 层、写作表述层和工程协议层的正式止损规则。
4. 总验收阶段对争议对象的最小记录字段、证据来源要求和下游消费语义。
5. 与 `结直肠腺体分割_plan_优化版/01_实验执行/11_总验收与止损/00_阶段总协议.md`、01_阶段门槛总表.md、03_中文保底条件.md、04_英文起步条件.md 的联动边界。
6. 独立 `回退条件`、固定回退顺序和重新放行条件，防止带着来源不清或协议冲突的对象进入总裁决。
7. 面向 stop_loss_decision.md、final_readiness_judgement.md 和 writing_entry_decision.md 的代码落地接口与字段链。
8. 本文件自己的 `冲突裁决记录`、强版 `文件质量自检` 和独立 `Diagnostics 闭环`。

### 1.2 当前文件不负责什么

当前文件不重新定义下面这些内容：

- 数据身份、split、标签口径、阈值来源、评估脚本和结果记录规范本体；这些内容以上游冻结协议为准。

- 中文保底、英文起步和写稿进入的档位定义本体；当前文件只提供它们必须继承的止损边界和降调前提。
- 正式论文文本、摘要句式和章节组织；当前文件只回答哪些对象还能保留为主线证据、哪些必须降级、删除或回退。

### 1.3 为什么当前文件必须独立存在

总验收阶段最容易出现的伪完成状态不是“完全没讨论止损”，而是下面这些情况：

- 某个模块已经跑了很多实验，于是默认应该保留在主线，但正式证据其实只在波动范围内。
- 外部对比已经接进主表，于是默认应该继续保留，但来源标记或公平边界并没有写清。
- `CRAG` 已经跑完，于是默认可以写成稳定跨 benchmark 支撑，但方向、强度和角色其实还不够。
- 中文或英文已经开始构思，于是默认局限性、降调和删除项可以先不处理。
- 工程协议已经出现分叉，于是默认后面回写表注就能补救，但事实上已经影响正式结果有效性。

如果没有一份独立止损协议底表，后续 stop_loss_decision.md、项目总状态判断和写稿进入判断就会重新落回“感觉上还能留一下”的口头协商，而不是正式裁决。

## 3. 本阶段唯一允许处理的变量

### 3.1 当前文件真正要证明什么

本文件真正要证明的，不是“项目里哪些内容看起来还不错”，而是：

> 在阶段门槛、结果包、外部对比、`CRAG` 和写作边界都已经冻结的前提下，所有争议对象都已经被诚实压成 retain / downgrade / remove / rollback 中的一档，不再存在“后面写稿时再决定”“先保留着也许能讲”的灰区。

因此，本文件通过的含义同时包括：

- 主线模块是否继续承担正式贡献已经成文。
- 外部对比哪些还能进入主表、哪些只能做引用对照已经成文。
- `CRAG` 到底是 A / B / C 哪种支撑，还是只能降到局限性说明已经成文。
- 中文与英文的允许表述边界已经被止损规则先行约束。
- 工程协议若已分叉，必须先回退到最后一个清楚冻结版本，而不是把问题带入总裁决。

### 3.2 当前文件不允许用什么替代正式裁决

当前明确不允许把下面这些情况写成“已经完成止损判断”：

- 只因为某个模块已经投入很多训练，就默认它应当保留。
- 只因为某个外部方法已经接入，就默认它有主表资格。
- 只因为 `CRAG` 已经跑过，就默认它可以支撑强泛化表达。
- 只因为中文或英文准备开写，就默认删减、降调和局限性可以后面再补。
- 只因为大部分对象都已经判定，就允许剩余争议对象继续留在“待定”状态。

### 3.3 当前文件与 `Gate_11` 的关系

本文件不是独立于总验收阶段门之外的附录，而是 `Gate_11` 的直接组成部分之一：

```text
Gate_11 = stage_matrix_ready
          AND project_status_fixed
          AND stop_loss_fixed
          AND cn_ready_fixed
          AND en_ready_fixed
          AND writing_entry_fixed
```

其中当前文件直接负责冻结：

```text
stop_loss_fixed = all_contested_targets_classified
                  AND action_boundary_fixed
                  AND rollback_boundary_fixed
                  AND downstream_stop_loss_schema_fixed
```

只有当所有争议对象都已落到唯一动作、没有待定项、没有来源不清项、没有协议分叉未回退项时，`stop_loss_fixed` 才能为真。

## 4. 阶段门控表达式

### 4.1 当前文件直接继承哪些冻结项

本文件直接继承并不得擅自改动的上位规则如下：

- 主任务：`2D gland segmentation`
- 主数据集：`GlaS`
- 第二 benchmark：`CRAG`
- `GlaS` 正式 split：train68 / val17 / testA60 / testB20
- `CRAG` 正式 split：train153 / val20 / test40
- 主指标优先级：F1 / Object Dice / Object Hausdorff
- 补充指标：Dice / IoU / HD95 / Boundary F1
- `best_selector = val_objdice_max`
- `GlaS threshold_source = val17`
- `CRAG threshold_source = val20`
- eval_cast_policy = logits/probabilities must be kept or cast to float32 before thresholding
- `boundary_metric_width = 3 px`
- `boundary_metric_impl = skimage.segmentation.find_boundaries(mode=inner) + binary_dilation`
- `connected_components_impl = scipy.ndimage.label`
- `connected_components_connectivity = 8`
- `postprocess_version = none_in_v1`
- `tta_version = none_in_v1`
- 正式种子集合：3407 / 1234 / 2025
- 结果来源标记：reproduced / † / *

## 6. 分层止损与回退细则

### 6.1 全项目统一触发条件

下面任意一条成立，就必须触发止损判断：

1. 收益接近波动范围。
2. 跨 benchmark 方向不稳。
3. 结果来源不清。
4. 公平性边界说不清。
5. 协议或评估口径发生分叉。

对应固定公式：

```text
need_stop_loss = near_variance
                 OR cross_benchmark_unstable
                 OR source_unclear
                 OR fairness_unclear
                 OR protocol_diverged
```

### 6.2 固定处理动作

一旦触发止损，不允许继续口头拖延，只允许落到下面四种动作之一：

- `retain`：保留在正文主线或正式主表。
- `downgrade`：降级到补充层、discussion、局限性说明或次级表格。
- `remove`：移出正文主线、正式主表和主要贡献叙述。
- `rollback`：回退到最后一个协议清楚、来源清楚且公平性成立的冻结版本。

### 6.3 明确禁止的处理方式

下面这些做法全部禁止：

- 因为已经投入大量训练成本，就继续保留不稳模块。
- 因为外部方法已经接入，就硬塞进主表。
- 因为 `CRAG` 已经跑完，就硬写强泛化。
- 因为写稿即将开始，就回避来源边界和公平性问题。
- 因为只剩少数对象未定，就先把项目总状态写成 `Pass`。

### 6.4 模块层止损规则

#### `LKMA`

下面任意一条成立，`LKMA` 必须进入止损判断：

- `F1` 和 `Object Dice` 的收益接近三次重复波动范围。
- 对象级结果没有形成稳定正向价值。
- 成本明显增加但收益弱且职责解释不成立。

固定处理规则如下：

- 证据稳定且职责清楚：`retain`
- 收益边缘但仍有局部价值：`downgrade`
- 主指标和对象级证据都不支持：`remove`

#### `Boundary Head`

下面任意一条成立，`Boundary Head` 必须进入止损判断：

- Boundary F1 / HD95 / Object Hausdorff 没有形成明确改善。
- 黏连分离或边界局部图无法支持其职责解释。
- 主指标下降且没有其他证据补回。

固定处理规则如下：

- 边界证据或对象级证据至少一类稳定成立：`retain`
- 仅局部案例受益但主线证据不足：`downgrade`
- 指标和图像都不支持：`rollback` 到不带 `Boundary Head` 的上一稳定版本

#### `Distance-aware Loss`

下面任意一条成立，`Distance-aware Loss` 必须优先止损：

- 收益只在单个 seed 上出现。
- 三次均值接近波动范围。
- 训练复杂度上升但结论不稳。
- 在 `CRAG` 上相对上一版没有形成一致方向。

固定处理规则如下：

- 只有在多 seed 和跨数据集方向都稳定时才允许 `retain`
- 只在 `GlaS` 局部成立时一律 `downgrade`
- 证据不稳或方向反转时直接 `remove`

### 6.5 外部对比层止损规则

下面任意一条成立，外部对比层必须止损：

- 第一批外部对比已经足够支撑主表。
- 新方法复现实验代价高且不能显著增强结论。
- 新方法与当前主张关系不大。
- 协议差异过大，无法做公平横比。

外部对比的处理动作只允许如下：

- 首批代表性方法且公平协议成立：`retain`
- 协议不同但仍有参考价值：`downgrade` 为 `†`、discussion 或补充表
- 成本失控且说服力有限：`remove`
- 来源链或评估链已分叉：`rollback`

下面这些内容禁止进入正式主表：

- 来源标记说不清的值。
- split、指标或测试流程明显不同的值。
- 只做了单次筛查且未满足进入条件的方法。
- 需要特殊后处理却没有明确标注的方法。

### 6.6 `CRAG` 层止损规则

`CRAG` 在当前项目中的固定角色是第二 benchmark 的补充验证层，而不是主证据来源。

下面任意一条成立，`CRAG` 结论必须降级或回退：

- 最终模型相对 baseline 的优势完全消失。
- 关键指标方向反转。
- 多 seed 或多指标结果不稳定。
- 无法给出可信解释。
- 结果来源或阈值来源说不清。

`CRAG` 的处理动作固定为：

- 与 `GlaS` 方向一致且能支撑补充验证：保留 A / B / C 中对应等级
- 只形成有限支撑：保留为补充验证并降低文字强度
- 方向反转或证据失真：`downgrade` 到局限性说明
- 协议分叉或来源失效：`rollback`

下面这些表述在止损触发后全部禁止：

- `strong generalization`
- `superior across benchmarks`
- `robust cross-dataset advantage`

### 6.7 写作层止损规则

下面任意一条成立，写作表述必须降调：

- 只有 `GlaS` 明显成立。
- `CRAG` 只形成有限支撑。
- 外部对比不全面。
- 任务内强对照没有被全面超过。
- 某个模块已经被 downgrade / remove / rollback。

这类情形下只允许保留下面一类表述：

- 主线在 `GlaS` 上成立。
- `CRAG` 提供补充或有限支持。
- 相对任务内强对照仍有差距或处于同一量级。
- 某些设计仅作为局部增强或备选分析，而非核心贡献。

下面这些做法全部禁止：

- 用单个最好 seed 暗示稳定提升。
- 用补充验证包装成主证据。
- 用不完整外部对比暗示全面领先。
- 用未保留模块写主要贡献。

### 6.8 `3-seed` 审稿应对与追加 reporting 细则

当前主协议关于 seed 的正式结论固定为：

- `main_seed_set = {3407, 1234, 2025}`
- `seed_reporting_mode = fixed_3_seed_main`
- 主表、阶段放行和总验收主判断优先基于这组冻结 `3-seed`

下面任意一条成立，才允许启动 `additional_reporting_seed_plan`：

1. 审稿人或编辑明确要求更多重复、方差解释或更稳的统计支撑。
2. 总验收发现差异长期落在 `near_variance` 附近，且写作层确实需要补充稳定性说明。
3. 外部强对照与当前主线差距很小，英文起步阶段需要补 rebuttal 级稳定性证据。

启动后必须同时满足下面五条：

1. 追加 reporting 只能在主协议结论已经冻结之后启动，不能倒逼修改已有 `3-seed` 主表。
2. 额外 seed 集合必须在新增运行前一次性冻结，并记录为 `additional_seed_plan`，不能边跑边改。
3. 追加范围必须统一施加到当前主线方法和对应最强直接比较对象，不能只补我方方法。
4. 主表继续保留 `3-seed` 正式结果，新增结果只能进入 supplementary、rebuttal note 或附加 reporting 小节。
5. 所有追加统计都必须显式保留 `main_seed_set`、`additional_seed_set`、`combined_seed_count` 和 `seed_escalation_reason`。

当前明确禁止的做法包括：

- 先看 `3-seed` 结果，再挑“更有利”的额外 seed。
- 只给最终方法补更多 seed，却不对关键直接比较对象同步补充。
- 把 `3-seed` 主表和 `3+n seed` 补充表混成同一组 `mean +- std`。
- 用追加 reporting 掩盖原始 `3-seed` 结论其实不稳这一事实。

当前建议的补充输出固定为：

- reports/stage_reports/seed_escalation_note.md
- reports/tables/additional_seed_summary.csv
- reports/tables/main_vs_additional_seed_compare.csv

其中最小记录字段至少要包含：

- `main_seed_set`
- `additional_seed_set`
- `combined_seed_count`
- `seed_escalation_reason`
- `additional_reporting_scope`
- `methods_covered`
- `same_protocol_confirmed`
- `same_eval_proto_confirmed`
- `same_threshold_policy_confirmed`
- reporting_destination = supplementary / rebuttal / appendix

如果上述任一字段缺失，追加 seed 结果不得进入正式写作资产。

### 6.9 工程与协议层回退规则

下面任意一条成立，必须 `rollback`：

- 协议中途被改动且未留冻结说明。
- 某批结果来源无法追溯。
- 评估口径出现分叉。
- 后处理或测试流程只在个别方法上启用。
- 表格与 run 级记录对不上。

触发回退后，处理顺序固定为：

1. 回退到最后一个协议清楚的冻结版本。
2. 重新标记后续结果的有效性。
3. 将不再公平的结果移出正式主表与主要叙述。
4. 只保留口径一致的结果进入总验收。
