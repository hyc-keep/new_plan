# 01–04 正式复现实验执行手册与状态记录

## 1. 本文件的目的

本文件规定如何对项目**已经完成的 01–04 阶段**做一次完整、独立、可比较的正式复现实验，并在每个阶段完成后记录真实产物与对比结论。

本文件是执行手册与状态索引，不是指标真源。所有实验数值只能来自本次重新运行脚本生成的 CSV、JSON、YAML、checkpoint、预测文件、独立复核产物和 Gate 报告；不得手工填写或修改任何结果数字。

## 2. 什么是本项目的“正式复现”

本项目的正式复现不是重新设计模型、修改指标，也不是只检查数据 hash 或跑一条新的预检配置。

正式复现的含义是：

```text
使用已经完成并曾通过 01–04 的原始代码、原始数据处理流程、原始正式配置、原始训练/测试/汇总入口
→ 在一个全新的独立复现批次中从 01 开始完整重新执行
→ 重新生成新的数据阶段产物、训练产物、测试结果、三 seed 聚合结果和阶段 Gate
→ 将“新复现结果”分别与“历史原实验结果”和“原计划验收要求”比较
→ 据真实比较结果判断：一致 / 可接受差异 / 未复现成功。
```

因此，本轮复现必须回答三个问题：

1. **流程是否可重新执行？** 原代码能否从原始输入完整运行至阶段产物。
2. **结果是否可重复？** 新结果是否与历史原实验结果一致，或在预先定义的可接受差异范围内。
3. **指标是否仍达标？** 新结果是否满足对应阶段原计划中已经锁定的验收指标与统计要求。

只有三个问题都由真实产物回答，才能称为本轮 01–04 复现成功。

## 3. 不属于正式复现的事项

以下事项可以作为工程诊断或准备工作，但**不属于本轮 01–04 正式复现实验结果**：

- 仅重新计算数据 hash、仅检查文件存在性；
- 只运行 runtime probe、只运行 smoke；
- 使用新写的 `*_chain.yaml`、新训练逻辑或新指标口径而没有证明其与历史正式流程等价；
- 使用历史 checkpoint、历史 CSV、历史 aggregate 直接生成“新结果”；
- 修改历史 metrics、checkpoint、`run_meta`、summary 或 Gate 数字；
- 只跑 02/03/04 而未从 01 的原始数据准备与划分流程重新开始。

此前在 `full_chain_reproduction_20260724` 下进行的 `*_chain.yaml` runtime 尝试，仅是新增工程框架的预检/失败证据；它不代表历史 01–04 的正式复现结果，也不得作为本手册中任何阶段的通过或失败结论。

## 4. 正式复现的总原则

### 4.1 使用对象

- 使用历史 01–04 已通过时对应的**原代码、原配置、原数据处理和原汇总入口**；
- 先通过 Git 提交、历史配置路径、历史报告中的命令和实际脚本逐项锁定这些对象；
- 新复现只能改变输出批次路径与本轮身份标记，不得改变模型、数据划分规则、seed、训练参数、测试口径、阈值或统计方法；
- 若原代码因环境变化无法直接运行，只允许做最小兼容修复；修复必须记录，并在结果比较时标记为“兼容修复复现”，不能冒充完全二进制一致复现。

### 4.2 新旧隔离

- 历史 `experiments/**`、历史结果表、历史 checkpoint 和历史 Gate 一律只读保留；
- 本轮每个阶段都写入新的复现批次目录，例如：

```text
experiments/reproduction_<batch_id>/01_data/
experiments/reproduction_<batch_id>/02_A1/<run_name>/
experiments/reproduction_<batch_id>/03_A2/<run_name>/
experiments/reproduction_<batch_id>/04_B1/<run_name>/
```

- 复现失败的目录、日志和报告必须保留；不得删除、覆盖或用第二次运行替换第一次失败证据；
- 如需重试，创建新的 `<batch_id>`，并在本文件记录“第几次复现、为何重试、与上一批次的关系”。

### 4.3 结果判断

每个阶段都必须同时给出三类比较：

| 比较对象 | 问题 | 结论来源 |
| --- | --- | --- |
| 新复现 vs 历史原实验 | 新旧结果是否一致或差异可解释 | 新旧真实 CSV / manifest / checkpoint / summary |
| 新复现 vs 原计划要求 | 本轮是否达到该阶段计划阈值、样本数、seed 数和统计要求 | 新结果与计划验收条款 |
| 新复现内部一致性 | 数据、config、run、checkpoint、测试、汇总和 Gate 是否回链一致 | 本轮 run_meta、manifest、独立复核与 Gate |

结论只允许使用以下状态：

- `pass`：流程完整，结果满足计划要求，且与历史结果一致或处于预先定义的可接受差异范围；
- `partial`：流程完成但有未解决差异、缺少某项比较或部分指标未达标；
- `fail`：运行失败、证据不完整、数据/配置/结果身份不闭合，或指标不满足计划要求；
- `blocked`：前一阶段未完成、正式对象尚未锁定或环境阻断；
- `pending_not_run`：尚未开始运行。

“新结果与历史数字不完全相同”不自动等于失败；但必须先按锁定的容差和统计规则比较，再决定是否可接受。没有比较依据时只能写 `partial`，不能自行宣称成功。
