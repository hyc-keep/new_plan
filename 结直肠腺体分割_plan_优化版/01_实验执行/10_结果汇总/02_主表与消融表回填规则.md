# 主表与消融表回填规则

## 1. 文件角色与执行边界

### 1.1 当前文件负责什么

这份文件不是“表头示意页”，而是 `10_结果汇总` 阶段里专门负责 `tables_ready` 子门的协议文件。

它当前只负责把下面七件事写死：

1. 哪些 run 级正式记录可以进入表格层，哪些只能停在补充层、参考层或失败案例层。
2. `GlaS` 主结果表、主线逐步消融表、首批外部对比表、`CRAG` 主结果表、`GlaS + CRAG` 联合总结表五类正式表的唯一身份。
3. 各类表格允许进入的行、固定列、固定排序、来源边界和表注规则。
4. reproduced / † / *、single_seed / mean +- std、TestA / TestB、A / B / C、retain / backup / delete 在表格层里的唯一合法用法。
5. 表格层与 run 级记录、图像层、`stage summary`、结果包检查表和下游总验收之间的字段映射关系。
6. 当前文件自己的独立 `回退条件`、`代码落地接口`、`冲突裁决记录`、`文件质量自检` 和独立 `Diagnostics 闭环`。
7. 当前文件作为 `结直肠腺体分割_plan_优化版/01_实验执行/10_结果汇总/00_阶段总协议.md` 中 `tables_ready` 子门展开协议的正式职责。

### 1.2 当前文件不负责什么

当前文件不重新定义下面这些内容：

- 哪些结果先算正式 run；这些由 `结直肠腺体分割_plan_优化版/01_实验执行/10_结果汇总/01_结果记录模板.md` 冻结。
- 哪些图片该导出、失败案例如何编号、正文候选图如何分组；这些由 `结直肠腺体分割_plan_优化版/01_实验执行/10_结果汇总/03_可视化与失败案例清单.md` 冻结。
- 各阶段为什么保留、降级或删除某个模块；这些由 `02-09` 阶段 handoff 和 `stage summary` 给出。
- 投稿结果包的最终状态分级；这些由 `结直肠腺体分割_plan_优化版/01_实验执行/10_结果汇总/05_投稿结果包检查表.md` 和 `11_总验收与止损` 下游文件负责裁决。

### 1.3 为什么当前文件必须独立存在

结果汇总阶段最容易出现的伪完成状态不是“没有表”，而是：

- 表已经做了，但行为什么能进、为什么不能进没有统一规则。
- 数字已经排上去了，却说不清它到底是 `single_seed` 还是 `mean +- std`。
- 引文值、复现值和特殊测试设置结果混在一张表里，只靠肉眼猜。
- 已删除模块仍然留在主线消融表里，让正文贡献叙事和阶段结论打架。
- `CRAG` 的补充验证被写成与 `GlaS` 同强度主结论。

如果表格层不独立冻结，后面 03_可视化与失败案例清单.md、04_stage_summary模板.md、05_投稿结果包检查表.md 和 11_总验收与止损/* 都会再次面对“结果很多，但主表资产边界不清”的灰区。

## 3. 本阶段唯一允许处理的变量

### 3.1 表格层的统一定义

从现在开始，表格层固定理解为：

```text
formal_tables
= glas_main_table
+ ablation_table
+ external_main_table
+ crag_main_table
+ glas_crag_joint_table
```

并且：

```text
tables_ready
= glas_main_ready
AND ablation_ready
AND external_ready
AND crag_ready
AND joint_ready
AND table_notes_ready
```

### 3.2 当前文件唯一合法输入资产

当前文件只允许消费下面这些正式资产：

- `official_result_record`
- `stage_pass`
- retain / backup / delete
- conclusion_grade = A / B / C
- result_source_type = reproduced / † / *
- aggregation = single_seed / mean +- std
- `boundary_metric_width = 3 px`
- `boundary_metric_impl = skimage.segmentation.find_boundaries(mode=inner) + binary_dilation`
- `connected_components_impl = scipy.ndimage.label`
- `connected_components_connectivity = 8`
- `metric_crosscheck_result`
- `seed_reporting_mode`
- `direct_comparison_eligible`

当前文件明确不允许：

- 在表格层重新定义 run 级字段。
- 在表格层重新计算没有来源说明的新数字。
- 先决定要写哪一行，再倒推修改阶段结论或来源标记。
- 用表格写法掩盖 GlaS / CRAG 角色差异或 TestA / TestB split 差异。

### 3.3 当前文件在 `Gate_J1` 中的角色

`结直肠腺体分割_plan_优化版/01_实验执行/10_结果汇总/00_阶段总协议.md` 把 `tables_ready` 写成正式子门。

当前文件就是这个子门的展开协议：

```text
tables_ready
= run_to_table_mapping_clear
AND row_eligibility_clear
AND column_schema_clear
AND source_boundary_clear
AND benchmark_role_clear
AND table_note_clear
```

如果 `tables_ready = false`，本阶段默认不得进入结果包就绪状态，也不得向 `11_总验收与止损` 交接主表层资产。

## 4. 阶段门控表达式

后面统一只承认下面九条总原则：

1. 先有正式 run 记录，再允许进表。
2. 先区分 GlaS / CRAG，再讨论联合总结表。
3. 先区分 TestA / TestB / test40，再讨论“主结果”。
4. 先区分 reproduced / † / *，再讨论公平比较。
5. 先确认 `metric_crosscheck_result != fail`，再允许把正式指标写进表。
6. 先区分 single_seed / mean +- std 与 fixed_3_seed_main / additional_reporting，再讨论稳定结论。
7. 先写保留 / 降级 / 删除结论，再写主线消融行。
8. 先保留失败与局限边界，再写正文主表。
9. 先把表注规则写死，再允许生成论文表格。

如果一张表只能告诉读者“哪个数字最好”，却不能说明来源、稳定性、benchmark 身份和模块结论，这张表默认不具备进入正式结果汇总层的资格。

## 5. 核心规则卡片

### 5.1 `GlaS` 主结果表规则

- 当前结论：`GlaS` 主结果表固定服务正文主结果层，必须分开报告 `TestA60` 与 `TestB20` 的对象级主指标
- 规则类型：`官方协议固定项 + 论文写作规则`
- 适用阶段：`10_结果汇总`
- 直接依据：`结直肠腺体分割_plan_优化版/01_实验执行/10_结果汇总/01_结果记录模板.md`、`结直肠腺体分割_plan_优化版/01_实验执行/00_总览与规范/04_评估口径与官方脚本对齐.md`
- 核心公式或定义参考：`glas_main_ready = rows_eligible AND tests_split_clear AND notes_clear`
- 采用原因：TestA / TestB 难度不同，主表若混成单值，会掩盖 malignant 子集上的真实差异
- 不采用的相邻方案：不采用 A/B 平均值替代 TestA / TestB；不采用只报 `Dice`
- 代码落点：reports/tables/glas_main_metrics.csv
- 运行记录字段：`method_name`, `split_role`, `result_source_type`, `aggregation`, `TestA_F1`, `TestA_ObjDice`, `TestA_ObjHaus`, `TestB_F1`, `TestB_ObjDice`, `TestB_ObjHaus`
- 验收方式：检查 `GlaS` 主表同时存在 TestA60 / TestB20 两组对象级三指标，且表注完整

### 5.2 `GlaS` 主结果表允许进入的行

- 当前结论：`GlaS` 主结果表只允许进入 `UNet`、`ResNet34-U-Net`、`current_mainline`、任务内 direct comparison 关键方法和首批外部对比中已形成正式结果的 `1-3` 个方法；若某行的 `metric_crosscheck_result = fail`，则该行不得进入正式主表
- 规则类型：`表格资格规则`
- 适用阶段：`10_结果汇总`
- 直接依据：`结直肠腺体分割_plan_优化版/02_路线与投稿/03_结直肠腺体分割_投稿层级自查与止损判断.md`、`结直肠腺体分割_plan_优化版/03_文献证据/06_我们项目的GlaS_CRAG对照主结果表模板.md`
- 核心公式或定义参考：`glas_main_row = stage_passed AND source_clear AND protocol_clear AND task_role_clear AND metric_crosscheck_pass`
- 采用原因：正文主表只能承载最核心的内部主线、代表性外部对比和任务内强锚点
- 不采用的相邻方案：不把所有跑过的方法全塞进主表；不把只做筛查的方法提前升级到正式主表
- 代码落点：reports/tables/glas_main_metrics.csv
- 运行记录字段：`can_enter_main_table`, `stage_pass`, `result_source_type`, `direct_comparison_eligible`, `supports_mainline`, `metric_crosscheck_result`
- 验收方式：检查主表中的每一行都能解释“为什么它值得占一行”；检查入表行不存在 `metric_crosscheck_result = fail`

### 5.3 `GlaS` 主结果表禁止进入的内容

- 当前结论：单次筛查值、指标口径不一致值、A/B 平均值、来源不清值和特殊设置未标记值禁止进入 `GlaS` 主结果表
- 规则类型：`来源边界规则`
- 适用阶段：`10_结果汇总`
- 直接依据：`结直肠腺体分割_plan_优化版/01_实验执行/08_外部对比/06_引用值与复现值标记规则.md`、`结直肠腺体分割_plan_优化版/03_文献证据/05_GlaS_CRAG_主结果数值速查表.md`
- 核心公式或定义参考：`glas_main_forbidden = single_screening OR metric_mismatch OR split_mismatch OR source_unclear OR special_setting_hidden`
- 采用原因：这些值即使看起来好看，也不构成公平主表证据
- 不采用的相邻方案：不以“后面表注再补”替代正式资格判断
- 代码落点：scripts/validate_sources.py, reports/tables/glas_main_metrics.csv
- 运行记录字段：`result_source_type`, `aggregation`, `has_special_test_setting`, `direct_comparison_eligible`
- 验收方式：检查不合格结果都已降级到参考区、补充区或 discussion

### 5.4 主线逐步消融表规则

- 当前结论：主线逐步消融表只允许按 `UNet -> ResNet34-U-Net -> + LKMA -> + Boundary Head -> + Distance-aware Loss` 的固定顺序回填
- 规则类型：`路线冻结规则`
- 适用阶段：`10_结果汇总`
