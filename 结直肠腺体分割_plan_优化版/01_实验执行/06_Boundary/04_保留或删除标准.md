# 保留或删除标准

本文件不是 `06_Boundary` 阶段里“看起来有点提升就先留着”的讨论稿，也不是跑完之后再补一段主观去留意见的说明文。

它在当前项目中的唯一职责是：

> 作为 `06_Boundary` 的正式阶段裁决 protocol，把 `结直肠腺体分割_plan_优化版\01_实验执行\06_Boundary\03_实验步骤.md` 已冻结的执行链、`结直肠腺体分割_plan_优化版\01_实验执行\06_Boundary\02_边界标签生成规则.md` 已冻结的边界目标协议、`结直肠腺体分割_plan_优化版\01_实验执行\06_Boundary\00_阶段总协议.md` 已冻结的 `Gate_D1` 与 handoff 语义，压成唯一合法的 `keep / backup / drop` 判定规则；从现在开始，`Boundary Head` 是否进入主线、是否只能保留为备选、以及何时必须回退到 `boundary_input_base`，都只能由这里写明的证据顺序、决策资产、止损边界、回退条件和交接字段决定，不能再靠单次最好值、局部好看样例或“理论上应该有用”的主观偏好改判。

从现在开始，只要讨论 `Boundary` 阶段“什么时候才允许写正式结论、哪些证据优先级最高、边界收益和对象级退化如何裁决、什么情况下必须回退、`07_Distance` 到底从哪里起步”，都必须以本文件为准；若与同层文件冲突，以 `结直肠腺体分割_plan_优化版/01_实验执行/06_Boundary/00_阶段总协议.md` 管阶段身份与 `Gate_D1`，以 `结直肠腺体分割_plan_优化版/01_实验执行/06_Boundary/01_设计依据.md` 管为什么边界监督值得单独验证，以 `结直肠腺体分割_plan_优化版/01_实验执行/06_Boundary/02_边界标签生成规则.md` 管边界目标协议，以 `结直肠腺体分割_plan_优化版/01_实验执行/06_Boundary/03_实验步骤.md` 管执行顺序，而本文件负责把这些上游冻结内容压成最终可比较、可止损、可交接给 `07_Distance` 的正式判定逻辑。

## 1. 文件角色与执行边界

### 1.1 当前文件负责什么

当前文件只负责冻结下面八件事：

1. `Boundary` 阶段的正式结论最早在什么资产条件下才允许书写。
2. `Boundary` 的正式证据必须按什么顺序读取，避免被单个数字、单张图或单个 split 带偏。
3. 什么情况下可以把 `Boundary Head` 判为 `keep` 并作为后续主线一部分继续保留。
4. 什么情况下只能降级为 `backup`，而不能冒充正式主线保留。
5. 什么情况下必须直接 `drop`，并停止继续为 `Boundary` 扩宽搜索边界。
6. 边界级收益、对象级退化、可视化和成本冲突时，应按什么固定顺序裁决。
7. 阶段结论必须输出哪些正式字段，才能被 `Gate_D1` 和 `07_Distance` 稳定消费。
8. 一旦结论不成立，应该按什么独立回退条件回到 `boundary_input_base`，而不是让下游阶段替当前文件兜底。

### 1.2 当前文件不负责什么

当前文件不重新定义下面这些内容：

- `Boundary` 为什么值得做，以及为什么它必须排在 `LKMA` 之后。
- 边界标签唯一来源、轮廓定义、宽度规则和三链同口径实现细则。
- `screening_main -> three_seed_main -> optional_one_backup -> stage_decision` 的执行顺序。
- 数据身份、split、输入尺寸、主 gland 标签定义、训练协议和评估协议本体。
- `Distance-aware Loss`、外部对比与 `CRAG` 阶段的结构、损失和后续职责。

这些职责分别以下列文件为准：

- `结直肠腺体分割_plan_优化版/01_实验执行/06_Boundary/00_阶段总协议.md`
- `结直肠腺体分割_plan_优化版/01_实验执行/06_Boundary/01_设计依据.md`
- `结直肠腺体分割_plan_优化版/01_实验执行/06_Boundary/02_边界标签生成规则.md`
- `结直肠腺体分割_plan_优化版/01_实验执行/06_Boundary/03_实验步骤.md`
- `结直肠腺体分割_plan_优化版/01_实验执行/01_数据协议/00_阶段总协议.md`
- `结直肠腺体分割_plan_优化版/01_实验执行/04_Baseline/02_训练协议.md`

### 1.3 为什么当前文件必须独立存在

因为 `Boundary` 阶段最容易出现的伪完成状态不是“完全没写结论”，而是：

- 口头上说只看正式三 seed，实际却拿 screening run 或单个 seed 的最好结果强行保留。
- 口头上说优先看边界收益，实际却只看对象级某个更顺眼的数字。
- 口头上说主版本不过线就止损，实际却不断补更多宽度、权重或相邻结构，偷换成持续搜索。
- 口头上说已经能交接到 `07_Distance`，实际却没有把 `decision_level / decision_reason / current_base_next` 写成正式字段。

如果没有一份独立判定文件，`结直肠腺体分割_plan_优化版/01_实验执行/06_Boundary/03_实验步骤.md` 会被迫替本文件承担“到底算不算保留”的职责，`07_Distance` 会被迫自行决定该从 `boundary_kept_base` 还是 `boundary_input_base` 起步，而 `Gate_D1` 也就无法真正形成可审计的阶段闭环。

## 3. 本阶段唯一允许处理的变量

### 3.1 当前文件唯一判定对象

- 当前结论：本文件唯一判定对象是“在 `current_base` 协议完全不动的前提下，已经按固定执行链完成的 `Boundary main_variant = decoder_final + w3 + BCE + lambda0.3` 是否值得进入后续主线，以及唯一合法备选版本是否只能作为辅助解释而不能取代主版本结论”。
- 规则类型：`工程冻结规则`
- 适用阶段：`06_Boundary`
- 直接依据：`结直肠腺体分割_plan_优化版/01_实验执行/06_Boundary/00_阶段总协议.md`、`结直肠腺体分割_plan_优化版/01_实验执行/06_Boundary/02_边界标签生成规则.md`、`结直肠腺体分割_plan_优化版/01_实验执行/06_Boundary/03_实验步骤.md`
- 核心公式或定义参考：`decision = f(boundary_metric_trend, object_metric_trend, seed_consistency, split_consistency, visual_support, cost_change)`
- 采用原因：只有把判定对象收缩为“已完成正式主链的 Boundary 主版本”，后续保留/删除结论才具有单变量与可追溯性。
- 不采用的相邻方案：不把本文件写成执行步骤文件；不把本文件写成标签协议文件；不把任何未授权扩表变体纳入正式主结论。
- 代码落点：[project_root]/scripts/compare_runs.py、[project_root]/scripts/summarize_stage.py
- 运行记录字段：`decision_level`, `decision_reason`, `seed_consistency`, `split_consistency`, `visual_support_level`, `cost_assessment`
- 验收方式：检查正式结论只围绕主版本三 seed 的正式资产形成；备选版本若存在，也只能作为辅助裁决输入。

### 3.2 当前文件直接继承哪些冻结项

本文件直接继承并不得擅自改动的上位规则如下：

- 主任务：`2D gland segmentation`
- 主数据集：`GlaS`
- 正式 split：`train68 / val17 / testA60 / testB20`
- `current_base` 唯一来源：`(lkma_decision_level == keep) ? baseline_plus_lkma : baseline`
- `Boundary` 主版本：`current_base + BoundaryHead(decoder_final, w=3, loss=bce, lambda=0.3)`
- 额外备选集合：`{w5, lambda02, lambda05}` 且最多一个
- 固定三 seed：`3407 / 1234 / 2025`
- `best_selector = val_objdice_max`
- `threshold_source = val17`
- `aug_profile_name = light_aug_v1`
- `eval_aug_enable = false`
- `eval_cast_policy = logits/probabilities must be kept or cast to float32 before thresholding`
- `boundary_metric_width = 3 px`
- `boundary_metric_impl = skimage.segmentation.find_boundaries(mode=inner) + binary_dilation`
- `connected_components_impl = scipy.ndimage.label`
- `connected_components_connectivity = 8`
- `postprocess_version`：当前轮待由研究定标、阶段锁定、Pre-check/Gate 冻结；`none_in_v3` 仅作历史 lineage，不是当前值。
- 执行链：`screening_main -> three_seed_main -> optional_one_backup -> stage_decision`

- 采用原因：`backup` 的意义是承认模块可能有方向价值，但当前证据强度不足以进入正式主线。

### 4.7 直接删除的判定条件

- 当前结论：若边界级收益只出现在单个 seed、均值变化落在波动范围内、对象级主指标出现明确退化、可视化不支持、或成本明显上升但无对应收益，则必须判为 `drop`。
- 规则类型：`工程冻结规则`

- 直接依据：`结直肠腺体分割_plan_优化版/01_实验执行/06_Boundary/00_阶段总协议.md`、`结直肠腺体分割_plan_优化版/01_实验执行/06_Boundary/03_实验步骤.md`、03_DCAN.md、04_Metrics-Reloaded.md
- 核心公式或定义参考：`drop = no_stable_gain or material_regression or unsupported_visuals or unjustified_cost`

### 4.8 “边界涨、对象级掉”裁决规则

- 当前结论：若 `Boundary F1` 上升，但 `Object Dice / F1` 或 `Object Hausdorff` 出现明确退化，当前版本不得直接 `keep`，最多只允许降为 `backup` 或直接 `drop`。

- 不采用的相邻方案：不因边界指标变好就忽略对象级恶化；不在对象级掉分时继续堆更多边界搜索。

- 验收方式：检查任何 `keep` 结论都没有出现“边界涨但对象级明显掉”的冲突模式；检查边界涨幅来自同一 `3 px` 宽度和冻结实现。
