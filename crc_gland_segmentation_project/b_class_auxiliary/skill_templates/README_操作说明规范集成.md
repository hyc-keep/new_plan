## 一、为什么需要这个规范

### 1.1 问题背景

在多阶段实验项目中，经常遇到以下问题：
- ❌ 几个月后重做实验，不记得命令怎么传参
- ❌ 新成员加入，看不懂代码也不知道怎么跑
- ❌ 配置改了，操作文档没更新，按旧文档跑会报错
- ❌ 文档写得模糊（"根据情况调整参数"），实际跑不通

### 1.2 解决方案

强制要求每个阶段交付时包含**标准化操作说明**，确保：
- ✅ 不看源码也能从头跑通
- ✅ 所有参数都有明确说明（必填/默认值）
- ✅ 包含明确的成功标准和常见坑
- ✅ 配置变更时同步维护

## 三、标准六章节结构（快速参考）

每份操作说明必须包含以下六个章节：

| 章节 | 核心内容 | 必需元素 |
|------|---------|---------|
| **一、这个阶段在做什么** | 阶段定位、核心产物、与前后阶段关系 | 阶段标识符（A1/A2/B1）、产出目录 |
| **二、先读哪些文件** | 推荐阅读顺序（≥3个文件） | 从总览到细节的学习路径 |
| **三、配置引用关系/运行顺序** | 技术架构图或流程图 | 配置层级树或脚本执行顺序 |
| **四、完整执行流程与命令** | 逐步命令清单 + 参数说明 | 每个参数标注"必填"或"默认值" |
| **五、跑完怎么确认成功** | 文件检查 + 指标范围 + gate 状态 | 参考指标附带容差说明 |
| **六、常见坑** | 真实踩坑经验（≥2条） | 错误现象 + 原因 + 正确做法 |

## 四、快速开始（编写新阶段操作说明）

### 4.1 复制模板

> 复制模板只复制文档结构，不继承 A2 的 run_name、指标、checkpoint、raw/mean±std 数字、aggregate 字段或 gate 结论。04 未开始时只允许建立计划/阶段锁定文件，不得创建正式 B1 运行产物。

```bash
cd /home/featurize/work/Paper/crc_gland_segmentation_project

# 创建新阶段目录（如果还不存在）
mkdir -p reports/stage_reports/implementation_tracking/04_Baseline

# 复制最相近的模板（训练阶段用 02 或 03，工具链阶段用 01）
cp reports/stage_reports/implementation_tracking/02_UNet流程验证/操作说明_如何运行本阶段.md \
   reports/stage_reports/implementation_tracking/04_Baseline/操作说明_如何运行本阶段.md
```

### 4.2 修改核心内容

打开新复制的操作说明，按顺序修改：

1. **章节一**：改阶段定位、核心产物、前置阶段
2. **章节二**：改关键文档列表（如果有新的配置说明）
3. **章节三**：改配置文件名（如 `A2_*` → `B1_*`）
4. **章节四**：改命令和参数（最重要，逐条核对）
5. **章节五**：改成功标准和参考指标
6. **章节六**：改常见坑（基于真实踩坑经验）

### 4.3 验证清单

- [ ] 找一个非开发者按文档从头跑一遍
- [ ] 所有命令都能直接复制粘贴运行
- [ ] 所有参数都有"必填"或"默认值"说明
- [ ] 包含明确的成功标准（文件 + 指标 + gate）
- [ ] 包含至少 2 条真实踩坑经验
- [ ] 没有"请参考源码"等推诿表述
