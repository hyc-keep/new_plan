# 设计依据

本文件不是 `05_LKMA` 阶段的背景说明，也不是“先试几个大核版本，再回头补一段动机”的备注。

它在当前目录中的唯一职责是：

> 作为 `05_LKMA` 的正式设计依据 protocol，把 `为什么 baseline 之后先验证 LKMA / LKMA 到底承担什么职责 / 为什么大核-大感受野思路适合当前腺体任务 / 为什么这里只允许做最小单变量验证` 一次写死；后续 `结直肠腺体分割_plan_优化版\01_实验执行\05_LKMA\02_插入位置与参数范围.md`、`结直肠腺体分割_plan_优化版\01_实验执行\05_LKMA\03_实验步骤.md`、`结直肠腺体分割_plan_优化版\01_实验执行\05_LKMA\04_保留或删除标准.md` 只能在这里冻结的设计前提下展开，不能各自再发明一套动机、职责边界或文献解释。

从现在开始，只要讨论 `LKMA` 阶段“为什么先做它、它算什么模块、它应该证明什么、为什么不能扩成大搜索”，都必须以本文件为准；若与同层细则冲突，以 `结直肠腺体分割_plan_优化版/01_实验执行/05_LKMA/00_阶段总协议.md` 管阶段门与交接，以本文件管设计理由与设计边界。

## 1. 文件角色与执行边界

### 1.1 当前文件负责什么

当前文件只负责冻结下面六件事：

1. 为什么 `LKMA` 是 baseline 之后、`Boundary` 之前的第一模块。
2. `LKMA` 在当前项目里的严格职责定义，即它解决什么、不解决什么。
3. 为什么“大核 / 大感受野 / 形态建模”与腺体分割任务相关。
4. 哪些文献能支持“值得先试 `LKMA`”，哪些文献只能提供相邻思路边界，不能直接等同于“完整复现原模型”。
5. 为什么当前阶段必须压成 `baseline -> baseline + single LKMA module` 的最小单变量验证。
6. 后续 05_LKMA/02-04 应当继承什么样的设计前提、证据优先级和叙述边界。

### 1.2 当前文件不负责什么

当前文件不重新定义下面这些内容：

- `ResNet34-U-Net` 作为正式 baseline 的结构身份、encoder 初始化来源和 skip 边界。
- `train_proto_v1 / eval_proto_v1` 的完整训练与评估 protocol。
- `LKMA` 的正式阶段门、handoff 资产与 `keep / backup / drop` 最终交接语义。
- `LKMA` 的具体插入位置、kernel 最终主版本和备选版本执行顺序。
- `Boundary Head`、`Distance-aware Loss` 与外部对比阶段的设计。

这些职责分别以下列文件为准：

### 1.3 为什么当前文件必须独立存在

因为 `LKMA` 阶段最容易出现的伪完成状态不是“没做”，而是：

- baseline 已经稳定，但没有先把“为什么先做 `LKMA`”写成正式理由。
- 一边引用大核文献，一边把 `LKMA` 偷换成完整 `VAN / WTConv / UniRepLKNet` 风格模块。
- 一边说自己验证的是形态与上下文增强，一边又把边界监督或距离约束写进 `LKMA` 职责。
- 一边说是单变量验证，一边把位置、kernel、模块家族和训练协议一起打开。

如果没有一份独立设计依据，后面的参数边界、执行步骤和保留/删除判定就会各自重写一套动机，导致 `LKMA` 到底是什么、为什么值得验证、为什么不能扩搜索再次分裂。

## 3. 任务难点与设计动机

### 3.1 当前文件唯一设计对象

- 当前结论：本文件唯一设计对象是“为什么在已正式放行的 `ResNet34-U-Net` baseline 上，先验证一个首版可插拔的大核形态建模块 `LKMA`”
- 规则类型：`路线层已锁定 + 工程冻结规则`
- 适用阶段：`05_LKMA`
- 直接依据：`结直肠腺体分割_plan_优化版/02_路线与投稿/02_结直肠腺体分割_分阶段实验路线与执行标准.md`、`结直肠腺体分割_plan_优化版/01_实验执行/05_LKMA/00_阶段总协议.md`
- 核心公式或定义参考：`design_question = should_validate( baseline + single LKMA module ) before boundary supervision`
- 采用原因：后续 `Boundary / Distance` 的起点必须建立在“第一模块为什么是 `LKMA`”已经被解释清楚的前提上
- 不采用的相邻方案：不把本文件写成参数边界表；不把本文件写成训练步骤表；不把本文件写成大核方法综述合集
- 代码落点：[project_root]/src/models/modules/lkma.py、[project_root]/configs/model/lkma_v1.yaml
- 运行记录字段：`lkma_version`、`lkma_role`、`lkma_impl_type`
- 验收方式：检查本文件所有结论都能回收到“为什么先做 `LKMA`”，而不是“如何跑某个具体变体”

### 3.2 当前文件直接继承哪些冻结项

本文件直接继承并不得擅自改动的上位规则如下：

- baseline 正式基座：`ResNet34-U-Net`
- baseline 唯一合法起点：B1 current_standard 的正式真实资产，消费边界为 `frozen_baseline_with_warning`；`original_gate_b1=false` 仅作历史审计事实
- `train_proto_version = train_proto_v1`
- `eval_proto_version` 由 04 阶段当前轮 Gate 冻结
- `eval_proto_lineage = eval_proto_v1 -> eval_proto_v3`（v1 仅为历史祖先）
- `eval_cast_policy = logits/probabilities must be kept or cast to float32 before thresholding`
- `boundary_metric_width = 3 px`
- `boundary_metric_impl = project_custom_erosion_xor_3x3_ones_8conn_border0_tol3px`
- `connected_components_impl = scipy.ndimage.label`
- `connected_components_connectivity = 8`
- `effective_batch_size = physical_batch_size * grad_accum_steps`
- `best_selector = val_objdice_max`
- `threshold_source = val17`
- 对象级主指标优先：`F1 / Object Dice / Object Hausdorff`
- 固定三 seed：`3407 / 1234 / 2025`

### 3.3 当前文件禁止重新打开哪些问题

本文件禁止借设计依据之名重新打开：

- 数据 split、输入尺寸、增强包、主 loss、optimizer、scheduler。
- baseline 结构身份、encoder 预训练策略与 `effective_batch_size`。
- `threshold` 前 `float32`、`Boundary F1 = 3 px`、`find_boundaries(mode=inner) + binary_dilation` 与 `scipy.ndimage.label` 的评估实现口径。
- `Boundary Head` 和 `Distance-aware Loss` 的模块设计与保留逻辑。
- `LKMA` 的最终插入位置、kernel 主版本和备选版本执行顺序。

换句话说，本文件只能冻结设计理由与设计边界，不能绕过阶段母协议重开实现细节。

## 4. 设计边界(做什么/不做什么)

### 4.1 为什么 baseline 之后先做 `LKMA`

- 当前结论：当前主线在 baseline 之后先验证 `LKMA`，因为腺体任务的第一层不足首先是整体形态表达和长程上下文不足，而不是一开始就缺显式边界监督
- 规则类型：`路线层已锁定 + 论文支持的候选范围`
- 适用阶段：`LKMA 设计层`
- 直接依据：`结直肠腺体分割_plan_优化版/02_路线与投稿/02_结直肠腺体分割_分阶段实验路线与执行标准.md`、04_MILD-Net.md
- 核心公式或定义参考：`priority_order = morphology_context -> boundary_separation -> distance_refinement`
- 采用原因：如果整体形态表达还不稳，直接加边界监督会把“主干上下文不足”与“边界头效果有限”混在一起
- 不采用的相邻方案：不采用 baseline 后直接做 `Boundary Head`；不采用先开 `Distance-aware Loss`；不采用多个模块并联起步
- 代码落点：[project_root]/src/models/modules/lkma.py、[project_root]/configs/experiment/lkma_c1_v1.yaml
- 运行记录字段：`stage_code`、`model_code`、`lkma_role`、`train_proto_version`、`eval_proto_version`、`eval_cast_policy`、`boundary_metric_width`、`boundary_metric_impl`、`connected_components_impl`、`connected_components_connectivity`
- 验收方式：检查后续 05_LKMA/02-04 与 `06_Boundary/*` 都继承本文件的阶段顺序解释

### 4.2 `LKMA` 在当前项目中的职责定义

- 当前结论：`LKMA` 在当前项目中固定承担 `形态感知 + 大感受野 + 上下文聚合` 职责，不承担显式边界分离、距离约束或完整 backbone 替换职责
- 规则类型：`工程冻结规则`
- 适用阶段：`LKMA`
- 直接依据：`结直肠腺体分割_plan_优化版/01_实验执行/05_LKMA/00_阶段总协议.md`、04_MILD-Net.md
- 核心公式或定义参考：`module_role(LKMA) != boundary_supervision`；`module_role(LKMA) != distance_regularization`
