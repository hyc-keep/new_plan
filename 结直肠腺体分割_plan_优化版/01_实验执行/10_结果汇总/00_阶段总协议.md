# 结果汇总阶段总协议

本文件不是 `10_结果汇总` 目录旧稿的补充说明，也不是“等所有实验都跑完之后再临时拼表、挑图、补总结”的整理便签。

它在当前项目中的唯一职责是：

> 作为 `10_结果汇总` 阶段的正式母协议，把 Run-to-table traceability / Main-table admissibility / Ablation-table admissibility / Figure-and-failure evidence packaging / Stage-summary aggregation / Submission-package readiness 一次写死；后续脚本、表格、图像整理和写稿准备只允许忠实实现这份协议，不允许一边看结果一边重新决定哪些数字能进主表、哪些模块还能算主贡献、哪些图只是好看但不能算证据、哪些资产还没齐却先往 `11_总验收与止损` 推进。

从现在开始，`10_结果汇总` 子目录下的其他文件都视为本文件的细化与执行展开；若与本文件冲突，以本文件为准回改。

## 1. 文件角色与执行边界

### 1.1 当前文件负责什么

当前文件只负责冻结下面九件事：

1. `10_结果汇总` 作为 `09_CRAG验证` 之后、`11_总验收与止损` 之前的正式证据汇总阶段的唯一身份、阶段目标和放行边界。
2. run 级结果进入正式结果层的唯一准入条件，也就是“先有完整协议记录，再谈主表与写作使用”的唯一规则。
3. `GlaS` 主结果表、主线逐步消融表、首批外部对比层、`CRAG` 主结果表和 `GlaS + CRAG` 联合总结表的唯一固定结构、资格门和来源边界。
4. reproduced / † / *、single_seed / mean +- std、TestA / TestB、GlaS / CRAG、A / B / C、ready / ready_with_minor_gaps / not_ready 这些写作和验收边界在结果汇总层的唯一合法用法。
5. 图像资产、失败案例、局限性证据和正文候选图的唯一打包规则，确保可视化同时能支撑优点与局限，而不是只保留“漂亮图”。
6. `stage summary` 汇总层、handoff 资产和投稿结果包检查层的唯一收口方式，确保 `11_总验收与止损` 能直接消费，不再依赖口头解释。
7. 本阶段最低交付物、正式阶段门、独立 `回退条件`、固定回退顺序和重新放行条件。
8. 本阶段的代码落地接口、字段链、冲突裁决记录、`文件质量自检` 和独立 `Diagnostics 闭环`。
9. 本阶段与 `08_外部对比`、`09_CRAG验证`、`11_总验收与止损` 之间的正式交接语义，防止“结果很多但正式资产还没成型”。

### 1.3 为什么当前文件必须独立存在

`10_结果汇总` 最容易出现的伪完成状态不是“完全没整理”，而是下面这些情况：

- 数字很多，但主表资格、来源边界和阶段通过状态没有一起冻结。
- `GlaS`、外部对比和 `CRAG` 已分别有结果，却没有统一成能直接写进正文的主表与联合表层。
- 表格已经做出来，但说不清某个值到底是 `single_seed`、`mean +- std`、`reproduced` 还是 † / *。
- 图像导出很多，但失败案例、局限性证据、样本 ID 和正文候选层仍然不完整。
- 各阶段已经收口，但 `stage summary` 没有统一汇总，导致最后总验收还要倒回去重查每一步决策。
- 结果包“看起来差不多能写”，却还没被诚实地压成 ready / ready_with_minor_gaps / not_ready 中的一档。

如果没有一份独立阶段总协议，后续 `11_总验收与止损`、联合主结果表、图像资产整理和论文实验章节都会再次面对“结果做过了，但到底哪些是真正正式资产、哪些只是内部参考值”的灰区。

## 3. 本阶段唯一允许处理的变量

### 3.1 当前阶段真正要证明什么

`10_结果汇总` 阶段真正要证明的，不是“我们已经积累了很多结果文件”，而是：

> 在数据协议、训练协议、评估协议、来源标记、主线结论、外部对比结论和 `CRAG` 结论都已经冻结的前提下，所有正式 run、表格、图像、失败案例、`stage summary` 和结果包状态已经被压成一套可追溯、可写稿、可复查、可被 `11_总验收与止损` 直接消费的正式证据层。

因此，本阶段通过的含义同时包括：

- 每个正式数字都能追溯到具体 run、协议版本、阈值来源和来源标记。
- `GlaS` 主表、主线消融表、外部层、`CRAG` 主表和联合总结表都已经形成正式回填规则与正式资产。
- 图像证据层同时包含成功案例、失败案例、局部放大、原图与真值、样本 ID 和错误类型。
- 各阶段保留 / 降级 / 删除结论已经通过统一 `stage summary` 汇总层收口。
- 投稿结果包状态已经老实落到 ready / ready_with_minor_gaps / not_ready 三档之一。
- `11_总验收与止损` 不再需要重回各阶段翻日志，就能对项目总状态、止损和写稿进入条件做正式裁决。

### 3.2 当前阶段不允许用什么替代正式结论

当前明确不允许把下面这些情况写成“结果汇总已完成”：

- 只有表格截图或几个最好数字，没有完整 run 级记录。
- 表格已经做出来，但 reproduced / † / *、single_seed / mean +- std、TestA / TestB 或 GlaS / CRAG 仍然混写。
- 已删除模块仍然出现在正文主线消融表或主结果表中。
- 图像很多，但失败案例、文字说明和样本级追溯信息不完整。
- `stage summary` 仍散落在各阶段，没有统一汇总和交接。
- 结果包“看起来已经能写”，却还没有正式状态和缺口说明。

### 3.3 为什么当前阶段必须在这里收口

原因是：

- `08_外部对比` 已经形成首批外部比较层和来源边界，本阶段必须把它真正压进主表和表注体系。
- `09_CRAG验证` 已经形成第二 benchmark 的正式 handoff，本阶段必须把它合并到联合结果表和结果包资产中。
- 路线层与投稿层已经明确：能否进入中文保底或英文起步，关键不再是继续扩路线，而是结果资产是否真正在形式和证据上收完整。

## 5. 如何判断设计有效(验收标准)

### 5.1 正式结果记录规则

- 核心公式或定义参考：`formal_result = protocol_info + per_seed_metrics + source_boundary + failure_note`
- 采用原因：没有 run 级记录支撑的数字，不具备进入主表和联合表的资格
- 不采用的相邻方案：不采用只摘最好数字；不采用只保留指标截图不保留 run 元信息；不采用跳过失败说明直接写结论
- 代码落点：[project_root]/reports/official_run_registry.md、[project_root]/reports/tables/*.csv、notes/*.md

## 7. 回退条件

### 7.1 独立回退触发条件

只要出现下面任意一条，本阶段都必须先回退修正，而不是直接进入 `11_总验收与止损`：

- 主表数字无法追到来源 run，或 run 级记录字段缺失。
- reproduced / † / *、single_seed / mean +- std、GlaS / CRAG、TestA / TestB 发生混写。
- 表格层与 `stage summary` 的保留 / 降级 / 删除结论互相矛盾。
- 图像层只有成功案例，没有失败案例，或失败案例没有文字说明。
- 联合结果表中的 `CRAG` 分级结论与 `09_CRAG验证` 正式 handoff 不一致。
- 结果包看似完整，但 `package_status` 仍无法诚实归入三档之一。

### 7.2 固定回退顺序

回退时统一按下面顺序排查：

1. 先检查来源 run 与正式注册层映射是否完整。
2. 再检查来源标记、稳定性标记和 split 边界是否统一。
3. 再检查图像与失败案例是否足以支撑表格结论与局限性表述。
4. 再检查 `stage summary` 是否与正式表格和模块结论一致。
5. 最后再判定结果包状态是否真的达到 `ready` 或只能落到 ready_with_minor_gaps / not_ready。

### 7.3 回退后的重新放行条件

发生回退后，只有同时满足下面条件，才允许重新声明 `10_结果汇总` 阶段通过：

- 问题来源已经按固定回退顺序定位并修复。
- 修复动作、影响范围和复验结果已写入注册层、summary 或结果包检查文档。
- 主表、联合表、失败案例和 `stage summary` 的正式资产重新对齐。
- 当前阶段重新满足第 `5` 节全部规则卡片与第 `6` 节的最低交付物、handoff 资产和下游起点要求。
