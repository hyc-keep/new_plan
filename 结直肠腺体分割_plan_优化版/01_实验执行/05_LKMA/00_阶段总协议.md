# LKMA 阶段总协议

本文件不是 `05_LKMA` 目录旧稿的补充说明，也不是“先做几组大核实验再回头归纳”的阶段记录。

它在当前项目中的唯一职责是：

> 作为 `05_LKMA` 阶段的正式主协议，把 `LKMA as the first post-baseline module / Single-variable large-kernel morphology modeling / Minimal search boundary / Keep-backup-drop decision with cost awareness` 一次写死；代码、实验和阶段汇总只允许忠实实现与验证这份协议，不允许一边训练一边重新决定 `LKMA` 是什么、插在哪里、核多大、是否继续扩搜索、以及什么情况下才允许保留。

从现在开始，`05_LKMA` 子目录下的其他文件都视为本文件的细化与执行展开；若与本文件冲突，以本文件为准回改。

## 计划 lineage 与下游 handoff

统一字段：`source_stage`、`source_manifest`、`source_protocol_version`、`source_run_name`、`consumer_stage`、`consumer_file`、`consumption_boundary`。当前消费边界是 04 baseline 冻结身份到 LKMA 当前候选及其下游 Boundary handoff；字段缺失或计划/config/run_name/run_meta/manifest/gate/handoff 不同步时 Gate blocked，回到治理/阶段锁定。

## 回退条件

结构、训练、评估或 handoff 任一字段漂移，或当前轮版本未由研究定标、阶段锁定、Pre-check、Gate 冻结时，停止执行并回退到最近冻结协议。

## 代码落地接口

模块注入落点为 `src/models/modules/lkma.py` 与模型配置；阶段门消费 run_meta、manifest 和汇总表，未完成字段校验不得生成下游入口。

## 冲突裁决记录

以全局参数冻结表和当前阶段 Gate 为准；历史 lineage 仅用于解释，不覆盖当前配置。

## 文件质量自检

检查 lineage 七字段、当前消费边界、配置与 run_meta 一致性、模块代码落点、回退触发和下游资产清单，并逐项回读本文件与同层细则。

## Diagnostics 闭环

落盘后复核标题层级、字段名、表格和旧协议残留；发现问题先回写，再重新运行正式文档检查。

## 1. 文件角色与执行边界

### 1.1 当前文件负责什么

当前文件只负责冻结下面六件事：

1. `LKMA` 作为 baseline 之后第一层正式模块增强的唯一身份与阶段目标。
2. `baseline -> baseline + LKMA` 的单变量比较边界、搜索边界和正式阶段门。
3. `LKMA v1` 的模块定义、插入位置、kernel 候选、复杂度记录和命名字段链。
4. `LKMA` 阶段何时允许保留、何时降级为 `backup`、何时必须删除并回退到纯 baseline。
5. `LKMA` 阶段最低交付物、正式资产、阶段 manifest 和对 `06_Boundary` 的交接语义。
6. `LKMA` 阶段的实现入口、冲突裁决、文件质量自检和 diagnostics 闭环。

### 1.2 当前文件不负责什么

当前文件不重新定义下面这些内容：

- `ResNet34-U-Net` 作为正式 baseline 的结构身份、encoder 初始化来源和 skip 边界。
- baseline 当前轮冻结的 `train_proto_version` 与 `eval_proto_version` 的完整训练与评估 protocol；上一轮 `eval_proto_v1 -> eval_proto_v3` 仅作为历史祖先保留。
- `Gate_B1_compare` 与 baseline 放行门 `Gate_B1` 的成立条件。
- `Boundary Head`、`Distance-aware Loss` 和外部对比阶段的模块设计。

这些职责分别以下列文件为准：

### 1.3 为什么当前文件必须独立存在

因为 `LKMA` 阶段最容易出现的伪完成状态不是“完全没跑”，而是：

- baseline 还没有作为正式 handoff 基座收口，就提前开始大核模块验证。
- 一边改位置、一边改 kernel、一边改训练细节，最后无法确认收益是不是来自 `LKMA`。
- screening run 已经跑通，却没有形成三 seed 正式比较、成本记录和 keep-backup-drop 结论。
- `Boundary` 想继承 `LKMA` 结果时，说不清应当继承 `baseline`、`baseline + kept LKMA` 还是某个非正式尝试版本。

如果没有一份独立阶段总协议，后续 `06_Boundary`、阶段汇总脚本和论文消融表都会再次出现“LKMA 已经做过，但到底按哪套变量边界、哪套资产链、哪一版结论继续”不一致的问题。

## 3. 本阶段唯一允许处理的变量

### 3.1 当前阶段真正要证明什么

`05_LKMA` 阶段真正要证明的，不是“加了一个大核模块后某组数字变了”，而是：

> 在 baseline 已经正式放行且训练、评估和比较协议全部冻结的前提下，`LKMA` 作为第一层 post-baseline 模块增强，是否能够以最小单变量设置提供可复查、可解释、可交接的形态与大感受野收益，并因此获得 `keep / backup / drop` 的正式阶段结论。

因此，本阶段通过的含义同时包括：

- `LKMA` 已经被压缩成真实的单模块、单位置、单主 kernel 的可解释比较。
- `baseline vs baseline + LKMA` 的差异来自模块本身，而不是训练或评估 protocol 漂移。
- `LKMA` 的正式资产、比较表、成本表、案例和 decision note 已经齐全到可被 `06_Boundary` 直接消费。

### 3.2 当前阶段不允许用什么替代正式结论

当前明确不允许把下面这些情况写成“LKMA 已完成”：

- 只跑了一个 screening run 就宣称模块有效。
- 同时试多个位置和多个 kernel，却仍把结论写成单变量比较。
- 只看单次高值或像素级指标，不看对象级主指标、可视化和成本。
- 只写“模块可能有用”，不形成 `keep / backup / drop` 正式结论和下游起点。

### 3.3 为什么当前阶段必须拆成多层阶段门

`LKMA` 最容易发生的错误是把“代码能跑”误写成“模块值得留下”。

因此，当前阶段必须把放行条件拆成：

- 上游基座门：负责确认当前比较建立在已经正式放行的 baseline 上。
- 运行完整门：负责确认 screening、三 seed 主版本和必要资产齐全。
- 比较就绪门：负责确认公平比较、对象级指标、可视化和成本证据全部到位。
- 决策门：负责确认 keep-backup-drop 结论已经形成且可被下游直接消费。
- 交接门：负责确认 `06_Boundary` 已经能读取唯一合法 `current_base`。

## 4. 设计边界(做什么/不做什么)

### 4.1 当前阶段直接继承哪些冻结项

本文件直接继承并不得擅自改动的上位规则如下：

- 主任务：`2D gland segmentation`
- 主数据集：`GlaS`
- 正式 split：`train68 / val17 / testA60 / testB20`
- 标签口径：`mask > 0`
- 输入通道：`RGB`
- 输入尺寸：`512 x 512`
- 归一化：`ImageNet mean/std`
- 增强版本：`light_aug_v1`
- 主损失：`L_seg = L_BCE + L_Dice`
- 优化协议：`AdamW + lr 1e-3 + weight_decay 1e-4 + ReduceLROnPlateau + epoch_max 120 + early_stopping 20 + AMP on`
- 主指标优先级：`F1 / Object Dice / Object Hausdorff`
- `best_selector = val_objdice_max`
- `threshold_source = val17`
- `train_proto_version = train_proto_v1`
- `eval_proto_version` 由 04 阶段当前轮 Gate 冻结
- `eval_proto_lineage = eval_proto_v1 -> eval_proto_v3`；`eval_proto_v1` 仅为历史祖先，不是当前正式运行身份
- `eval_cast_policy = logits/probabilities must be kept or cast to float32 before thresholding`
- `boundary_metric_width = 3 px`
- `boundary_metric_impl = project_custom_erosion_xor_3x3_ones_8conn_border0_tol3px`（按 04_Baseline 的 protocol_v3 项目定义）
- `connected_components_impl = scipy.ndimage.label`
- `connected_components_connectivity = 8`
- `postprocess_version`：当前轮待由研究定标、阶段锁定、Pre-check/Gate 冻结；`none_in_v3` 仅作历史 lineage，不是当前值。
- baseline 正式基座：`ResNet34-U-Net`
- baseline encoder 初始化：`official ImageNet-1K supervised weights`
- fixed three seeds：`3407 / 1234 / 2025`

### 4.2 当前阶段唯一合法起点

当前阶段唯一合法起点固定为：

- 项目消费的是已经冻结的 `B1 current_standard`，而不是任意 `R34UNet` 试验版。
- `04_Baseline` 的原始 `Gate_B1=false` 与 `handoff_ready_for_c1=false` 必须保留为历史审计事实，不得手改为 `true`。
- 在当前治理决策下，05 允许在带 warning 的冻结基线边界下正式执行，但不得把该入口写成 B1 稳定性完全通过。

```text
baseline_ready_for_lkma = frozen_conditional_baseline
current_baseline = B1 current_standard
baseline_status = valid_with_stability_warning
stability_warning = true
original_gate_b1 = false
consumption_boundary = frozen_baseline_with_warning
formal_handoff = conditional_handoff
```

这意味着：

- 当前文件不得绕过 baseline 资产身份，直接把任何 `R34UNet` 试验版当成 `LKMA` 比较基座。
- `LKMA` 的所有正式 run 都必须能追溯到 B1 current_standard 的真实资产、问题登记和冻结字段链。
- `baseline_ready_for_lkma` 是当前阶段的治理消费状态，不是对原始 `Gate_B1` 的改写。

### 4.3 当前阶段唯一变量

本阶段唯一允许处理的核心变量只有“在正式 baseline 上追加一个可插拔 `LKMA` 模块的结构协议”：

- `LKMA v1` 的模块定义
- `LKMA` 的插入位置
- `LKMA` 的 kernel 候选
- `LKMA` 是否带来可接受的复杂度成本
- `LKMA` 是否值得进入后续主线

本阶段明确不允许借机改动：

- baseline backbone
- encoder 预训练策略
- decoder 主干
- 数据协议
- 输入尺寸
- 主增强包
- 主 loss
- optimizer / scheduler / lr
- `best checkpoint` 选择器
- 阈值来源
- 边界分支
- 距离约束
- 外部对比与 `CRAG` 正式验证

### 4.4 当前阶段正式阶段门

当前文件固定采用下面的阶段门语义：

```text
Gate_C1 = baseline_ready_for_lkma
          AND lkma_main_runs_complete
          AND lkma_compare_ready
          AND lkma_assets_ready
          AND lkma_decision_ready
          AND handoff_ready
```

这意味着：

- `LKMA` 阶段不是“先跑完再说”，而是必须在上游、运行、比较、决策和交接同时闭环后才允许进入 `06_Boundary`。
- 即使主版本三 seed 已完成，只要 keep-backup-drop 结论或 handoff 资产还没形成，也不得视为阶段通过。

## 5. 如何判断设计有效(验收标准)

下面所有核心规则都写成正式协议格式；后续实现、实验和阶段汇总时，只允许把这些规则忠实落地，不允许擅自补新的隐含边界。

### 5.1 第一模块身份与作用定义

- 当前结论：后 baseline 的第一模块固定为 `LKMA`，其角色固定为 `形态感知 + 大感受野 + 上下文聚合`，而不是边界分离、距离约束或完整新 backbone
- 规则类型：`路线层已锁定 + 工程冻结规则`
- 适用阶段：`05_LKMA` 启动时生效，后续是否继承取决于本阶段结论
- 直接依据：`结直肠腺体分割_plan_优化版/02_路线与投稿/02_结直肠腺体分割_分阶段实验路线与执行标准.md`、`结直肠腺体分割_plan_优化版/03_文献证据/05_腺体任务经典论文/04_MILD-Net.md`、`结直肠腺体分割_plan_优化版/01_实验执行/05_LKMA/01_设计依据.md`
- 核心公式或定义参考：当前任务难点固定为 `整体形态 + 长程上下文 + 复杂结构连贯性`，模块职责不包括显式 `boundary supervision`
- 采用原因：腺体分割在 baseline 之后优先需要验证的是形态和感受野增强是否值得保留，而不是立刻叠加边界监督
- 不采用的相邻方案：不把 `LKMA` 写成“默认必保留模块”；不把它写成完整 `VAN/RepLKNet` 主干替换；不把它和 `Boundary` 合并为同一阶段
- 代码落点：[project_root]/src/models/modules/lkma.py、[project_root]/configs/model/lkma_v1.yaml
- 运行记录字段：`lkma_version`、`lkma_role`、`lkma_impl_type`
- 验收方式：检查阶段文档、配置和实现都明确 `LKMA` 只负责形态与感受野增强，没有顺手引入边界头、距离头或额外分支

### 5.2 `LKMA v1` 模块定义

- 当前结论：`LKMA v1` 固定为 `single-branch large-kernel depth-wise spatial block + optional 1x1 channel mixing + residual path` 的可插拔模块，不采用完整 `VAN / WTConv / RepLKNet / UniRepLKNet` 原版结构
- 规则类型：`工程冻结规则`
- 适用阶段：`05_LKMA`

- 核心公式或定义参考：首版核心计算对象固定为 `Y = X + PWConv(DWConv_k(X))` 或其等价轻量变体，其中 `DWConv_k` 是唯一大核空间混合主体
- 采用原因：当前项目需要的是可解释、可插拔、易落地的首版模块，而不是在 `LKMA` 阶段直接复现重型大核文献全结构
- 不采用的相邻方案：不采用完整 `LKA attention gating`；不采用完整 `WT + IWT` 小波链；不采用整套 backbone 替换
- 代码落点：[project_root]/src/models/modules/lkma.py
- 运行记录字段：`lkma_impl_type`、`kernel_size`、`use_pwconv`、`use_residual`
- 验收方式：检查 lkma.py 主体仍是单模块大核空间混合，且没有演变成多分支 attention 或小波多级分解整链

### 5.3 插入位置规则

- 当前结论：首轮正式主位置固定为 `bottleneck`；最多只允许再补一个高层备选位置 `decoder_stage1`
- 规则类型：`论文支持的候选范围 + 工程冻结规则`

- 核心公式或定义参考：位置候选集合固定为 `P = {bottleneck, decoder_stage1}`
- 采用原因：`bottleneck` 最接近全局语义与整体形态聚合，是首轮最合理主位点；若主位点证据不足，最多只补一个靠近重建路径的高层位点
- 不采用的相邻方案：不采用多 encoder stage 同时插入；不采用多 decoder stage 同时插入；不采用全网络铺满 `LKMA`

### 5.4 kernel 候选与首轮默认值

## 7. 回退条件

### 7.1 独立回退触发条件

只要出现下面任意一条，本阶段都必须先回退修正，而不是直接进入 `06_Boundary`：

- baseline 实际尚未满足 `Gate_B1`，却已经启动正式 `LKMA` 比较。
- 主版本三 seed 不完整，或正式 run 的核心资产缺失。
- `insert_position`、`kernel_size`、`train_proto_version`、`threshold_source` 或 `eval_cast_policy / boundary_metric_width / boundary_metric_impl / connected_components_impl / connected_components_connectivity` 在 config.yaml、run_meta.yaml、结果表、stage manifest 和汇总文档之间无法对齐。
- `LKMA` 平均对象级主指标明显低于 baseline，且可视化不支持形态增益。
- `TestB` 明显更差，或者某个 seed 波动失控且没有排查记录。
- 成本明显上升但没有形成清晰收益证据，却仍试图给出 `keep` 结论。
- 主版本不过线，却想通过扩大搜索空间强行保留模块。
- keep-backup-drop 结论和 `06_Boundary` 的 `current_base` 说明不一致。

### 7.2 固定回退顺序

回退时统一按下面顺序排查：

1. 检查插入位置、通道与输出尺寸是否正确。
2. 检查 kernel、padding 和 residual 连接是否正确。
3. 检查 baseline 训练与评估协议是否真正继承。
4. 检查复杂度代价是否被正确记录。
5. 只有前面都确认无误后，才允许决定是否补唯一一个备选版本，还是直接降级为 `backup / drop`。

### 7.3 回退后的重新放行条件

发生回退后，只有同时满足下面条件，才允许重新声明 `LKMA` 阶段通过：

- 问题来源已经按固定回退顺序定位并修复。
- 修复动作、影响范围和复验结果已写入 [project_root]/notes/debug_note.md、阶段总结或 decision note。
- config.yaml、run_meta.yaml、结果表、可视化和 manifest 重新对齐。
- 当前阶段重新满足第 `5` 节全部规则卡片与第 `6` 节的最低交付物、交接资产和下游起点要求。
