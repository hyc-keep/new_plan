# 结果记录模板

## 1. 文件角色与执行边界

### 1.1 当前文件负责什么

这份文件不是“随手登记实验数字”的备忘录，而是 `10_结果汇总` 阶段里专门负责 run 级正式记录准入门的协议文件。

它当前只负责把下面六件事写死：

1. 什么样的 run 才有资格进入 `10_结果汇总` 的正式结果层。
2. 单次正式 run、三次重复聚合、文献 direct comparison 三类记录分别必须写哪些字段。
3. GlaS / CRAG、val / testA / testB / test、reproduced / † / *、single_seed / mean +- std 这些边界在 run 级记录里如何显式落盘。
4. 主表资格、消融资格、失败案例留痕和结果来源标记在结果记录层如何先冻结，再交给后面的表格和结果包文件消费。
5. 结果记录与 `official_run_registry`、主表回填、失败案例清单、`stage summary` 和投稿结果包检查之间的字段映射关系。
6. 当前文件自己的独立 `回退条件`、`代码落地接口`、`冲突裁决记录`、`文件质量自检` 和 `Diagnostics 闭环`。

### 1.2 当前文件不负责什么

当前文件不重新定义下面这些内容：

- 数据集 split 身份、标签口径和数据资产来源；这些以上游数据协议为准。
- 模型结构、训练协议、模块去留和阶段通过结论；这些由 `02-09` 阶段正式 handoff 给出。
- 主表、消融表、图像清单、`stage summary` 和结果包状态的最终版式；这些由同层其它文件负责展开。
- `11_总验收与止损` 的总裁决、中文保底和英文起步条件；当前文件只负责给下游准备干净、可追溯的输入记录。

### 1.3 为什么当前文件必须独立存在

结果汇总阶段最容易出现的伪完成状态不是“完全没记”，而是：

- 有几个好看的数字，但没有 run 级来源链。
- 有 `mean +- std`，却找不到对应 per-seed 记录。
- 有主表候选行，却说不清它到底是 `reproduced`、`†` 还是 `*`。
- 有失败案例图片，却没有把它和具体 run、具体 split、具体错误类型连起来。
- 有 `CRAG` 结论等级，却没有对应的 run 记录支撑。

如果 run 级记录模板不独立冻结，后面 02_主表与消融表回填规则.md、03_可视化与失败案例清单.md、04_stage_summary模板.md 和 05_投稿结果包检查表.md 都会再次面对“数字看起来能用，但到底能不能正式消费”的灰区。

## 3. 本阶段唯一允许处理的变量

### 3.1 正式结果记录的统一定义

从现在开始，任何想进入正式结果汇总层的记录都固定理解为：

```text
official_result_record
= protocol_info
+ splitwise_metrics
+ source_boundary
+ failure_trace
+ table_eligibility
```

其中：

- `protocol_info`：`run_name`、阶段、版本链、seed、`best_selector`、阈值来源、checkpoint、增强 profile、评估实现和统计口径等协议信息。
- `splitwise_metrics`：按正式 split 身份拆开的主指标与补充指标。
- `source_boundary`：reproduced / † / *、公平性说明、是否可 direct comparison。
- `failure_trace`：失败现象、异常样本、是否需要回退。
- `table_eligibility`：能否进主表、能否进消融表、是否只留补充层。

### 3.2 当前文件唯一合法输入资产

当前文件只允许消费下面这些上游正式资产：

- 正式 run 的 config.yaml、run_meta.yaml、指标 csv、`best.ckpt` 路径。
- 各阶段已经冻结的 `stage_pass`、模块去留和结论等级。
- 外部对比阶段已经写清的 reproduced / † / * 与公平性边界。
- `CRAG` 阶段已经写清的 A / B / C 结论等级。

当前文件明确不允许：

- 在结果记录阶段重新算没有说明来源的新数字。
- 用测试集回调阈值再写入正式记录。
- 先决定表格行，再倒推要补哪些 run 字段。

### 3.3 当前文件在 `Gate_J1` 中的角色

`结直肠腺体分割_plan_优化版/01_实验执行/10_结果汇总/00_阶段总协议.md` 里把 `registry_ready` 写成 `Gate_J1` 的正式子门。

当前文件就是这个子门的展开协议：

```text
registry_ready
= run_fields_complete
AND split_identity_clear
AND source_boundary_clear
AND failure_trace_present
AND table_flags_present
```

如果 `registry_ready = false`，后面的主表、消融表、失败案例和结果包状态都默认不得宣称正式完成。

## 4. 阶段门控表达式

后面统一只承认下面七条总原则：

1. 先记录协议，再记录数字。
2. 先记录 per-seed，再记录 `mean +- std`。
3. 先区分 GlaS / CRAG，再做联合汇总。
4. 先区分 val / testA / testB / test，再做表格回填。
5. 先区分 reproduced / † / *，再讨论 direct comparison。
6. 先记录失败说明和异常边界，再写“是否可进主表”。
7. 先把 run 级记录写完整，再让同层文件消费，不允许跳过记录层直接拼表。

如果最后只留下一个“最好数字”，或者只有表格里的一行，没有 run 级正式记录，这个结果默认不具备进入正式结果汇总层的资格。

## 5. 核心规则卡片

### 5.1 正式 run 准入规则

- 当前结论：任何进入正式结果层的结果，必须先有完整 run 级记录，且记录链能回指 run_meta.yaml、指标文件、checkpoint 和可视化目录
- 规则类型：`工程冻结规则`
- 适用阶段：`10_结果汇总`
- 直接依据：`结直肠腺体分割_plan_优化版/01_实验执行/10_结果汇总/00_阶段总协议.md`，`结直肠腺体分割_plan_优化版/01_实验执行/00_总览与规范/03_命名与结果记录规范.md`
- 核心公式或定义参考：`formal_run = metadata + metrics + source_type + note`
- 采用原因：没有 run 级记录支撑的数字不能稳定回填主表，也不能在总验收时复查
- 不采用的相邻方案：不采用只保留截图、只保留 best seed、只保留汇总表一行
- 代码落点：reports/official_run_registry.md
- 运行记录字段：`run_name`, `stage_code`, `stage_name`, `dataset`, `model_name`, `config_version`, `seed`, `checkpoint_path`
- 验收方式：检查每个正式数字都能追到唯一 run、唯一 checkpoint 和唯一版本链

### 5.2 字段链完整性规则

- 当前结论：run 级正式记录至少要同时覆盖命名字段、版本字段、评估字段、来源字段、失败字段和表格资格字段
- 规则类型：`工程冻结规则`
- 适用阶段：`10_结果汇总`
- 直接依据：`结直肠腺体分割_plan_优化版/01_实验执行/00_总览与规范/03_命名与结果记录规范.md`，`结直肠腺体分割_plan_优化版/01_实验执行/00_总览与规范/04_评估口径与官方脚本对齐.md`
- 核心公式或定义参考：`field_complete = naming + versions + eval + source + notes + eligibility`
- 采用原因：很多返工不是因为指标错，而是字段不全导致后面无法筛表、无法追溯和无法审计
- 不采用的相邻方案：不采用只记 `run_name + metrics` 的极简记录；不采用只在自然语言里含混带过版本和来源
- 代码落点：reports/official_run_registry.md，reports/tables/*.csv
- 运行记录字段：`config_version`, `train_proto_version`, `aug_profile_name`, `eval_aug_enable`, `eval_proto_version`, `eval_cast_policy`, `model_version`, `loss_version`, `postprocess_version`, `boundary_metric_width`, `boundary_metric_impl`, `connected_components_impl`, `connected_components_connectivity`, `metric_crosscheck_result`, `seed_reporting_mode`, `result_source_type`, `is_mean_std`
- 验收方式：检查字段链能够被同层表格和下游总验收直接消费，不需要回各阶段补口头说明

### 5.3 `GlaS` 分 split 记录规则

- 当前结论：`GlaS` 正式记录必须分开保留 `val17`、`TestA60`、`TestB20`，不得写成一个总测试值
- 规则类型：`官方协议固定项 + 工程冻结规则`
- 适用阶段：`10_结果汇总`
- 直接依据：`结直肠腺体分割_plan_优化版/01_实验执行/01_数据协议/03_GlaS划分协议.md`，`结直肠腺体分割_plan_优化版/01_实验执行/00_总览与规范/04_评估口径与官方脚本对齐.md`
- 核心公式或定义参考：`glas_record = val17 + testA60 + testB20`
- 采用原因：`TestA` 和 `TestB` 难度不同，把两者压成单值会掩盖困难子集上的对象分离与边界问题
- 不采用的相邻方案：不采用 TestA / TestB 平均值替代分 split 结果；不采用 `GlaS` 只记一个 `test`
- 代码落点：reports/official_run_registry.md，reports/tables/glas_main_metrics.csv
- 运行记录字段：`split_role`, `threshold_source`, `F1`, `ObjDice`, `ObjHaus`, `Dice`, `IoU`, `HD95`, `Boundary F1`
- 验收方式：检查 `GlaS` 记录中同时存在 val17 / TestA60 / TestB20 三段，并且主指标完整

### 5.4 `CRAG` 正式记录规则

- 当前结论：`CRAG` 正式记录固定为 `val20 + test40` 两段，且必须保留对 baseline 的相对结论与 A / B / C 结论等级
- 规则类型：`工程冻结规则 + handoff 规则`
- 适用阶段：`10_结果汇总`
- 直接依据：`结直肠腺体分割_plan_优化版/01_实验执行/01_数据协议/04_CRAG划分协议.md`，`结直肠腺体分割_plan_优化版/01_实验执行/09_CRAG验证/03_结果解释规则.md`
- 核心公式或定义参考：`crag_record = val20 + test40 + conclusion_grade`
- 采用原因：`CRAG` 是补充验证层，除了指标本身，还必须把“相对 baseline 的意义强度”一起记录下来
- 不采用的相邻方案：不采用只记测试数字、不记结论等级；不采用把 `CRAG` 写成与 `GlaS` 完全同强度的主证据
- 代码落点：reports/official_run_registry.md，reports/tables/crag_main_metrics.csv
- 运行记录字段：`compared_with_baseline`, `conclusion_grade`, `supports_mainline`
- 验收方式：检查 `CRAG` 记录里同时有指标、说明和结论等级，且与 `09_CRAG验证` handoff 一致

### 5.5 三次重复聚合规则

- 当前结论：完成三次正式重复后，必须额外生成聚合记录；聚合不能替代 per-seed 记录，只能建立在 per-seed 之上
- 规则类型：`工程冻结规则`
- 适用阶段：`03_UNet稳定性` 之后的正式结果汇总
- 直接依据：`结直肠腺体分割_plan_优化版/01_实验执行/03_UNet稳定性/02_mean_std汇总规则.md`，`结直肠腺体分割_plan_优化版/01_实验执行/00_总览与规范/03_命名与结果记录规范.md`
- 核心公式或定义参考：`mean_std_record = per_seed_records + mean +- std + stability_note`
- 采用原因：后面的主表、消融表和投稿判断都需要稳定性证据，不能用 best seed 冒充均值结论
- 不采用的相邻方案：不采用 best seed 代替 `mean +- std`；不采用不同协议 run 混算；不采用 `GlaS TestA` 和 `TestB` 混算
- 代码落点：reports/official_run_registry.md，reports/tables/*.csv
- 运行记录字段：`seeds`, `seed_reporting_mode`, `main_seed_set`, `additional_seed_set`, `combined_seed_count`, `is_mean_std`, `aggregation_scope`, `relative_to_baseline`, `stability`
- 验收方式：检查聚合记录能逐项回指三次正式 seed 记录，且 split 和协议一致

### 5.6 文献值与 external direct comparison 记录规则

- 当前结论：对引文值和外部方法，必须额外记录来源文件、公平性说明、特殊测试设置和是否可 direct comparison
- 规则类型：`来源边界规则`
- 适用阶段：`08_外部对比` 之后进入 `10_结果汇总` 的对照记录
- 直接依据：`结直肠腺体分割_plan_优化版/01_实验执行/08_外部对比/06_引用值与复现值标记规则.md`，`结直肠腺体分割_plan_优化版/03_文献证据/05_GlaS_CRAG_主结果数值速查表.md`
- 核心公式或定义参考：`direct_comparison_record = value_source + fairness_note + special_setting + eligibility`
- 采用原因：direct comparison 的最大风险是“数字写进表里了，但根本不公平”
- 不采用的相邻方案：不采用只记文献数字不记来源；不采用 `†` 与 `*` 混成同一类；不采用 `TA-Net` 的 A/B 平均值伪装成 TestA / TestB
- 代码落点：reports/tables/direct_comparison.csv，reports/tables/external_main_table.csv
- 运行记录字段：`paper_or_repo_source`, `fairness_note`, `has_special_test_setting`, `direct_comparison_eligible`, `can_enter_main_table`
- 验收方式：检查每条引文或外部记录都能解释“为什么能比”或“为什么只能条件纳入”

### 5.7 失败说明与主表资格规则

- 当前结论：每条正式记录必须同时写失败说明与表格资格，不能跳过异常边界直接写“能进主表”
- 规则类型：`工程冻结规则 + 止损规则`
- 适用阶段：`10_结果汇总`
- 直接依据：`结直肠腺体分割_plan_优化版/01_实验执行/10_结果汇总/03_可视化与失败案例清单.md`，`结直肠腺体分割_plan_优化版/01_实验执行/10_结果汇总/05_投稿结果包检查表.md`
- 核心公式或定义参考：`table_eligible = protocol_clear AND source_clear AND failure_trace_present`
- 采用原因：很多结果不是指标差，而是来源边界不清、失败说明缺失，最终不具备正式写作资格
- 不采用的相邻方案：不采用“数字好就先进表，说明后补”；不采用完全不记失败案例的“纯主表视角”
- 代码落点：reports/official_run_registry.md，notes/failure_case_notes.md
- 运行记录字段：`success_failure_summary`, `abnormal_case`, `can_enter_main_table`, `can_enter_ablation_table`
- 验收方式：检查每条正式记录都有失败说明或明确写 `none_observed`，并且表格资格有原因可追

## 8. 回退条件

### 8.1 独立回退触发条件

只要出现下面任意一条，当前记录就必须先回退修正，而不是直接进入主表或结果包：

- run 级记录无法追到 run_meta.yaml、指标文件或 `best.ckpt`
- `GlaS` 把 TestA / TestB 合写，或 `CRAG` 缺少 `conclusion_grade`
- reproduced / † / * 没有明确写清
- `mean +- std` 没有对应 per-seed 原始记录
- 失败说明为空，或者异常写法无法支撑后面的失败案例层
- 记录层字段和同层表格层字段语义打架

### 8.2 固定回退顺序

回退时统一按下面顺序排查：

1. 先检查来源 run、版本链和 checkpoint 是否完整。
2. 再检查 split 身份、阈值来源和 `best_selector` 是否统一。
3. 再检查 reproduced / † / *、公平性说明和 direct comparison 资格是否清楚。
4. 再检查失败说明、主表资格和消融资格是否可追溯。
5. 最后再让同层文件重新消费修复后的记录层。

### 8.3 回退后的重新放行条件

发生回退后，只有同时满足下面条件，才允许重新声明记录层通过：

- 字段链已补齐，且与上游协议一致。
- 分 split 指标、来源边界和失败说明已恢复完整。
- 主表资格与同层表格规则重新对齐。
- 当前记录重新满足第 `5` 节全部规则卡片和第 `6` 节全部模板要求。

## 9. 代码落地接口

### 9.1 正式 run 注册入口

- 代码文件：scripts/validate_sources.py
- 入口类/函数：`collect_official_runs()`、`validate_run_record_fields()`
- 输入：config.yaml、run_meta.yaml、指标 csv、checkpoint 路径、可视化目录、失败说明
- 输出：reports/official_run_registry.md
- `dtype`：指标字段为 `float`；身份、来源和状态字段为 string / bool
- 依赖配置：`run_name`, `config_version`, `train_proto_version`, `eval_proto_version`, `eval_cast_policy`, `threshold_source`, `boundary_metric_width`, `boundary_metric_impl`, `connected_components_impl`, `connected_components_connectivity`, `result_source_type`
- 前置断言：缺版本链、缺 split 身份、缺 checkpoint 或缺来源标记的 run 不得进入注册层
- 运行产物：正式 run 注册文档、字段校验结论

### 9.2 聚合记录构建入口

- 代码文件：scripts/compare_runs.py
