# 深度学习结直肠腺体分割论文研究与投稿入门指南

> 本文先解释深度学习论文实验的完整逻辑，再把这套逻辑映射到当前结直肠腺体分割项目。

## 0. 先看懂我们论文的完整实验流程

你可以先把一篇深度学习分割论文理解成下面这条证据链：

```text
研究问题
  -> 文献与方法定标
  -> 固定数据、标签、split 和评估协议
  -> 用 UNet 跑通完整实验系统
  -> 用多个 seed 验证 UNet 稳定性
  -> 确定内部正式 Baseline
  -> 在 Baseline 上逐个加入自己的研究模块
  -> 做消融实验，解释每个模块的贡献
  -> 在同一数据集上与外部模型公平对比
  -> 在第二 benchmark 上做补充验证
  -> 汇总统计、可视化和失败案例
  -> 写论文并投稿
```

当前项目对应为：

```text
A1：UNet 流程验证
  目标：确认数据、训练、测试、指标和可视化链路正确

A2：UNet 稳定性
  目标：用固定多个 seed 建立最小稳定参照

B1：ResNet34-U-Net 正式 Baseline
  目标：建立后续研究共同使用的内部结构起点

C1：ResNet34-U-Net + LKMA
  目标：验证大范围上下文建模是否改善腺体分割

C2：ResNet34-U-Net + LKMA + Boundary Head
  目标：验证边界监督是否改善边界和黏连问题

C3：继续加入 Distance-aware Loss（如果前一阶段结果支持）
  目标：验证距离或形状约束是否带来额外收益

外部对比：UNet++、DeepLabV3+、Attention U-Net 等
  目标：证明完整方法相对于其他代表性模型的竞争力

CRAG 验证：在第二个相关 benchmark 上验证
  目标：观察方法是否不仅适用于 GlaS
```

### 0.1 论文中每个模型的角色

在一篇论文中，不同模型不是“全部都叫 baseline”，而是承担不同角色：

| 模型或实验 | 在当前项目中的身份 | 它要回答的问题 |
|---|---|---|
| UNet | 流程基线、最小模型参照 | 最基本的分割系统能达到什么水平？ |
| ResNet34-U-Net | 内部正式 Baseline | 后续创新从什么结构起点开始？ |
| Baseline + LKMA | 第一项改进 | LKMA 是否带来额外贡献？ |
| Baseline + LKMA + Boundary | 第二项改进 | 边界监督是否进一步解决任务难点？ |
| 完整方法 `Ours` | 论文最终方法 | 所有保留模块组合起来是否有效？ |
| UNet++、DeepLabV3+ 等 | 外部对比模型 | Ours 相对于其他代表性模型如何？ |
| CRAG 上的 Baseline/Ours | 第二 benchmark 验证 | 方法在另一个相关数据集上是否仍成立？ |

这里最重要的是：

```text
Baseline 是内部研究参照；
外部模型是论文竞争性参照；
Ours 是你的完整方法。
```

### 0.2 外部对比到底怎么做

外部对比通常是在**同一个数据集**上进行，而不是先把数据集换掉。

例如在 GlaS 上，固定相同的：

- train/val/TestA/TestB 划分；
- 图像预处理和输入尺寸；
- 训练预算和主要训练协议；
- 验证集模型选择规则；
- 测试阈值来源；
- 指标实现和报告方式。

然后分别运行：

```text
UNet
UNet++
ResNet34-U-Net
DeepLabV3+
Attention U-Net
其他可复现的腺体分割方法
Ours = ResNet34-U-Net + LKMA + Boundary Head + Distance-aware Loss
```

最后比较：

```text
Ours 是否优于这些代表性方法？
Ours 的提升是否在 TestA 和 TestB 都合理？
Ours 是否在多个 seed 下方向一致？
Ours 的优势是否对应边界、黏连或对象分离难点？
```

所以，外部对比的核心不是“把每个模块都塞进所有模型”，而是：

> 在相同任务和尽可能公平的实验协议下，把完整方法 `Ours` 与其他公开代表性方法放在同一张比较表中。

### 0.3 “把模块放到其他模型上”属于什么实验

例如：

```text
UNet + LKMA
DeepLabV3+ + LKMA
ResNet34-U-Net + LKMA
```

这不是外部模型对比的基本要求，而是更进一步的**模块泛化性实验**或**跨主干验证**。它回答：

> LKMA 的效果是否只依赖 ResNet34-U-Net，还是换一个主干也可能有效？

这类实验有价值，但不是当前第一优先级。当前优先级应是：

```text
先证明模块在内部正式 Baseline 上有效
 -> 再做完整方法的外部对比
 -> 资源充足时再做模块跨主干验证
```

### 0.4 “换到 CRAG”属于什么实验

CRAG 不是外部模型，而是第二 benchmark。它和 GlaS 的关系是：

```text
GlaS：主 benchmark，承载主实验结论
CRAG：第二 benchmark，承载补充验证结论
```

在 CRAG 上重新训练并测试 Baseline 和 Ours，回答：

> 在另一个相关结直肠腺体数据集上，Ours 是否仍然优于自己的 Baseline？

这叫第二数据集验证或跨 benchmark 验证。

如果在 GlaS 训练后直接到 CRAG 测试，则是更严格的跨数据集泛化实验，必须单独说明，不能和“在 CRAG 上重新训练测试”混在一起。

### 0.5 什么结果才足以支持“方法有效”

不是只要 Ours 有一个指标最高就可以投稿。更完整的证据应该包括：

1. `Ours` 相比内部 Baseline 在主要指标上有改善。
2. 多个 seed 下提升方向基本一致，而不是一次偶然高分。
3. 消融实验能说明提升来自哪个模块。
4. 外部对比表明 Ours 相对代表性模型具有竞争力。
5. 困难案例可视化支持研究动机，例如边界、黏连、小腺体或形态不规则案例改善。
6. TestA/TestB 或不同 benchmark 上没有严重相互矛盾的退化。
7. 没有数据泄漏、测试集调参或不公平训练预算。
8. 论文如实报告限制，例如当前 B1 的稳定性 warning。

因此更准确的结论是：

> 如果完整方法在预先规定的实验协议下，稳定改善内部 Baseline，并在外部方法和第二 benchmark 上得到相互支持的证据，那么可以认为它在当前实验范围内有效，并具备写论文投稿的实验基础。

但不能把它直接扩写成“对所有结直肠病理图像都有效”。

### 0.6 当前项目最终可能形成的论文实验表

#### 内部消融表

```text
M0 = ResNet34-U-Net
M1 = M0 + LKMA
M2 = M1 + Boundary Head
M3 = M2 + Distance-aware Loss
```

这张表回答每个模块是否贡献。

#### 外部方法对比表

```text
UNet
UNet++
ResNet34-U-Net
DeepLabV3+
Attention U-Net
任务内公开方法
Ours = M3 或最终保留版本
```

这张表回答完整方法相对于其他模型是否有竞争力。

#### 第二 benchmark 表

```text
CRAG 上的 ResNet34-U-Net
CRAG 上的 Ours
必要时加入少量外部模型
```

这张表回答方法换到相关数据集后是否仍有改善。

三张表的作用不同，不能互相替代：

```text
消融表：解释模块贡献
外部对比表：证明竞争力
第二 benchmark 表：补充验证适用范围
```

### 0.7 最后用一句话记住论文逻辑

```text
先用 UNet 证明实验系统可信，
再用 ResNet34-U-Net 建立内部 Baseline，
然后逐个加入自己的模块形成 Ours，
用消融证明每个模块有贡献，
用同一数据集上的 UNet++/DeepLabV3+/Attention U-Net 等证明竞争力，
再用 CRAG 等第二 benchmark 做补充验证，
最后综合统计、可视化、失败案例和限制写论文。
```

## 1. 先建立总认识：深度学习研究到底在做什么

深度学习论文研究不是“找一个网络，训练一下，看哪个分数高”。它本质上是在回答一个经过控制的科学问题：

> 在明确的数据和评估条件下，某种方法是否能够解决某类问题？如果有效，效果来自哪里？是否稳定？适用于哪些场景？

一项完整研究至少包含下面五个对象：

```text
研究问题：要解决什么困难？
方法假设：为什么你设计的东西可能有效？
实验系统：如何保证数据、训练和评估可信？
比较基准：相对于什么对象判断提升？
证据链：如何证明提升真实、稳定、可解释？
```

例如，结直肠腺体分割中的研究问题可以是：

```text
现有模型在相邻腺体黏连、边界模糊和形态变化场景下容易分割错误。
因此，尝试引入更大范围的上下文建模、边界监督或距离约束，
以改善腺体对象分离和边界质量。
```

这里有一个非常重要的区别：

- **“代码能跑”是工程问题。**
- **“指标变高”是实验现象。**
- **“为什么变高、是否稳定、是否由目标模块造成”才是研究问题。**

论文要证明的是第三层，而不是只展示第一层或第二层。

## 2. 一般深度学习研究的完整模块

一项深度学习医学图像分割研究，通常可以分成以下模块。它们不是互相独立的文件，而是一条有先后关系的证据链。

| 模块 | 要回答的问题 | 主要作用 |
|---|---|---|
| 研究问题 | 为什么要做这项研究 | 确定任务难点和研究目标 |
| 文献调研 | 别人已经做到什么程度 | 确定方法来源、差距和实验对象 |
| 数据协议 | 用什么数据、标签和划分 | 保证输入和比较条件固定 |
| 流程基线 | 整套代码链是否可信 | 排查数据、训练、评估和导出问题 |
| 稳定性实验 | 结果是否依赖偶然随机状态 | 评估可重复性 |
| 正式 Baseline | 后续方法从什么强度起步 | 建立统一比较基座 |
| 方法设计 | 你具体增加了什么 | 回应研究问题 |
| 消融实验 | 每个模块是否真的贡献 | 解释增益来源 |
| 外部对比 | 相比其他代表性方法如何 | 证明竞争力 |
| 泛化验证 | 换数据或场景是否仍有效 | 评估迁移和鲁棒性 |
| 统计与误差分析 | 结果是否可信、哪里有效 | 支撑结论和讨论 |
| 论文写作 | 如何完整表达证据 | 将实验转成可审查的学术论证 |
| 投稿与审稿 | 研究是否符合期刊要求 | 完成学术传播和修改 |

下面逐项解释为什么需要它们。

## 3. 第一步：提出研究问题，而不是先挑网络

### 3.1 正确的起点

错误起点是：

```text
我想用一个新模块，所以找一个模型加进去。
```

正确起点是：

```text
任务中最严重的错误是什么？
现有方法为什么会出现这个错误？
什么类型的信息可能帮助解决它？
如何设计实验验证这个想法？
```

对结直肠腺体分割，常见研究问题包括：

- 腺体边界模糊，如何提高边界定位能力？
- 相邻腺体黏连，如何增强对象分离能力？
- 小腺体容易漏检，如何改善小目标召回？
- 腺体形态变化大，如何增强上下文和形状建模？
- 染色和组织结构变化，如何提升跨样本稳定性？

### 3.2 研究问题要能对应实验

如果你声称研究边界问题，实验中至少应该出现：

- 边界相关指标。
- 边界可视化。
- 与边界模块对应的消融。
- 对黏连和边界困难案例的错误分析。

如果只报告一个 Dice，而没有边界证据，就很难证明你真的解决了边界问题。

### 3.3 研究假设

研究问题之后要形成一个可验证假设：

> 如果增加某种信息或结构，模型应该在某种困难场景下改善某类指标。

例如：

```text
假设：大核上下文能够让模型看到更大范围的组织结构，
因此可能改善形态不规则和相邻腺体场景。
```

```text
假设：显式边界监督能够让模型关注腺体轮廓，
因此可能改善 Boundary F1、HD95 和黏连分离。
```

注意，“可能改善”是研究假设，不是提前写好的结果。实验可能验证它，也可能否定它。

## 4. 第二步：文献调研到底要做什么

文献调研不是简单收集论文标题、年份和最高指标。它要完成四项工作。

### 4.1 了解任务背景

需要弄清楚：

- 结直肠腺体分割为什么重要。
- GlaS、CRAG 等数据集是什么。
- 数据规模、标注形式和官方评价如何定义。
- 任务中最常见的失败模式是什么。

### 4.2 找到方法来源

不同方法需要不同来源：

- U-Net、ResNet：经典基础模型论文。
- LKMA：大核或上下文建模论文。
- Boundary Head：边界监督、边界感知或多任务论文。
- Distance-aware Loss：距离图、形状约束或几何损失论文。
- Object F1、Object Dice：任务官方评价或任务论文。

### 4.3 确定实验对象

文献可以帮助你确定：

- 什么模型适合作为正式 baseline。
- 哪些模型适合作为外部对比。
- 哪些模块值得做消融。
- 哪些方法只是引用结果，哪些可以自己复现。

### 4.4 找到研究空白

研究空白不是“没有人用过我的模块”这么简单，至少要说明：

- 现有方法在哪个任务难点上仍不足。
- 现有方法的实验或结构有什么局限。
- 你的方法为什么有机会改善这个问题。

文献阅读建议建立表格，至少记录：

| 字段 | 记录内容 |
|---|---|
| 论文 | 标题、作者、年份、期刊或会议 |
| 任务 | 数据集和分割任务 |
| 模型 | 主干、decoder、附加模块 |
| 监督 | 二值 mask、边界、距离图等 |
| 训练 | loss、optimizer、输入尺寸、增强 |
| 评估 | Dice、Object Dice、HD95 等 |
| 代码 | 是否公开、仓库地址、可复现程度 |
| 适配 | 与当前数据和工程的差异 |
| 角色 | baseline、模块来源、外部对比或背景 |

## 5. 第三步：数据协议为什么必须在模型之前固定

模型实验的前提是数据可信。如果数据、标签或 split 在实验过程中不断变化，后面的模型比较就没有统一基础。

### 5.1 需要固定的内容

- 数据集身份。
- 图像和标签的对应关系。
- 标签二值化规则。
- 训练集、验证集和测试集划分。
- 输入通道和颜色顺序。
- 输入尺寸。
- 归一化。
- 训练增强。
- 测试后处理和 TTA 是否启用。

当前项目的主数据集是 GlaS，主线使用固定的 train、val、TestA、TestB 角色；CRAG 作为第二 benchmark 做补充验证。

### 5.2 train、validation、test 分别做什么

```text
Train：用于更新模型参数。
Validation：用于选择 checkpoint、调整协议和冻结阈值。
Test：只用于最后报告泛化结果。
```

最重要的规则是：

> 不能根据 TestA 或 TestB 的结果反向调整模型、阈值或实验方案，然后仍把它当作独立测试结果。

### 5.3 为什么数据协议变化会影响论文

如果你改了：

- split。
- 标签处理。
- 输入尺寸。
- 归一化。
- 测试后处理。

那么新结果可能已经不再和旧结果属于同一实验协议。必须记录版本，并把新旧结果分开。

## 6. 第四步：什么是流程基线，为什么先跑 UNet

### 6.1 流程基线不是正式创新基线

“基线”这个词在实际项目中有两种常见含义，必须区分。

#### 流程基线

它的目标是检查：

```text
数据 -> 模型 -> loss -> 训练 -> checkpoint -> 评估 -> 可视化
```

是否全部正确。

它不一定是论文最终最主要的比较对象。

#### 正式 Baseline

它的目标是作为后续方法的统一参照，要求：

- 结构合理。
- 结果稳定。
- 任务匹配。
- 可复现。
- 能承载后续单变量改进。

当前项目中：

```text
普通 UNet = 流程基线和最小模型参照
ResNet34-U-Net = 正式结构 Baseline
```

### 6.2 UNet 是什么

UNet 是经典 encoder-decoder 分割网络。

```text
输入图像
  -> Encoder：提取越来越抽象的语义特征
  -> Bottleneck：最深层特征
  -> Decoder：逐步恢复空间分辨率
  -> 输出前景 logit
```

它通过 skip connection 把 encoder 的浅层空间信息传给 decoder：

```text
F_decoder = Decode(Concat(Up(F_deeper), F_encoder_same_level))
```

因此它同时利用：

- 深层语义：哪里可能是腺体。
- 浅层定位：腺体边界在哪里。

### 6.3 UNet 在研究流程中的作用

#### 作用一：验证数据链

检查：

- 图像和 mask 是否对应。
- mask 是否正确二值化。
- resize 是否同步。
- RGB 是否正确。
- split 是否读对。

#### 作用二：验证训练链

检查：

- 前向传播是否正常。
- loss 是否能计算。
- 反向传播是否正常。
- checkpoint 是否保存和加载。
- 是否出现 NaN 或全背景预测。

#### 作用三：验证评估链

当前项目的预测和评估链可以写成：

```text
logits
 -> sigmoid
 -> threshold
 -> binary mask
 -> connected components
 -> object/pixel metrics
```

#### 作用四：建立最小参照

UNet 回答：

> 在不加入复杂结构的情况下，当前数据集和当前协议能达到什么水平？

后续所有模型都需要知道自己相对这个最小参照发生了什么变化。

### 6.4 为什么不能一开始直接用复杂 Baseline

理论上可以，但会增加定位风险。

如果复杂模型结果异常，你无法快速判断是：

- 数据问题。
- 标签问题。
- 预训练权重问题。
- skip 对接问题。
- loss 问题。
- 评估问题。
- 论文代码适配问题。
- 复杂模型本身的问题。

先跑 UNet 的目的不是认为它最强，而是建立一个低复杂度、易排查的参照系统。

可以把它理解成医学实验中的对照检测：在使用复杂设备或复杂治疗前，先确认基础检测链没有错误。

---

## 7. 第五步：Baseline 到底是什么，如何确定

### 7.1 Baseline 的正式定义

在深度学习研究中，Baseline 是一个经过明确协议训练和评估的参考方法，用来回答：

> 如果不加入你的研究创新，合理的现有方法在这个任务上能达到什么水平？

因此 Baseline 不是：

- 随便一个网络。
- 只跑一次的模型。
- 论文里最高的那个数字。
- 和你的方法训练条件完全不同的模型。
- 不能复查结构和配置的黑盒代码。

### 7.2 Baseline 是怎么确定的

通常经过下面五步确定。

#### 第一步：确定任务类型

结直肠腺体分割属于 2D 医学图像语义/实例相关分割任务，因此需要选择分割模型，并关注对象级和边界级评价。

#### 第二步：查找任务领域常用模型

阅读：

- 任务经典论文。
- 经典分割模型论文。
- 近期任务内论文。
- 公开代码和复现报告。

#### 第三步：确定 Baseline 角色

要先决定你选择的是：

```text
流程基线？
正式结构基线？
外部对比方法？
任务内直接对比方法？
```

不同角色不必是同一个模型。

#### 第四步：检查是否适合当前工程

模型需要满足：

- 输入和标签可以适配。
- 结构来源清楚。
- 代码可以复现或合理实现。
- 训练资源可接受。
- 能在固定协议下重复。
- 后续可以在其上增加研究模块。

#### 第五步：在当前数据集上验证

Baseline 不是因为论文里有名就自动成立。它必须在你的数据、你的代码和你的评估协议下跑通，并接受公平比较。

### 7.3 Baseline 是不是一定要复现别人的论文实验

答案取决于你声称的研究身份。

| 你的说法 | 对复现的要求 |
|---|---|
| “我们复现了某论文方法” | 应尽量按原文和官方代码逐项复现 |
| “我们采用某论文结构作为 baseline” | 可以做任务适配实现，但要写清差异 |
| “我们使用经典 UNet 作为工程基线” | 不需要声称复现原始训练 recipe |
| “表中引用了论文结果” | 必须标为引用值，不能写成自己的复现结果 |

### 7.4 完整复现、任务适配和工程实现

#### 完整复现

尽量保持：

- 数据集和 split。
- 预处理和增强。
- 输入尺寸。
- 模型结构。
- loss。
- optimizer 和训练策略。
- 后处理。
- 指标实现。

只有在这些内容足够一致时，才适合说“复现该论文结果”。

#### 任务适配实现

当论文代码不能直接使用时，可以实现相同方法思想，但应记录：

- 原论文结构来源。
- 官方代码是否可得。
- 当前实现与原文的差异。
- 沿用的参数。
- 自行确定的工程参数。
- 因数据集不同而作的适配。

准确表述是：

> 我们实现了该方法面向当前任务的适配版本。

不要写成：

> 我们逐行复现了原作者代码。

#### 工程基线

工程基线的目标是建立可控的内部参照，不一定对应某篇论文完整的原始实验方案。

例如当前项目的 UNet 使用统一的输入、`BCE + Dice`、AdamW 和项目评估链。它可以称为“基于经典 UNet 的工程基线”，但不能说这些参数全部来自 U-Net 原论文。

### 7.5 Baseline 有年限要求吗

没有绝对的“必须近五年”或“必须近三年”规则。

应区分：

- 经典基础模型：U-Net、ResNet 等可以很早，仍然是合理基础。
- 正式结构 baseline：应选择当前任务中成熟、稳定、可解释的方法。
- 外部对比方法：最好包含近期且有代表性的模型。
- 研究空白依据：需要重点阅读近年论文，说明问题仍有研究价值。

年份只是参考因素，不能凌驾于任务匹配性、公平性和可复现性之上。

### 7.6 如何判断 Baseline 是否合格

可以用下面八个问题检查：

1. 它是否与当前分割任务匹配？
2. 结构和来源是否清楚？
3. 是否能在当前数据集上训练和测试？
4. 是否能固定协议并重复训练？
5. 是否能与后续方法只保留一个主要差异？
6. 是否有合理的任务动机？
7. 是否有完整指标、可视化和错误分析？
8. 是否能够承担论文主线的比较起点？

## 8. 第六步：为什么正式 Baseline 还要和 UNet 对比

这是你目前最关键的疑问之一。

### 8.1 正式 Baseline 不是天然正确

选择 ResNet34-U-Net 并不等于证明它一定比 UNet 好。它只是一个候选正式结构，需要在当前任务上验证。

如果不比较，逻辑只是：

```text
我选择了 ResNet34-U-Net
```

如果比较后，逻辑才是：

```text
在同一数据、训练和评估协议下，
ResNet34-U-Net 相比普通 UNet 的对象级表现更好或不低于，
因此选择它作为后续实验的正式结构基座。
```

### 8.2 这不是为了证明 Baseline 是创新

Baseline 和 UNet 的比较不是为了把 ResNet34-U-Net 写成你的创新，而是为了确认：

- 这个基座足够强。
- 这个基座值得后续模块继承。
- 后续模块的提升不会只是从一个过弱模型上获得的假增益。

### 8.3 当前项目的结构关系

```text
普通 UNet
= 最小 encoder-decoder

ResNet34-U-Net
= ResNet34 encoder + U-Net style decoder

UNet -> ResNet34-U-Net
= 主要研究差异为 plain encoder -> residual encoder
```

Baseline 阶段不应该顺手加入 LKMA、Boundary Head、Distance-aware Loss 或复杂后处理，否则就无法判断差异来自哪里。

### 8.4 正式比较的原则

UNet 和 ResNet34-U-Net 应尽量保持一致：

- 数据集和 split。
- 输入尺寸。
- 归一化和增强。
- loss。
- optimizer、学习率和 scheduler。
- epoch 上限和 early stopping。
- best checkpoint 选择。
- threshold 来源。
- 后处理。
- seed 集合。
- 指标实现。

比较单位不能是：

```text
UNet 最好的一次结果 vs ResNet34-U-Net 最好的一次结果
```

而应是：

```text
UNet 三个 seed 的 mean +- std
vs
ResNet34-U-Net 三个 seed 的 mean +- std
```

TestA 和 TestB 必须分开统计。

## 9. 第七步：正式 Baseline 之后如何改进

### 9.1 改进实验的基本逻辑

正式 baseline 建立后，才开始验证你的研究方法：

```text
Baseline
 -> 加入一个模块
 -> 比较
 -> 做消融
 -> 决定保留或删除
```

不是：

```text
多个模块一起加入
 -> 分数变高
 -> 直接宣布有效
```

### 9.2 单变量原则

如果研究 LKMA，就尽量只改变 LKMA：

```text
M0 = ResNet34-U-Net
M1 = M0 + LKMA
```

如果研究 Boundary Head，就在已确定的基座上只加入边界分支：

```text
M2 = M1 + Boundary Head
```

如果研究 Distance-aware Loss：

```text
M3 = M2 + Distance-aware Loss
```

每一步都要记录：

- 新增了什么。
- 为什么新增。
- 其他协议是否不变。
- 主要指标如何变化。
- 哪些错误案例变化。
- 是否出现副作用。

### 9.3 当前项目的模块动机

#### LKMA

目标是增强较大范围的上下文建模，可能帮助：

- 形态变化。
- 腺体周围组织关系。
- 较大范围结构判断。

#### Boundary Head

目标是显式强化边界学习，可能帮助：

- 边界模糊。
- 相邻腺体黏连。
- 边缘定位。

#### Distance-aware Loss

目标是引入距离或形状信息，可能帮助：

- 区域内部和边界的几何关系。
- 腺体对象分离。
- 形态一致性。

这些是研究假设，不是自动成立的结果。

### 9.4 改进后是否有效的判断层级

#### 工程有效

模型能训练、测试、保存、导出，没有明显实现错误。

#### 指标有效

主指标在统一协议下提升，或没有明显退化。

#### 稳定有效

多个 seed 下方向基本一致，不是单次偶然高值。

#### 任务有效

提升对应腺体任务难点，例如黏连减少、边界改善、小腺体漏检减少。

#### 科学有效

消融、外部对比、错误案例和统计分析共同支持结论。

如果只有一次 Dice 提高，只能说“观察到一次指标提升”，还不能直接说“方法有效”。

## 10. 第八步：消融实验、外部对比和泛化验证

### 10.1 消融实验是什么

消融实验就是逐个拆分方法，观察每个部分的作用。

如果完整方法为：

```text
M_full = ResNet34-U-Net + LKMA + Boundary Head + Distance-aware Loss
```

至少需要：

```text
M0 = ResNet34-U-Net
M1 = M0 + LKMA
M2 = M1 + Boundary Head
M3 = M2 + Distance-aware Loss
```

消融要回答：

- LKMA 是否有效？
- Boundary Head 是否增加了额外收益？
- Distance-aware Loss 是否值得保留？
- 是否存在模块之间的冲突？
- 哪个模块主要改善区域，哪个主要改善边界？

### 10.2 外部对比是什么

外部对比是和其他公开方法比较，用于说明你的方法在同一任务上具备竞争力。

它和内部 Baseline 的角色不同：

```text
内部 Baseline：解释你的方法增加了什么
外部对比：说明你的方法相对于其他代表性方法如何
```

当前项目计划的第一批外部对比包括：

- UNet++。
- DeepLabV3+。
- Attention U-Net。

任务内直接对比方法可以根据复现条件选择，例如 DCAN、MILD-Net、DEA-Net、TA-Net 等。

如果直接采用论文中的数字，必须标记为引用值；只有自己在统一协议下跑出的结果，才是复现值。

### 10.3 第二 benchmark 验证

CRAG 用于补充观察方法在另一个同任务 benchmark 上的表现。

但它不能自动证明：

- 适用于所有病理图像。
- 适用于所有医院和染色条件。
- 已经完成真正跨域泛化。

必须说明是在 CRAG 上重新训练测试，还是 GlaS 训练后直接跨数据集测试。

---

## 11. 第九步：什么才算实验结果可信

### 11.1 结果可信的最低条件

至少应具备：

- 固定数据协议。
- 训练、验证和测试角色清楚。
- best checkpoint 来自验证集。
- threshold 来自验证集并固定到测试集。
- TestA 和 TestB 分开报告。
- 保留每个 seed 的原始结果。
- 报告 mean 和 std。
- 有对象级和像素级指标。
- 有预测可视化和失败案例。
- 配置、日志、checkpoint 和结果文件可对应。

### 11.2 结果不能只看平均值

平均值回答总体趋势，标准差回答波动情况，逐样本结果回答错误发生在哪里。

三者都需要：

```text
per-seed raw results
+ mean +- std
+ per-sample/error analysis
```

标准差不是显著性检验，也不能证明模型在所有随机种子下都稳定；它只是描述当前固定 seed 集合中的波动。

### 11.3 指标要和任务问题对应

结直肠腺体分割不能只报 Dice。建议分层：

```text
主指标：F1 / Object Dice / Object Hausdorff
补充指标：Dice / IoU / HD95 / Boundary F1
```

通常：

- F1、Dice、IoU 越高越好。
- Hausdorff、HD95 越低越好。
- Boundary F1 越高越好。

最终仍要以项目冻结的指标定义和实现为准。

## 12. 第十步：从研究到论文投稿的完整路线

### 第 1 阶段：研究准备

产出：

- 研究问题。
- 任务难点。
- 文献地图。
- 候选 Baseline。
- 方法假设。
- 实验矩阵。

此时不要急着写“最终结果”，因为还没有实验依据。

### 第 2 阶段：数据准备

产出：

- 数据清单。
- 图像标签对应检查。
- 固定 split。
- 标签转换规则。
- 输入和归一化规则。
- 数据质量报告。

### 第 3 阶段：流程验证

使用 UNet 完成：

```text
smoke check
 -> 单 seed 正式训练
 -> 验证集评估
 -> checkpoint 选择
 -> threshold 冻结
 -> TestA/TestB 测试
 -> 可视化
 -> 错误分析
```

这一阶段的结论是：

> 研究系统已经跑通，数据、训练和评估链能够工作。

不是：

> 本文方法已经有效。

### 第 4 阶段：稳定性验证

使用固定多个 seed 训练同一 UNet，报告：

- per-seed 结果。
- mean +- std。
- 异常 seed。
- 失败案例。

这一阶段的结论是：

> 最小参照是否足够稳定，可以承担后续比较。

### 第 5 阶段：建立正式 Baseline

选择并实现正式 Baseline：

```text
ResNet34 encoder + U-Net style decoder
```

只改变结构主变量，训练和评估协议继承上游。

然后做：

```text
UNet vs ResNet34-U-Net
```

这一阶段的结论是：

> ResNet34-U-Net 是否足以作为后续模块的正式比较基座。

### 第 6 阶段：方法实验和消融

按单变量顺序：

```text
Baseline
 -> + LKMA
 -> + Boundary Head
 -> + Distance-aware Loss（可选）
```

每个阶段决定：

- 保留。
- 删除。
- 降级为补充实验。
- 需要回退排查。

### 第 7 阶段：外部对比和第二 benchmark

完成：

- 通用分割模型对比。
- 任务内方法对比。
- GlaS 主结果。
- CRAG 补充验证。

### 第 8 阶段：结果汇总

至少准备：

- 主结果表。
- 消融表。
- 外部对比表。
- per-seed 结果。
- mean +- std。
- 可视化图。
- 失败案例图。
- 复杂度和速度统计。
- 实验配置和版本信息。

### 第 9 阶段：论文写作

#### Abstract

回答：做了什么、为什么做、怎么做、效果如何。

#### Introduction

回答：任务重要性、现有不足、研究问题、本文贡献。

#### Related Work

按方法方向组织：UNet、上下文建模、边界监督、距离约束、腺体分割方法。

#### Method

写清：

- 总体框架。
- Baseline 主干。
- LKMA。
- Boundary Head。
- Distance-aware Loss。
- 总损失函数。
- 推理过程。

#### Experiments

写清：

- 数据集和 split。
- 预处理和增强。
- 训练配置。
- 评估指标。
- Baseline 选择原因。
- 消融设置。
- 外部对比协议。
- 多 seed 统计方式。

#### Results

回答：

- 完整方法是否优于 Baseline。
- 每个模块是否有贡献。
- 哪些指标改善。
- TestA/TestB 是否一致。
- 哪些困难案例改善。

#### Discussion

讨论：

- 为什么可能有效。
- 对哪些样本有效。
- 哪些指标没有改善。
- 计算代价。
- 数据规模限制。
- 跨数据集限制。

#### Conclusion

只写实验真正支持的结论，不要把两个 benchmark 写成“适用于所有病理图像”。

### 第 10 阶段：期刊选择和投稿

期刊选择要综合考虑：

- 任务匹配度。
- 医学图像、数字病理或计算病理收稿范围。
- 方法创新强度。
- 实验完整度。
- 近期相近文章。
- 是否要求代码、数据和伦理声明。
- 文章格式和篇幅。

投稿材料通常包括：

- 主文稿。
- 图表和补充材料。
- Cover Letter。
- Highlights 或 Key Points。
- 数据和代码声明。
- 伦理、利益冲突和基金信息。

## 13. 当前项目路线的完整解释

当前项目不是“先随便找一个 Baseline，再不断加模块”，而是分层建立证据。

```text
阶段 1：数据协议
    作用：固定数据、标签、split、输入和评估前提

阶段 2：UNet 流程验证
    作用：确认数据链、训练链、评估链、可视化链可信

阶段 3：UNet 稳定性
    作用：确认最小参照不依赖一次偶然训练

阶段 4：ResNet34-U-Net 正式 Baseline
    作用：在相同协议下建立更有代表性的结构基座

阶段 5：+ LKMA
    作用：验证大范围上下文建模是否有效

阶段 6：+ Boundary Head
    作用：验证边界监督是否改善边界和黏连

阶段 7：+ Distance-aware Loss
    作用：验证几何距离和形状约束是否提供额外收益

阶段 8：外部对比
    作用：证明方法相对其他代表性模型的竞争力

阶段 9：CRAG 验证
    作用：提供第二 benchmark 的补充证据

阶段 10：结果汇总和投稿
    作用：将完整证据组织成论文
```

### 13.1 当前项目中 UNet 的身份

```text
UNet = 流程验证模型 + 最小可复现基线
```

它主要证明：

- 数据能否正确进入模型。
- 训练和评估代码是否正常。
- 评价指标是否真实落地。
- 预测结果和标签是否对齐。

### 13.2 当前项目中正式 Baseline 的身份

```text
ResNet34-U-Net = 正式结构 Baseline
```

它主要证明：

- 更成熟的 residual encoder 是否值得作为主干。
- 后续改进是否建立在一个足够有代表性的起点上。
- 后续模块的增益是否不是因为起始模型过弱。

### 13.3 当前项目中后续方法的身份

```text
ResNet34-U-Net + LKMA
ResNet34-U-Net + LKMA + Boundary Head
ResNet34-U-Net + LKMA + Boundary Head + Distance-aware Loss
```

这些才是围绕研究问题展开的改进方法，不应在 Baseline 阶段提前混入。

## 14. 当前项目的建议执行顺序

你已经完成普通 UNet，接下来不要直接跳到最终方法，建议按下面顺序检查。

### 第一步：确认 UNet 是完整闭环，而不只是训练结束

至少应有：

- 固定 split。
- 训练日志。
- best checkpoint 和 last checkpoint。
- 验证集指标。
- TestA 和 TestB 指标。
- 预测 mask。
- 可视化。
- 错误案例。
- 指标实现和配置记录。

### 第二步：完成 UNet 多 seed 稳定性

使用当前固定 seed 集合：

```text
3407 / 1234 / 2025
```

得到每个 seed 和 mean +- std 结果。

### 第三步：训练 ResNet34-U-Net

尽量保持以下内容不变：

- 数据和 split。
- 输入尺寸。
- 归一化。
- 增强。
- loss。
- optimizer。
- scheduler。
- checkpoint 选择。
- threshold。
- 评估实现。
- seed 集合。

只把 encoder 换成 ResNet34，decoder 继续使用 U-Net 风格结构。

### 第四步：比较 UNet 与 ResNet34-U-Net

同时检查：

- TestA 的 Object F1、Object Dice、Object Hausdorff。
- TestB 的 Object F1、Object Dice、Object Hausdorff。
- Dice、IoU、HD95、Boundary F1。
- 三个 seed 的均值和标准差。
- 可视化和失败案例。
- 结构、训练和评估字段是否一致。

### 第五步：再进入 LKMA

只有在正式 Baseline 的结构和比较条件清楚后，才执行：

```text
ResNet34-U-Net -> ResNet34-U-Net + LKMA
```

此时 LKMA 才是主要新增变量，结果才具有明确归因。

## 15. 常见错误理解

### 错误一：Baseline 就是随便一个能跑的模型

不对。它必须任务匹配、来源清楚、协议公平、结果可复查，并且能够承担后续比较。

### 错误二：Baseline 必须是最新论文的方法

不对。经典 U-Net、ResNet 仍可作为基础；正式 Baseline 和外部对比需要结合任务、年份、稳定性和复现条件综合选择。

### 错误三：Baseline 必须完整复现论文代码

只有当你声称“复现该论文结果”时，才需要尽量完整复现。作为工程或结构基线时，可以做任务适配，但必须诚实说明。

### 错误四：普通 UNet 和正式 Baseline 重复了，应该删掉 UNet

不对。两者角色不同：

```text
UNet：验证系统和建立最小参照
正式 Baseline：建立后续创新实验的正式起点
```

### 错误五：只要新方法 Dice 高，就说明研究成功

不对。还要看公平性、稳定性、对象级指标、边界质量、消融和失败案例。

### 错误六：可以用后续模块挽救一个不合格 Baseline

不建议。应先弄清 Baseline 失败是工程问题、协议问题还是结构问题。否则后续改进的增益无法解释。

### 错误七：测试集不好，就调 threshold 或挑 checkpoint

不允许。checkpoint 和 threshold 应在验证集上确定，再固定到测试集。

### 错误八：只保留最好的一次训练

不够。正式结论至少需要保留 per-seed 结果、均值、标准差和错误案例。

## 16. 最终理解

深度学习研究的完整逻辑可以压缩成：

```text
先提出问题
 -> 读文献，知道别人做了什么
 -> 固定数据和评价协议
 -> 用简单模型验证实验系统
 -> 用多个 seed 证明基本结果稳定
 -> 选择并验证正式 Baseline
 -> 在 Baseline 上一次加入一个改进
 -> 用消融解释每个模块的贡献
 -> 用外部对比证明竞争力
 -> 用第二数据集观察补充泛化
 -> 汇总统计、可视化和失败案例
 -> 写论文并投稿
```

对应到当前项目：

```text
UNet
= 先证明实验系统可信

ResNet34-U-Net
= 建立正式结构参照

LKMA / Boundary / Distance
= 在正式参照上验证具体研究假设

外部对比和 CRAG
= 补充证明竞争力和任务外验证表现
```

所以，研究不是“先找一个模型然后加东西”，而是：

> 先建立可信的比较起点，再通过控制变量和多层证据证明你的方法到底解决了什么问题。
